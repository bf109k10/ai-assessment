RAG / Retrieval-Augmented Generation

Поиск релевантных знаний и добавление их в контекст

Простыми словами

RAG — это когда модель не пытается всё помнить сама, а сначала ищет нужные фрагменты в документах, коде, wiki, базе знаний или тикетах, а потом использует найденное для ответа.

Когда использовать
  • Нужно ответить по документации.
  • Нужно найти API/класс/метод в коде.
  • Нужно подтянуть knowledge base.
  • Нужно уменьшить hallucination.
Ограничения
  • Может найти устаревший документ.
  • Может найти похожий, но не тот фрагмент.
  • Не задаёт поведение агента.
  • Требует хорошего indexing/chunking.

Важно для assessment

RAG отвечает за факты, но не за правила поведения. Если RAG нашёл старую инструкцию, а Rules говорят другое — Rules выше. Если нужно текущее состояние системы, лучше проверить через MCP/tools.

🧠

Memory Bank

Долговременная память проекта, пользователя или workflow

Простыми словами

Memory Bank — это сохранённые устойчивые знания: какие решения уже приняли, какие правила у проекта, какие предпочтения у пользователя, какие команды работают, какие подходы не подходят.

Когда использовать
  • Нужна continuity между сессиями.
  • Есть долгий проект.
  • Нужно помнить решения.
  • Нужно не спрашивать одно и то же.
Ограничения
  • Может устареть.
  • Может быть слишком общей.
  • Может конфликтовать с текущими файлами.
  • Не заменяет проверку фактов.

Memory Bank vs RAG

Memory Bank — это curated память о важных вещах. RAG — поиск по источникам. Memory помогает помнить “что мы решили”, RAG помогает найти “что написано в документации”.

🛠

MCP / Tools

Доступ к внешним системам, данным и действиям

Простыми словами

MCP и tools дают агенту возможность не только рассуждать, но и действовать: открыть файл, сходить в API, прочитать Jira, проверить CI, посмотреть логи, выполнить поиск.

Когда использовать
  • Нужно проверить реальное состояние.
  • Нужно прочитать файл/лог/тикет.
  • Нужно выполнить действие.
  • Нужно получить свежие данные.
Ограничения
  • Нужны permissions.
  • Tools могут быть read-only.
  • Опасные действия требуют осторожности.
  • Subagent может не иметь доступа.

Важно

MCP не делает агента “умнее” сам по себе. Он даёт доступ к данным и действиям. Качество зависит от того, какие tools доступны, какие права есть и как агент интерпретирует результат.

✅

Checkpoint

Структурный снимок состояния задачи

Простыми словами

Checkpoint — это не просто summary, а структурированная точка сохранения: цель, что уже сделано, какие решения приняты, какие есть риски и что делать дальше.

Когда использовать
  • Длинная задача.
  • Переход между сессиями.
  • Нужно передать работу другому агенту.
  • Перед context compression.
Что включать
  • Цель.
  • Done / not done.
  • Decisions.
  • Risks / blockers.
  • Next steps.

Checkpoint vs Summary

Summary просто кратко пересказывает. Checkpoint сохраняет состояние работы так, чтобы можно было продолжить без потери логики.

🧾

Summary / Compression

Сжатие истории, когда контекст переполняется

Простыми словами

Compression — это уменьшение объёма контекста. Summary — один из способов compression: длинный диалог превращается в краткое содержание.

Когда использовать
  • Длинный чат.
  • Много деталей и файлов.
  • Нужно продолжить позже.
  • Окно контекста ограничено.
Риски
  • Теряются нюансы.
  • Может исказить решение.
  • Может забыть edge case.
  • Не заменяет исходные файлы.

Лучший формат

Для рабочих задач лучше использовать structured checkpoint: цель, контекст, сделано, решения, команды, риски, next steps.

Skills / Playbooks / Commands

SKILL.md · .cursor/commands/ · прогрессивное раскрытие · лаконичность

Три формата в разных инструментах

Claude Code: SKILL.md в папке скилла. Cursor: .cursor/commands/name.md (slash-команды). Cline: .clinerules/workflows/. Принцип один: пошаговый runbook, который AI выполняет при вызове.

SKILL.md структура (Claude Code)
  • YAML frontmatter: name + description.
  • name: строчные + дефисы, max 64 символа.
  • description: от 3-го лица — что делает + когда (max 1024).
  • Тело: до 500 строк; всё остальное — доп. файлы.
  • Прогрессивное раскрытие: SKILL.md → reference files (1 уровень).
Best practices (Claude docs)
  • Лаконичность — не объяснять очевидное.
  • Description из 3-го лица: не “я помогу”, а “Обрабатывает...”.
  • Ссылки глубиной 1 уровень (no deep nesting).
  • Свобода: high (текст) / medium (псевдокод) / low (точный скрипт).
  • Evaluation-driven: сначала eval, потом документация.
  • Тестировать с Haiku, Sonnet, Opus.

Rule vs Command vs Skill vs Subagent (A3/A4 Cursor)

Rule: всегда применяется в проекте. Command: slash-команда, вызывается явно (/mr-self-review, /catalog-field-change). Skill: переиспользуемая экспертиза, агент применяет сам. Subagent: специализированный агент для workstream (A4). Skill не нарушает Rule.

Cursor Command структура (A3)
  • Input: параметры задачи.
  • Research: что найти в Ask mode.
  • Plan: что подтвердить до edit.
  • Implementation: шаги в Agent mode.
  • Validation: что запустить.
  • Guardrails: явные запреты.
  • Output: что вернуть.
Антипаттерны Skills
  • SKILL.md объясняет очевидное (тратит токены).
  • Description "помогает с файлами" — не обнаружим.
  • Глубокая вложенность ссылок (A → B → C).
  • Зависящие от времени инструкции.
  • Слишком много вариантов без дефолта.

AGENTS.md / CLAUDE.md

Проектная инструкция · иерархия · .cursorignore · глобальный vs проектный

Разные инструменты — разные файлы

Cursor читает AGENTS.md в корне + .cursor/rules/. Claude Code использует CLAUDE.md (проектный) + ~/.claude/CLAUDE.md (глобальный). Cline: .clinerules/. OpenAI Codex: AGENTS.md. Принцип: более специфичный файл переопределяет общий.

Что класть в AGENTS.md / CLAUDE.md
  • Команды тестов, build, lint, import.
  • Архитектурные правила и stack.
  • Definition of Done.
  • Запрещённые технологии/действия.
  • Source of truth (где данные, как обновлять).
  • Workflow: как вносить изменения.
Что НЕ класть
  • Секреты и токены.
  • Большие логи и generated dumps.
  • Часто меняющиеся данные (→ RAG/MCP).
  • Огромные docs (→ ссылки или RAG).
  • A4/A5 scope в A3 артефакте.

.cursorignore и контекстные фильтры

.cursorignore (Cursor) — файл в корне проекта, исключает файлы из AI контекста (синтаксис как .gitignore): node_modules, generated JSON, dist, secrets, бинарники. Claude Code: аналогично через exclusion patterns. Зачем: уменьшает шум, защищает чувствительные файлы, фокусирует контекст на нужном. Меньше токенов = лучше качество ответа.

AGENTS.md vs Rules vs Skills

AGENTS.md = роль агента и workflow (всегда активен). Rules (.mdc) = что нельзя/обязательно (высший приоритет). Skill = как выполнять (по вызову). Command = конкретный workflow (явный вызов). Конфликт AGENTS.md + Rules: Rules побеждает.

⚖️

Rules / Policy

.cursor/rules/*.mdc · .clinerules/ · always-apply vs auto-attach vs conditional

Что это

Rules — обязательные ограничения: code style, safety, compliance, запреты на технологии, архитектурные стандарты. Rules отвечают на вопрос “что нельзя/обязательно”, а не “как делать”. Применяются автоматически — выше Skill и Command в иерархии.

Cursor Rules (.cursor/rules/)
  • Файлы .mdc в .cursor/rules/.
  • Always-apply: в каждом чате (глобальные стандарты).
  • Auto-attach: к конкретным файловым паттернам.
  • Conditional: через @ в чате.
  • Видны агенту высокоприоритетно.
Примеры правил
  • Нельзя добавлять DB/Docker/real auth в демо.
  • Stack: Java 17, Quarkus, React 19, RTK Query.
  • Generated JSON — не редактировать вручную.
  • Preserve sourceFile/sourceSheet/sourceRow.
  • Не добавлять зависимости без approval.

Rules vs AGENTS.md vs Command (финальная таблица)

Rule (.mdc) = что всегда запрещено/обязательно → автоматически. AGENTS.md = роль и workflow агента → всегда. Command (.cursor/commands/) = как выполнять процесс → явный вызов. Skill = экспертиза → агент выбирает сам. Rule самый сильный из них при конфликте.

Subagents

Делегирование подзадач специализированным агентам

Простыми словами

Subagent — отдельный агент или режим, которому главный агент передаёт часть задачи: найти причину, проверить гипотезу, собрать факты, сделать review.

Когда полезны
  • Сложная задача.
  • Параллельный анализ.
  • Нужна специализация.
  • Нужно разделить поиск и решение.
Ограничения
  • Может видеть не весь контекст.
  • Может иметь ограниченные tools.
  • Может ошибиться.
  • Главный агент должен проверить итог.

Главный принцип

Subagent не заменяет ответственность главного агента. Он возвращает findings, а главный агент объединяет, проверяет конфликт с Rules/AGENTS.md и принимает финальное решение.

Workflow / Агентский цикл

Perceive → Plan → Act → Observe — как агент строит задачи

Агентский цикл

Агент не выполняет задачу одним шагом. Он воспринимает ситуацию, планирует шаги, выполняет действие через tool/MCP, наблюдает результат и повторяет цикл до достижения цели или остановки.

Типы workflow
  • Linear: шаги строго последовательно.
  • Branching: if/else ветки по условию.
  • Parallel: несколько шагов одновременно.
  • Loop: повтор до достижения цели.
  • DAG: граф зависимостей шагов.
Human-in-the-loop
  • Пауза перед критическим действием.
  • Запрос approval у пользователя.
  • Подтверждение опасной операции.
  • Итерация по обратной связи.
  • Обязательно при деплое / удалении данных.

Обработка ошибок в цикле

Хороший агент: при ошибке tool — retry с backoff; при неверном результате — Reflexion (самокритика и новая попытка); при тупике — остановиться и спросить человека, не продолжать бесконечный цикл.

Prompt Engineering

Как строить инструкции для получения лучших результатов от LLM

Простыми словами

Prompt Engineering — создание инструкций для LLM. От качества промпта зависит всё: структура ответа, точность, безопасность, формат. Применяется и к system prompt, и к user prompt, и к AGENTS.md.

Техники
  • Zero-shot: прямой вопрос без примеров.
  • Few-shot: 1–5 примеров в промпте.
  • Chain-of-Thought: "шаг за шагом".
  • Role prompting: "Ты эксперт в...".
  • Structured output: JSON/XML формат.
  • Negative constraints: "Не делай X".
Что включать в хороший промпт
  • Роль и контекст агента.
  • Чёткая задача и ограничения.
  • Формат ожидаемого ответа.
  • Примеры (few-shot) если нужно.
  • Что явно запрещено делать.
  • Как обрабатывать неизвестное.

Важно для assessment

System prompt задаёт роль и поведение. CoT помогает с многошаговыми рассуждениями. Structured output снижает риск нераспарсиваемых ответов. Few-shot повышает точность для специфичных форматов. Temperature=0 даёт предсказуемость при работе с tools.

Agentic Patterns

ReAct, CoT, Plan-then-Execute, Reflexion — архитектура рассуждений

Основные паттерны

Паттерны агентов — это архитектурные подходы к тому, как агент рассуждает и действует. Разные задачи требуют разных паттернов.

Паттерны и когда
  • ReAct: tools + итерация → для большинства агентов.
  • CoT: сложная логика/math → рассуждения вслух.
  • Plan-then-Execute: план → исполнение → длинные предсказуемые задачи.
  • Reflexion: самокритика + retry → когда первый ответ плохой.
  • ToT: дерево вариантов → исследование решений.
ReAct цикл подробнее
  • Thought: что делать и зачем.
  • Action: вызов tool с аргументами.
  • Observation: результат tool.
  • → повтор до финального ответа.
  • Большинство LLM-агентов используют именно ReAct.

Plan-then-Execute vs ReAct

Plan-then-Execute: задача хорошо определена, шаги известны заранее — агент строит полный план и выполняет. ReAct: результат каждого шага влияет на следующий, нужна адаптация по ходу. В реальности часто комбинируют: высокоуровневый план + ReAct на каждом шаге.

Vector DB & Embeddings

Как RAG хранит и ищет знания под капотом

Как работает

Embedding — числовое представление текста (вектор размерностью 768–3072). Похожие по смыслу тексты → близкие векторы. Vector DB хранит векторы и умеет быстро искать ближайшие по cosine similarity / ANN (Approximate Nearest Neighbor).

Indexing pipeline
  • Документ → Chunking.
  • Chunk → Embedding model.
  • Вектор → Vector DB.
  • Метаданные (дата, тип, источник).
  • Примеры DB: Pinecone, Chroma, pgvector.
Retrieval pipeline
  • Запрос → Embedding.
  • Similarity search → Top-K кандидатов.
  • Reranking (cross-encoder, опционально).
  • Inject top-1..3 chunk'ов в контекст.
  • Dense / Sparse / Hybrid retrieval.

Chunking стратегии

Fixed-size (по N токенов с overlap), по абзацам/секциям (семантические границы), code-aware (по функциям/классам), semantic chunking (разбивка по перепаду смысла). Плохой chunking → плохой retrieval. Overlap 10–20% помогает не терять контекст на границах.

Context Window & Tokens

Что умещается в голове модели и как этим управлять

Простыми словами

Context window — максимальный объём текста, который LLM держит "в голове" за один вызов. Всё суммируется в токенах: system prompt, AGENTS.md, история чата, RAG-фрагменты, tool results, user message, ответ модели.

Что съедает контекст
  • System prompt / AGENTS.md.
  • История чата (растёт линейно).
  • RAG-фрагменты (top-K × размер chunk).
  • Tool results (могут быть огромными).
  • Прикреплённые файлы.
Как управлять
  • Summary / Checkpoint истории.
  • Умный RAG: только top-K нужных chunk'ов.
  • Truncation старых сообщений.
  • Subagents (параллельная разгрузка).
  • Sliding window (скользящее окно).
  • Tool results → только нужная часть.

Важно для assessment

При переполнении контекста модель "забывает" старые части. Токены = стоимость + качество. GPT-4o: 128K токенов. Claude 3.5: 200K. Gemini 1.5 Pro: 1M. Но больше не всегда лучше — "lost in the middle": модель хуже использует середину длинного контекста.

Agent Card

Формальный контракт агента: что, как, с чем и при каких условиях (A4)

Что это и зачем

Agent Card — документ, задающий рабочий контракт агента. Без него агент может выйти за scope, вызвать запрещённый tool, нарушить безопасность или провалить quality check. Это не просто описание — это техническое соглашение команды с агентом.

Обязательные поля Agent Card
  • Name: имя агента.
  • Purpose: что делает.
  • Inputs: что принимает.
  • Outputs: что возвращает.
  • Allowed tools: разрешённые инструменты.
  • Forbidden actions: явный список запретов.
  • Approval gates: когда стоп → человек.
  • Quality checks: как проверяем результат.
  • Risks: scope creep, data exposure, wrong tool.
Approval gates (A4)
  • После research → перед реализацией.
  • Перед изменением API контракта.
  • Перед принятием финального diff.
  • Снижают риск неправильных tool calls.
  • Не заменяют тесты — дополняют.

Каверзный момент

Забыть Forbidden actions = критическая ошибка. Агент должен знать не только что он делает, но и что явно запрещено: реальные платежи, production данные, внешние write-действия без approval, DB/auth/deployment в тренировочном демо.

Internal MCPs: jira / graylog / atp

Netcracker-специфичные MCP-серверы и cross-MCP investigation workflow

Три внутренних MCP-сервера

В Netcracker есть три основных internal MCP: jira-mcp (Jira + Confluence), graylog-mcp (логи), atp-mcp (тест-системы ATP). Все запускаются через npx из npm — не нужен local build.

Сценарии использования
  • jira-mcp: найти задачу, прочитать описание, найти x-request-id, подготовить plan/MR description.
  • graylog-mcp: найти логи по x-request-id, timeline, top errors, root cause candidates.
  • atp-mcp: тест-сценарии, failed execution, скриншоты, logs, datasets.
Cross-MCP workflow: Jira→Graylog
  • jira-mcp → найти x-request-id в тикете.
  • graylog-mcp → найти логи по x-request-id.
  • Собрать timeline + top errors + affected services.
  • AI → investigation summary (hypotheses).
  • Human → решение и next actions.

MCP Security — правило №1

Реальные токены/пароли — только в локальном конфиге (user-level ~/.cursor/mcp.json). В Git, Confluence, артефактах — только плейсхолдеры (<JIRA_BEARER_TOKEN>). Auto-run для write tools — выключен. A2: только read-only Confluence smoke test. Write action → только с явным approval. Три типа MCP серверов: internal, vendor-managed, community/open-source — начинать с allowlist approach.

Hallucination & AI Verification

Как AI ошибается, как проверять результат и R-C-T-F-Q структура промпта

Что такое hallucination

Hallucination — AI уверенно выдаёт технически правдоподобный, но неверный ответ: несуществующий класс/метод/endpoint, запрещённая библиотека, неверный fix, тест на мок вместо реального behaviour, MR description с выдуманными изменениями.

Как проверять AI-ответ
  • Сверить с реальными файлами/кодом.
  • Запустить тесты и build.
  • Проверить diff вручную.
  • Спросить: "на каких файлах основан ответ?"
  • Self-review с конкретными категориями.
  • "LGTM" без evidence = бесполезный review.
R-C-T-F-Q структура промпта (A0)
  • Role: кто AI (Senior Java Developer).
  • Context: stack, файлы, ограничения, diff.
  • Task: что сделать.
  • Format: формат ответа.
  • Quality bar: критерии хорошего ответа.

Правила для разработчика (A0/A1)

Research first, then implementation. Сначала AI находит релевантные файлы и объясняет flow, потом план, потом реализация после approval. Нельзя сразу просить менять код. Всегда запускать реальные validation commands (тесты, build, smoke check) — не доверять словам AI.

Типовые вопросы для assessment и хорошие ответы

Чем RAG отличается от Memory Bank?

RAG ищет релевантные фрагменты во внешних источниках. Memory Bank хранит устойчивый контекст и прошлые решения. RAG — “найти факт”, Memory — “помнить, что уже решили”.

Чем Skill отличается от Rule?

Skill описывает, как выполнять задачу. Rule задаёт ограничения: что можно, нельзя и какой стандарт соблюдать. При конфликте Rule выше Skill.

Что такое Checkpoint?

Checkpoint — структурный снимок состояния: цель, контекст, что сделано, решения, риски, next steps. Он лучше обычного summary для продолжения работы.

Может ли subagent использовать MCP?

Зависит от платформы и permissions. Иногда у subagent те же tools, иногда ограниченные, иногда tools доступны только главному агенту.

Что делать, если RAG противоречит Rule?

Rule выше, потому что RAG даёт факты, а Rule задаёт ограничения поведения. Нужно проверить, не устарел ли найденный документ.

Что делать, если Memory противоречит файлу?

Для актуального факта файл или tool-result обычно надёжнее. Memory могла устареть. Но нужно проверить, что файл действительно актуален.

Зачем нужен AGENTS.md?

Чтобы агент понимал проектный workflow: как запускать тесты, что нельзя менять, как оформлять изменения, какие команды и стандарты использовать.

Почему одна модель работает по-разному?

Потому что различаются system prompt, rules, context, tools, MCP, RAG, memory, temperature, permissions, IDE implementation и качество retrieved data.

Что важнее: текущий чат или Rules?

Текущий чат задаёт задачу, но Rules задают рамки. Если пользователь просит нарушить Rule, агент должен объяснить ограничение и предложить безопасный вариант.

Что такое MCP Protocol и как он работает?

Model Context Protocol — открытый стандарт (Anthropic, 2024) для подключения агентов к внешним сервисам. Три примитива: Resources (данные), Prompts (шаблоны), Tools (функции). Агент отправляет tool_use → MCP Server выполняет → возвращает tool_result. Один протокол для любых серверов без кастомных интеграций.

Как работает embedding в RAG?

Текст → embedding model → числовой вектор (768–3072 измерений). Похожие по смыслу тексты → близкие векторы. При поиске запрос тоже превращается в вектор и сравнивается с векторами в Vector DB через cosine similarity. Топ-K ближайших chunk'ов попадают в контекст.

Что такое ReAct паттерн?

ReAct = Reason + Act. Агент чередует шаги: Thought (что делать и зачем) → Action (вызов tool с аргументами) → Observation (результат tool). Повторяет цикл до финального ответа. Большинство современных AI-агентов работают по этому паттерну.

Как агент строит workflow?

Агент планирует шаги, определяет нужные tools, выполняет шаги последовательно или параллельно, обрабатывает ошибки (retry/Reflexion), при критических действиях останавливается для human approval. Цикл: Perceive → Plan → Act → Observe → повтор до цели.

Что такое hallucination и как бороться?

Hallucination — модель уверенно выдаёт неверный факт. Методы снижения: RAG (грундируем на реальных данных), tool-use (проверяем через API/файлы), structured output, temperature снижение, явное "скажи 'не знаю' если не уверен" в промпте.

Что такое chunking в RAG?

Chunking — разбивка документов на фрагменты для индексации. Стратегии: fixed-size (по токенам с overlap), по абзацам, code-aware (по функциям), семантический. Плохой chunking = плохой retrieval. Overlap 10–20% помогает не терять контекст на границах.

Чем zero-shot отличается от few-shot?

Zero-shot: задача без примеров — модель полагается только на обучение. Few-shot: 1–5 примеров input→output в промпте, модель обучается "на лету". Few-shot повышает точность для специфичных форматов и нестандартных задач.

Что такое human-in-the-loop?

Паттерн: агент останавливает workflow и запрашивает подтверждение человека перед критическим действием (удаление данных, деплой, финансовые операции). Снижает риск автономных ошибок. Обязателен при необратимых операциях.

Чем MCP отличается от обычного REST API?

Обычный API — кастомная интеграция: свой формат, своя документация, свой SDK. MCP — один стандартный протокол для всех серверов. Агент с MCP-клиентом динамически узнаёт доступные tools (capability discovery) и работает с любым MCP-сервером без дополнительного кода.

Что такое temperature у LLM?

Temperature контролирует случайность ответов. Temperature=0: детерминированный, всегда самый вероятный токен. Temperature=1+: творческий, разнообразный. Для агентов с tools рекомендуется низкая temperature (0–0.2) для предсказуемости и правильных tool calls.

Что такое reranking в RAG?

После первичного retrieval top-K кандидатов cross-encoder reranker переоценивает каждый попарно с запросом — более точно, чем cosine similarity. Результат: top-1–3 действительно релевантных chunk'а из 20 кандидатов. Дороже, но точнее.

Когда нужен Plan-then-Execute vs ReAct?

Plan-then-Execute: задача хорошо определена, шаги известны заранее, нужна предсказуемость. ReAct: результат каждого шага влияет на следующий, много неопределённости, нужна адаптация. На практике часто комбинируют: высокоуровневый план + ReAct внутри каждого шага.

Что такое "lost in the middle" эффект?

Модель хуже использует информацию из середины длинного контекста — лучше всего помнит начало (system prompt) и конец (последнее сообщение). Решение: важное класть в начало или конец, использовать структурированные маркеры, не надеяться на "найдёт сам" в середине.

Что такое Reflexion паттерн?

Reflexion — агент генерирует ответ, затем критически оценивает его ("что пошло не так?"), генерирует вывод о причине ошибки и повторяет попытку с учётом самокритики. Полезен, когда первый ответ явно неудовлетворительный и есть критерий качества.

Как контролировать что subagent не нарушит Rules?

Передавать subagent'у релевантные Rules/AGENTS.md явно в задаче. Главный агент проверяет результат subagent'а на соответствие Rules перед применением. Subagent не имеет автономного доступа к Rules главного агента — нужно явно включать.

Зачем нужен Chain-of-Thought (CoT)?

CoT заставляет модель "думать вслух" — пошагово рассуждать перед ответом. Снижает ошибки на сложных логических, математических и многошаговых задачах. Достигается добавлением "Think step by step" или few-shot примерами с рассуждениями.

Чем Cline skill отличается от Cline workflow?

Skill = on-demand expertise для области (mr-self-review, анализ логов). Workflow = последовательность шагов для типовой задачи (catalog-field-change). Skill отвечает "как проверять/анализировать", workflow — "какие шаги выполнять". Хранятся в .cline/skills/ и .clinerules/workflows/.

Чем Cursor command отличается от Cursor rule?

Rule = persistent project context, применяется всегда (.cursor/rules/) — задаёт что можно/нельзя, code style, архитектуру. Command = slash-команда для конкретного workflow (.cursor/commands/) — описывает какие шаги выполнять. Rule постоянный, command вызывается явно.

Что такое x-request-id в Netcracker workflow?

Correlation ID для трассировки запроса через систему. Находят в Jira ticket (description/comments) через jira-mcp → используют graylog-mcp для поиска логов по этому ID → строят timeline инцидента. Ключевой артефакт cross-MCP расследования.

Что такое 7-этапный AI workflow (A3)?

Research → Plan → Approval → Implementation → Validation → Review → Documentation. Важно не пропускать этапы: нельзя сразу менять код (риск неверных файлов), нельзя пропускать validation (код может выглядеть правильно, но ломать build), нельзя пропускать self-review.

Что такое agentic workflow vs one-off feature (A4)?

One-off feature — выполнить задачу один раз. Agentic workflow — переиспользуемая процедура с workstreams, approval gates, quality checks, subagents. A4 результат: не просто реализованный /shop/bundles, а workflow catalog-sales-journey который можно применить для любого нового sales journey.

Чем AGENTS.md отличается от CLAUDE.md?

Это два названия одного подхода в разных инструментах. Cursor и OpenAI Codex читают AGENTS.md. Claude Code использует CLAUDE.md (проектный) и ~/.claude/CLAUDE.md (глобальный). Cline — .clinerules/. Принцип одинаков: файл объясняет агенту роль, workflow, ограничения и команды в проекте.

Зачем нужен .cursorignore?

Исключает файлы из AI-контекста (синтаксис как .gitignore): node_modules, generated JSON, dist, бинарники, secrets. Меньше шума → лучше качество ответа, меньше токенов. Важно: это не .gitignore — оба файла существуют независимо. Claude Code реализует то же через exclusion patterns в настройках.

Что такое always-apply Rule vs auto-attach Rule?

Always-apply (.cursor/rules/): применяется в каждом чате автоматически — глобальные стандарты проекта. Auto-attach: применяется только к файлам, подходящим под паттерн (например, *.java). Conditional: включается явно через @ в чате. Always-apply — наивысший приоритет в иерархии правил.

Как Claude выбирает нужный Skill из 100+ доступных?

По полю description в YAML frontmatter SKILL.md. При старте только name+description всех скиллов грузятся в системный промпт. SKILL.md читается только когда skill релевантен. Поэтому description должно: быть от 3-го лица, содержать ключевые слова, описывать КОГДА использовать. "Помогает с файлами" — плохой description, Claude не выберет такой skill.

Что такое прогрессивное раскрытие в SKILL.md?

Архитектурный принцип: SKILL.md содержит обзор и ссылки на детальные файлы (FORMS.md, reference.md, scripts/). Claude читает SKILL.md когда skill активирован, а дополнительные файлы — только по мере необходимости. Это экономит токены. Ключевое правило: ссылки только 1 уровня глубины (SKILL.md → файл, не SKILL.md → A.md → B.md).

В чём разница между degrees of freedom в Skill?

High freedom (текстовые инструкции): несколько подходов допустимы, контекст определяет лучший (code review). Medium freedom (псевдокод/шаблоны): есть предпочтительный паттерн, вариации допустимы (generate report). Low freedom (точные скрипты): задача хрупкая, нужна согласованность (DB migration — точная команда, без флагов). Чем хрупче задача, тем ниже свобода.

⚠️ Каверзные вопросы: где легко ошибиться на assessment

Эти вопросы звучат просто, но типичный ответ — неверный. Знать термин недостаточно: нужно понять почему стандартная реакция ошибочна.

🔴 Subagent может редактировать файлы сам?

Нет. По умолчанию subagent read-only (A4 safe model). Файлы меняются ТОЛЬКО в main Agent flow после approval. Subagent возвращает findings → главный агент решает → человек подтверждает → только тогда edit.

🔴 AI self-review написал "LGTM" — достаточно?

Нет. Хороший self-review проверяет конкретные категории с evidence: scope, тесты, контракты, generated files, security/privacy. "LGTM" без ссылок на файлы — формальность, не реальная проверка. Нужно ещё и вручную проверить.

🔴 Можно закоммитить mcp.json с реальными токенами?

НИКОГДА. Реальные токены, пароли, credentials — только в локальном конфиге (~/.cursor/mcp.json). В Git, Confluence, артефактах — только плейсхолдеры <JIRA_BEARER_TOKEN>. Нарушение — security incident.

🔴 Generated JSON из Excel можно редактировать вручную?

Нет. Excel/source data — source of truth. JSON должен генерироваться через import с трейсабилити (sourceFile, sourceSheet, sourceRow, importedAt). Ручные правки ломают трейсабилити и делают state неверифицируемым.

🔴 Skill говорит установить библиотеку, Rule запрещает. Что делать?

Rule выше. Skill — способ выполнения, не право нарушать policy. Нужно адаптировать Skill: найти способ без новой зависимости, или получить явный approval в исключительном случае.

🔴 Пользователь: "быстро поменяй без тестов". Что делать?

Не игнорировать AGENTS.md. Текущий запрос не отменяет проектный workflow. Объяснить риск, предложить минимальный smoke check. AGENTS.md/Rules выше сиюминутного запроса пользователя.

🔴 MCP tool вернул "success" — задача выполнена?

Нет. Tool result нужно интерпретировать и верифицировать. "Success" может быть неполным или ложным. Нужно проверить: тесты, build, реальное состояние системы. Tool result = данные для рассуждения, не окончательный вердикт.

🔴 Можно сразу просить AI менять файлы?

Нет (A0/A1/A3). Research first, then plan, then approval, then implementation. Цена пропуска: изменение неверных файлов, scope creep, сломанный API контракт. Cursor: Ask mode → планирование → Agent mode после подтверждения.

🔴 Контекст переполнен — продолжать?

Нет без управления. При переполнении модель "забывает" ранние части. Решения: structured checkpoint, summary/compression, subagent, убрать лишние attachments. Нельзя надеяться что модель "запомнит сама".

🔴 AI написал "все тесты прошли" — верить?

Нет. Всегда запускай реальные validation commands самостоятельно. AI может иметь устаревший контекст или ошибиться. Только реальный build pass/test green = факт. Это ключевое правило A0.

🔴 Что должен включать Agent Card?

Девять полей: Name, Purpose, Inputs, Outputs, Allowed tools, Forbidden actions, Approval gates, Quality checks, Risks. Отсутствие Forbidden actions — критическая ошибка. Агент без ограничений — небезопасный агент.

🔴 RAG нашёл документ — можно использовать?

Проверить сначала. Документ может быть устаревшим (старая версия, другой продукт). Проверить метаданные (дата, версия). Если противоречит Rules — Rules выше. Для актуального состояния системы лучше проверить через MCP/tool.

🔴 Какие данные нельзя отправлять в AI?

Никогда: секреты, токены, пароли, production URLs с credentials, реальные данные клиентов, личные данные, коммерчески чувствительная информация. Использовать анонимизированные примеры и плейсхолдеры.

🔴 Чем Cursor Ask mode отличается от Agent mode?

Ask mode: research и планирование — читает файлы, объясняет, предлагает план. Agent mode: выполняет изменения после подтверждения. Правило A1/A3: сначала Ask/Plan mode, потом Agent mode только после согласованного плана.

🔴 Зачем нужны approval gates если есть тесты?

Они не заменяют друг друга. Тесты проверяют корректность кода. Approval gates — контрольные точки для человека: подтвердить план, API изменение, финальный diff. Вместе: тесты = автоматика, gates = человеческий контроль ключевых решений.

🔴 Для чего нужен x-request-id в Netcracker?

Correlation ID для cross-system tracing. Находят в Jira (description/comments) → ищут в Graylog логи → строят timeline. Cross-MCP workflow: jira-mcp (найти ID) → graylog-mcp (найти логи) → AI (investigation summary) → Human (решение).

🔴 Memory сказала одно, актуальный файл — другое. Что выбрать?

Актуальный файл/tool-result для фактов. Memory может устареть. Но проверь: не является ли файл тоже устаревшей копией? Для актуального состояния системы: tool-result > Memory > RAG. Для правил поведения: AGENTS.md/Rules сильнее Memory.

🔴 Reusable workflow vs один раз сделал задачу (A4)?

Reusable workflow — это runbook, который другой разработчик может применить к следующей похожей задаче. Один раз = нет ценности для команды. A4 артефакт должен доказывать переиспользуемость: описать inputs, steps, checks, constraints так, чтобы это работало для следующего sales journey.

🔴 SKILL.md в 1000 строк — это нормально?

Нет — нарушает принцип лаконичности. Лимит: 500 строк для тела SKILL.md. Всё сверх этого должно быть в отдельных reference файлах (FORMS.md, reference.md, scripts/). Иначе каждый раз когда skill активен, весь SKILL.md занимает токены контекста. Прогрессивное раскрытие: Claude читает доп. файлы только по мере необходимости.

🔴 Я написал Rule про стек проекта — этого достаточно для агента?

Нет. Rule говорит "что нельзя". Для полной конфигурации нужны: AGENTS.md (роль, команды, workflow агента), Rules (запреты, стандарты), Commands (конкретные процессы). Rule без AGENTS.md — агент знает ограничения, но не знает роль и workflow. AGENTS.md без Rules — не знает ограничений.

🔴 .cursorignore не нужен — node_modules же в .gitignore?

Это разные системы. .gitignore не влияет на Cursor AI контекст. Cursor индексирует всё кроме явно исключённого в .cursorignore. node_modules без .cursorignore = тысячи файлов в AI контексте = шум, медлительность, плохие ответы. Обязательно иметь оба файла для нормальной работы.

🔴 Description скилла: "Helps process documents efficiently and quickly" — хорошо?

Нет — три проблемы. 1) Расплывчато: не понятно какие документы, 2) нет ключевых слов для обнаружения, 3) нет указания КОГДА использовать. Хорошо: "Extracts text and tables from PDF files, fills forms, merges documents. Use when working with PDF files or user mentions PDFs, forms, document extraction." — конкретно + триггеры.

Типовой ответ: “Я разделяю слои: правила поведения — в AGENTS.md/Rules, знания — в Memory/RAG, действия — через MCP/tools, повторяемые процессы — в Skills, передачу работы — через Subagents.”
Главный риск: смешать всё в один слой. Тогда агенту сложнее понять, что является правилом, что фактом, что памятью, а что просто способом выполнения.
Для подготовки: умей объяснить разницу не по определению, а по кейсам: “где хранить?”, “что выше?”, “что делать при конфликте?”, “когда проверять через tool?”.
