Hooks — принуждение на уровне shell

Hooks — shell-скрипты (или prompt-проверки), которые выполняются до или после действий агента. Жёстче Rules: hook может заблокировать tool call на системном уровне. Cursor: .cursor/hooks.json. Claude Code: .claude/settings.json.

Что такое Hooks и зачем они нужны

Hooks = enforcement

Rules — инструкции агенту ("не делай X"). Агент может их проигнорировать при длинном контексте. Hooks — системный уровень: shell-скрипт выполняется до/после каждого tool call и может вернуть ненулевой exit code → действие блокируется.

Cursor: hooks.json

Проект: .cursor/hooks.json и скрипты в .cursor/hooks/. User: ~/.cursor/hooks.json. События в camelCase (preToolUse, beforeShellExecution, afterFileEdit…). Блокировка через JSON permission: "deny" или exit code 2; опция failClosed.

Claude Code: settings.json

.claude/settings.json (project) или ~/.claude/settings.json (user). События в PascalCase: PreToolUse, PostToolUse, Stop. Matcher на имя tool (Bash, Write). Блокировка — ненулевой exit code скрипта.

Когда применять

  • Секреты не должны попасть в commit — никогда
  • Production DB нельзя изменять без approval
  • Все tool calls должны логироваться (compliance)
  • Внешние команды ограничены (security sandbox)

Как работает Hook — flow выполнения

1

Agent решает

Агент хочет вызвать tool (Bash, Write, Read...)

2

PreToolUse hook

Shell-скрипт запускается с инфо о вызове. Может заблокировать.

3

Tool выполняется

Только если hook вернул exit 0. Иначе — агент получает ошибку.

4

PostToolUse hook

Запускается после tool. Для аудита, логирования, проверки результата.

5

Результат агенту

Tool output + hook stdout добавляются в context агента.

Ключевой механизм: hook возвращает ненулевой exit code → tool call отменяется. Агент видит ошибку и не может продолжить запрещённое действие — это и есть принуждение.

Типы Hooks

🛑

PreToolUse

Запускается ДО выполнения tool call. Может заблокировать.

до выполнения

Применение: проверка секретов перед git commit, валидация SQL перед запросом, требование approval перед destructive action.

Что получает скрипт: JSON с tool name, tool input (аргументы), session info — через stdin или env переменные.

# scripts/check-secrets.sh #!/bin/bash # Читаем tool input из stdin (JSON) INPUT=$(cat) COMMAND=$(echo "$INPUT" | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('input',{}).get('command',''))") # Проверяем на паттерны секретов if echo "$COMMAND" | grep -qiE "(password|secret|api_key|token)="; then echo "BLOCKED: potential secret in command" >&2 exit 1 # ненулевой exit → tool блокируется fi exit 0 # OK → tool выполняется
📋

PostToolUse

Запускается ПОСЛЕ выполнения tool call. Для аудита и логирования.

после выполнения

Применение: логировать все tool calls в audit log, проверять результат на нарушения, отправлять метрики.

Важно: PostToolUse не может отменить уже выполненное действие, но может добавить информацию в context агента.

# scripts/audit-log.sh #!/bin/bash INPUT=$(cat) TOOL=$(echo "$INPUT" | python3 -c "import sys,json; print(json.load(sys.stdin).get('tool_name',''))") TIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ") echo "[$TIMESTAMP] tool=$TOOL" >> ~/.claude/audit.log exit 0
🔔

Stop

Запускается когда агент завершает работу (перед остановкой).

при остановке

Применение: отправить уведомление о завершении задачи, обновить статус в трекере, создать checkpoint.

# scripts/notify-done.sh #!/bin/bash INPUT=$(cat) # Можно отправить webhook, обновить Jira, создать файл checkpoint echo "Task completed at $(date)" >> ~/agent-log.txt exit 0

📖 Claude Code — Hooks

Права доступа: скрипты должны быть исполняемыми (chmod +x). В Cursor command-hook может вернуть JSON {"permission":"deny","agent_message":"…"} вместо exit 1; exit 2 тоже блокирует. В Claude Code достаточно exit 1 на PreToolUse.

Тонкости и нюансы

Hooks vs Rules — в чём реальная разница

Rules — текстовые инструкции агенту ("никогда не делай X"). В теории агент может их проигнорировать при слишком длинном context window (эффект "lost in the middle"). Hooks — системный уровень: скрипт выполняется в OS, агент физически не получает результат tool если hook вернул exit 1. Это настоящее enforcement, а не просьба.

Cursor vs Claude Code — разный синтаксис, одна идея

Имена событий: Cursor preToolUse / Claude PreToolUse. Имена shell-tool: Cursor Shell / Claude Bash. Cursor — плоский массив с command в записи; Claude — вложенный hooks[]. Cursor: matcher = JavaScript regex (не POSIX). Для только терминала в Cursor удобнее узкое событие beforeShellExecution вместо общего preToolUse.

Что агент получает в stdin

Hook скрипт получает JSON через stdin:

{ "tool_name": "Bash", "tool_use_id": "toolu_abc123", "input": { "command": "git add . && git commit -m 'fix'" }, "session_id": "session_xyz" }

Hook stdout попадает в context агента

Всё что скрипт выводит в stdout — агент видит как дополнительный контекст. Это можно использовать: например, hook проверяет код и добавляет в context "найдено 3 предупреждения безопасности: [список]". stderr — только для системных логов, агент не видит.

User vs project — где лежит конфиг

Cursor: project .cursor/hooks.json (в git) · user ~/.cursor/hooks.json — пути к скриптам относительно корня проекта или ~/.cursor/ соответственно.
Claude Code: project .claude/settings.json · user ~/.claude/settings.json. Project перекрывает user при конфликте. Командные security-hooks версионируй в репозитории.

Примеры реальных hooks

Запрет опасных Bash команд

#!/bin/bash INPUT=$(cat) CMD=$(echo "$INPUT" | python3 -c \ "import sys,json; print(json.load(sys.stdin) .get('input',{}).get('command',''))") # Запрет на drop таблиц и force push DANGEROUS="(DROP TABLE|DROP DATABASE|force-push|--force)" if echo "$CMD" | grep -qiE "$DANGEROUS"; then echo "Blocked: dangerous command detected" exit 1 fi exit 0

Требование approval для write в prod

#!/bin/bash INPUT=$(cat) PATH=$(echo "$INPUT" | python3 -c \ "import sys,json; print(json.load(sys.stdin) .get('input',{}).get('path',''))") # Блокируем write в prod/configs if echo "$PATH" | grep -q "production\|prod/"; then echo "Blocked: write to production requires manual review" exit 1 fi exit 0

Полный audit log

#!/bin/bash # PostToolUse — логируем ВСЕ tool calls INPUT=$(cat) TOOL=$(echo "$INPUT" | python3 -c \ "import sys,json; print(json.load(sys.stdin) .get('tool_name','unknown'))") TS=$(date -u +"%Y-%m-%dT%H:%M:%SZ") PROJECT=$(basename "$(pwd)") echo "$TS | $PROJECT | $TOOL" >> \ ~/.claude/tools-audit.log exit 0

Проверка .env перед read

#!/bin/bash # PreToolUse для Read — блокируем .env INPUT=$(cat) FPATH=$(echo "$INPUT" | python3 -c \ "import sys,json; print(json.load(sys.stdin) .get('input',{}).get('path',''))") if echo "$FPATH" | grep -qE "\.env$|\.env\."; then echo "Blocked: reading .env files is restricted" exit 1 fi exit 0

Best practices — Делай / Не делай

✅ Делай

  • Используй PreToolUse для security-критичных проверок (секреты, prod, destructive)
  • Пиши хуки идемпотентными — повторный вызов не ломает систему
  • Логируй через PostToolUse все tool calls в audit файл
  • Добавляй понятный echo "Blocked: причина" перед exit 1 — агент увидит причину
  • Храни скрипты в git: .cursor/hooks/ + hooks.json или scripts/ + .claude/settings.json
  • Тестируй hook локально: echo '{"tool_name":"Bash","input":{"command":"drop table"}}' | bash scripts/check.sh

❌ Не делай

  • Не заменяй хуки правилами там, где нужно жёсткое принуждение — Rules LLM может "забыть"
  • Не добавляй тяжёлую логику в PreToolUse — это блокирует агента, должно быть быстро
  • Не пиши хуки без chmod +x — скрипт не выполнится
  • Не используй exit 1 для "рекомендаций" — только для реальных блокировок
  • Не логируй секреты в audit log — обрабатывай содержимое command перед записью
  • Не забывай: PostToolUse не отменяет уже выполненное — только PreToolUse может заблокировать

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Hook?
Hook — скрипт (или prompt-проверка в Cursor), выполняемый автоматически до или после действия агента. Claude Code: .claude/settings.json, ненулевой exit code блокирует PreToolUse. Cursor: .cursor/hooks.json, JSON permission или exit code. Это enforcement вне LLM: агент не выполняет tool, если hook отклонил.
Junior Что такое PreToolUse Hook?
Тип Hook, запускаемый до выполнения tool call. Получает JSON через stdin: tool_name, input (аргументы), session_id. Exit 0 → tool выполняется. Exit 1 → tool блокируется, stdout скрипта передаётся агенту как объяснение. Применение: валидация, проверка безопасности, approval-gates.
Junior Что такое PostToolUse Hook?
Тип Hook, запускаемый после выполнения tool call. Действие уже произошло — отменить нельзя. Stdout скрипта добавляется в context агента. Применение: audit logging (запись всех tool calls), анализ результата, уведомления. PostToolUse exit 1 сигнализирует о проблеме пост-фактум, но не откатывает действие.
Junior Где хранится конфигурация Hooks?
Cursor: .cursor/hooks.json (project) или ~/.cursor/hooks.json (user), скрипты в .cursor/hooks/. Claude Code: .claude/settings.json или ~/.claude/settings.json, часто скрипты в scripts/. Project-конфиг версионируй в git — команда получает одинаковые hooks при клонировании.
Junior Чем Hook отличается от Rule?
Rule — текстовая инструкция в LLM context: агент читает и старается следовать, но при длинном контексте может «забыть». Hook — shell-скрипт на OS-уровне: агент физически не получает результат tool call если Hook вернул exit 1. Это не LLM-уровень, а системный. Hook нельзя обойти из промпта.
Junior Что такое shell hook скрипт?
Обычный shell/Python/bash скрипт, который Hook вызывает при каждом matching tool call. Получает JSON через stdin, возвращает exit code и stdout. Пример: check_secrets.sh читает путь файла из stdin, grep-ает на паттерны API keys, возвращает exit 1 если найдены. Скрипт должен быть исполняемым: chmod +x script.sh.
Junior Какие типы событий (hook types) доступны в Claude Code?
4 типа: PreToolUse — до выполнения tool; PostToolUse — после выполнения tool; Stop — когда агент завершает работу; SubagentStop — когда завершает работу subagent. Каждый тип настраивается отдельно с matcher (какой tool) и command (что запустить).
Junior Что происходит если Hook возвращает exit 1?
Для PreToolUse: tool call отменяется. Агент получает stdout скрипта как объяснение почему заблокировано. Агент может попробовать альтернативный подход. Для PostToolUse: tool уже выполнился, exit 1 — сигнал о проблеме пост-фактум, агент знает что что-то пошло не так и может компенсировать.
Junior Есть ли Hooks в Cursor IDE?
Да. В Cursor — .cursor/hooks.json (schema version: 1), command- и prompt-hooks на события вроде preToolUse и beforeShellExecution. Блокировка через JSON-ответ (permission: "deny") или exit code. Это OS-уровень, как в Claude Code, но другие имена событий и tools (Shell вместо Bash). Rules и CI остаются дополнением, не заменой hooks.
Junior Как включить или выключить конкретный Hook?
Cursor: удали запись из массива в hooks.json — файл перечитывается при сохранении (иногда нужен restart Cursor). Claude Code: удали запись из .claude/settings.json — обычно со следующей сессии. JSON без комментариев: отключение = удаление объекта из массива. Скрипты на диске можно оставить.
Middle — 10 вопросов
Middle Когда Hook необходим, а Rule недостаточно?
Rule недостаточно когда нарушение неприемлемо ни при каких обстоятельствах: запись секретов в репо, deploy в production без approval, удаление production данных. LLM может «не заметить» Rule при длинном контексте. Hook необходим для: security compliance, regulatory requirements, irreversible actions. Правило: всё что нельзя допустить ни разу — только Hook.
Middle PreToolUse для validation vs PostToolUse для logging — в чём разница в применении?
PreToolUse = prevention: проверка до выполнения, может заблокировать. Используй для: проверки секретов, lint check, approval gates, rate limiting. Должен быть быстрым (<500ms). PostToolUse = observation: анализ после, не может отменить. Используй для: audit logging, метрик, добавления context для агента. Может быть медленнее — не блокирует агента.
Middle Как написать hook скрипт проверяющий файл перед записью?
Структура PreToolUse hook для Write tool: #!/bin/bash\nINPUT=$(cat)\nFILE=$(echo "$INPUT" | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['input']['path'])")\nCONTENT=$(echo "$INPUT" | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['input']['content'])")\nif echo "$CONTENT" | grep -qE '(password|api_key|secret)\s*=\s*["'\''`][^"'\''`]+'; then\n echo 'ERROR: Detected possible secret in file'; exit 1\nfi\nexit 0
Middle Как именно Hook блокирует действие агента технически?
Механизм: Claude Code запускает hook скрипт как subprocess перед tool call. Передаёт JSON через stdin (tool name, args, session). Если subprocess завершился с exit code != 0 → Claude Code не выполняет tool call и передаёт агенту stdout скрипта как ошибку. Агент получает сообщение типа «Action blocked: [stdout hook]» и реагирует на него. Это OS-уровень, не LLM-уровень.
Middle Как использовать Hooks для security compliance?
Паттерны: (1) No-secrets: PreToolUse на Write — сканирование контента на API keys, passwords; (2) No-prod: PreToolUse на Bash — блокировка команд с prod-хостами; (3) Approval gate: PreToolUse на Deploy — проверка флага в файле/env что деплой авторизован; (4) Audit trail: PostToolUse на все tools — запись в лог файл. Специализированные инструменты лучше: trufflehog, detect-secrets вместо самописного grep.
Middle Чем Hooks отличаются от Permissions (allow/deny) в settings.json?
Permissions — статический список разрешённых/запрещённых tools и путей. Пример: "allow": ["Bash(git *)", "Write(src/*)"]. Это declarative: или разрешено или нет. Hooks — dynamic: shell-скрипт с произвольной логикой. Может проверять содержимое файла, время суток, внешний API, флаги deployment. Permissions — простые ограничения; Hooks — сложная условная логика.
Middle Как тестировать Hooks до выхода в production?
Стратегия: (1) Запускай скрипт вручную с тестовым stdin: echo '{"tool_name":"Write","input":{"path":"test.env","content":"key=value"}}' | ./scripts/check_secrets.sh; (2) Тестируй граничные случаи: файл с настоящим секретом, без секрета, edge cases; (3) Проверяй exit codes: убедись что exit 1 возвращается именно когда нужно; (4) Staging environment с теми же hooks перед production.
Middle Какой performance impact от Hooks на скорость агента?
Каждый PreToolUse Hook добавляет latency к каждому tool call. Если hook занимает 2 секунды, а агент делает 30 tool calls — 60 секунд overhead. Оптимизация: (1) PreToolUse должен выполняться <500ms; (2) фильтруй по matcher — не запускай тяжёлый hook для каждого Read; (3) кешируй результаты для неизменившихся файлов; (4) тяжёлый анализ → PostToolUse или отдельный CI step.
Middle Как настроить Hooks для audit logging всех tool calls?
PostToolUse Hook с matcher ".*" (все tools): #!/bin/bash\nTIMESTAMP=$(date -u +"%Y-%m-%dT%H:%M:%SZ")\nINPUT=$(cat)\necho "$TIMESTAMP $INPUT" >> ~/.claude/audit.log\nexit 0. Добавь ротацию логов (размер или дата). Логируй: timestamp, tool_name, input summary, session_id. Для серьёзного audit: отправляй в centralized logging (ELK, Datadog).
Middle Приведи примеры реальных Hooks: no-secrets check и lint check
No-secrets PreToolUse (Write): сканирует content файла grep'ом на паттерны api_key=, password=, secret= — блокирует если найдено. Лучше использовать detect-secrets или gitleaks. Lint PreToolUse (Write для *.py): после записи файла — запускает flake8 или ruff, если ошибки → exit 1 с сообщением. Важно: lint check лучше как PostToolUse (быстрое PreToolUse) или post-write callback.
Senior — 5 вопросов
Senior Как спроектировать архитектуру Hooks для enterprise compliance?
Многоуровневая система: (1) Global Hooks (~/.claude/settings.json) — universal security для всех проектов: secrets scan, audit log; (2) Project Hooks (.claude/settings.json) — домен-специфичные: billing approval gate, prod-protection; (3) Shared scripts repository — библиотека проверенных hook скриптов для всей организации; (4) CI/CD duplication — те же проверки в pipeline как fallback. Project перекрывает global — проектные нужды приоритетнее.
Senior Чем Hooks отличаются от OS-level access controls и когда что использовать?
OS controls (AppArmor, SELinux, filesystem permissions, containers) — работают независимо от агента, не знают о семантике AI задачи. Hooks — контекстно-зависимы: знают tool_name, session_id, конкретные аргументы. Hooks подходят для: бизнес-логики (deploy только по пятницам), context-aware блокировок. OS controls — для: изоляции процессов, filesystem sandboxing, network policies. Архитектура: оба слоя вместе — defence in depth.
Senior Как диагностировать failing Hook который не блокирует то что должен?
Чеклист: (1) скрипт исполняемый? (ls -la scripts/ → chmod +x); (2) путь в settings.json корректный? (абсолютный vs относительный); (3) matcher совпадает с tool name? (регистр важен: «Write» не «write»); (4) JSON в settings.json валидный? (синтаксическая ошибка игнорирует весь файл); (5) скрипт возвращает правильный exit code? (echo 'test_input' | ./script.sh; echo $?); (6) логика условия верна? Добавь set -x в скрипт для debug.
Senior Как реализовать цепочку Hooks (hooks composition) для многошаговой проверки?
Два паттерна: (1) Single orchestrator script: один hook скрипт последовательно вызывает несколько проверок, возвращает первую ошибку или exit 0; (2) Multiple hooks on same event: несколько записей в settings.json для одного matcher — все выполняются, первый non-zero блокирует. Рекомендация: orchestrator лучше для связанных проверок (secrets → lint → size limit); multiple records — для независимых проверок разных команд/доменов.
Senior Как использовать Hooks для координации multi-agent систем?
PreToolUse Hooks как coordination layer: (1) Locking: проверяет файл блокировки перед критическими операциями — блокирует если другой агент держит lock; (2) State validation: агент A читает state, Hook проверяет что state консистентен перед тем как агент B продолжит; (3) Sequencing: PostToolUse пишет в shared state, следующий агент проверяет через PreToolUse. Ограничение: Hooks синхронные, нет встроенной message passing между агентами.
Lead — 5 вопросов
Lead Как выстроить governance Hooks политику в команде?
Governance framework: (1) Mandatory Hooks список документирован в AGENTS.md — какие Hooks обязательны и почему; (2) Owner per hook — каждый скрипт имеет ответственного; (3) PR review для .claude/settings.json — изменения проходят review; (4) Hook scripts в shared repo — централизованная библиотека, не копирование по проектам; (5) Alert на hook failures — если обязательный Hook сломан, команда должна знать.
Lead Какие Hooks обязательны для production-окружения?
Минимальный обязательный набор: (1) Secrets scanner (PreToolUse, Write) — никаких credentials в коде; (2) Production guard (PreToolUse, Bash) — блокировка destructive команд на prod хостах без явного approval; (3) Audit log (PostToolUse, all) — полный trail всех tool calls; (4) File scope guard (PreToolUse, Write) — запись только в разрешённые директории. Дополнительно для финтех/медтех: PII detection, HIPAA/PCI compliance checks.
Lead Как включить Hooks в onboarding процесс?
Onboarding с Hooks: (1) .claude/settings.json версионируется в репо — новый разработчик клонирует → Hooks активны автоматически; (2) hook скрипты в .claude/scripts/ или scripts/hooks/ также в репо; (3) onboarding checklist: «убедись что hook скрипты исполняемые (chmod +x)»; (4) тест: попробуй записать файл с fake API key — должен быть заблокирован; (5) документация в AGENTS.md: какие Hooks активны и что они проверяют.
Lead Как построить audit trail через Hooks?
Полноценный audit trail: (1) PostToolUse на все tools (matcher: .*); (2) Логируй: timestamp (UTC), tool_name, session_id, input summary, user; (3) Append-only лог файл: ~/.claude/audit.log или centralized; (4) Log rotation: daily files, retention policy (30/90 дней); (5) Для compliance: отправка в SIEM (Splunk, Datadog) в реальном времени; (6) Алерты на suspicious patterns: массовые deletes, prod access в нерабочее время.
Lead Как распределить ответственность за Hooks между командами?
Матрица ответственности: (1) Security team: owns secrets scanner, audit trail — пишут, review, мониторят; (2) Platform/DevOps: owns prod guards, deployment Hooks; (3) Feature teams: owns domain-specific Hooks для своих сервисов; (4) Tech Lead: финальный review всех изменений в .claude/settings.json. Shared hook scripts repository: Platform team владеет, но принимает PR от всех команд.
Architect — 5 вопросов
Arch Как Hooks образуют security enforcement layer в AI-системе?
Hooks — второй после OS controls слой в AI security stack: [OS/Container isolation] → [Claude Code Hooks] → [LLM Rules/AGENTS.md] → [User]. Hooks уникальны тем что: знают семантику AI-действий (tool_name, аргументы), работают в real-time до/после каждой операции, программируемы бизнес-логикой. Архитектурный принцип: Hooks не заменяют OS security — они добавляют application-level semantic layer поверх infrastructure controls.
Arch Как спроектировать pipeline архитектуру для Hooks?
Hook pipeline для enterprise: (1) Global pre-checks (~/.claude/settings.json): secrets, audit; (2) Project checks (.claude/settings.json): domain logic, prod guards; (3) Orchestrator pattern: один entry-point script вызывает модульные проверки последовательно; (4) Plugin architecture: hook script загружает проверки из hooks.d/ директории — команды добавляют проверки без изменения основного скрипта; (5) shared library: общие функции (parse_stdin, log_action) в sourced файле.
Arch Как Hooks и Rules образуют defence in depth для AI агента?
Два уровня одной политики: Rules (LLM-level) — educates агента: «не пиши секреты, это опасно»; срабатывает в 95% случаев, но не надёжен при context saturation. Hooks (OS-level) — enforces: физически блокирует запись если паттерн найден; срабатывает в 100% случаев независимо от состояния LLM. Принцип: критические политики ВСЕГДА дублируются на обоих уровнях. Rule объясняет, Hook гарантирует.
Arch Как использовать Hooks для observability в production AI-системе?
Observability через Hooks: (1) Metrics: PostToolUse считает tool calls по типам → Prometheus/Datadog; (2) Tracing: session_id как trace ID, tool calls как spans; (3) Anomaly detection: Hook отправляет в stream processor, ML модель детектирует аномалии (необычно много deletes, ночная активность); (4) Business KPIs: сколько PR создано агентом, сколько файлов изменено, deploy frequency через AI. Hooks — идеальная injection точка для observability без изменения агента.
Arch Как выработать cross-team стандарты для Hooks?
Стандартизация на уровне организации: (1) Shared hooks library: git репо с проверенными скриптами, SemVer версионирование; (2) Hook contract: стандарт stdin/stdout формата, exit code semantics; (3) Testing framework: набор тестовых JSON fixtures для каждого hook; (4) Rollout process: новый обязательный Hook → pilot в одной команде → gradual rollout → всем; (5) Exception process: как команда запрашивает исключение из обязательного Hook с justification.
