Skills

Переиспользуемые инструкции для конкретных типов задач. В Claude Code — SKILL.md с YAML frontmatter, агент может выбрать автоматически. В Cursor — Commands (/command-name). Сравнение в конце страницы.

Что такое Skills и зачем они нужны

Skills = переиспользуемые инструкции

Не ограничения (это Rules), а пошаговые инструкции как выполнить конкретный тип работы. Code review, migration, debugging, onboarding — у каждого своя структура.

  • Стандартизируют процессы команды
  • Вызываются явно или автоматически по description
  • Могут описывать как использовать MCP tools
  • Версионируются в репозитории

Файловая структура (Claude Code)

Project-level

.claude/ └── commands/ ├── code-review.md ← /code-review ├── db-migration.md ← /db-migration └── onboarding.md ← /onboarding

Версионируются в git. Доступны всей команде.

Global-level

~/.claude/ └── commands/ ├── my-review.md ← личный стиль └── quick-debug.md

Личные skills. Работают во всех проектах. Не попадают в git.

Формат SKILL.md — полная структура

Полный шаблон

--- name: "code-review" description: "Thorough code review: correctness, security, design" # ↑ ≤100 символов, 3rd person. # По description агент решает # применять ли skill автоматически. version: "1.2" author: "team" --- # Code Review Skill ## Trigger Use when reviewing code, checking PR, or auditing implementation quality. ## Steps 1. Check correctness and edge cases 2. Verify tests are meaningful 3. Look for security issues 4. Evaluate design and readability ## Output format **[Severity]** Issue — file:line Severity: Critical > Major > Minor > Suggestion End with top-3 priority fixes. ## Constraints - Do NOT change code, only review - If no issues — say so explicitly - List total count by severity

Поля frontmatter

ПолеОбяз.Описание
name✓Slug для вызова. kebab-case, ≤50 символов.
description✓≤100 символов. Агент читает для автовыбора.
version—Версионирование изменений.
author—Команда / автор.

Как вызывать

# Явный вызов /code-review @.claude/commands/code-review.md # Из CLAUDE.md — список доступных Available commands: @.claude/commands/code-review.md @.claude/commands/db-migration.md # Автовыбор (по description) # Пользователь: "сделай review кода" # Агент: видит description, применяет
Description — ключевое поле. По нему агент решает применять ли Skill. Размытый description → skill игнорируется. Начинай с глагола: "Perform...", "Analyse...", "Generate..."

Cursor vs Claude Code — формат Skills

🖱 Cursor — Skills

Файл .cursor/skills/SKILL.md или путь указанный в fullPath. Агент читает skill по команде или при упоминании. Вызов: явно в задаче или через @skill-name.

# SKILL.md ## Trigger When user asks to create a PR ## Steps 1. Run git status 2. Create branch 3. Commit changes 4. Push and open PR ## Output PR URL returned to user
🤖 Claude Code — Skills

Файл skills/SKILL.md или в директории проекта. Загружается через settingSources: ["project"]. Агент вызывает skill по имени через Skill tool.

# SKILL.md (Claude Code) ## Trigger create-pr, open PR, push changes ## Steps 1. git status → check changes 2. git checkout -b feature/name 3. git commit -m "..." 4. git push origin HEAD 5. gh pr create --title "..." ## Return PR URL and branch name

Lifecycle вызова Skill

1

Trigger

Пользователь: /skill-name, или агент сам по description.

2

Load

SKILL.md загружается в context к текущей задаче.

3

Apply

Агент следует steps. Может вызывать MCP tools согласно инструкциям.

4

Output

Результат строго в формате описанном в Skill.

Тонкости использования

Автовыбор агентом по description — и почему description критичен

В Claude Code агент читает description всех доступных Skills и может применить подходящий без явного вызова. Пользователь пишет "сделай review" — агент находит Skill с description: "code review..." и применяет. Если description расплывчатый ("помоги с кодом") — агент его проигнорирует. Правило: начинай с глагола в 3-м лице, описывай конкретно что делает Skill.

Skills vs Rules — частая ошибка

"Никогда не коммить секреты" — это Rule, не Skill. Skill описывает как выполнить задачу. Rule описывает что нельзя. Если кладёшь политику в Skill — она применится только при явном вызове. Критические ограничения безопасности должны быть в Rules (alwaysApply:true) или в Hooks.

Context overhead при загрузке

SKILL.md целиком загружается в context при вызове. Большой Skill (500+ строк) = много токенов = дорого + агент хуже следует. Держи Skills сфокусированными: один Skill — одна задача. Если задача большая — раздели на sub-skills или используй ссылки внутри.

Skills + MCP — стандартизация workflow

Skill может содержать инструкции как использовать конкретные MCP tools. Это стандартизирует то как команда работает с реальными данными.

# В Skill для performance debug: 1. Use graylog-mcp: get errors last 24h 2. Use metrics-mcp: check p95 response time 3. Compare with baseline before the change

Degrees of Freedom — явно указывай что фиксировано

Лучшие Skills явно указывают что агент должен делать строго, а где может адаптировать:

## Degrees of freedom High: tone and wording (агент свободен) Medium: level of detail per issue Low: output format (строго по шаблону)

Без этого агент либо слишком жёсткий, либо игнорирует нужные части.

Skill может ссылаться на другие файлы

В Claude Code внутри Skill можно ссылаться на файлы через @. Агент прочитает их как дополнительный контекст при выполнении Skill.

# В SKILL.md: Before reviewing, read: @docs/architecture.md @docs/coding-standards.md Then apply these review criteria...

Best practices

Делай

  • Один Skill — одна задача: code-review, pr-template, debug
  • Description с глагола в 3-м лице, ≤100 символов
  • Output format — покажи шаблон ответа внутри Skill
  • Constraints — что агент НЕ должен делать
  • Degrees of Freedom — что фиксировано, что гибко
  • Версионируй: version: "1.0"

Не делай

  • God Skill 500+ строк — разбей на несколько
  • Rules в Skills — политику пиши в .mdc, не здесь
  • Размытый description: "помоги с кодом"
  • Хранить данные — Skill это инструкции, не Memory
  • Дублировать AGENTS.md контент

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Skill в Cursor / Claude Code?
Skill — SKILL.md файл с YAML frontmatter и Markdown инструкциями, описывающий repeatable workflow агента. Примеры: code review, DB migration, onboarding checklist. В отличие от Rule (политика) — Skill описывает как выполнить конкретный процесс. Агент может применять Skill автоматически (по description) или при явном вызове.
Junior Что такое SKILL.md файл?
Markdown файл с YAML frontmatter. Структура: ---\nname: code-review\ndescription: "Performs thorough PR code review..."\n---\n## Trigger\nWhen to apply...\n## Steps\n1. ...\n## Output\nWhat to return. Frontmatter description — самое важное поле: агент использует его для автоматического выбора Skill.
Junior Где хранятся Skills в проекте?
Project-level: .claude/skills/skill-name/SKILL.md или .claude/commands/name.md — версионируются в git, доступны всей команде. Global-level: ~/.claude/skills/ — личные Skills, работают во всех проектах, не попадают в git. В Cursor: Skills-подобное поведение через .cursor/rules/ и Commands (.cursor/commands/).
Junior Как вызвать Skill явно?
В Claude Code: через /команду если Skill настроен как command, или описав задачу в чате — агент автоматически выберет подходящий Skill по description. В Cursor: Skill-подобные инструкции вызываются через @rule-name или /command-name. Явный вызов гарантирует применение конкретного Skill.
Junior Чем Skill отличается от Rule?
Rule — ограничение/политика: «никогда не делай X», применяется автоматически. Skill — инструкция/процесс: «вот как выполнить Y шаг за шагом». Rule ограничивает поведение агента. Skill расширяет возможности — описывает как правильно делать задачу. При конфликте: Rule > Skill (ограничение имеет приоритет над способом выполнения).
Junior Чем Skill отличается от Command?
Skill (Claude Code): может применяться автоматически агентом по description без явного вызова пользователем. Command (Cursor/Claude Code): вызывается только явно через /имя — агент сам не инициирует. Skill = агент инициирует. Command = пользователь инициирует. Для задач которые должны срабатывать без напоминания — Skill предпочтительнее.
Junior Что такое skill trigger (автоматический вызов)?
Auto-trigger — механизм когда агент применяет Skill без явного вызова. Работает через semantic matching: агент читает description всех доступных Skills, сравнивает с текущей задачей, если совпадение — применяет Skill. Это ключевое преимущество Skills перед Commands: пользователю не нужно помнить имя команды.
Junior Как агент находит и выбирает нужный Skill?
При получении задачи агент: (1) читает descriptions всех доступных SKILL.md файлов; (2) семантически сравнивает задачу с каждым description; (3) если description совпадает — применяет Skill как часть ответа. Качество description критично: чёткий, конкретный description = правильный выбор Skill; расплывчатый = неверный выбор или игнорирование.
Junior Что входит в структуру SKILL.md?
Обязательные элементы: Frontmatter: name, description (≤100 символов, когда применять). Тело: ## Trigger (условия применения), ## Steps (пошаговый процесс), ## Output (что вернуть пользователю). Опционально: ## Prerequisites, ## Examples, ## Announce (как агент сообщает о применении Skill).
Junior Как создать свой первый Skill?
1. Создай директорию .claude/skills/my-skill/. 2. Создай SKILL.md с frontmatter: ---\nname: my-skill\ndescription: "Use when..."\n---. 3. Опиши ## Trigger (когда применять), ## Steps (что делать). 4. Тестируй: задай агенту задачу, совпадающую с description — агент должен применить Skill. 5. Уточни description если Skill применяется слишком широко или не применяется.
Middle — 10 вопросов
Middle Когда создавать Skill, а когда достаточно добавить инструкцию в AGENTS.md?
AGENTS.md: общий контекст, постоянные инструкции, «как устроен проект». Skill: конкретный повторяемый процесс с чёткими шагами, который нужен не всегда, а при определённых задачах. Правило: если инструкция используется для каждого запроса → AGENTS.md. Если это специфический workflow (code review, migration, deploy) который нужен только иногда → Skill.
Middle Как написать description который хорошо работает для автовыбора?
Требования к description: (1) ≤100 символов; (2) третье лицо или trigger-форма: «Use when reviewing PR or auditing code quality»; (3) конкретный trigger: не «Does code analysis» а «Performs thorough code review checking correctness, security, tests. Use when reviewing existing code or PR»; (4) добавь «NOT for» для исключения неверных trigger: «NOT for writing new code or tests»; (5) тестируй description с реальными задачами.
Middle Как Skills помогают стандартизировать повторяемые workflow команды?
Skills — «записанный процесс» команды. Вместо того чтобы каждый разработчик помнил 10-шаговый процесс code review или migration — Skill содержит эти шаги и агент выполняет их консистентно. Преимущества: новые разработчики мгновенно следуют командным процессам; процессы версионируются в git; изменение процесса = обновление SKILL.md.
Middle Когда использовать Skill, а когда Command — trade-offs?
Skill: задача должна запускаться автоматически при определённых условиях, пользователь не хочет помнить команду, агент сам знает когда применить. Command: задача запускается только по запросу пользователя, нужен явный контроль когда выполняется. Skill лучше для: постоянных workflow. Command лучше для: одноразовых или редких задач с явным намерением.
Middle Как Skill влияет на context window при загрузке?
Agent-requested Skill загружается в context только когда агент решает применить. Он не присутствует в каждом запросе (в отличие от alwaysApply Rule). Если SKILL.md большой (2000+ токенов) — при загрузке он занимает место в window. Оптимизация: держи Steps лаконичными, Examples вынеси в отдельные файлы с ссылками, используй иерархическую структуру.
Middle Как организовать Skills по доменам для большой команды?
Структура директорий: .claude/skills/\n├── code-quality/\n│ ├── code-review/SKILL.md\n│ └── refactoring/SKILL.md\n├── database/\n│ ├── migration/SKILL.md\n│ └── query-optimization/SKILL.md\n└── deployment/\n └── release/SKILL.md. Каждый домен — поддиректория. Ответственный за домен — владелец соответствующих SKILL.md файлов.
Middle Как версионировать Skills при изменении командных процессов?
Skills — living documents, не set-and-forget. Процесс обновления: (1) изменился process (новый security checklist) → PR обновляет SKILL.md; (2) reviewer проверяет что новые шаги отражают реальный process; (3) можно добавить version в frontmatter: version: "2.1"; (4) Skills review включать в retrospective или при architecture change. Stale Skill = агент выполняет устаревший процесс.
Middle Как организовать Skills в командном репозитории?
Project Skills → .claude/skills/ в репо, версионируются в git. Доступны всей команде при клонировании. Global Skills → ~/.claude/skills/, личные, не в git. Документирование в AGENTS.md: список ключевых Skills с описанием когда использовать. PR workflow для Skills: новый Skill — PR → review description и шаги → merge.
Middle Как Skills помогают в onboarding новых сотрудников?
Skills = encoded knowledge команды. Новый разработчик: клонирует репо → получает все процессы в .claude/skills/. Не нужно читать wiki — Skills применяются автоматически при нужных задачах. Специальный onboarding/SKILL.md: когда агент видит «я новый разработчик» — применяет гид по проекту. Результат: ускорение time-to-productivity нового члена команды.
Middle Как проверить что Skill работает и применяется корректно?
(1) Задай задачу точно совпадающую с description — Skill должен примениться. (2) Задай похожую но другую задачу — Skill не должен примениться. (3) Проверь каждый шаг: выполняет ли агент все ## Steps или пропускает. (4) Проверь Output: соответствует ли ## Output разделу. (5) Тест «граничный случай»: задача с неполными данными — Skill должен запросить недостающее.
Senior — 5 вопросов
Senior Как спроектировать библиотеку Skills для большой организации?
Архитектура: (1) Core Skills — universal для всех команд: code-review, security-audit, documentation-update; в shared repo; (2) Domain Skills — специфичные для домена: billing-migration, ML-experiment-tracking; в domain repo; (3) Team Skills — локальные процессы команды: их .claude/skills/; (4) Discovery механизм: README.md в .claude/skills/ с каталогом; (5) Skills вреестре: какие Skills есть, кто owner, когда обновлены.
Senior Как реализовать Skills composition — когда один Skill вызывает другой?
Прямого вызова Skill из Skill нет в Claude Code. Паттерны: (1) Reference: в ## Steps ссылка «Apply [security-audit Skill] for step 3»; (2) Shared prerequisites: выносить общие шаги в include-файл, ссылаться через @path; (3) Conditional logic: один Skill с ## If DB context / ## If API context; (4) Orchestrator Skill: высокоуровневый Skill описывает последовательность применения других Skills. Тест composition: agent должен выполнить полный workflow без потери шагов.
Senior Как диагностировать ситуацию когда Skill не срабатывает когда должен?
Диагностика по порядку: (1) description слишком общий/специфичный? → Переформулируй; (2) задача пользователя не совпадает с description семантически? → Попробуй разные формулировки задачи; (3) Skill файл в правильной директории? (.claude/skills/ vs .claude/commands/); (4) frontmatter валидный? → Проверь YAML синтаксис; (5) другой Skill перехватывает задачу? → Проверь descriptions на overlap; (6) Skill слишком большой? → Context window при загрузке может давить.
Senior Как строить Skills для complex multi-step workflow?
Для сложных workflows: (1) Numbered steps: явная нумерация 1..N, агент не пропускает; (2) Checkpoints: в конце каждого блока шагов — «Verify: [список что должно быть готово] before proceeding»; (3) Prerequisites section: требования к tools, доступам, данным; (4) Graceful degradation: «If tool X unavailable — mark as MANUAL REQUIRED»; (5) Announce section: агент объявляет применение Skill и текущий шаг. Тест: выполни Skill на реальной задаче и проверь все N шагов.
Senior Чем Skills отличаются от fine-tuning модели?
Fine-tuning: изменяет веса модели, закодированные паттерны становятся частью «памяти» модели, дорого, требует данных и compute, постоянно. Skills: добавляют инструкции в context при конкретных задачах, модель не меняется, бесплатно обновить (edit SKILL.md), версионируются в git. Skills предпочтительнее для: team-specific процессов, часто меняющихся правил. Fine-tuning: для постоянного domain-specific знания встроенного в модель.
Lead — 5 вопросов
Lead Как выстроить governance Skills в команде?
Framework: (1) Owner per Skill — кто отвечает за актуальность; (2) PR review для SKILL.md — новые Skills и изменения через PR; (3) Skills catalog в AGENTS.md — список с описанием когда использовать; (4) Quarterly Skills review — актуальны ли процессы, не стали ли stale; (5) Approval для новых Skills: кто может добавить в .claude/skills/ — только team lead или через PR vote.
Lead Как Skills становятся knowledge base организации?
Skills = encoded organizational knowledge. Когда опытный разработчик документирует свой process в SKILL.md — знание перестаёт быть tribal. Стратегия: (1) onboarding: «задокументируй процессы которым тебя учили» → новые SKILL.md; (2) при уходе разработчика: обязательная передача знаний через Skills; (3) постмортемы: новые Skill шаги из каждого инцидента; (4) Skills как living documentation — актуальнее wiki.
Lead Как использовать Skills в onboarding процессе?
Onboarding через Skills: (1) project-onboarding/SKILL.md: применяется автоматически когда агент видит запрос «помоги освоиться в проекте»; (2) domain-specific Skills объясняют команде как работать с конкретными системами; (3) первая неделя нового разработчика: работа с агентом активирует нужные Skills; (4) измерение: время до первого PR у нового разработчика; (5) feedback loop: нет Skills для типичной задачи → создай Skill.
Lead Как поддерживать актуальность Skills со временем?
Skills rot prevention: (1) Process change → Skill update: чеклист при изменении процессов; (2) Expiry dates: frontmatter reviewed: "2025-Q1" — quarterly проверка; (3) Owner accountability: owner Skills отвечает за актуальность; (4) Agent feedback: когда агент обнаруживает что шаг Skill не работает — создаёт issue; (5) CI check: не обновлявшиеся 6+ месяцев Skills помечаются.
Lead Как распределить Skills между командами в крупной организации?
Матрица ответственности: (1) Platform team: infrastructure Skills (deployment, monitoring, DB ops); (2) Feature teams: domain Skills (billing logic, user management); (3) QA/Security: testing Skills, security review Skills; (4) Shared Skills repo: cross-cutting Skills, владеет Platform; (5) Skills discoverability: каждая команда публикует свои Skills в organisational catalog — другие команды могут переиспользовать.
Architect — 5 вопросов
Arch Как Skills реализуют концепцию reusable AI workflow components?
Skills как software components: Interface = description (когда применять), Implementation = ## Steps (как выполнять), Contract = ## Output (что вернуть). Reusability: один Skill используется несколькими командами (code-review Skill применяется для Java, Python, Go одинаково). Composability: Skills ссылаются друг на друга как модули. Versioning: SKILL.md в git = semantic versioning для AI workflows.
Arch Как Skills library становится организационным активом?
Organizational skill asset: (1) Skills содержат уникальные знания организации — лучшие практики, domain knowledge; (2) при масштабировании команды: новые агенты сразу имеют доступ к Skills; (3) M&A scenario: Skills library = quick knowledge transfer; (4) ROI: время сэкономленное агентами, применяющими правильные Skills, vs время на написание процессов вручную; (5) competitive advantage: уникальные Skills = уникальная эффективность команды.
Arch Как встроить Skills versioning в CI/CD pipeline?
CI/CD для Skills: (1) Lint SKILL.md: проверка YAML frontmatter, наличие обязательных секций; (2) Description quality check: длина ≤100 символов, наличие trigger clause; (3) Skill testing: sandbox запуск агента с тестовыми задачами, проверка применения нужных Skills; (4) Breaking change detection: diff Skill Steps → алерт если изменились; (5) Distribution: pipeline публикует обновлённые Skills в shared repo.
Arch Как Skills обеспечивают cross-team knowledge sharing?
Knowledge sharing через Skills: (1) Platform team пишет deployment Skill — все feature teams используют без изучения infra; (2) Security team пишет security-audit Skill — агенты всех команд автоматически применяют security practices; (3) Skills API: Skill как public interface к экспертизе команды — другие команды «вызывают» экспертизу через Skill; (4) Inner source модель для Skills: любая команда может предложить PR к Skill другой команды.
Arch Как спроектировать discoverable Skills ecosystem?
Discoverability requirements: (1) Skills registry: центральный index всех Skills с categorization, owner, last updated; (2) Rich descriptions: description + extended description в ## Overview; (3) Tagging: теги в frontmatter (domain, tool, language) для фильтрации; (4) Auto-documentation: CI генерирует Skills catalog из SKILL.md файлов; (5) Recommendation engine: агент предлагает релевантные Skills при onboarding или при задачах в новом домене.
