Checkpoint / Summary / Compression

Как сохранять состояние длинной задачи, чем checkpoint лучше summary, и как не потерять важные решения при compaction.

На главнуюК assessment

Что такое Checkpoint, Summary и Compression

Checkpoint = точка возобновления

Структурированная запись состояния задачи в конкретный момент: что сделано, какие решения приняты, что осталось и конкретно следующий шаг. Создаётся намеренно — человеком или агентом.

  • Отвечает на: «Откуда продолжать?»
  • Содержит: Goal, Done, Decisions, Next step, Evidence
  • Создаётся: явно, перед паузой или compaction
  • Аналог: git commit с подробным описанием

Summary = краткий пересказ

Сжатый пересказ того что происходило. Бывает ручным (агент по запросу) и автоматическим (генерируется при compaction). Теряет детали — это его природа.

  • Отвечает на: «Что происходило?»
  • Содержит: общий смысл, высокоуровневые выводы
  • Создаётся: автоматически при compaction или по запросу
  • Аналог: краткое содержание главы книги

Compression / Compaction = сжатие

Технический процесс уменьшения объёма context window. Старая история заменяется summary. Неизбежен при длинных задачах. Checkpoint — инструмент защиты от потерь при compaction.

  • Триггер: context window ~80% заполнен
  • Результат: старые turns → summary (~5–10k токенов)
  • Потери: конкретные детали, решения, отклонённые пути
  • Аналог: старые письма уничтожены, остался только пересказ

Summary и Compact — не одно и то же

Summary = «перескажи»

Краткое содержание разговора или задачи — для человека, документации или переноса в новый чат.

Compact = «сожми контекст, освободи место»

Механизм управления context window: старая часть чата заменяется сжатым summary, чтобы работа в том же треде могла продолжиться.

При compaction часть деталей может потеряться. Для важных задач делай явный checkpoint в файле или Memory Bank — не полагайся только на авто-summary после compact.

Главное отличие: summary — пересказ для человека и переноса смысла; compact — инженерия context window в текущем чате. Для точного продолжения задачи summary часто слабоват — нужен checkpoint (next step, decisions, evidence).

Ключевое отличие Checkpoint от Summary

Summary говорит «что было» — он описательный. Checkpoint говорит «откуда продолжать» — он операционный. Summary создаётся системой автоматически и оптимизирован для сжатия. Checkpoint создаётся намеренно и оптимизирован для возобновления работы. Плохой Checkpoint: «Мы работали над auth». Хороший Checkpoint: «Next: implement refresh token в api/auth/refresh.py, JWT secret в env, тест в tests/test_auth.py:45».

Summary vs Compact — частая путаница

«Сделай summary» ≠ /compact и ≠ авто-compaction: пересказ в ленте не обязан освободить context window. Compact выбрасывает старые turns и подставляет сжатое summary — модель в следующих шагах уже не видит исходные детали. Summary удобно копировать в доку или новый чат; compact — чтобы не рвать длинный тред. Оба могут терять детали → явный checkpoint в файле.

С нуля: что такое Compaction / Compression — Claude Code / Cursor

Checkpoint из предыдущего блока нужен в первую очередь из‑за compaction — ниже разберём механику сжатия. Концепция одинакова в Claude Code и Cursor.

Уровень 0 — простыми словами

Представь что у тебя есть лист бумаги на котором ты записываешь разговор с агентом. Каждый вопрос, каждый ответ, каждый файл который агент прочитал — всё записывается. Лист конечный. Когда он заканчивается — агент берёт резинку и стирает старые записи, оставляя только краткий пересказ. Это и есть compaction — автоматическое сжатие старой истории. Проблема: в процессе стирания теряются детали. Checkpoint — это когда ты сам записываешь главное на отдельный лист перед тем как агент возьмётся за резинку.

Что именно теряется при compaction

❌ Теряется (не попадает в summary)
  • Конкретные строки кода которые смотрели
  • «Мы пробовали решение A — оно не работает»
  • Точные имена файлов и пути
  • Промежуточные команды и их вывод
  • Отклонённые гипотезы (агент проверит снова)
  • Почему выбрали подход B а не A
  • Конкретные номера портов, версии, URL
✅ Остаётся в summary
  • Общий смысл задачи
  • Высокоуровневые выводы
  • Текущий статус («работаем над auth»)
  • System prompt и AGENTS.md/CLAUDE.md (не compaction-ятся)
  • Rules (alwaysApply — перезагружаются)

Виды сжатия контекста

1. Автоматический compaction (срабатывает сам) Триггер: context window близок к лимиту (~80% и выше — ориентир) Что происходит: LLM суммаризирует старую историю Cursor: в breakdown контекста появляется «Summarized conversation»; кольцо у поля ввода показывает заполнение Claude Code: SystemMessage {subtype: "compact_boundary"} в SDK Контроль: минимальный (инструкции что сохранять — в AGENTS.md / CLAUDE.md) 2. Ручное сжатие (ты выбираешь момент) Claude Code (IDE): команда /compact в чате — то же суммаризирование, но по твоему сигналу Cursor (IDE Agent): встроенной /compact нет — см. блок «Cursor — compaction» ниже Cursor CLI: /summarize или /compress — сжать диалог вручную Лучший момент: перед новым крупным этапом задачи (после context checkpoint в файл) 3. Новая сессия (самое жёсткое сжатие) Что происходит: history = пуста, только system + rules + AGENTS.md Контроль: полный — ты решаешь что передать Cursor: New Chat; опционально @Past Chats или вставка context checkpoint Claude Code: новая сессия / --resume / checkpoint в Memory Bank

Что НЕ compaction-ится — стабильный контекст

Не теряются при любом compaction: CLAUDE.md / AGENTS.md — перезагружается при каждом запросе Rules (alwaysApply) — всегда в context Memory Bank файлы — агент читает их явно в начале сессии Checkpoint (в файле) — агент читает его если ты передал → Вывод: важные решения дублируй в CLAUDE.md / Memory Bank → Они переживут любое количество compaction'ов
🖱 Cursor — compaction

Когда окно контекста почти заполнено, Cursor сам сжимает старые turns в summary. В Agent-чате IDE нет /compact, /compress и /summarize — эти slash-команды относятся к Cursor CLI (см. ниже).

  • Авто: кольцо контекста у поля ввода → breakdown → сегмент Summarized conversation
  • Ручной аналог /compact в IDE: (1) попроси агента сжать историю с фокусом («сохрани: файлы, решения, next step») и продолжай в том же чате; (2) New Chat + context checkpoint в файле или вставкой; (3) @Past Chats — подтянуть прошлый диалог в новый чат без копипасты всей ленты
  • Cursor CLI: /summarize / /compress — ручной compact (сжать контекст сессии, освободить окно); не то же самое, что попросить «перескажи» в IDE. Сброс — /clear (/new-chat)
  • Стабильный слой: Rules, AGENTS.md, Skills — не заменяются summary
  • Инструкции compactor'у: секция в AGENTS.md «Summary instructions» / что всегда сохранять при сжатии
Перед «жёстким» сжатием (New Chat или длинная пауза): 1. «Сохрани context checkpoint в docs/progress/task.md по шаблону» 2. New Chat → @docs/progress/task.md или вставка текста → Это замена ручного /compact + checkpoint из схемы выше

Claude Code SDK даёт полный контроль через параметры и хуки.

Два разных «Checkpoint» — не путать

В Cursor в интерфейсе есть слово Checkpoint, но это не тот context checkpoint, о котором идёт речь на этой странице. Одинаковое название — разная задача.

Что сохраняетСмысл задачи: goal, done, decisions, next step, рискиСнимок файлов на диске до правок агента в этой сессии
ЗачемПродолжить работу после compaction, New Chat, паузы, handoffОткатить код, если агент пошёл не туда (undo правок, не Git)
Где живётФайл, Memory Bank, вставка в новый чатЛокально в Cursor; в ленте сообщений у хода агента

После restoreАгент снова «знает» задачу из текста checkpointФайлы как до этого хода; история чата и summary compaction не откатываются

Типичная ошибка

Нажал Restore Checkpoint в Cursor и ждёшь, что агент «вспомнит» все решения из середины длинного чата. UI checkpoint откатывает только изменённые файлы. Чтобы не потерять логику задачи при compaction или новом чате — нужен отдельный context checkpoint (файл / Memory Bank). Оба инструмента полезны: UI — когда код испорчен; context — когда испорчен или переполнен контекст.

🖱 Cursor — Restore Checkpoint

Перед значимыми правками Agent сохраняет snapshot. В истории чата у сообщения — восстановление состояния репозитория на момент до следующих edit'ов агента.

  • Не заменяет Git — для коммитов и веток по-прежнему Git
  • Не переносит задачу в новый чат — для этого context checkpoint
  • Можно комбинировать: Restore UI checkpoint (код) + тот же context checkpoint в AGENTS.md (решения)
🤖 Claude Code — file checkpointing

У Claude Code тоже есть откат файлов (checkpointing кода) — это отдельно от /compact и от текстового handoff в CLAUDE.md / Memory Bank.

  • /compact — сжатие истории чата, не откат файлов
  • Context checkpoint — твой markdown / memory-bank, как в академии
  • 📖 Checkpointing (files)

Checkpoint vs Compaction — как работают вместе

Checkpoint и compaction — не одно и то же. Compaction — это проблема (потеря деталей). Checkpoint — это решение. Вот как они взаимодействуют.

Сценарий без Checkpoint — что происходит при compaction Turn 1–40: Агент работает, исследует, принимает решения... «Выбрали PostgreSQL вместо MongoDB — потому что нужен JOIN» «Файл api/auth.py строка 78 — тут root cause» «Пробовали solution A — не работает из-за race condition» ↓ COMPACTION: context → summary ↓ Summary: «Работали над auth, исследовали базу данных, что-то починили» Turn 41: Агент не знает: - почему PostgreSQL ← может предложить MongoDB снова - где именно баг ← начнёт искать заново - что уже пробовали ← повторит solution A Сценарий с Checkpoint — защита от потери Turn 35: Создаём Checkpoint (до того как context переполнится) Goal: fix auth session bug Done: api/auth.py:78 root cause найден Decision: PostgreSQL (нужен JOIN для sessions) Tried + failed: solution A (race condition на line 92) Next: implement fix в middleware.py ↓ Turn 40: COMPACTION (summary теряет детали) ↓ Turn 41: Агент читает Checkpoint: ✓ знает где баг (api/auth.py:78) ✓ знает почему PostgreSQL ✓ не повторит solution A ✓ знает следующий шаг

Три стратегии работы с compaction

🛡 Стратегия 1
Checkpoint before compaction

Создай Checkpoint до того как context переполнился. Лучшее время: когда context на 60–70%.

Context 70% → создай Checkpoint ↓ Continue работу ↓ Compaction → summary потеряет детали ↓ Checkpoint в Memory Bank → читай его
📋 Стратегия 2
AGENTS.md + Memory Bank

Не только чат: checkpoint.md в Memory Bank — state задачи; принятые навсегда решения — в AGENTS.md / memory.md / decisions.md. Файлы и AGENTS не подменяются conversation summary.

memory-bank/checkpoint.md ← текущая задача (временно) memory-bank/decisions.md ← устойчивые решения AGENTS.md ← workflow + @include memory # AGENTS.md (фрагмент) ## Permanent decisions - DB: PostgreSQL (JOIN needed) ← выживет любой compaction
🔄 Стратегия 3
PreCompact hook (Claude Code)

Через SDK: хук PreCompact архивирует полную историю до сжатия.

hooks.json: { "PreCompact": { "command": "python archive.py" } } ← archive.py сохраняет полный transcript в файл до того как summary заменит историю

Когда делать Checkpoint — правило 60%

Не жди когда context переполнится — к тому времени compaction уже может начаться. Практическое правило: если работаешь над задачей и уже прошло много turns → создай Checkpoint сейчас. В Claude Code SDK: следи за токенами в ResultMessage.usage. В Cursor: если чат уже очень длинный — пора создавать Checkpoint.

Идеальный checkpoint template

# Checkpoint

## Goal
Что хотим получить.

## Current state
Где сейчас находимся.

## Done
- Что уже проверено/сделано.
- Какие команды запускали.
- Какие файлы смотрели.

## Decisions
- Какие решения приняты.
- Почему выбрали именно так.

## Evidence
- Логи, файлы, ссылки, результаты tools.

## Risks / blockers
- Что может быть не так.
- Что требует approval.

## Next steps
1. Следующий шаг.
2. Что проверить после него.
3. Как понять, что задача завершена.

Когда делать checkpoint

Обязательно

  • Перед паузой.
  • Перед compaction.
  • Перед передачей subagent'у.
  • После важного решения.
  • Перед destructive action.

Желательно

  • После incident investigation.
  • После refactoring.
  • Перед PR summary.
  • Перед сменой гипотезы.

Тот же чат или новый — когда что выбирать

Context checkpoint создают в файл (репозиторий, Memory Bank) — так он работает и в текущем чате, и после New Chat. Вопрос не «куда писать checkpoint», а где продолжать работу после того как он сохранён.

Оставайся в том же чате

  • Ещё есть запас по context ring, мало summarized history.
  • Checkpoint — маркер этапа, следующий шаг сразу в этом треде.
  • Откат файлов через UI Checkpoint — только здесь.

Не полагайся на checkpoint только как на сообщение в ленте — после compaction оно может уйти в summary.

Открывай новый чат

  • Контекст переполнен или уже сжимался — чистый старт дешевле, чем ещё 30 turns.
  • Смена фазы задачи или смена «режима» (explore vs implement).
  • Cursor: @Past Chats — если нужен кусок старого диалога, но не вся лента.

Шаблон первого сообщения: «Продолжи задачу. Контекст в @…/checkpoint.md. Начни с Next step, не повторяй Done.»

Порядок действий (универсально) 1. Создать / обновить checkpoint в файле (не только в чате) 2. Оценить context ring + качество ответов агента ↓ запас есть, этап тот же? → тот же чат + @файл при необходимости ↓ переполнение / новый этап / дрейф? → New Chat + @checkpoint.md 3. Решения навсегда → Memory Bank (decisions.md) и/или AGENTS.md / CLAUDE.md; не копи завершённые checkpoint в memory.md

Один файл — два сценария

Один и тот же checkpoint.md можно читать в старом чате («обнови прогресс и иди дальше») и в новом (единственный источник правды о задаче). Новый чат не «восстанавливает» checkpoint автоматически — ты явно даёшь @ или вставку. В старом чате после compaction без файла агент опирается на summary и может потерять детали.

Примеры

Плохой summary

“Мы разбирали CI, там что-то падало, нужно дальше смотреть.”

Хороший checkpoint

“Goal: найти причину job X. Done: проверены logs A/B. Decision: infra не причина. Next: проверить test Y после commit abc.”

Тонкости и нюансы Checkpoint / Summary

Checkpoint ≠ полный backup

Checkpoint сохраняет решения и прогресс, но не весь context window. При resume агент начинает с summary, а не с полной историей диалога. Детали, не попавшие в summary, теряются навсегда.

Слишком частые checkpoints убивают скорость

Каждый checkpoint — это LLM-вызов для генерации summary. Для коротких задач overhead превышает пользу. Делай checkpoint только после реально значимых этапов, а не каждые 5 минут.

Summary теряет детали

LLM при compaction суммаризирует — конкретные строки кода, точные команды, имена файлов могут не попасть в summary. Решение: явно указывай «включи в summary: конкретные файлы и команды» при запросе checkpoint.

Checkpoint не защищает от drift

Если агент ушёл не туда до checkpoint — checkpoint зафиксирует неверный прогресс. Checkpoint полезен только при валидированном состоянии. Сначала убедись что прогресс правильный, затем делай checkpoint.

AGENTS.md, CLAUDE.md и Memory Bank — что переживает compaction

История чата при compaction сжимается в summary. Не в чате храни то, что нельзя потерять — три слоя с разными ролями (подробнее: Memory Bank, структура файлов):

  • AGENTS.md / CLAUDE.md — в контексте каждого запроса: как работать в репо, workflow, лимиты, Summary instructions, ссылки на файлы Memory Bank. Сюда — короткие постоянные нормы, не дневник задачи.
  • Memory Bank — checkpoint.md — тот же смысл, что context checkpoint: goal, done, decisions, next текущей задачи. Обновляется по ходу работы; после done — выжимка в decisions.md, сам checkpoint архивируют или удаляют.
  • Memory Bank — memory.md / decisions.md — устойчивые факты и решения проекта между сессиями. На диске, summary их не стирает; агент подтягивает через протокол в AGENTS.md или @memory-bank/….

Не путать: «персистентный checkpoint» — не один файл. Временный прогресс → checkpoint.md; навсегда → Memory Bank + AGENTS.md. Дублировать одно и то же в трёх местах не нужно — один источник правды на факт.

Cursor UI Checkpoint ≠ context checkpoint

Restore Checkpoint в ленте Agent чата откатывает файлы на диске, а не «сохранённое состояние задачи» в тексте. Для compaction и New Chat используй context checkpoint в файле; UI checkpoint — когда нужно отменить пачку правок агента.

Best practices — Делай / Не делай

✅ Делай

  • Создавай checkpoint после каждого валидированного этапа.
  • Явно указывай что включить в summary (файлы, команды, решения).
  • Держи прогресс задачи в checkpoint.md (Memory Bank или docs/progress/), не только в ленте чата.
  • После валидации этапа — устойчивые решения в Memory Bank (decisions.md) и при необходимости в AGENTS.md / CLAUDE.md.
  • Делай checkpoint перед рискованными или необратимыми операциями.
  • Проси агента назвать что было сделано перед созданием checkpoint.

❌ Не делай

  • Не делай checkpoint слишком часто — overhead перевешивает пользу на коротких задачах.
  • Не доверяй summary без проверки критичных деталей — всегда верифицируй.
  • Не используй чат как единственное хранилище — дублируй в файл; постоянные факты — в Memory Bank / AGENTS.md, не в checkpoint.md.
  • Не создавай checkpoint если агент ушёл не туда — сначала откорректируй направление.

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Checkpoint в контексте AI агентов?
Checkpoint — это структурированная запись текущего состояния задачи: что уже сделано, какие решения приняты, что осталось сделать и какой следующий шаг. Checkpoint создаётся в момент паузы в работе, чтобы продолжить с той же точки позже — без потери контекста. Аналог из разработки: commit с описанием что сделано. Разница с простой заметкой: Checkpoint структурирован так, чтобы агент мог по нему восстановить полный контекст задачи. Не путать с кнопкой Restore Checkpoint в Cursor — та откатывает файлы на диске, а не текстовый handoff задачи.
Junior Есть ли в Cursor команда /compact как в Claude Code?
В Agent-чате Cursor IDE встроенной команды /compact нет. Сжатие истории происходит автоматически, когда контекст почти заполнен (в breakdown видно «Summarized conversation»). Ручные аналоги: попросить агента сделать структурированное summary с фокусом на решения и next step; открыть New Chat и передать context checkpoint из файла или через @Past Chats; в Cursor CLI — /summarize или /compress. В Claude Code в IDE есть явный /compact.
Junior Зачем нужен Checkpoint — какую проблему он решает?
Checkpoint решает проблему потери контекста при compaction, паузах и передаче задачи. Без Checkpoint: агент сжимает длинный диалог (compaction) → теряет детали принятых решений и промежуточный прогресс. Новая сессия начинается с нуля. С Checkpoint: даже после compaction агент читает Checkpoint и знает: цель, что сделано, ключевые решения, что дальше. Checkpoint = survivor механизм для длинных задач.
Junior Чем Summary отличается от Compact (compaction)?
Summary — «перескажи»: краткое содержание для человека, документации или вставки в новый чат. Compact / compaction — «сожми контекст»: старая часть чата заменяется сжатым summary, чтобы освободить context window и продолжить в том же треде. Compact может использовать summary внутри, но цель compact — уменьшить активный контекст, а не просто пересказать. При compaction детали могут потеряться → для важных задач явный checkpoint в файле. Исключение по имени: в Cursor CLI /summarize и /compress — это ручной compact, не «summary для человека». В IDE фраза «сделай summary» compact обычно не запускает. Claude Code: /compact; Cursor IDE: авто-compaction.
Junior Чем Checkpoint отличается от Summary?
Summary — краткое содержание: пересказывает что было. Checkpoint — структурная точка продолжения: фиксирует состояние достаточно детально для продолжения работы. Summary отвечает на 'что происходило', Checkpoint на 'откуда продолжать'. Пример: Summary — 'мы исследовали проблему с CI'. Checkpoint — 'Goal: fix CI job X. Done: проверены logs A/B (не причина). Decision: infra ok. Blocker: нужен access к test env Y. Next: запустить тест Z с флагом --verbose'.
Junior Когда нужно делать Checkpoint?
Обязательно: перед любой паузой в работе, перед compaction (context window заполняется), перед передачей задачи subagent'у, после принятия важного архитектурного решения, перед destructive действием. Желательно: после завершения каждого значимого этапа задачи (investigation → implementation → testing), после incident investigation, перед PR submission. Правило: если ты делаешь паузу более 30 минут или задача переходит к другому агенту → Checkpoint обязателен.
Junior Что нужно писать в Checkpoint?
Структура эффективного Checkpoint: (1) Goal — что нужно достичь (одна строка). (2) Status — текущий статус (In Progress / Blocked / Done). (3) Done — что уже выполнено с конкретными фактами (файлы, commits, проверенные гипотезы). (4) Key decisions — принятые архитектурные/технические решения с обоснованием. (5) Current context — важные факты, открытия, аномалии. (6) Blockers — что мешает продолжить. (7) Next step — конкретное следующее действие. (8) References — ссылки на файлы, PR, tickets. Без этих полей Checkpoint неполный.
Junior Как Checkpoint помогает при compaction контекста?
Compaction — автоматическое или ручное сжатие длинного диалога когда context window заполняется. При compaction: детали теряются, принятые решения могут исчезнуть, прогресс неочевиден. С Checkpoint: до compaction агент создаёт Checkpoint → после compaction читает его → восстанавливает полный контекст задачи. Checkpoint — 'страховочная сетка' перед compaction. Best practice: при длинных задачах делать Checkpoint каждые 30–50% context window — не ждать пока compaction случится.
Junior Что теряется без Checkpoint при compaction?
Без Checkpoint при compaction теряется: (1) Промежуточные решения (почему выбрали подход A, а не B). (2) Исследованные и отклонённые гипотезы (не придётся проверять снова). (3) Точный прогресс (что именно готово, что нет). (4) Контекст блокеров (что не работало и почему). (5) Важные открытия в процессе (например, найденный edge case). Агент начнёт заново или допустит те же ошибки. Summary от compaction не содержит этих деталей — он слишком общий.
Junior Где хранить Checkpoint?
Опции хранения Checkpoint: (1) В чате — написать Checkpoint как сообщение, он будет в conversation history. Простой вариант, но потеряется при очистке чата. (2) В Memory Bank — memory-bank/checkpoint.md или memory-bank/task-name-checkpoint.md. Персистентно между сессиями. (3) В репозитории — docs/progress/feature-name-checkpoint.md. Хорошо для командной работы. (4) В issue/ticket — как comment в Jira/GitHub Issue. Рекомендация: для личных задач → чат. Для командных → репозиторий или ticket.
Junior Как передать Checkpoint в новую сессию агента?
Передача Checkpoint: (1) Если Checkpoint в Memory Bank → новая сессия автоматически читает Memory Bank в начале. (2) Если Checkpoint в файле → включить файл через @mention или попросить агента прочитать его: 'Прочитай checkpoint в docs/progress/auth-feature.md и продолжи с него'. (3) Если в чате предыдущей сессии → скопировать текст Checkpoint в начало нового чата. Формат при передаче: 'Продолжи задачу. Checkpoint: [содержание]'. Агент должен начать с 'Я вижу что сделано X, следующий шаг Y. Начинаю с него.'
Junior Что такое structured state в контексте Checkpoint?
Structured state — формализованное, машиночитаемое представление состояния задачи. Это когда Checkpoint оформлен не как свободный текст, а со стандартными полями (Goal, Done, Decisions, Next). Преимущества structured state: агент легче parsit и использует поля; команда имеет предсказуемый формат; можно автоматически обрабатывать (например, генерировать отчёт о прогрессе). Пример: вместо 'мы почти дошли до решения' → 'Status: 70% complete. Next: implement auth middleware (ref: backend/auth/middleware.py)'.
Middle — 10 вопросов
Middle Context checkpoint — продолжать в том же чате или открыть новый?
Сначала сохрани checkpoint в файл — это работает в обоих сценариях. Тот же чат: контекст не забит, тот же этап задачи, нужна только страховка или лёгкое сжатие без смены треда; откат кода через UI Restore Checkpoint — только в текущей сессии. Новый чат: кольцо контекста почти полное, уже есть Summarized conversation, смена крупного этапа, дрейф агента, долгая пауза или другая задача — чистый prefix (Rules + AGENTS.md + @checkpoint.md) надёжнее длинной ленты. Не храни единственную копию checkpoint только в истории сообщений — после compaction она деградирует. Постоянные решения — в Memory Bank (decisions.md) и/или AGENTS.md, текущая задача — в checkpoint.md.
Middle Как структурировать эффективный Checkpoint для сложной технической задачи?
Эффективная структура Checkpoint для технической задачи: [# Checkpoint — ] с полями: Goal (1 строка), Status (In Progress/Blocked/Done), Done (bulleted список с конкретными артефактами: файлы, commits, тесты, PR), Key Decisions (каждое решение + rationale: 'выбрали Redis вместо Memcached потому что TTL + persistence'), Current State (важные факты: 'auth service на порту 8080, DB на 5432'), Blockers (если есть: 'нет доступа к prod logs'), Next Step (одно конкретное действие с файлом и функцией если применимо), References. Объём: 200–400 слов — достаточно для полного восстановления контекста.
Middle Когда лучше делать Checkpoint — в середине задачи или только в конце?
Checkpoint в конце задачи — неправильно: если задача не завершена, а compaction уже произошёл — поздно. Правильный подход: Progressive Checkpointing — создавай Checkpoint в конце каждого значимого этапа. Для трёхэтапной задачи: Checkpoint 1 после investigation (что нашли, что исключили), Checkpoint 2 после implementation (что изменили, какие решения), Checkpoint 3 перед deploy (полный статус, что протестировано). Плюс: обязательный Checkpoint при любой паузе >30 мин. Частые маленькие Checkpoints > один большой в конце.
Middle Как использовать паттерн Checkpoint + Memory Bank?
Паттерн: Checkpoint в Memory Bank = персистентный state между сессиями. Реализация: при конце рабочей сессии → агент создаёт/обновляет memory-bank/checkpoint.md. При начале новой сессии → агент читает Memory Bank и находит checkpoint.md → восстанавливает контекст → продолжает. Дополнительно: при завершении задачи → из checkpoint извлекаются устойчивые знания (решения, конвенции) → переносятся в memory-bank/decisions.md. Checkpoint = temporary state; Memory Bank decisions = permanent knowledge. Lifecycle: checkpoint создаётся → обновляется по ходу задачи → архивируется при завершении.
Middle Чем Summary от compaction отличается от Checkpoint и как их использовать вместе?
Автоматический Summary от compaction: генерируется системой, общий, теряет детали, не структурирован под продолжение работы. Checkpoint: создаётся намеренно агентом/пользователем, структурирован, содержит всё необходимое для продолжения. Использование вместе: (1) Перед compaction → создать Checkpoint вручную. (2) После compaction → система создаёт автоматический Summary. (3) Если Checkpoint уже есть в Memory Bank → Summary менее критичен. (4) Если Checkpoint не был создан → Summary + дополнительные вопросы агенту для восстановления контекста. Правило: Checkpoint > Summary всегда. Автоматический Summary — последний резерв, не замена.
Middle Как различаются автоматические и ручные Checkpoint — когда каждый уместен?
Автоматический Checkpoint: триггерится системой по условию (context window > 80%, прошло 30 минут, компания сделала определённый шаг). Плюсы: не требует памяти пользователя. Минусы: может пропустить момент или иметь неполный контекст. Ручной Checkpoint: пользователь явно просит 'создай checkpoint'. Плюсы: точно в нужный момент, агент добавляет максимум контекста. Минусы: требует дисциплины. Рекомендация: установить автоматический Checkpoint как safety net (по объёму context), дополнять ручными Checkpoint перед критическими моментами (перед deploy, perед паузой). Гибрид: лучшая стратегия.
Middle Как использовать Checkpoint для multi-step задач с зависимыми этапами?
Multi-step Checkpoint strategy: (1) Task decomposition first — разбить задачу на этапы с чёткими milestone. (2) Checkpoint per milestone — каждый Checkpoint фиксирует: этап завершён, что стало известно, input для следующего этапа. (3) Dependency tracking — в Checkpoint явно указать что данный этап зависит от предыдущего решения. Пример: 'Checkpoint 2 depends on: decision from CP1 — use PostgreSQL'. (4) Branching — если в одном этапе две гипотезы → фиксировать обе в Checkpoint с результатом проверки. (5) Gate criteria — каждый Checkpoint содержит 'Definition of Done' для этого этапа.
Middle Что включать в Checkpoint — решения, прогресс или следующий шаг?
Checkpoint должен содержать все три: Решения (наиболее критичны — без них агент повторит уже отклонённые пути), Прогресс (конкретные артефакты: файлы изменены, тесты пройдены — не 'мы много сделали') и Следующий шаг (один конкретный action — избегает неопределённости при resume). Приоритет включения при нехватке места: (1) Следующий шаг + key decisions (минимум для продолжения). (2) + Прогресс (что уже не нужно делать). (3) + Контекст и blocker. Плохой Checkpoint: 'мы работали над auth' — ничего конкретного. Хороший: 'реализован JWT middleware (auth/middleware.py), следующий: добавить refresh token endpoint'.
Middle Чем Checkpoint отличается от обновления AGENTS.md?
AGENTS.md — постоянная документация: правила, конвенции, архитектурные решения проекта. Обновляется редко, когда принятое решение становится постоянной нормой. Checkpoint — временный state текущей задачи: прогресс, решения именно этой задачи, что дальше делать. Checkpoint для этой задачи, AGENTS.md для всего проекта. Flow: в ходе задачи → Checkpoint. После завершения → если решение стало конвенцией → обновить AGENTS.md. Пример: Checkpoint 'используем PostgreSQL для этой фичи' → после завершения → AGENTS.md 'проект использует PostgreSQL, prefer JSONField over separate tables for settings'.
Middle Как использовать Checkpoint для передачи задачи subagent'у?
Subagent handoff через Checkpoint: (1) Main agent создаёт Checkpoint с полным контекстом задачи. (2) В task description для subagent включить Checkpoint: 'Контекст задачи: [checkpoint содержание]'. (3) Subagent начинает работу с чтения Checkpoint: 'Вижу что сделано X, моя задача Y'. (4) Subagent создаёт свой mini-Checkpoint при завершении: что он сделал, что нашёл. (5) Main agent получает subagent Checkpoint + объединяет с общим Checkpoint. Best practice: не передавать весь длинный диалог subagent'у — только структурированный Checkpoint. Это экономит context window и даёт subagent'у чистый фокус.
Middle Когда Checkpoint недостаточен и нужны дополнительные механизмы?
Checkpoint недостаточен при: (1) Очень длинных задачах (недели) — Checkpoint устаревает; нужны обновления Memory Bank и периодические review сессии. (2) Множественных параллельных задачах — один Checkpoint на задачу, нужен task registry. (3) Командных задачах — Checkpoint в личном чате не виден коллегам; нужен в git репозитории или ticket. (4) Audit requirements — регуляторные требования к трассировке решений; нужен формальный decision log, не только Checkpoint. (5) Когда задача по сути завершена — завершённая задача заслуживает документацию в AGENTS.md/docs, а не только Checkpoint.
Senior — 5 вопросов
Senior Как проектировать checkpoint систему для complex multi-agent workflow?
Multi-agent checkpoint architecture: (1) Hierarchical checkpoints — main agent имеет master checkpoint, каждый subagent имеет свой. При merge subagent findings → main checkpoint обновляется. (2) Checkpoint protocol — стандартный формат для всех агентов в workflow: одинаковые поля, одинаковые паттерны reference к артефактам. (3) Central checkpoint store — shared storage (Memory Bank файл или git repo) где все агенты могут читать/писать checkpoints. Конкуренция: optimistic locking или sequential writes. (4) Checkpoint events — при каждом критическом шаге → emit checkpoint event. Monitoring подписывается на эти события. (5) Checkpoint replay — возможность воспроизвести workflow от любого checkpoint (state machine semantics). (6) TTL для checkpoints — автоматически архивировать checkpoints для завершённых задач.
Senior Как диагностировать потерю state в workflow без checkpoint?
Диагностика state loss: (1) Признаки — агент повторяет уже сделанные шаги, задаёт вопросы на которые уже был ответ, принимает противоречащие ранее принятым решения. (2) Root cause анализ — посмотреть conversation history: где произошёл compaction? Что было до и после? (3) Recovery — если есть Memory Bank → читать оттуда. Если есть файлы в repo → читать изменения через git log. Если нет → reconstruction через ручную беседу с агентом. (4) Postmortem — почему не было checkpoint? Добавить в workflow: (a) автоматический checkpoint trigger при context > 70%, (b) mandatory checkpoint перед любой паузой. Prevention > recovery.
Senior Как выбрать granularity Checkpoint — слишком частые vs слишком редкие?
Checkpoint granularity trade-off: Too frequent → overhead, много времени на checkpoint вместо работы. Too rare → риск потери контекста. Оптимальная granularity зависит от: (1) Риска задачи — destructive операции (delete, prod deploy) → checkpoint перед каждым. Readonly операции → реже. (2) Сложности задачи — простая задача (1 час) → 1 checkpoint в конце. Сложная (несколько дней) → checkpoint per major milestone. (3) Context window usage — автоматически при 60–70% usage. (4) Reversibility — необратимые действия → checkpoint обязателен. Reversible → можно пропустить. Heuristic: если риск потери > 30 минут работы → делай checkpoint сейчас.
Senior Как проектировать auto-resume workflow на основе Checkpoint?
Auto-resume design: (1) Checkpoint as entry point — каждый workflow run начинается с: 'есть ли checkpoint для этой задачи?' Если да → load и resume. Если нет → start fresh. (2) Idempotent steps — каждый шаг workflow проверяет 'уже ли сделано' перед выполнением (на основе checkpoint). Пример: 'PR создан? Если да в checkpoint → skip create_pr, move to review'. (3) State machine model — workflow = набор states. Checkpoint = current state + transition history. Resume = перейти в нужное state. (4) Checkpoint validation — при resume проверить что checkpoint не устарел (referenced artifacts still exist, decisions still valid). (5) Partial resume — если checkpoint частично невалиден → resume с последнего валидного state. Реализация: checkpoint.json в task directory с state field.
Senior Как валидировать целостность Checkpoint перед использованием?
Checkpoint integrity validation: (1) Referenced artifacts check — все упомянутые файлы, PR, commits в checkpoint — существуют ли? Если файл удалён → checkpoint частично невалиден. (2) Decision consistency — решения в checkpoint не противоречат актуальному состоянию Rules/AGENTS.md? Если Rules изменились → требуется reconciliation. (3) Timestamp freshness — если checkpoint создан давно → верифицировать ключевые факты (порт, URL, version). (4) Completeness check — все обязательные поля заполнены? (Goal, Next Step обязательны). (5) Schema validation — если checkpoint имеет defined schema → validate против неё. При validation failure: не silent fail → явно сообщить user 'checkpoint устарел в части X, вот что нужно уточнить'.
Lead — 5 вопросов
Lead Как ввести стандарты Checkpoint в командной разработке с AI агентами?
Командные стандарты Checkpoint: (1) Шаблон — единый Checkpoint template для команды (Goal, Status, Done, Decisions, Next). Хранится в репозитории как docs/templates/checkpoint.md. (2) When-to-checkpoint policy — документированные триггеры: перед паузой, перед compaction, при передаче задачи. (3) Storage convention — Checkpoint для командной задачи → в репозитории (docs/progress/) или в task tracker. Не в личном чате. (4) Review — для длинных задач → weekly sync включает review checkpoint (прогресс, blockers, решения). (5) Onboarding — новый engineer проходит практику: выполни задачу с Checkpoint per milestone. (6) Accountability — каждый Checkpoint имеет author и timestamp. Решения прослеживаемы.
Lead Когда руководителю требовать Checkpoint от команды — как это помогает управлению?
Checkpoint как management tool: (1) Требовать при: задача > 2 дней, задача блокирует других, задача в prod без rollback. (2) Standup replacement для async работы — вместо ежедневного standup: обновлённый checkpoint = статус. (3) Transparency — руководитель видит реальный прогресс, принятые решения, blocker — без долгих встреч. (4) Risk early warning — если checkpoint показывает blocker или нет Next Step → поговорить сразу. (5) Knowledge transfer — при болезни/отпуске инженера → другой продолжает по checkpoint. (6) Post-mortem material — checkpoint истории = timeline того что происходило в задаче. Checkpoint как management artifact: требовать в Definition of Done для сложных задач.
Lead Как использовать Checkpoint как project management инструмент для long-running AI задач?
Checkpoint в project management: (1) Milestone tracking — Checkpoint per milestone = реальный progress indicator, не '%complete'. (2) Decision log — из checkpoints извлекаются ключевые решения → ADR (Architecture Decision Records). (3) Retrospective data — checkpoint история = что заняло больше времени чем ожидалось, где были blockers → для оценки будущих задач. (4) Handoff documentation — при смене assignee → checkpoint = onboarding документ для нового. (5) Stakeholder reporting — выжимка из checkpoint = status report (без технических деталей). (6) Resource planning — если checkpoint показывает много blockers → нужна помощь/ресурсы. Integration: checkpoint → автоматически обновляет ticket status в Jira/GitHub Issues.
Lead Как обучать команду Checkpoint практикам — какой план?
Training plan: (1) Theory (30 мин) — зачем checkpoint, что теряется без него, разница checkpoint vs summary. Реальный пример: потеря 4 часов работы без checkpoint. (2) Template walkthrough — разбор шаблона поле за полем, что писать в каждое. (3) Практика 1 — взять реальную завершённую задачу и ретроспективно написать checkpoint как будто в середине задачи. (4) Практика 2 — выполнить небольшую задачу с агентом, делая checkpoint каждые 30 минут. (5) Peer review — поменяться checkpoints, оценить: можно ли продолжить работу по этому checkpoint? (6) Integration в process — checkpoint добавляется в Definition of Done. Metric: % задач с checkpoint.
Lead Как использовать Checkpoint в SLA для long-running AI задач?
Checkpoint в SLA контексте: (1) Checkpoint frequency SLA — для задач > 4 часов: checkpoint каждые 2 часа или при каждом milestone. Нарушение SLA → эскалация. (2) Checkpoint completeness — checkpoint должен содержать все обязательные поля. Incomplete checkpoint = не checkpoint. (3) Recovery SLA — при потере context (compaction без checkpoint) → время на восстановление из checkpoint ≤ 15 минут. (4) Knowledge transfer SLA — по checkpoint другой engineer должен уметь продолжить задачу без дополнительных объяснений. (5) Audit SLA — checkpoint хранятся N лет для соответствия compliance требованиям. Enforcement: automated check — есть ли свежий checkpoint при PR creation для long-running задач.
Architect — 5 вопросов
Arch Как рассматривать Checkpoint как state machine snapshot для reliable agent systems?
State machine perspective: AI workflow = state machine. States: (Started, Investigation, Implementation, Testing, Review, Done). Transitions: triggered by milestones, decisions, results. Checkpoint = snapshot конкретного state: все переменные, решения, что привело к этому состоянию. Architectural implications: (1) Explicit state definition — задокументировать какие states возможны и что означает checkpoint в каждом из них. (2) Transition guards — transition из одного state в другой → automatic checkpoint creation. (3) State validation — при resume из checkpoint → validate что referenced state still valid. (4) Replay capability — из начального state, применив sequence of transitions из checkpoints, воспроизвести весь workflow. Это дает observability, auditability и reproducibility.
Arch Как проектировать Checkpoint архитектуру для distributed multi-agent системы?
Distributed checkpoint architecture: (1) Distributed state problem — несколько агентов работают параллельно, у каждого свой context. Нужен механизм для создания global consistent checkpoint (Chandy-Lamport algorithm из distributed systems). (2) Coordinator pattern — designated coordinator agent отвечает за global checkpoint: запрашивает local checkpoints у всех агентов, объединяет. (3) Vector clocks — для упорядочивания checkpoint событий в distributed system. (4) Checkpoint store — shared storage (git repo, S3, Redis) с concurrency control. (5) Partial failure handling — если один агент недоступен при checkpoint → либо wait, либо checkpoint без него с пометкой 'partial'. (6) Recovery protocol — при failure агента → другой агент читает его последний checkpoint и берёт на себя задачу.
Arch Как интегрировать Checkpoint с event sourcing паттерном?
Checkpoint + Event Sourcing: Event sourcing — хранить не текущий state, а sequence of events, из которых state восстанавливается. Синергия с Checkpoint: (1) Events = каждое действие агента (tool call, decision, file change) логируется как event. (2) Checkpoint = snapshot state в определённый момент для быстрого restore (без replay всех events от начала). (3) Hybrid approach: sparse checkpoints + dense event log. Restore = load последний checkpoint + replay events после него. (4) Audit trail = event log + checkpoints = полная история + быстрый restore. (5) Time travel debugging — восстановить state на любой момент времени: load checkpoint before X + replay events up to X. Implementation: каждое agent action → append-only event log. Checkpoint → snapshot current state hash.
Arch Как строить fault-tolerant AI workflow с Checkpoint механизмом?
Fault-tolerant workflow design: (1) Checkpoint before every risky operation — перед любым destructive или long-running действием → checkpoint. При failure → restore и retry или rollback. (2) At-least-once semantics — некоторые operations могут выполниться дважды (при retry) → idempotent operations обязательны. Checkpoint отслеживает что уже выполнено → skip on retry. (3) Saga pattern — длинная distributed transaction = series of local transactions. Каждая local transaction → checkpoint. При failure → compensating transaction + restore from checkpoint. (4) Circuit breaker — если workflow часто падает на одном шаге → checkpoint + circuit breaker → не retry бесконечно, эскалируй. (5) Dead letter queue — failed checkpoints → для анализа почему workflow упал. (6) Health checks — периодически verify checkpoint integrity → alerting при corruption.
Arch Как проектировать Checkpoint versioning для эволюционирующегося workflow?
Checkpoint versioning: (1) Schema versioning — checkpoint format версионируется: v1, v2. При migration: старые checkpoints читаются через compatibility layer. (2) Backward compatibility — новая версия workflow может читать checkpoints старой версии (с migration если нужно). (3) Migration strategy — при schema change: автоматическая миграция checkpoint при first read. Или: explicit migration run для всех существующих checkpoints. (4) Version field — каждый checkpoint содержит schema_version поле. (5) Deprecation — старые schema versions поддерживаются N месяцев, потом forced migration или rejection. (6) Breaking changes — если workflow structure радикально изменилась → старые checkpoints невалидны → explicit handling (start fresh или manual recovery). Реализация: checkpoint.schema.json + version registry + migration scripts.
