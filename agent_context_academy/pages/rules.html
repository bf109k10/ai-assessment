Rules — политика агента

Rules — жёсткие инструкции поведения агента: что всегда делать, что никогда, как вести себя с конкретными файлами. Четыре типа (always / auto-attached / conditional / manual) с разным поведением загрузки и приоритетом.

Что такое Rules и чем отличаются от AGENTS.md

Rules = политика

Если AGENTS.md описывает workflow и контекст (рекомендации), то Rules — жёсткая политика. Примеры:

  • "Никогда не коммить файлы с секретами"
  • "Всегда добавлять unit тест к каждому PR"
  • "При работе с *.java — использовать только JPA"
  • "Не устанавливать npm пакеты без approval"

Cursor: .cursor/rules/*.mdc

Файлы с расширением .mdc в директории .cursor/rules/. Содержат YAML frontmatter + Markdown body. Cursor читает их и включает в context по правилам активации.

Claude Code: аналоги

В Claude Code понятие "Rules" реализуется через:

  • CLAUDE.md секции с жёсткими инструкциями
  • Hooks — shell-скрипты, которые принуждают к соблюдению (PreToolUse, PostToolUse)
  • Permissions — разрешения в .claude/settings.json

Формат .mdc файла — архитектура

Полный формат

--- description: "No secrets in commits" globs: [] # [] = all files alwaysApply: true # всегда в context --- # No Secrets Rule Never commit files containing: - API keys, tokens, passwords - .env files with real values - Private certificates or keys ALWAYS check before suggesting git add/commit. If a secret is found — stop and warn the user.
# Auto-attached rule для Java файлов --- description: "Java coding standards" globs: ["**/*.java", "**/*.kt"] alwaysApply: false --- Use JPA for all database operations. No raw SQL. Method length max 30 lines. Always add Javadoc to public methods.

Поля frontmatter

ПолеОбяз.Описание
description✓Краткое описание правила. Используется AI для понимания когда применять.
globs—Паттерны файлов для auto-attach. [] = все файлы. ["**/*.java"] = только Java.
alwaysApply—true — правило ВСЕГДА в context независимо от файлов. false — только при match.
Naming convention: называй файлы описательно: no-secrets.mdc, java-standards.mdc, pr-checklist.mdc. Не rule1.mdc.
Лимиты: в description не более 80 символов. Не все версии Cursor поддерживают сложные globs. Тестируй на реальных файлах.
🖱 Cursor — Rules (.mdc)

Файлы .cursor/rules/*.mdc. YAML frontmatter + Markdown body. Три режима активации: alwaysApply, globs, description-only.

--- description: "No secrets in commits" globs: [] alwaysApply: true --- Never commit API keys or .env files.
🤖 Claude Code — аналоги Rules

В Claude Code «жёсткие правила» реализуются через секции CLAUDE.md + Hooks (принуждение) + settings.json (permissions).

# CLAUDE.md — секция правил ## Hard Rules - Never commit secrets - Always run tests before commit - No npm installs without approval # hooks.json — принуждение PreToolUse: проверяет перед действием

4 типа Rules — как работает активация

Always

alwaysApply: true

Всегда в context. Независимо от файлов и задачи.

Когда использовать: критическая политика которую агент ВСЕГДА должен соблюдать — безопасность, запреты на секреты, обязательные шаги workflow.

Стоимость: каждый запрос потребляет токены на этот файл. Держи маленьким (<100 строк).

alwaysApply: true globs: [] # Пример: правило безопасности, PR-чеклист, team conventions

Auto Attached

globs: ["**/*.java"]

Включается автоматически при match с файлами в context.

Когда использовать: language/framework-специфичные стандарты. Правило для Java файлов не нужно при работе с Python.

Примеры globs:

  • ["**/*.java", "**/*.kt"] — Java/Kotlin файлы
  • ["src/billing/**"] — конкретный модуль
  • ["**/*.sql"] — SQL файлы
  • ["**/tests/**", "**/*Test*.java"] — тестовые файлы

Agent Requested

description-only

Агент сам решает когда включить, на основе description.

Когда использовать: специализированные гайды которые нужны только иногда — "deployment checklist", "database migration guide". Агент читает description и включает если задача релевантна.

Важно: хороший description критичен — по нему агент решает включать ли правило.

# alwaysApply: false, globs: [] → # агент сам решает на основе description description: "Database migration checklist for schema changes"

Manual

@rule-name

Только когда пользователь явно упоминает через @mention.

Когда использовать: большие гайды которые нужны редко — "архитектурный review", "security audit guide". Не засоряют context в обычной работе.

Использование: @rule-name или @.cursor/rules/deploy-guide.mdc в чате.

Примеры реальных правил

Безопасность (alwaysApply)

--- description: "Security: no secrets in code" alwaysApply: true --- NEVER include in code or commits: - API keys, tokens, passwords - .env files with real values - Connection strings with credentials If asked to add such values → suggest .env and add .env to .gitignore first.

Java стандарты (auto-attached)

--- description: "Java/Kotlin code standards" globs: ["**/*.java", "**/*.kt"] alwaysApply: false --- - Use JPA for all DB operations, no raw SQL - Max method length: 30 lines - Public methods require Javadoc - Use Optional, not null returns - Exceptions: checked for IO, unchecked for logic

PR чеклист (alwaysApply)

--- description: "PR creation checklist" alwaysApply: true --- Before suggesting PR creation: 1. Tests passing (`mvn test`) 2. No hardcoded secrets 3. PR description: Problem→Fix→Tests→Risk 4. DB migration has rollback plan 5. API changes → architecture approval

Deploy guide (manual)

--- description: "Production deployment procedure" alwaysApply: false globs: [] --- # Production Deploy Checklist 1. Feature flag OFF before deploy 2. DB migration in separate step 3. Canary 5% → 25% → 100% 4. Monitor errors for 15 min each step 5. Rollback: `kubectl rollout undo`

Тонкости и нюансы

alwaysApply Rules попадают в system-prefix

Rules с alwaysApply: true не живут в user-чате — IDE подмешивает их в начало context window вместе с platform system prompt. Для модели это ведёт себя как system instructions (роль system / стабильный prefix), хотя файлы лежат в .cursor/rules/ и версионируются в git. Это не то же самое, что скрытый platform prompt: вы контролируете Rules, но не platform safety. Подробнее: System instructions и карта Контекст агента.

alwaysApply: true — не злоупотреблять

Каждое alwaysApply правило потребляет токены при КАЖДОМ запросе. Если у тебя 5 таких правил по 200 строк — это 1000 токенов overhead на каждый вопрос, даже простой. Правило: в alwaysApply только то, что реально критично всегда. Остальное — auto-attached или manual.

Конфликт Rules с AGENTS.md

Rules имеют приоритет над AGENTS.md. Если AGENTS.md говорит "используй Maven" а rule говорит "используй Gradle для billing/" — rule побеждает. Правило: не дублируй важные правила в AGENTS.md если они уже в Rules, чтобы не создавать конфликтующие инструкции.

Claude Code Hooks — жёстче чем Rules

В Claude Code правила можно усилить через Hooks — shell-скрипты выполняющиеся до/после tool calls. Например, PreToolUse hook для Bash может проверять, не содержит ли команда git commit файлы с секретами. Hooks — это enforcement на уровне системы, не suggestion.

# .claude/settings.json — пример hook { "hooks": { "PreToolUse": [{ "matcher": "Bash", "hooks": [{ "type": "command", "command": "scripts/check-secrets.sh" }] }] } }

Globs — подводные камни

Glob ["**/*.java"] матчит все Java файлы рекурсивно. Но если агент работает только с pom.xml — Java rule не включится. Убедись что globs покрывают реальные точки входа: включи и конфигурационные файлы (pom.xml, build.gradle) если они релевантны.

# Лучше для Java-проекта: globs: ["**/*.java", "**/*.kt", "**/pom.xml", "**/build.gradle"]

Rules scope — только Cursor

.cursor/rules/*.mdc читает только Cursor Agent. Claude Code использует CLAUDE.md + Hooks + permissions в .claude/settings.json. Если в команде используются оба инструмента — нужно дублировать критические правила в оба формата или хранить общую истину в AGENTS.md/CLAUDE.md.

Best practices создания Rules

Структура правила

  • Начинай с NEVER / ALWAYS — агент чётко понимает директиву
  • Краткость: 20-50 строк идеал, не роман
  • Один файл — одна тема: no-secrets.mdc, не all-rules.mdc
  • Description ≤ 80 символов, конкретный: "Java DB: no raw SQL"
  • Примеры помогают: покажи плохой и хороший вариант

Анти-паттерны

  • God rule — одно правило на всё → агент путается
  • Все правила как alwaysApply → token overhead
  • Расплывчатые инструкции: "пиши хороший код" — не правило
  • Противоречие между разными .mdc файлами
  • Дублировать то же самое что есть в AGENTS.md

Для команды

  • Держи Rules в репо (версионируются как код)
  • PR review для изменений в rules/
  • Документируй в AGENTS.md: "ключевые правила в .cursor/rules/"
  • Периодически аудит: устарели ли правила?
  • Тестируй что правила реально работают на примерах

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Rule в Cursor?
Rule — файл .cursor/rules/*.mdc с YAML frontmatter и Markdown-телом. Содержит инструкции для агента на уровне политики: стандарты кода, ограничения, соглашения команды. Cursor автоматически читает все файлы из .cursor/rules/ при запуске сессии.
Junior Что такое .mdc формат файла?
Markdown-файл с YAML frontmatter. Пример структуры: ---\ndescription: "Enforce Java style"\nglobs: ["**/*.java"]\nalwaysApply: false\n---\n# Правило\nВсегда используй JPA, не raw SQL. Frontmatter определяет когда и как Rule применяется; тело — сами инструкции в свободном Markdown.
Junior Что такое alwaysApply: true в frontmatter?
Директива, при которой Rule включается в каждый запрос к агенту независимо от открытых файлов и контекста. Потребляет токены при каждом вызове, поэтому используется только для критически важных политик (безопасность, обязательные стандарты). Остальные правила лучше переводить на auto-attached (globs).
Junior Что такое auto-attached rule?
Rule с заполненным полем globs (например ["**/*.py"]). Cursor автоматически добавляет его в контекст, когда в диалоге открыты файлы, совпадающие с паттернами. Более токено-эффективен чем alwaysApply: срабатывает только когда реально нужен.
Junior Как создать Rule в Cursor?
Создай файл в директории .cursor/rules/ с расширением .mdc. Добавь YAML frontmatter (description, globs или alwaysApply). Напиши инструкции в Markdown-теле. Cursor подхватит файл автоматически — перезапуск не нужен. Команда: Cursor → Settings → Rules → New Rule, или создай файл вручную.
Junior Где хранятся Rules в Cursor?
В директории .cursor/rules/ в корне проекта. Это обычная папка в git-репозитории, поэтому Rules версионируются вместе с кодом. Каждый разработчик, клонировавший репо, автоматически получает все правила команды.
Junior Что такое frontmatter в .mdc файле?
Блок YAML в начале файла, обёрнутый в ---. Три ключевых поля: description — для кого этот Rule (используется agent-requested); globs — список паттернов файлов для auto-attach; alwaysApply — true/false, включать ли всегда. Без frontmatter Rule трактуется как manual (только по явному @mention).
Junior Чем Rule отличается от AGENTS.md?
Rule — политика: постоянное ограничение или стандарт («никогда не коммить секреты», «всегда использовать JPA»). AGENTS.md — контекст и workflow: описание проекта, структура, что избегать в этом конкретном репо. Приоритет: Rules > AGENTS.md при конфликте.
Junior Что такое agent-requested rule?
Rule с непустым description и alwaysApply: false (без globs). Агент читает descriptions всех таких Rules и сам решает, запросить ли конкретный Rule для текущей задачи. Description должен быть ≤80 символов и чётко объяснять когда этот Rule нужен.
Junior Что такое manual rule?
Rule без frontmatter или с пустым description/globs/alwaysApply. Применяется только при явном @rule-name в чате. Полезно для специализированных инструкций, которые нужны редко и в очень конкретном контексте. Не попадает в context автоматически — никакого token overhead.
Middle — 10 вопросов
Middle Когда использовать alwaysApply vs globs?
alwaysApply: true — только для критической политики которая должна соблюдаться абсолютно всегда (no-secrets, обязательный стиль всего кода). Globs — для языко- или модульно-специфичных правил (Java-стиль только для *.java, frontend-соглашения для src/components/**). alwaysApply = token overhead × N запросов; globs = токены тратятся только при работе с нужными файлами.
Middle Каков приоритет Rules относительно AGENTS.md и других слоёв?
Полная иерархия по убыванию приоритета: Hooks (OS) > System Prompt > Rules (alwaysApply) > Rules (auto-attached) > AGENTS.md > current chat > Memory > Skills. При конфликте: более высокий уровень побеждает. Rules — политика, AGENTS.md — контекст, Claude Code Hooks — абсолютный enforcement на уровне системы.
Middle Как оптимизировать token overhead от alwaysApply Rules?
1) Переводи тематически узкие Rules с alwaysApply на auto-attached (globs). 2) Объединяй связанные alwaysApply-правила в один файл вместо 10 отдельных. 3) Держи тело alwaysApply-правила лаконичным — только суть, без примеров. 4) Правило: не более 3–5 alwaysApply Rules; остальные — auto-attached или agent-requested.
Middle Как Claude Code заменяет Cursor Rules если нет .mdc?
В Claude Code нет .mdc файлов. Аналоги: (1) Секции MUST/NEVER/ALWAYS в CLAUDE.md — мягкое текстовое принуждение; (2) Hooks в .claude/settings.json — PreToolUse/PostToolUse shell-скрипты для жёсткого enforcement; (3) Permissions в settings.json — allow/deny для конкретных tools. Hooks дают настоящее принуждение: агент физически не может выполнить заблокированное действие.
Middle Как писать эффективный description для agent-requested Rule?
Description читается агентом при каждом запросе для принятия решения «применять ли». Требования: ≤80 символов, конкретный (плохо: «code style», хорошо: «Java: always JPA, no raw SQL, no Lombok»), написан в третьем лице или как триггер («Use when editing Java files»). Расплывчатый description = агент не знает когда запрашивать Rule = Rule никогда не применяется.
Middle Что такое принцип single responsibility для Rules?
Один .mdc файл — одна тема. no-secrets.mdc, java-style.mdc, db-access.mdc — три отдельных файла, не один. Это позволяет: применять только нужные (auto-attached по globs), легко обновлять одно правило не трогая остальные, избегать конфликтов. «God rule» с 50 инструкциями — антипаттерн: agent confusion + максимальный token overhead.
Middle Как правильно версионировать Rules в git?
Rules хранятся в .cursor/rules/ — обычная директория, не в .gitignore. PR включает изменение Rules → code review. Рекомендации: добавить дату изменения в description; owner per .mdc (кто отвечает); CHANGELOG для критических Rules; CI-проверка что обязательные Rules присутствуют. Новый разработчик клонирует репо → Rules сразу активны.
Middle Как диагностировать конфликт между двумя Rules?
Признаки: агент выдаёт непоследовательное поведение, «балансирует» между двумя инструкциями. Диагностика: (1) перечисли все alwaysApply Rules; (2) ищи противоречия (краткость vs подробность, разные стили); (3) нет deterministic winner для Rules одного уровня. Решение: объедини противоречащие правила в один файл с условиями или удали менее важное.
Middle Как использовать Rules для командных стандартов кодирования?
Rules — централизованный источник истины для всей команды. Примеры: java-style.mdc (форматирование, naming), test-conventions.mdc (AAA-паттерн, naming тестов), no-secrets.mdc (никаких паролей в коде). Преимущества перед wiki/README: Rules применяются автоматически, не требуют от разработчика помнить о правилах.
Middle Как проверить что Rule работает как ожидается?
1) Спроси агента напрямую: «Какие Rules применяются к этому файлу?». 2) Создай тестовый кейс: попроси агента сделать то, что Rule должен запрещать. 3) Для auto-attached: открой matching файл и проверь что Rule появляется в context. 4) Проверь YAML frontmatter на валидность (нет опечаток в ключах). 5) Убедись что файл в .cursor/rules/, не в другой директории.
Senior — 5 вопросов
Senior Как спроектировать архитектуру Rules для крупного проекта с несколькими командами?
Многоуровневая архитектура: (1) Глобальные Rules (alwaysApply) — security, compliance, обязательные для всех: no-secrets.mdc, no-direct-prod.mdc. (2) Доменные Rules (auto-attached по globs) — стиль для Java/Python/frontend. (3) Модульные Rules (agent-requested) — специфика платёжного модуля, DB-доступа. Принципы: owner per file, quarterly audit, не более 5 alwaysApply, один файл = одна ответственность.
Senior Как диагностировать ситуацию когда агент систематически нарушает Rule?
Причины по убыванию вероятности: (1) Rule не попадает в context (не alwaysApply, globs не совпадают, файл не в .cursor/rules/); (2) Lost in the middle — Rule есть в context, но тонет среди большого количества токенов; (3) Конфликт с другим Rule или с инструкцией пользователя; (4) Формулировка правила неоднозначна. Диагностика: спроси агента воспроизвести Rule своими словами — если не может, Rule не в context.
Senior Что такое эффект «lost in the middle» для Rules и как с ним бороться?
LLM хуже обрабатывает информацию из середины длинного контекста. Если Rules занимают 5000+ токенов и расположены в середине context window — агент может «не видеть» отдельные правила при длинной задаче. Решение: (1) минимизируй количество alwaysApply Rules; (2) критические правила выноси в начало (Cursor использует начало system prompt); (3) для абсолютно критичных правил — дублируй в Hooks (Claude Code).
Senior Что такое Rules inheritance паттерн и как реализовать?
Паттерн: базовые Rules + специализированные, переопределяющие базу. В Cursor нет встроенного наследования, реализуется через: (1) base-style.mdc (alwaysApply) + java-style.mdc (globs *.java) — специфика добавляется поверх базы; (2) явные ссылки в теле Rule: «расширяет base-style.mdc, добавляет...». Риск: при конфликте «базы» и «специализации» — нет deterministic winner, нужно явно прописать приоритет.
Senior Что такое Rules debt и как проводить аудит?
Rules debt — накопленные устаревшие, конфликтующие или избыточные правила. Симптомы: агент следует неактуальному правилу; два Rules противоречат друг другу; 20 alwaysApply Rules при работающих 3. Аудит (раз в квартал): (1) список всех .mdc — проверить актуальность каждого; (2) поиск конфликтов (grep по похожим темам); (3) token cost analysis — сколько токенов потребляет набор Rules; (4) sunsetting: помечай устаревшие # DEPRECATED перед удалением.
Lead — 5 вопросов
Lead Как выстроить governance Rules в команде?
Governance model: (1) Owner per .mdc — каждый файл имеет ответственного; (2) PR review для Rules — изменения в .cursor/rules/ проходят review как код; (3) Обязательные Rules задокументированы в AGENTS.md с объяснением зачем; (4) Quarterly audit — встреча команды, проверка актуальности всех Rules; (5) Новые Rules — только через PR, с обоснованием и примером нарушения которое хотим предотвратить.
Lead Как использовать Rules для onboarding новых разработчиков?
Rules — автоматический учитель для новичков. Стратегия: (1) onboarding.mdc (alwaysApply, временный) — «ты в новом проекте, ключевые соглашения: ...»; (2) Style Rules автоматически корректируют ошибки новичков без manual review; (3) AGENTS.md ссылается на Rules как источник стандартов; (4) onboarding checklist: «клонировал репо → .cursor/rules/ должны загрузиться → проверь что видишь N правил». Результат: снижение onboarding friction.
Lead Как организовать PR review для изменений в Rules?
Rules = code. Изменения в .cursor/rules/ требуют PR review. Checklist для reviewer: (1) новое правило не конфликтует с существующими; (2) тип Rule соответствует нужде (зачем alwaysApply, а не globs?); (3) description ≤80 символов, конкретный; (4) изменение протестировано — агент ведёт себя как ожидается; (5) owner указан. Рекомендация: назначь Rule-reviewer (Tech Lead или Principal Engineer).
Lead В чём разница между командными и индивидуальными Rules? Как управлять обоими?
Командные Rules: в .cursor/rules/ в репо, версионируются, применяются ко всем → изменения через PR. Индивидуальные: в Cursor Settings → Rules (personal) или локальный .mdc в .gitignore → не влияют на команду. Cursor Business: Team Rules через admin panel — не могут быть отменены разработчиком. Правило: личные предпочтения — individual; кодовые стандарты — team; compliance — Team Rules в Cursor Business или Hooks.
Lead Как найти баланс между enforcement через Rules и доверием агенту?
Over-restricted Rules → агент не может делать разумные задачи; Under-restricted → нарушения стандартов. Баланс: (1) Rules запрещают конкретно опасные действия (секреты, direct DB в prod), а не ограничивают общую свободу; (2) правило «targeted prohibition»: не «можно только X», а «нельзя конкретно Y»; (3) тестируй каждое правило — не мешает ли оно легитимным задачам; (4) Rules — living documents: если правило блокирует valid use case, обнови его.
Architect — 5 вопросов
Arch Как Rules вписываются в многослойную архитектуру AI-системы?
Rules — слой policy & conventions в AI-стеке: ниже System Prompt (задаёт роль агента), выше AGENTS.md (контекст проекта). В архитектуре: [Hardware] → [OS] → [Claude Code Hooks] → [System Prompt] → [Rules (alwaysApply)] → [Rules (auto-attached)] → [AGENTS.md] → [Chat context]. Rules — единственный слой управляемый командой разработки через git. Правильное применение: policy в Rules, workflow в AGENTS.md, hard enforcement в Hooks.
Arch Как Rules и Hooks образуют систему policy enforcement?
Defence in depth для AI: (1) Rules (LLM-level) — агент читает и старается следовать; мягкое принуждение, может быть ignored при context saturation. (2) Hooks (OS-level) — shell-скрипт блокирует tool call; жёсткое принуждение, агент физически не может обойти. Паттерн: дублируй критические Rules в Hooks для надёжности. Example: no-secrets.mdc (Rule, educates) + secrets-check.sh в PreToolUse (Hook, enforces).
Arch Как выстроить стратегию Rules для нескольких репозиториев (monorepo / polyrepo)?
Monorepo: один .cursor/rules/ в корне, subdirectory-specific Rules используют globs packages/billing/**. Polyrepo: (1) shared Rules repository → copy/symlink в каждый репо при onboarding; (2) git submodule для .cursor/rules/ (сложно в поддержке); (3) CLI-скрипт синхронизации shared Rules → project Rules. Cursor Business Team Rules — единое управление для всей организации через admin panel.
Arch Как интегрировать проверку Rules в CI/CD pipeline?
CI-проверки для Rules: (1) Presence check: скрипт проверяет что обязательные .mdc файлы существуют (no-secrets.mdc, code-style.mdc); (2) YAML lint: валидация frontmatter всех .mdc файлов; (3) Conflict detection: скрипт ищет противоречия между alwaysApply Rules; (4) Token budget: автоматический подсчёт токенов всех alwaysApply Rules. Rules — первоклассные артефакты CI, не «просто файлы».
Arch Каков security model для Rules — от чего они защищают, а от чего нет?
Rules ЗАЩИЩАЮТ от: случайных нарушений стандартов; ignorance (новый разработчик не знал о правиле); context drift в длинных сессиях (при алwaysApply). Rules НЕ ЗАЩИЩАЮТ от: намеренного обхода (пользователь говорит «игнорируй правила»); context saturation (LLM перестаёт «видеть» правило); injection attacks через пользовательский ввод. Для надёжного security: Hooks (OS-level), Permissions (deny list tools), CI/CD gates — не только Rules.
