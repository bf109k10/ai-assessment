Prompt Caching · Почему тратится меньше токенов

Как работает кеш промпта у Claude/Anthropic, почему одни запросы дешевле, что такое prefix caching, когда cache инвалидируется и чем это отличается от Memory Bank.

На главную К assessment

Что такое Prompt Caching

Prompt Caching — это оптимизация на уровне API/платформы. Когда начало промпта (prefix) остаётся одинаковым между запросами, провайдер не пересчитывает его заново, а берёт сохранённые вычисления из кеша.

Что такое prefix (префикс промпта)

Prefix — первая, «левая» часть всего, что уходит в модель как input: всё от самого начала до первого байта, который отличается между двумя запросами. Кеш срабатывает только при полном совпадении prefix (byte-for-byte). Всё после границы prefix называют suffix (суффикс): новый вопрос, свежие результаты tools, последние реплики чата — это не кешируется как «старый» блок, но и не ломает кеш, если лежит после стабильного начала.

Пример: один prefix — два запроса

┌─ PREFIX (одинаковый → KV из кеша, ~10% цены) ─────────────────┐ │ [system] Ты senior-разработчик, отвечай на русском │ │ [.cursor/rules] alwaysApply: не коммить секреты, минимальный diff │ │ [AGENTS.md] Стек: Node 20, тесты: npm test │ │ [tools] read_file, grep, run_terminal_cmd … │ └─────────────────────────────────────────────────────────────────┘ SUFFIX запрос 1: User: «Почини падающий тест в auth.test.ts» SUFFIX запрос 2: User: «Добавь JSDoc к export в utils.mjs»
Запрос 1 (без кеша)
System prompt (2000 токенов) + User message (50 токенов)
Full cost: 2050 токенов
Запрос 2 (prefix кеширован)
System prompt — из кеша (×0.1 цена)
Новый message
Cost: ~50 + 200 токенов ≈ -90%
Ключевая идея: Платформа сохраняет вычисленные KV-state (Key-Value состояния) для prefix'а. При следующем запросе с тем же prefix — эти состояния берутся из кеша. Это быстрее (меньше latency) и дешевле (меньше compute).

Как это работает технически

KV-Cache (Key-Value Cache)

LLM при обработке токенов вычисляет Key и Value матрицы для attention механизма. Эти вычисления можно сохранить для неизменной части промпта.

  • System prompt + AGENTS.md + Rules = стабильный prefix
  • KV-state для этого prefix кешируется на сервере
  • Следующий запрос с тем же prefix → берётся из кеша
  • Только новые токены (user message, tool results) обрабатываются полностью

Что кешируется у Anthropic (Claude)

  • Минимум 1024 токена для кеширования (Claude 3)
  • Можно явно разметить cache_control: {type: "ephemeral"}
  • TTL кеша: ~5 минут для ephemeral
  • Цена cached tokens: ~10% от full price
  • Cache hit → экономия до 90% на prefix
  • Cache miss → полная цена, но кеш создаётся

Что должно быть стабильным (в начале промпта) для кеша

1

System prompt (самый стабильный)

Роль агента, safety rules, базовые инструкции. Меняется редко → отличный кандидат для кеша.

2

AGENTS.md / проектный контекст

Стек технологий, workflow, команды тестов. Меняется редко в течение сессии → хорошо кешируется.

3

RAG-документы / большие справочники

Если добавить их в начало один раз и они не меняются — следующие запросы дешевле.

4

User messages / Tool results (последние)

Всегда новые — не кешируются. Но за счёт кеша prefix'а, стоимость только этих токенов.

Инвалидация кеша — главная ловушка

Важно: Кеш работает ТОЛЬКО если prefix совпадает byte-for-byte. Любое изменение в начале промпта = инвалидация всего кеша после этой точки.

Что ломает кеш

  • Изменение порядка tool definitions
  • Добавление timestamp в system prompt
  • Изменение любого токена в начале
  • Разные system prompts в разных сессиях
  • Динамический контент в начале (дата, случайное)
  • Истечение TTL (~5 минут для ephemeral)

Что сохраняет кеш

  • Одинаковый system prompt во всех запросах
  • Стабильный порядок tools
  • Документы/RAG в начале (перед user messages)
  • Быстрые последующие запросы (в TTL)
  • Нет динамики в prefix'е

Пример: правильный порядок для максимального кеша

Плохо — кеш сломан

System: You are a Java developer.
Current time: 14:32:01 ← меняется!
AGENTS.md: ...
---
User: Fix the bug

Timestamp в prefix'е → каждый запрос = full cost

Хорошо — кеш работает

System: You are a Java developer.
AGENTS.md: [стабильный контент]
RAG Docs: [стабильные документы]
--- ← граница кеша
User: Fix the bug (14:32:01 ← здесь OK)

Prefix стабилен → cached. Время только в user message.

Prompt Caching vs Memory Bank vs RAG — не путать!

МеханизмЧто этоГде живётЦельИнвалидация
Prompt Caching Кеш вычислений KV-state для одинакового prefix'а Сервер провайдера (Anthropic, OpenAI) Экономия токенов и latency Изменение prefix, истечение TTL
Memory Bank Хранилище смысловой информации: решения, предпочтения, знания о проекте Файл/DB в проекте или IDE Continuity между сессиями Явное обновление/удаление
RAG Поиск и добавление релевантных документов в контекст Vector DB + document store Factual grounding из большой базы Переиндексация документа
Checkpoint Структурный снимок состояния задачи Файл в чате или проекте Handoff, продолжение работы Ручное обновление
Assessment trap: "Prompt caching — это как Memory?" — Нет! Это низкоуровневая инфраструктурная оптимизация. После истечения TTL кеш теряется. Модель ничего "не помнит" из кеша — это только saved computation.

Практическая польза для разработчика

Когда важен prompt caching

  • Большой AGENTS.md / CLAUDE.md (экономия значительная)
  • Много RAG-документов в context
  • Iterative разработка в одной сессии
  • API-интеграции с повторными вызовами
  • Agentic loop: каждый шаг дешевле

Как использовать на практике

  • Стабильный system prompt — не добавляй динамику
  • Документы/контекст — в начало, перед user messages
  • Не меняй порядок tool definitions между запросами
  • Используй claude.ai / Cursor — они управляют кешем автоматически
  • В API — явная разметка cache_control

Числа (Claude 3.5 Sonnet)

  • Cache write: 125% от base price (один раз)
  • Cache read: 10% от base price (каждый раз)
  • Минимум для кеша: 1024 токена
  • TTL: ~5 минут (ephemeral)
  • Выгода: при 5+ повторных запросах = окупается

Тонкости и нюансы Prompt Caching

Любое изменение prefix сбрасывает кеш полностью

Кеш работает по принципу byte-for-byte совпадения prefix'а. Добавил один символ в system prompt — весь кеш инвалидируется. Поэтому динамические данные (timestamp, session ID, случайные нonce) никогда не должны быть в system prompt. Только в user message — после стабильного prefix'а.

TTL кеша — ~5 минут, не сессия

Anthropic KV-cache живёт около 5 минут при активном использовании. После паузы — инвалидируется. Это означает: кеш работает при быстрых последовательных запросах (agentic workflow), но не между сессиями пользователя. Рассчитывать на кеш для "утренних" запросов — ошибка.

Prompt caching ≠ Memory — принципиально разные вещи

Caching = saved compute state (attention matrices). Пропали — ничего не помнит. Memory Bank = смысловое хранилище знаний (файл или БД). Кеш прозрачен для агента, он не знает кешировано ли. Memory — явный контекст который агент читает. Путать их на assessment — грубая ошибка.

Структура промпта критична для попадания в кеш

Для максимального кеша: [стабильный system prompt] → [AGENTS.md] → [RAG документы] → [user message с динамикой]. Нарушение порядка — потеря кеша. Если поменяешь порядок RAG документов между запросами — кеш промахнётся. Order matters для byte-for-byte matching.

Кеш работает на провайдере — не на клиенте

KV-cache хранится на стороне Anthropic API (или другого провайдера). Cursor/Claude Code не управляют кешем напрямую — они оптимизируют структуру промпта. При смене провайдера или модели — кеш сбрасывается. Feature доступна на Anthropic Sonnet/Opus, не на всех моделях.

Best practices — Делай / Не делай

✅ Делай

  • Размещай стабильный system prompt первым, динамику — в user message
  • Держи порядок RAG документов постоянным между запросами
  • Используй кеш для agentic workflows: повторные вызовы с одним system prompt
  • Проверяй через usage API: cached_input_tokens vs input_tokens
  • Объединяй AGENTS.md + Rules в один стабильный блок перед динамикой

❌ Не делай

  • Не добавляй timestamp, session ID или рандом в system prompt
  • Не меняй порядок tools или документов между похожими запросами
  • Не рассчитывай на кеш после длинных пауз (TTL ~5 минут)
  • Не путай caching экономию с Memory Bank — это разные механизмы
  • Не ожидай кеш на всех моделях — только Anthropic Sonnet/Opus и аналоги

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое prompt caching?
Prompt caching — инфраструктурная оптимизация LLM провайдера: если несколько запросов начинаются с одинакового текста (prefix), провайдер вычисляет attention один раз и сохраняет промежуточное KV-состояние. При последующих запросах с тем же prefix берёт из кеша, а не пересчитывает. Результат: cached токены стоят ~10% от обычной цены, latency первого token значительно снижается. Это прозрачная оптимизация — программист не меняет код, только структурирует prompt определённым образом.
Junior Что такое cache prefix?
Cache prefix — начальная часть prompt, которая остаётся одинаковой между запросами. Чем длиннее стабильный prefix — тем больше токенов кешируется. Типичный prefix: system prompt + Rules + AGENTS.md + инструменты. Всё это не меняется между запросами одной сессии. Динамическая часть (user message, conversation history) идёт ПОСЛЕ стабильного prefix — так кеш работает эффективно. Правило: стабильное → в начало prompt; динамическое → в конец.
Junior Как работает кеш в LLM на техническом уровне?
LLM обрабатывает каждый токен через attention механизм, создавая Key-Value (KV) матрицы для каждого слоя. При prompt caching провайдер сохраняет эти KV матрицы для prefix токенов. При следующем запросе с тем же prefix — KV берётся из кеша, вычисляются только новые токены. Это byte-for-byte сравнение: любое изменение в prefix инвалидирует кеш. Провайдер (Anthropic, OpenAI) управляет кешем на своей стороне — разработчик не имеет прямого контроля.
Junior Что такое cache hit и cache miss?
Cache hit — запрос начался с точно такого же prefix как предыдущий → провайдер использует сохранённые KV матрицы, prefix токены не пересчитываются. Дешевле и быстрее. Cache miss — prefix изменился (даже на 1 символ) → полный пересчёт, кеш не используется. Разница в цене: у Anthropic кешированные токены ~10% стоимости обычных. При cache hit: первый output token (TTFT) значительно быстрее. Мониторить: API response включает cache_creation_input_tokens и cache_read_input_tokens.
Junior Зачем нужно prompt caching — какова практическая польза?
Три главных выгоды: (1) Cost reduction — если system prompt 10 000 токенов и кешируется в 90% запросов → экономия ~90% стоимости prefix токенов. При большом масштабе это тысячи долларов в месяц. (2) Latency reduction — time-to-first-token (TTFT) снижается в 2–5x для длинных prefix. Пользователь видит ответ быстрее. (3) Longer system prompts — при высоком cache hit rate можно позволить себе более детальные Rules/AGENTS.md без пропорционального роста стоимости. Пример: 100K токенов system prompt с кешем = почти бесплатный prefix.
Junior Что такое cached tokens?
Cached tokens — токены из prefix которые уже вычислены и сохранены в кеше провайдера. В API response возвращается статистика: prompt_tokens (всего), cached_tokens (из кеша), new_tokens (пересчитаны заново). Cached tokens стоят значительно дешевле (Anthropic: ~10% цены input tokens). Важно понимать: cached tokens всё равно считаются в рамках context window limit — кеш экономит вычисления и деньги, но не место в context.
Junior Как кеш снижает стоимость запросов?
Экономия происходит потому что провайдер не пересчитывает attention для cached tokens. Пример: system prompt = 8000 токенов. При цене $3/1M input tokens без кеша: 8000 × $3/1M = $0.024 за запрос только на prefix. С кешем (90% hit rate): 8000 × $0.3/1M = $0.0024 — в 10 раз дешевле. При 1000 запросов в день: $24 vs $2.4. Реальная экономия зависит от hit rate и размера prefix. Инструмент для расчёта: Anthropic Calculator на их сайте.
Junior Что такое stable prefix и почему он критичен?
Stable prefix — часть prompt которая не меняется между запросами. Кеш работает только если prefix идентичен byte-for-byte. Если prefix меняется в каждом запросе — кеш не работает никогда. Что должно быть в stable prefix: system prompt, Rules, AGENTS.md содержимое, tool descriptions, few-shot examples. Что НЕ должно быть в prefix: timestamp, user name (если меняется), любые динамические данные. Архитектурный принцип: static/slow-changing context → в начало; dynamic context → в конец.
Junior Когда кеш сбрасывается?
Кеш инвалидируется при: (1) Любом изменении prefix — даже один символ → full miss. (2) Истечении TTL — Anthropic: ~5 минут неиспользования. (3) Обновлении модели — новая версия Claude имеет пустой кеш. (4) Смене API ключа / организации. Практически: если добавить timestamp в system prompt → кеш никогда не hit (новый timestamp каждую секунду). Если добавить user_name в начало system prompt (меняется для разных пользователей) → кеш не работает при multi-user сценарии.
Junior Кто управляет кешем — разработчик или платформа?
Платформа (провайдер) управляет кешем автоматически. Разработчик не имеет прямого контроля: нельзя принудительно заполнить кеш, нельзя явно очистить, нельзя задать TTL. Разработчик управляет косвенно: через структуру prompt (stable prefix в начале) и через cache_control флаги (у Anthropic: явная пометка breakpoints). В Cursor и Claude Code кеш работает автоматически за кулисами — при длинных разговорах с одинаковым system prompt. Можно проверить эффективность: смотреть cache_read_input_tokens в API usage.
Middle — 10 вопросов
Middle Как структурировать prompt для максимального cache hit rate?
Правило: стабильное → начало, динамическое → конец. Идеальный порядок: [System instructions] → [Rules / AGENTS.md] → [Tool descriptions] → [Few-shot examples] → [Retrieved context (RAG)] → [Conversation history] → [Current user message]. Каждый слой менее стабильный чем предыдущий. Conversation history меняется с каждым turn, но всё что перед ней кешируется. Ошибка: system prompt в начале, потом дата/время — это ломает весь кеш для всего что после.
Middle Почему добавление timestamp или даты в начало system prompt ломает кеш?
Кеш требует byte-for-byte совпадения prefix. Timestamp меняется каждую секунду → prefix никогда не совпадает → 100% cache miss. Даже если в system prompt написать 'Today is Monday, July 1, 2026' — завтра это изменится и кеш сбросится. Решение: убери временные данные из system prompt (и вообще из prefix). Если нужна дата → добавляй в user message (после stable prefix). Если нужен current context (текущая задача, имя пользователя) — тоже в user message, не в system prompt.
Middle Как Rules и AGENTS.md влияют на кеш и где их лучше размещать?
Rules и AGENTS.md — отличный кандидат для кеша: они меняются редко (только при обновлении конфигурации). Если включить их в system prompt или его начало — они кешируются и не пересчитываются каждый запрос. Placement: rules → сразу после базовых system instructions. При смене Rules (commit в .cursor/rules/) кеш сбросится для всех новых запросов — это нормально. Бонус: можно позволить длинные детальные Rules без пропорционального роста cost, потому что кеш покрывает их стоимость.
Middle Как различать stable и dynamic части prompt при проектировании системы?
Классификация: Static (никогда не меняется) — core instructions, persona, output format rules → в начало. Slow-changing (меняется при deploy/обновлении) — Rules, AGENTS.md, tool descriptions → после static. Session-stable (меняется раз в сессию) — user context, project name → после slow-changing. Turn-dynamic (меняется каждый turn) — conversation history, retrieved RAG chunks → после session. Request-dynamic (меняется для каждого request) — текущий user message, timestamp → в самый конец. Тест: если это поле может быть разным для двух запросов к одному агенту → оно dynamic, должно быть в конце.
Middle Как работает кеширование в multi-turn диалоге?
В multi-turn диалоге: system prompt (stable prefix) + все предыдущие messages (growing history). После каждого turn conversation history растёт. Что кешируется: stable prefix (system prompt, rules) — всегда при hit. Growing history: провайдер может кешировать checkpoint в истории (обычно последний turn где был cache write). Практика: Anthropic кеширует до N breakpoints в conversation. При длинном диалоге кешируется начало истории, новые turns не кешированы. Важно: RAG chunks которые вставляются между turns — если одинаковые → тоже могут кешироваться.
Middle Что такое cache TTL и как он влияет на стратегию использования?
TTL (Time to Live) — время хранения кеша без использования. У Anthropic: ~5 минут. Если пользователь не делал запросов 5+ минут → кеш истёк → следующий запрос будет cache miss (полный пересчёт). Это влияет на UX: если пользователь долго думает перед следующим сообщением → latency выше при первом ответе после паузы. Стратегии: (1) Warm-up запрос — пустой/minimal запрос перед началом сессии для заполнения кеша. (2) Keep-alive pattern — periodic minimal запрос для поддержания кеша активным (только если оправдано cost). В production: учитывай TTL в SLA для latency.
Middle Как связаны кеш и context window?
Кеш не расширяет context window — он только экономит вычисления. Cached tokens всё равно занимают место в context window limit (например, 200K у Claude). Кеш экономит деньги и latency, но не увеличивает 'память' модели. Связь: при очень длинном context window (>100K токенов) — кеш критически важен для экономии, иначе каждый запрос очень дорог. Большие Rules/AGENTS.md (10-50K токенов) имеет смысл держать постоянно в prompt только при высоком cache hit rate — иначе дорого.
Middle Как измерить эффективность кеша в своей системе?
Метрики для measurement: (1) Cache hit rate = cache_read_tokens / (cache_read_tokens + cache_creation_tokens) × 100%. Цель: >80%. (2) Cost savings = (обычная цена prefix) - (actual cost с кешем). (3) TTFT improvement = avg time-to-first-token без кеша vs с кешем. (4) Cache miss patterns — при каких условиях происходят misses? Инструменты: API usage logs с cache statistics, custom dashboards. Anthropic API возвращает cache_creation_input_tokens (при заполнении кеша) и cache_read_input_tokens (при hit) — логируй оба поля для мониторинга.
Middle Какие anti-patterns кеширования нужно избегать?
Anti-patterns: (1) Dynamic data в prefix — timestamp, random ID, per-request data в system prompt. (2) User-specific prefix — если prefix содержит username/email → каждый user имеет свой кеш, hit rate = 1/N users. (3) Частые Rules updates — если Rules обновляются каждый deploy → кеш сбрасывается при каждом deploy. (4) Неупорядоченный контент — если порядок tools/rules меняется между запросами → byte mismatch → miss. (5) Ignore caching в design — не думать о кеше при проектировании prompt структуры → упустить 80% экономии. Rule: review prompt structure для caching при каждом изменении system prompt.
Middle Как использовать кеш для few-shot примеров?
Few-shot examples — отличный кандидат для кеша: они одинаковые для всех запросов. Если поместить examples в system prompt (до conversation history) → они кешируются. При большом числе few-shot примеров (20-50 examples) это может быть 5-20K токенов → значительная экономия. Правило размещения: static instructions → rules → few-shot examples → dynamic context → user message. Если few-shot примеры меняются в зависимости от задачи (dynamic selection) → они dynamic → в конец prompt. Dynamic few-shot = меньше cache savings, но лучшее quality.
Senior — 5 вопросов
Senior Как проектировать prompt архитектуру для максимального cache hit rate в production системе?
Production prompt architecture для кеширования: (1) Layered prompt structure — явные слои от самого стабильного к самому динамичному. Задокументировать слои в ADR. (2) Single source of truth для system prompt — не строить system prompt из разных частей в разных местах кода → нестабильный prefix. (3) Version-controlled prompt — system prompt в git, изменения через PR. При merge → кеш сбросится (запланировано), не внезапно. (4) A/B testing aware — при A/B тестировании prompt вариантов: разные пользователи → разные prefixes → раздельные кеши (снижает общий hit rate). (5) Prompt registry — централизованное место для всех prompts, версионирование, cache hit monitoring. (6) Warming strategy — при deploy новой версии prompt → warm-up запросы для заполнения кеша перед peak traffic.
Senior Как диагностировать деградацию кеша — cache hit rate упал с 85% до 30%?
Диагностика: (1) Проверь логи недавних изменений в system prompt — что изменилось? Даже небольшое изменение ломает весь кеш. (2) Проверь динамические поля — не добавилось ли что-то новое в prefix (timestamp, user context). (3) Посмотри на cache_creation_input_tokens → если вдруг выросли → кеш переписывается чаще. (4) Проверь размер prefix — если prefix вырос, возможно TTL теперь истекает до следующего запроса при low traffic. (5) Деплой? — новая версия модели сбрасывает все кеши, hit rate будет 0% пока не заполнится. (6) Traffic pattern change — если пользователи стали делать паузы >5 минут → больше TTL expiry → больше misses. Инструмент: correlation между deploy events и cache hit rate timeline.
Senior Как использовать кеш в high-throughput системах с тысячами запросов в секунду?
High-throughput caching considerations: (1) Shared cache — одинаковый system prompt для всех запросов даёт глобальный cache hit: первый запрос заполняет кеш, все последующие используют. (2) Concurrent fills — если много запросов пришло одновременно до заполнения кеша → все cache miss. Провайдер обычно coalesces — один fill, остальные ждут или тоже miss. (3) Rate limits — при cache miss full prompt обрабатывается → выше token throughput → может нарушить rate limits провайдера. Планируй capacity с учётом worst-case (0% hit rate). (4) Cost spikes — после deploy (полный miss) → cost spike. Smooth через gradual rollout. (5) Latency SLA — cache miss может 2–5x увеличить latency → иметь circuit breaker и user-facing degraded mode.
Senior Что такое cache warming и как реализовать для production?
Cache warming — заранее заполнить кеш перед началом traffic. Зачем: после deploy или TTL expiry первые запросы будут cache miss с высокой latency и стоимостью. Реализация: (1) Post-deploy hook — после успешного deploy запустить warm-up script: N запросов с production system prompt к провайдеру. Каждый запрос триггерит cache fill. (2) Synthetic requests — dummy user messages которые не влияют на production, только для прогрева кеша. (3) Shadow traffic replay — replay реальных запросов из прошлого для прогрева. (4) Canary-based warming — new version сначала получает 1% traffic, кеш наполняется → потом 100%. Мониторинг: cache hit rate должен достигнуть >70% в течение 5 минут после warm-up.
Senior Как кеш ведёт себя при обновлении модели и как минимизировать disruption?
При обновлении модели (например, claude-3-5-sonnet-20241022 → новая версия) кеш сбрасывается полностью — KV матрицы несовместимы между версиями модели. Disruption: первые N минут после перехода на новую модель → 100% cache miss → рост cost и latency. Стратегии минимизации: (1) Gradual model rollout — сначала 5% запросов на новую модель. Кеш новой модели наполняется при малом traffic. (2) Blue-green deployment — держать обе версии активными, переключать трафик постепенно. (3) Off-peak timing — обновлять модель в ночное время (низкий traffic = меньший impact. (4) Cache warming script — сразу после смены версии. (5) Cost budget — зарезервировать бюджет для post-migration period с низким hit rate.
Lead — 5 вопросов
Lead Как использовать кеш как cost optimization инструмент — что отслеживать и на что влиять?
Cost optimization через кеш: (1) Baseline — измерь текущую стоимость: prompt tokens per request × price. (2) Cache hit rate target — установить SLO: hit rate ≥ 80% для steady state. (3) Savings tracking — dashboard: ежедневная экономия от кеша vs baseline. (4) Prompt discipline — review process для изменений system prompt: любое изменение → cache invalidation → оценить cost impact. (5) Prefix length optimization — длиннее stable prefix = больше кешированных токенов = больше экономии. Поощрять перемещать контент в stable prefix. (6) TTL awareness — если hit rate низкий из-за TTL (sparse traffic) → либо accept, либо periodic warm-up запросы. ROI: при 10K requests/day с 8K-token system prompt → кеш экономит ~90% prefix cost. Числа должны быть видны на финансовом дашборде AI costs.
Lead Как выстроить мониторинг cache efficiency в команде?
Monitoring framework: (1) Metrics — cache_hit_rate, cache_read_tokens, cache_creation_tokens, cost_per_request (cached vs uncached), TTFT percentiles. (2) Dashboard — real-time: hit rate за последний час; trend: hit rate по дням/неделям; cost breakdown: cached vs uncached cost. (3) Alerts — если hit rate падает ниже threshold (например, <60%) → alert команде. (4) Deployment correlation — overlay deploy events на hit rate timeline. (5) Weekly report — cache efficiency summary с recommendations. Инструменты: Grafana + Prometheus, или DataDog, или custom в Metabase. Responsibility: assigned owner for cache monitoring, reviews metrics weekly.
Lead Как ввести кеш-friendly prompt engineering guidelines в стандарты команды?
Guidelines для команды: (1) Prompt Structure Rule — обязательный порядок: [static system] → [rules] → [tools] → [few-shot] → [dynamic context] → [user message]. Проверяется в PR review. (2) No dynamic data in prefix — запрет: timestamps, user IDs, per-request random data в system prompt. Linter rule если возможно. (3) Prompt change impact assessment — при изменении system prompt → оценить cache hit rate impact, добавить в PR description. (4) Документация — каждый prompt файл содержит комментарий: 'Stable prefix ends here — everything below is dynamic'. (5) Cache hit rate как team metric — в sprint review: посмотреть cache efficiency, обсудить деградации. Обучение: провести 30-мин workshop по prompt caching для новых engineers.
Lead Как спланировать кеш-бюджет в AI проекте?
Cache budgeting: (1) Model costs → разбить на: prefix cost (без кеша), suffix cost, output cost. (2) Forecast cache savings — при планируемом hit rate 80% prefix cost снижается в 10x → adjusted budget. (3) Worst-case planning — всегда держать бюджет для периодов 0% hit rate (после deploy, model update, low traffic). Резерв: 20% от total prompt cost. (4) Scale projections — при росте DAU hit rate обычно улучшается (одинаковый prefix у всех пользователей) → учесть в unit economics. (5) Experiment cost — A/B тесты с разными prompts → разные кеши → снижение hit rate на период → учесть. Правило: AI cost budget = (base prompt cost × (1 - hit_rate × 0.9)) + output cost.
Lead Как управлять кешем при rolling deployment новых версий Rules?
Rolling deployment и кеш: при обновлении Rules в .cursor/rules/ или AGENTS.md старый кеш инвалидируется — следующие запросы будут cache miss пока новый кеш не наполнится. Управление: (1) Batch Rules updates — не обновлять Rules часто по одному изменению. Аккумулировать изменения, один deployment. (2) Off-peak releases — деплоить обновления Rules в моменты низкого трафика. (3) Preview impact — перед деплоем посчитать размер стабильного prefix после изменения (больше prefix = дольше warm-up). (4) Cache warm-up post-deploy — скрипт отправки нескольких dummy запросов после деплоя. (5) Feature flags для Rules — если новые Rules нужны только части пользователей → conditional inclusion без инвалидации всего кеша. Policy: major Rules changes → notify team 24h ahead, schedule off-peak deploy.
Architect — 5 вопросов
Arch Как проектировать кеш архитектуру в production AI системе с множественными провайдерами?
Multi-provider cache architecture: (1) Provider abstraction layer — унифицированный интерфейс над разными LLM провайдерами. Cache semantics могут различаться (Anthropic vs OpenAI cache TTL, breakpoints). (2) Prompt normalization — стандартный формат prompt независимо от провайдера. Cache key = hash(normalized_prompt). (3) Provider-side vs application-side — провайдер-side кеш (автоматический) + application-side кеш (explicit, longer TTL). (4) Semantic caching — application-side: если два запроса семантически эквивалентны → возвращать cached response без вызова LLM (осторожно: может давать stale ответы). (5) Multi-tier caching — L1: in-memory (ms, small), L2: Redis (seconds, medium), L3: provider cache (minutes, large). (6) Invalidation strategy — при смене системного контекста → broadcast invalidation по всем уровням кеша.
Arch Как проектировать кеш совместно с CDN паттерном для AI сервисов?
CDN + AI cache pattern: (1) Edge caching — для детерминированных запросов (один и тот же question → один и тот же answer) можно кешировать ответы на edge. TTL: короткий для factual, длинный для static. (2) Cache key design — key = hash(model + system_prompt_version + user_message). Нельзя включать user_id (нарушение privacy). (3) Stale-while-revalidate — отдать cached ответ немедленно, в фоне обновить через LLM. (4) Cache bypass для personalized — запросы с personal context не кешируются на edge. (5) Geographic distribution — разные edge locations могут иметь разные caches (eventual consistency). (6) Privacy considerations — cached AI responses не должны содержать PII. Audit: что кешируется на edge, retention policy.
Arch Как обеспечить кеш isolation между разными tenants в multi-tenant AI системе?
Tenant cache isolation: (1) Namespace isolation — каждый tenant имеет свой cache namespace. Cross-tenant sharing ТОЛЬКО для global static prefix (общий system prompt). (2) Shared vs private prefix — общий system prompt для всех tenants → shared cache (maximum efficiency). Tenant-specific context → tenant-isolated cache. (3) Cache poisoning prevention — tenant A не может влиять на кеш tenant B. Изоляция на уровне cache key (hash включает tenant_id). (4) Cost attribution — track cache savings per tenant для billing. (5) Tenant-specific TTL — enterprise tenants могут иметь longer TTL (более высокая активность = кеш живёт дольше). (6) Audit — каждый cache hit логируется с tenant_id. Compliance: доказать что данные tenant A никогда не попали к tenant B через кеш.
Arch Как учитывать кеш как нефункциональное требование при проектировании AI системы?
NFR для кеширования: (1) Availability — система должна работать при cache backend failure (graceful degradation: cache miss = higher cost/latency, но не отказ). (2) Performance SLO — cache hit: TTFT < 500ms; cache miss: TTFT < 2s. Specify в SLA. (3) Cost SLO — target cache hit rate ≥ 80% для steady state. Exceeding budget при miss-heavy scenarios → alert. (4) Consistency — cached response должен быть валиден: проверять что cached ответы не устарели при Rules update. (5) Security NFR — никаких cross-tenant cache leaks, PII не кешируется. (6) Observability NFR — cache hit rate, latency, cost должны быть measurable и monitored. Architectural decision record (ADR): документировать cache strategy как explicit design decision с rationale и trade-offs.
Arch Как учитывать кеш при rolling deployment в zero-downtime окружении?
Zero-downtime deployment с кешем: (1) Blue-green с cache warm-up — green environment получает traffic только после warm-up (hit rate > threshold). Blue держится как fallback. (2) Canary + cache segregation — canary (1% traffic) использует новую версию prompt → отдельный кеш namespace. При полном rollout → merge или flush + refill основного кеша. (3) Version tagging — cache key включает prompt_version. При rollout: старый трафик → cache v1, новый → cache v2 (постепенный переход). (4) Rollback strategy — если rollback к предыдущей версии → старый кеш мог ещё быть активен (TTL). Ускоренный rollback. (5) Cache invalidation broadcast — при rollback → explicit invalidation нового кеша, возврат к старому. (6) Metric gates — automated rollout gated on cache hit rate metric: если < 60% после N минут → auto pause rollout.
