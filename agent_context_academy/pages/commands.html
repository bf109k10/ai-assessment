Commands

Commands — переиспользуемые Markdown-инструкции вызываемые явно через /command-name. Cursor хранит их в .cursor/commands/, Claude Code — в .claude/commands/. Формат отличается, смысл одинаковый. Отличие от Skills: Commands всегда явные, Skills в Claude Code могут применяться автоматически.

Что такое Commands и зачем они нужны

Commands = явные инструкции по требованию

Markdown-файл с пошаговыми инструкциями. Вызывается через /имя в чате — тогда и только тогда файл загружается в context как дополнительные инструкции агенту.

  • Шаблон PR description
  • Чеклист перед релизом
  • Пошаговый debug workflow
  • Инструкция по рефакторингу
  • Стандарт написания тестов

Формат и структура — Cursor и Claude Code

🖱 Cursor — .cursor/commands/

Расположение: .cursor/commands/*.md
Имя команды = имя файла без расширения.
pr-template.md → /pr-template
Frontmatter необязателен. Plain Markdown.

# .cursor/commands/pr-template.md # PR Template When creating a PR description, use: ## Problem [What issue does this address?] ## Solution [How was it solved?] ## Changes - [Key changes list] ## Tests - [What was tested] ## Risk - [Risks + rollback plan] ## Checklist - [ ] Tests passing - [ ] No secrets exposed - [ ] Docs updated
Вызов: введи /pr-template в чате → Cursor загружает файл в context → агент следует инструкциям.

Файловое дерево

.cursor/ └── commands/ ├── pr-template.md ← /pr-template ├── debug.md ← /debug ├── pre-release.md ← /pre-release └── refactor.md ← /refactor

⚠ Только project-level. Глобальных Commands в Cursor нет.

🤖 Claude Code — .claude/commands/

Расположение: .claude/commands/*.md или ~/.claude/commands/ (global)
Имя команды = поле description в YAML frontmatter.
Frontmatter обязателен. Поддерживает $ARGUMENTS.

# .claude/commands/pr-template.md --- description: "Create a structured PR description" --- # PR Template Create a PR description using this structure. Arguments: $ARGUMENTS (optional: feature/bugfix/hotfix) ## Problem [What issue does this address?] ## Solution [How was it solved?] ## Changes - [Key changes list] ## Tests - [What was tested] ## Risk - [Risks + rollback plan] ## Checklist - [ ] Tests passing - [ ] No secrets exposed - [ ] Docs updated
Вызов: /pr-template или /pr-template bugfix с аргументами. $ARGUMENTS в файле заменяется на введённый текст.

Файловое дерево

.claude/ └── commands/ ├── pr-template.md ← /pr-template ├── debug.md ← /debug └── pre-release.md ← /pre-release ~/.claude/ └── commands/ ← глобальные (все проекты) └── my-style.md

✓ Есть глобальный уровень: ~/.claude/commands/

Lifecycle вызова Command

1

Вызов

Пользователь вводит /command-name в чате (опционально с аргументами).

2

Load

Платформа находит файл в commands/ и загружает его в context текущего чата.

3

Apply

Агент следует инструкциям из файла. Может вызывать MCP tools если они упомянуты.

4

Output

Результат в формате описанном в Command. Структурированно по шаблону.

Ключевое правило: Commands никогда не применяются автоматически — ни в Cursor, ни в Claude Code. Агент не читает их без явного вызова. Если нужно что-то применять автоматически — используй Rules (Cursor) или добавь в AGENTS.md / CLAUDE.md.

Примеры реальных Commands

📝

pr-template

Шаблон описания Pull Request

/pr-template
# PR Template Create PR description: **Problem**: [issue being fixed] **Solution**: [how it was solved] **Tests**: [what was tested] **Risk**: [impact + rollback plan] **Checklist**: tests pass / no secrets / docs updated
✅

pre-release

Чеклист перед выкаткой в production

/pre-release
# Pre-release Checklist Verify before releasing to production: - [ ] All CI checks passing - [ ] Feature flags configured correctly - [ ] DB migrations tested in staging - [ ] Rollback plan documented - [ ] Monitoring alerts configured - [ ] Team notified in #deployments If ANY item fails — DO NOT proceed. Fix first.
♻️

refactor

Безопасный рефакторинг шаг за шагом

/refactor
# Safe Refactor Workflow 1. Write characterisation tests BEFORE any change 2. Make ONE small change at a time 3. Run tests after EACH change — must stay green 4. Do NOT change behaviour, only structure 5. Use IDE rename tools, not manual search/replace 6. Commit each logical step separately
🐛

debug — с $ARGUMENTS (Claude Code)

Дебаггинг с передачей описания проблемы

/debug issue description
--- description: "Systematic debug workflow for a given issue" --- # Debug Workflow Issue to debug: $ARGUMENTS 1. Reproduce: confirm we can reproduce reliably 2. Isolate: binary search — where does it break? 3. Hypothesize: list 3 most likely root causes 4. Test each: minimal reproducible example 5. Fix root cause, not symptom 6. Add regression test

Тонкости использования

Имя команды по-разному задаётся в Cursor и Claude Code

В Cursor имя команды = имя файла без расширения. pr-template.md → /pr-template. Frontmatter не нужен вообще. В Claude Code имя = поле description в YAML frontmatter. Файл без frontmatter в Claude Code работает, но description обязателен для корректной работы. Это частая путаница при переносе Commands между платформами.

$ARGUMENTS — только в Claude Code

Claude Code поддерживает плейсхолдер $ARGUMENTS в тексте Command. Всё что пользователь написал после /command-name подставляется на место $ARGUMENTS. Cursor этот механизм не поддерживает — агент получает весь Command файл и задачу пользователя рядом, и сам интерпретирует.

# Claude Code: /debug login page not loading # → $ARGUMENTS = "login page not loading" Issue to debug: login page not loading 1. Reproduce: confirm we can reproduce...

Command vs Rule — куда что класть

Частая ошибка: писать политику в Command вместо Rule.
Command: "как делать PR" — пошаговая инструкция, вызывается по требованию.
Rule: "не делай PR без тестов" — ограничение, применяется автоматически.
Если правило должно применяться всегда — это Rule. Если только когда пользователь просит — Command.

@mentions внутри Command — включение контекста

Внутри Command файла можно ссылаться на другие файлы через @. Агент прочитает их как часть контекста при выполнении.

# .cursor/commands/refactor.md Before refactoring, read: @docs/architecture.md @docs/adr/ADR-001-no-raw-sql.md Then apply these steps...

Commands не применяются автоматически — ни в одной платформе

В отличие от Rules (которые могут быть alwaysApply) и от Claude Code Skills (которые агент может выбрать по description) — Commands никогда не активируются без явного /command-name от пользователя. Это осознанный дизайн: Commands для действий по требованию, не фоновая политика.

Глобальный уровень только в Claude Code

В Cursor Commands существуют только на project-level (.cursor/commands/). Хочешь одинаковые Commands во всех проектах — придётся копировать руками или использовать template репо. В Claude Code есть ~/.claude/commands/ — глобальный уровень, работает во всех проектах без копирования.

Best practices

Делай

  • Короткие имена файлов: pr.md, debug.md, release.md
  • Начни с самых частых задач команды: PR template, pre-release checklist
  • Структурированный output — чеклисты, секции, формат
  • Документируй Commands в AGENTS.md: "Available: /pr, /debug, /release"
  • Используй $ARGUMENTS в Claude Code для гибкости
  • @mention нужных файлов внутри Command

Не делай

  • Политику в Command — "не делай X" → это Rule
  • Слишком общие имена: help.md — непонятно когда вызывать
  • Большие Commands 500+ строк — агент теряет нить
  • Хранить данные — Commands это инструкции, не Memory
  • Дублировать содержимое AGENTS.md — там уже есть workflow

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Command в Cursor?
Command (Slash Command) — файл .cursor/commands/*.md с инструкцией для агента. Вызывается явно через /имя-команды в chat. При вызове содержимое .md файла передаётся агенту как инструкция. Cursor предлагает autocomplete при вводе / — показывает список доступных Commands с описаниями.
Junior Как вызвать Command через /slash синтаксис?
В поле чата Cursor набери / — появится список доступных Commands. Выбери нужный или введи имя полностью: /code-review. Cursor загрузит содержимое соответствующего .md файла в контекст агента. Агент выполнит описанный в файле процесс. Без / — Command не активируется.
Junior Где хранятся Commands в Cursor?
Cursor: .cursor/commands/*.md (project-level, версионируются в git). Claude Code: .claude/commands/*.md (project) или ~/.claude/commands/*.md (global, личные). Project Commands доступны всей команде. Global Commands (~/.claude/) доступны во всех проектах для конкретного пользователя.
Junior Что такое .md файл для Command?
Обычный Markdown файл с инструкцией для агента. В Cursor: просто Markdown без frontmatter — содержимое передаётся агенту как prompt. В Claude Code: может содержать YAML frontmatter (description, allowed-tools) и переменную $ARGUMENTS для подстановки аргументов. Имя файла = имя команды (без .md).
Junior Чем Command отличается от Skill?
Command: вызывается только явно через /имя пользователем. Skill (Claude Code): может применяться автоматически агентом по description. Command = пользователь инициирует выполнение. Skill = агент сам решает применить. Для задач требующих явного намерения — Command. Для автоматически срабатывающих процессов — Skill.
Junior Чем Command отличается от Rule?
Rule — постоянная политика, применяется автоматически при каждом запросе (alwaysApply) или при работе с определёнными файлами (globs). Command — шаблон задачи, вызывается явно по запросу. Rule говорит «как вести себя всегда». Command говорит «вот инструкция для конкретного типа задачи». Для постоянных стандартов — Rule. Для повторяемых задач — Command.
Junior Как создать новый Command в Cursor?
1. Создай файл в директории .cursor/commands/. 2. Имя файла = имя команды: code-review.md → /code-review. 3. Напиши Markdown инструкцию: описание, шаги, expected output. 4. Command появится в autocomplete при вводе /. 5. Версионируй в git — команда получит Command при клонировании.
Junior Что такое slash trigger?
Slash trigger — ввод /имя-команды в chat для активации Command. Cursor и Claude Code используют / как prefix для Commands. Без / — агент воспринимает текст как обычный запрос, Command не активируется. В Claude Code: $ARGUMENTS в Command файле захватывает текст после имени команды: /fix-bug NullPointerException → $ARGUMENTS = "NullPointerException".
Junior Какие встроенные Commands есть в Cursor?
Cursor имеет встроенные команды: /edit (редактировать выделенный код), /fix (исправить ошибки), /explain (объяснить код), /generate (сгенерировать код). Пользовательские Commands из .cursor/commands/ добавляются к встроенным. При одинаковом имени: пользовательский Command имеет приоритет.
Junior Как передать аргументы в Command (в Claude Code)?
В Claude Code: использовать переменную $ARGUMENTS в .md файле. Пример Command: Fix the following bug: $ARGUMENTS. Вызов: /fix-bug NullPointerException in UserService. Результат: $ARGUMENTS заменяется на «NullPointerException in UserService». В Cursor: формального $ARGUMENTS нет — агент видит текст после /имя-команды в context.
Middle — 10 вопросов
Middle Когда использовать Command, а не просто написать задачу в chat?
Command оправдан когда: (1) задача повторяется (каждый PR проходит review по одинаковому процессу); (2) процесс многошаговый — легко что-то забыть без структуры; (3) нужна консистентность — все в команде делают одинаково; (4) согласован командой — хочешь зафиксировать «правильный» способ. Разовая уникальная задача → просто chat. Повторяемый структурированный процесс → Command.
Middle Как писать эффективный Command для code generation?
Структура Command для code gen: 1. Чёткий шаблон: «Сгенерируй [тип компонента] с следующими требованиями: $ARGUMENTS». 2. Стандарты: «Используй наш style guide, добавь unit тесты, следуй паттернам из existing code». 3. Output format: «Верни: файл компонента + тест файл + описание что создано». 4. Пример в Command: показывает структуру ожидаемого output. Конкретность = предсказуемость генерации.
Middle Как Commands помогают в написании документации?
Документация через Commands: /document-function — принимает функцию, генерирует docstring/JSDoc; /update-readme — обновляет README по изменениям в API; /generate-changelog — создаёт changelog из git commits. Преимущества: единый стиль документации, агент следует шаблону из Command, не нужно помнить что включать — Command содержит checklist.
Middle Как настроить Commands для review workflow?
PR review Command: /pr-review содержит: 1. Correctness check (логика, edge cases); 2. Security review (input validation, auth); 3. Test coverage (что не покрыто); 4. Performance (N+1, unnecessary loops); 5. Code style (соответствие team standards); 6. Output: summary + список issues по severity. Дополнительно: /security-review, /performance-review как отдельные Commands.
Middle Как организовать Commands в командном репозитории?
Организация: 1. .cursor/commands/ — все team Commands в git; 2. Именование: area-action.md (db-migrate.md, api-document.md); 3. README в .cursor/commands/: список Commands с кратким описанием; 4. Личные Commands → ~/.claude/commands/, не в репо; 5. AGENTS.md: секция «доступные Commands и когда использовать».
Middle Как Commands взаимодействуют с текущим context (открытые файлы, выделенный код)?
При вызове Command агент видит: (1) содержимое Command .md файла; (2) открытые в редакторе файлы (Cursor передаёт их в context); (3) выделенный код (если есть); (4) текст пользователя после /command-name. Command может ссылаться на context: «Примени к выделенному коду». Это позволяет Commands быть контекстно-зависимыми без явного указания файлов.
Middle Как Commands обеспечивают консистентность повторяемых задач?
Консистентность через Commands: каждый developer вызывает /code-review → агент всегда выполняет одни и те же шаги → review всегда включает security, tests, style. Без Command: каждый помнит разный набор проверок, junior пропускает security check, senior забывает документацию. Command = «команда договорилась как делать и зафиксировала».
Middle Как тестировать Commands перед выкаткой команде?
(1) Создай Command на feature branch. (2) Вызови с типичными входными данными — выполняет ли все шаги? (3) Тестируй edge cases: пустые аргументы, граничные случаи. (4) Проверь Output: соответствует ли ожидаемому формату? (5) Спроси коллегу попробовать — «интуитивно ли поведение?». (6) Для Claude Code: тест с реальным $ARGUMENTS. (7) PR с примером output — reviewer может оценить качество.
Middle Как версионировать Commands при изменении процессов?
Commands — код, не документация. При изменении процесса деплоя → обновление /deploy Command входит в migration checklist. Практики: (1) дата последнего обновления в Command файле; (2) CHANGELOG секция в README commands/; (3) тег deprecated в имени при устаревании (deploy-old.md → deploy.md); (4) CI check: не обновлявшиеся 6+ месяцев Commands — alert.
Middle Когда использовать Command vs Skill — ключевые trade-offs?
Command: ✓ явное намерение пользователя, ✓ Cursor поддерживает нативно, ✗ нет автозапуска, пользователь должен помнить имя. Skill (Claude Code): ✓ автоматический trigger по description, ✓ пользователь не помнит имя — агент выбирает сам, ✗ только Claude Code (не Cursor). Практика: Commands для Cursor-проектов; Skills для Claude Code workflows где нужен auto-trigger.
Senior — 5 вопросов
Senior Как спроектировать библиотеку Commands для организации?
Архитектура Commands library: (1) Standard Commands — универсальные для всех проектов: code-review, security-audit, document-api; в shared template репо, копируются при создании нового проекта; (2) Domain Commands — специфичные для домена; (3) Naming convention: domain-action; (4) Commands catalog: внутренний каталог всех Commands с примерами; (5) Version management: major.minor для Commands которые разные команды синхронизируют.
Senior Как использовать Commands для complex automation?
Многошаговая автоматизация через Commands: (1) Pipeline Command: /full-release — run tests → build → check security → create changelog → tag → notify; (2) Conditional logic в Command: «If $ARGUMENTS contains 'hotfix' — skip integration tests»; (3) Tool orchestration: Command перечисляет tools agenta (allowed-tools: [Bash, Write, WebFetch]); (4) Checkpoints: «После шага 3 выведи summary и жди подтверждения». Сложный Command = маленькая программа для агента.
Senior Как диагностировать ситуации когда Command выполняется некорректно?
Типичные проблемы: (1) Агент выполнил только часть шагов → lost in the middle: длинный Command, средние шаги пропускаются; решение: нумерация + verification checklist в конце; (2) Output не соответствует ожидаемому → Output раздел Command неточный; добавь пример; (3) Command применяется не к тому файлу → Command должен явно указывать «к чему применять»; (4) $ARGUMENTS не подставляется → проверь Claude Code settings, не Cursor.
Senior Как строить Commands composition — цепочка Commands?
Прямого вызова Command из Command нет. Паттерны: (1) Orchestrator Command: /full-audit описывает применить сначала /security-check, затем /performance-check, затем /doc-check; (2) Shared content: выносить общие шаги в отдельный .md, ссылаться через @path/shared-steps.md; (3) Parameter passing: результат Command A передаётся как $ARGUMENTS в Command B; (4) DRY principle для Commands: не дублировать шаги.
Senior Чем Commands отличаются от MCP tools и когда что использовать?
MCP tools: код написанный разработчиком, выполняет конкретную техническую операцию (query DB, call API, search files). Commands: Markdown-инструкция для агента как выполнить workflow используя уже доступные tools. MCP: низкоуровневый building block — atomарное действие. Command: высокоуровневый workflow — последовательность действий. Комбинация: Command описывает процесс, MCP tools — его реализация.
Lead — 5 вопросов
Lead Как выстроить governance Commands в команде?
Governance: (1) Owner per Command — кто отвечает за актуальность; (2) PR review для Commands — новые Commands через PR с тестом output; (3) Commands catalog в AGENTS.md — список доступных Commands с примерами; (4) Deprecation process: устаревшие Commands помечаются, не удаляются без уведомления; (5) Стандарты именования: domain-action.md — команда понимает назначение без чтения файла.
Lead Как Commands становятся инструментом командной продуктивности?
Commands как продуктивность: (1) стандартизируют «правильный способ» выполнения задач; (2) снижают cognitive load: не нужно помнить 15-шаговый review процесс — просто /pr-review; (3) накапливают team knowledge: лучшие практики закодированы в Commands; (4) измерение value: сравни время review до/после команды Commands; (5) Commands как competitive advantage: команда с хорошей Commands library быстрее выполняет стандартные задачи.
Lead Как использовать Commands в onboarding процессе?
Onboarding через Commands: (1) /onboarding-checklist: первые задачи нового разработчика; (2) /setup-dev-env: пошаговая настройка окружения; (3) первая неделя: senior показывает какие Commands есть и когда использовать; (4) в AGENTS.md: «для code review используй /pr-review, для документации /document»; (5) onboarding checklist включает: «попробуй /pr-review на любом PR», «создай свой первый Command».
Lead Как поддерживать актуальность Commands со временем?
Commands rot prevention: (1) Change triggers: при изменении процесса деплоя, API, архитектуры — обновление Commands входит в Definition of Done; (2) Review cadence: раз в квартал проверять все Commands; (3) Last-modified tracking: дата в файле или git blame; (4) Stale detection: CI алерт для Commands без изменений 6+ месяцев; (5) Team feedback: разработчик обнаружил что Command не работает → создаёт issue (не просто игнорирует).
Lead Как выработать стандарты написания Commands для команды?
Commands style guide: (1) Структура: описание (1 строка) → Prerequisites → Steps → Output; (2) Именование: domain-action, максимум 3 слова через дефис; (3) Шаги: явная нумерация, каждый шаг = одно действие; (4) Output: явно описать что агент вернёт; (5) Специфичность: Commands должны быть конкретными — «создай backup: pg_dump -h HOST > /backups/$(date).sql», а не «создай backup».
Architect — 5 вопросов
Arch Как Commands реализуют programmatic interface к AI?
Commands = публичный API агента для команды. Аналогия: Commands как REST endpoints — /deploy = POST /deploy, /pr-review = POST /review. Интерфейс: имя команды = endpoint, $ARGUMENTS = request body, Output = response. Стабильный интерфейс: разработчики знают какие Commands доступны и что ожидать от каждого. Версионирование: /deploy-v2 при breaking change.
Arch Как Commands вписываются в developer workflow automation?
Commands в workflow: (1) Development cycle: /generate-component → разработка → /pr-review → /document → merge; (2) IDE integration: Commands вызываются прямо из Cursor без переключения контекста; (3) Standardized workflows: команда следует одинаковым процессам независимо от опыта разработчика; (4) Automation hooks: git hooks → /release-notes автоматически. Commands = programmatic workflow layer над AI.
Arch Как Commands и Hooks образуют automated workflow?
Commands + Hooks = полный automation pipeline: Command описывает что делать (агент выполняет шаги), Hook обеспечивает безопасность каждого шага (блокирует опасные tool calls). Пример: /deploy Command запускает deployment process → PreToolUse Hook проверяет каждый Bash call на prod-safety → PostToolUse Hook логирует каждое действие в audit trail. Разделение обязанностей: Command = workflow logic, Hook = safety guarantees.
Arch Как интегрировать Commands в CI/CD pipeline?
CI/CD интеграция Commands: (1) CI lint: проверка всех .md файлов в .cursor/commands/ — наличие обязательных секций, YAML frontmatter (Claude Code); (2) Integration tests: sandbox запуск агента с каждым Command + тестовым input, проверка output; (3) Breaking change detection: diff Command Steps → уведомление команды; (4) Distribution: pipeline синхронизирует shared Commands в новые репозитории; (5) Documentation generation: из Commands автоматически генерируется Commands Reference в wiki.
Arch Как Commands становятся productivity infrastructure организации?
Commands как infrastructure: (1) Institutional knowledge: лучшие практики закодированы в Commands, не теряются при уходе людей; (2) Scale: 100 команд используют одинаковые Core Commands — обновление одного файла улучшает продуктивность всей организации; (3) Standardization: Commands делают «правильный путь» самым лёгким путём; (4) Measurement: через PostToolUse Hooks измеряй ROI Commands (сколько раз вызван, время выполнения, качество output); (5) Platform product: Commands library = внутренний продукт, Platform team = его владелец.
