Формула для assessment

Сначала приоритет, потом конкретность, потом свежесть. System выше всех. Rules/AGENTS.md задают рамки. User request задаёт задачу. MCP/tool-result часто даёт самые свежие факты. Skill — способ выполнения, но не право нарушать правила.

Учебник по ключевым темам

Эта часть нужна для подготовки: не просто знать термины, а уметь объяснить, когда что использовать, чем отличается одно от другого и какие есть ограничения.

RAG / Retrieval-Augmented Generation

Поиск релевантных знаний и добавление их в контекст
Для чего это: чтобы агент отвечал по актуальным фактам из документов/кода/wiki, а не по памяти модели.

Простыми словами

RAG — это когда модель не пытается всё помнить сама, а сначала ищет нужные фрагменты в документах, коде, wiki, базе знаний или тикетах, а потом использует найденное для ответа.

Когда использовать
  • Нужно ответить по документации.
  • Нужно найти API/класс/метод в коде.
  • Нужно подтянуть knowledge base.
  • Нужно уменьшить hallucination.

Важно для assessment

RAG отвечает за факты, но не за правила поведения. Если RAG нашёл старую инструкцию, а Rules говорят другое — Rules выше. Если нужно текущее состояние системы, лучше проверить через MCP/tools.

🧠

Memory Bank

Долговременная память проекта, пользователя или workflow
Для чего это: чтобы агент помнил принятые решения и не переспрашивал одно и то же между сессиями.

Простыми словами

Memory Bank — это сохранённые устойчивые знания: какие решения уже приняли, какие правила у проекта, какие предпочтения у пользователя, какие команды работают, какие подходы не подходят.

Когда использовать
  • Нужна continuity между сессиями.
  • Есть долгий проект.
  • Нужно помнить решения.
  • Нужно не спрашивать одно и то же.

Memory Bank vs RAG

Memory Bank — это curated память о важных вещах. RAG — поиск по источникам. Memory помогает помнить “что мы решили”, RAG помогает найти “что написано в документации”.

🛠

MCP / Tools

Доступ к внешним системам, данным и действиям
Для чего это: чтобы агент мог не только рассуждать, но и проверять/менять реальное состояние систем.

Простыми словами

MCP и tools дают агенту возможность не только рассуждать, но и действовать: открыть файл, сходить в API, прочитать Jira, проверить CI, посмотреть логи, выполнить поиск.

Когда использовать
  • Нужно проверить реальное состояние.
  • Нужно прочитать файл/лог/тикет.
  • Нужно выполнить действие.
  • Нужно получить свежие данные.

Важно

MCP не делает агента “умнее” сам по себе. Он даёт доступ к данным и действиям. Качество зависит от того, какие tools доступны, какие права есть и как агент интерпретирует результат.

✅

Checkpoint

Структурный снимок состояния задачи
Для чего это: чтобы не потерять цель, решения и next steps при длинной задаче или перед сжатием контекста.

Простыми словами

Checkpoint — это не просто summary, а структурированная точка сохранения: цель, что уже сделано, какие решения приняты, какие есть риски и что делать дальше.

Когда использовать
  • Длинная задача.
  • Переход между сессиями.
  • Нужно передать работу другому агенту.
  • Перед context compression.
Что включать
  • Цель.
  • Done / not done.
  • Decisions.
  • Risks / blockers.
  • Next steps.

Checkpoint vs Summary

Summary просто кратко пересказывает. Checkpoint сохраняет состояние работы так, чтобы можно было продолжить без потери логики.

🧾

Summary / Compression

Сжатие истории, когда контекст переполняется
Для чего это: чтобы уместить длинную историю чата в контекстное окно без потери сути.

Простыми словами

Compression — это уменьшение объёма контекста. Summary — один из способов compression: длинный диалог превращается в краткое содержание.

Когда использовать
  • Длинный чат.
  • Много деталей и файлов.
  • Нужно продолжить позже.
  • Окно контекста ограничено.
Риски
  • Теряются нюансы.
  • Может исказить решение.
  • Может забыть edge case.
  • Не заменяет исходные файлы.

Лучший формат

Для рабочих задач лучше использовать structured checkpoint: цель, контекст, сделано, решения, команды, риски, next steps.

🎓

Skills

Инструкции “как выполнять” типовые задачи
Для чего это: чтобы агент выполнял повторяемую задачу по проверенному сценарию, а не изобретал заново.

Простыми словами

Skill — это специализированная инструкция или playbook. Например: как создавать PDF, как делать code review, как анализировать лог, как оформлять отчёт.

Когда использовать
  • Повторяемая задача.
  • Нужен шаблон выполнения.
  • Нужно снизить хаос.
  • Нужны best practices.

Skill vs Rule

Rule говорит “что можно/нельзя”. Skill говорит “как делать”. Если Skill противоречит Rule, Skill нужно адаптировать.

📄

AGENTS.md

Проектная инструкция для агента
Для чего это: чтобы агент знал правила и workflow конкретного репозитория без повторных объяснений.

Простыми словами

AGENTS.md — это файл, который объясняет агенту, как работать в конкретном репозитории: роль, структура проекта, команды, тесты, ограничения, workflow.

Что класть
  • Команды сборки/тестов.
  • Архитектурные правила.
  • Definition of Done.
  • Как оформлять изменения.
Что не класть
  • Секреты и токены.
  • Большие логи.
  • Огромные docs.
  • Часто меняющиеся данные.

AGENTS.md vs Rules

AGENTS.md — общий проектный workflow. Rules — жёсткие ограничения или стандарты. Они могут пересекаться, но лучше архитектурно разделять.

👥

Subagents

Делегирование подзадач специализированным агентам
Для чего это: чтобы разгрузить контекст главного агента, делегировав часть работы специализированному агенту.

Простыми словами

Subagent — отдельный агент или режим, которому главный агент передаёт часть задачи: найти причину, проверить гипотезу, собрать факты, сделать review.

Когда полезны
  • Сложная задача.
  • Параллельный анализ.
  • Нужна специализация.
  • Нужно разделить поиск и решение.

Главный принцип

Subagent не заменяет ответственность главного агента. Он возвращает findings, а главный агент объединяет, проверяет конфликт с Rules/AGENTS.md и принимает финальное решение.

Типовые вопросы для assessment и хорошие ответы

Чем RAG отличается от Memory Bank?

RAG ищет релевантные фрагменты во внешних источниках. Memory Bank хранит устойчивый контекст и прошлые решения. RAG — “найти факт”, Memory — “помнить, что уже решили”.

Чем Skill отличается от Rule?

Skill описывает, как выполнять задачу. Rule задаёт ограничения: что можно, нельзя и какой стандарт соблюдать. При конфликте Rule выше Skill.

Что такое Checkpoint?

Checkpoint — структурный снимок состояния: цель, контекст, что сделано, решения, риски, next steps. Он лучше обычного summary для продолжения работы.

Может ли subagent использовать MCP?

Зависит от платформы и permissions. Иногда у subagent те же tools, иногда ограниченные, иногда tools доступны только главному агенту.

Что делать, если RAG противоречит Rule?

Rule выше, потому что RAG даёт факты, а Rule задаёт ограничения поведения. Нужно проверить, не устарел ли найденный документ.

Что делать, если Memory противоречит файлу?

Для актуального факта файл или tool-result обычно надёжнее. Memory могла устареть. Но нужно проверить, что файл действительно актуален.

Зачем нужен AGENTS.md?

Чтобы агент понимал проектный workflow: как запускать тесты, что нельзя менять, как оформлять изменения, какие команды и стандарты использовать.

Почему одна модель работает по-разному?

Потому что различаются system prompt, rules, context, tools, MCP, RAG, memory, temperature, permissions, IDE implementation и качество retrieved data.

Что важнее: текущий чат или Rules?

Текущий чат задаёт задачу, но Rules задают рамки. Если пользователь просит нарушить Rule, агент должен объяснить ограничение и предложить безопасный вариант.

Типовой ответ: “Я разделяю слои: правила поведения — в AGENTS.md/Rules, знания — в Memory/RAG, действия — через MCP/tools, повторяемые процессы — в Skills, передачу работы — через Subagents.”
Главный риск: смешать всё в один слой. Тогда агенту сложнее понять, что является правилом, что фактом, что памятью, а что просто способом выполнения.
Для подготовки: умей объяснить разницу не по определению, а по кейсам: “где хранить?”, “что выше?”, “что делать при конфликте?”, “когда проверять через tool?”.
