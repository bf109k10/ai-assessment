Построение агентских систем

Как из отдельных subagents собрать рабочую систему: декомпозиция задачи, выбор параллельной или последовательной схемы, дерево зависимостей, распределение файлов, чек-лист проектирования и готовые шаблоны task description.

На главную← Subagents

Что такое агентская система и зачем её строить

Один subagent — это исполнитель одной подзадачи. Агентская система — это несколько агентов, объединённых оркестратором в единый рабочий процесс: кто-то ищет, кто-то пишет, кто-то проверяет. Задача построения — правильно разложить работу и связать участников.

Агентская система = команда + прораб

Оркестратор (main agent) — как прораб на стройке: он не кладёт кирпичи сам, а раздаёт задачи специалистам и собирает результат.

  • Orchestrator — декомпозирует задачу, раздаёт, агрегирует, принимает финальное решение.
  • Workers (subagents) — каждый со своей ролью и изолированным контекстом.
  • Протокол возврата — стандартный формат findings, чтобы результаты можно было собрать.

Зачем строить систему, а не один агент

  • Масштаб — задача не влезает в один context window.
  • Параллелизм — независимые части идут одновременно → быстрее.
  • Изоляция шума — воркер читает 40 файлов, main получает summary.
  • Специализация — security / performance / tests как отдельные роли.

Когда система НЕ нужна

  • Задача на 1–2 файла — overhead координации больше выигрыша.
  • Шаги сильно зависят друг от друга (нет чего параллелить).
  • Контекст критичен и его тяжело передать воркеру.
  • Один агент справится за <10 turns.

Главный принцип построения

Система оправдана тогда, когда выигрыш от параллелизма или изоляции контекста перевешивает overhead координации (написание task descriptions + verification + агрегация). Начинай с одного агента; выделяй воркеров только там, где видна явная независимая часть.

Шаг 1. Декомпозиция задачи

Построение начинается не с агентов, а с разбиения задачи на атомарные подзадачи и определения зависимостей между ними. Это фундамент — от него зависит, что можно параллелить.

Задача: "Добавить rate limiting в публичное API" ── Декомпозиция на атомарные подзадачи ────────────── T1 Исследовать текущую структуру middleware # explore T2 Найти где хранить счётчики (Redis? in-memory?) # explore T3 Реализовать middleware rate limiter # implement T4 Написать тесты для лимитера # implement T5 Проверить безопасность (bypass, header spoofing) # review T6 Обновить документацию API # implement ── Дерево зависимостей (DAG) ──────────────────────── T1 ─┐ ├─→ T3 ─┬─→ T4 T2 ─┘ ├─→ T5 └─→ T6 T1, T2 — независимы → параллельно T3 — ждёт T1+T2 (нужны findings) T4, T5, T6 — независимы после T3 → параллельно

Шаг 2. Параллельно или последовательно

Из дерева зависимостей напрямую следует топология системы. Правило простое: независимые подзадачи → параллельно, зависимые → pipeline. На практике почти всегда получается гибрид: параллельные «волны» между точками синхронизации.

Параллельно (fan-out)

Несколько воркеров запускаются одновременно, main агрегирует.

Orchestrator ├─→ Worker A ─┐ ├─→ Worker B ─┤ одновременно └─→ Worker C ─┘ ↓ Агрегация findings

Условия: подзадачи независимы, разные наборы файлов, примерно одного размера.

Плюс: скорость. Минус: cost × N.

Последовательно (pipeline)

Output одного воркера = input следующего.

Planner ──→ spec ↓ Researcher ──→ facts ↓ Coder ──→ code ↓ Reviewer ──→ report

Условия: каждый шаг зависит от предыдущего.

Плюс: дешевле, предсказуемо. Минус: нет ускорения.

Гибрид — параллельные волны (реальный случай)

── Волна 1 (parallel) ── T1 explore ║ T2 explore ↓ точка синхронизации: main собрал findings ── Волна 2 (single) ── T3 implement (нужны T1+T2) ↓ точка синхронизации ── Волна 3 (parallel) ── T4 tests ║ T5 security ║ T6 docs ↓ Финальная агрегация + verification

Каждый шаг использует результат предыдущегоPipeline

Важно качество одного результата, задача неоднозначнаBest-of-N (см. паттерны на Subagents)
Cost критиченСначала один агент; параллель только при явном выигрыше

Параллелизм ускоряет не всегда

Если один воркер заканчивает за 30 сек, а второй за 5 минут — параллель даёт выигрыш лишь до размера самого долгого. А если main потом тратит больше времени на reconciliation противоречивых findings, чем воркеры на работу — параллель навредила. Балансируй размеры подзадач.

Шаг 3. Распределение файлов между воркерами

Ключевое правило параллельной системы: параллельные воркеры работают на непересекающихся наборах файлов. Иначе — гонка записи и потеря изменений.

✅ Непересекающиеся наборы

Worker A: api/auth/** Worker B: api/billing/** Worker C: frontend/** # Пересечений нет → безопасно параллелить

❌ Пересечение файлов

Worker A: api/auth/service.py Worker B: api/auth/service.py ← тот же файл! # last-write-wins: B перезапишет правки A

Last-write-wins на shared файле

Если два параллельных воркера редактируют один файл — тот, кто запишет последним, затрёт изменения первого. Никакого автоматического merge нет. Решение: (1) отдай shared файл одному воркеру; (2) или обрабатывай его последовательно через main; (3) или дай воркерам readonly, а правки собери в main.

Read-write разделение снижает конфликты

Частый безопасный паттерн: параллельные воркеры только читают (readonly: true) и возвращают findings/патчи как текст, а физические правки применяет один main agent последовательно. Так параллелизм ускоряет исследование без риска гонок записи.

Шаг 4. Чек-лист проектирования системы

Проверенный порядок построения — от разбиения задачи до проверки результата. Пройди по пунктам перед запуском воркеров.

Чек-лист построения агентской системы 1. Декомпозиция ▢ Разбить задачу на атомарные подзадачи ▢ Построить дерево зависимостей (что от чего зависит) 2. Топология ▢ Независимые → параллельно, зависимые → pipeline ▢ Определить точки синхронизации (волны) 3. Роли ▢ Назначить роль каждому воркеру (explorer / coder / reviewer) ▢ Explorer и reviewer → readonly 4. Файлы и границы ▢ Непересекающиеся наборы файлов для параллельных воркеров ▢ Files + Scope + Constraints для каждого (см. Subagents) 5. Протокол возврата ▢ Единый output format findings (structured, с evidence: file:line) 6. Агрегация и verification ▢ Main собирает findings, разрешает противоречия ▢ Main проверяет findings против Rules / AGENTS.md перед применением 7. Cost-бюджет ▢ Оценить токены на воркера; не запускать при явном перерасходе ▢ Сначала один агент, если задача мала

Роли-специализации

  • Explorer — ищет, анализирует, readonly, возвращает findings.
  • Implementer — пишет код по spec + findings, узкий scope.
  • Reviewer — проверяет реализацию против требований и Rules, readonly.
  • Security / Perf — специализированные reviewer'ы.

Протокол возврата findings

  • Structured: Summary → Findings → Evidence → Next steps.
  • Конкретные ссылки: file:line, не «нашёл проблему».
  • Confidence level: высокая / требует проверки.
  • Контроль размера: ключевое в summary, детали в appendix.

Шаг 5. Готовые шаблоны task description

Копипастные briefing-шаблоны для трёх базовых ролей системы researcher → implementer → reviewer. Подставь Files / Scope под свою задачу.

① Explorer (readonly)

Role: Code Explorer Goal: Найти где реализован rate limiting и как устроен middleware Files: api/middleware/**, api/config/*.py Scope: только middleware и точки его подключения; не бизнес-логика Constraints: readonly; не редактировать; не запускать команды Output: Summary + список {file:line, роль компонента} + точки расширения

② Implementer

Role: Backend Implementer Goal: Реализовать middleware rate limiter (100 req/min per IP) Context: findings Explorer'а (передать сюда), Python 3.11, Redis доступен Files: api/middleware/ratelimit.py (создать), api/app.py (подключить) Constraints: не менять публичный API-контракт; не трогать тесты чужих модулей Output: изменённые файлы + краткое описание решения

③ Reviewer (readonly)

Role: Security Reviewer Goal: Проверить лимитер на bypass: header spoofing, IPv6, прокси X-Forwarded-For Files: api/middleware/ratelimit.py Constraints: readonly; только анализ; не предлагать рефакторинг вне найденного Output: JSON [{issue, file:line, severity, fix_hint}], max 10
Оркестрация этой тройки: Explorer (readonly) → main передаёт findings в Implementer → Reviewer (readonly) проверяет результат. Explorer и Reviewer можно запускать параллельно с другими независимыми воркерами; Implementer — между ними, в точке синхронизации.

Шаг 6. Как строить саму оркестрацию

Оркестрация — это логика главного агента: цикл «декомпозировать → запустить волну → дождаться → собрать → проверить → следующая волна». Ниже — как этот цикл выглядит на практике и как его закодировать в briefing оркестратора.

Цикл оркестратора — шаг за шагом

ORCHESTRATION LOOP (логика main agent) 1. PLAN Декомпозировать задачу → список подзадач + DAG зависимостей # зафиксировать в TodoWrite / плане ↓ 2. SELECT Взять готовую «волну»: подзадачи без невыполненных зависимостей ↓ 3. DISPATCH Запустить воркеров волны: • независимые → параллельно (батч Task calls, background) • зависимые → по одному каждому — Role + Files + Scope + Constraints + Output ↓ 4. AWAIT Точка синхронизации: дождаться ВСЕХ воркеров волны # не завершать turn, пока есть незакрытые background воркеры ↓ 5. COLLECT Прочитать findings, привести к общему виду ↓ 6. VERIFY Проверить findings против Rules / AGENTS.md, разрешить противоречия, отбросить нарушающее правила ↓ 7. DECIDE Применить результат / запустить tie-breaker / скорректировать план ↓ есть ещё волны? ──да──► к шагу 2 │ нет ▼ 8. FINALIZE Собрать итог, применить правки одним исполнителем, отчитаться

Что такое «волна» (wave)?

Волна — это группа подзадач, которые можно запустить одновременно, потому что все их зависимости уже выполнены. Оркестратор идёт по дереву зависимостей слоями: сначала запускает все задачи без зависимостей (волна 1), ждёт их завершения (точка синхронизации), затем запускает те, что зависели от первой волны (волна 2), и так далее.

Проще говоря: волна = один «залп» параллельных воркеров. Внутри волны воркеры работают параллельно и независимо; между волнами — обязательная пауза, где оркестратор собирает результаты и решает, что запускать дальше.

Задача: добавить rate limiting Волна 1 [T1 изучить middleware] ║ [T2 выбрать хранилище] # параллельно ▼ точка синхронизации — main собрал findings обеих задач Волна 2 [T3 реализовать лимитер] # одна задача, ждала T1+T2 ▼ точка синхронизации Волна 3 [T4 тесты] ║ [T5 security] ║ [T6 документация] # снова параллельно ▼ Финальная агрегация + проверка на Rules

Почему «волны», а не «всё сразу»: T3 нельзя запустить пока не готовы T1 и T2 — иначе воркер получит пустой вход. Волны = способ соблюсти порядок зависимостей и при этом максимально распараллелить независимое.

Оркестратор сам НЕ делает работу воркеров

Соблазн — «заодно сам поправлю пару файлов». Тогда контекст main раздувается ровно тем шумом, ради изоляции которого и строилась система. Роль оркестратора: планировать, диспетчеризовать, агрегировать, верифицировать и принимать решение. Исполнение — воркерам.

Синхронизация: как ждать воркеров и какие есть варианты

Точка синхронизации — это «барьер»: оркестратор останавливается и дожидается воркеров волны, прежде чем идти дальше. Внутри волны воркеры бегут параллельно, но волну 2 нельзя начать, пока волна 1 не отдала findings.

Волна 1: worker A ──┐ worker B ──┤ бегут параллельно worker C ──┘ ▼ [БАРЬЕР: ждём всех, собираем findings] ▼ Волна 2: worker D (получает findings волны 1)

Главное правило синхронизации

Оркестратор не должен завершать turn, пока есть незакрытые background-воркеры. Иначе их результат теряется, и агрегация будет неполной. Явно дожидайся всех запущенных воркеров волны.

🖱 Cursor — как задать синхронизацию

Режим задаётся параметром run_in_background в вызове Task tool, а барьер — тем, что main явно читает результаты всех background-воркеров в конце turn.

// Foreground — ждать сразу (pipeline, «волна из одного») { "subagent_type":"generalPurpose", "run_in_background":false } // Background — параллельная волна: несколько Task calls // в ОДНОМ сообщении с run_in_background:true { "subagent_type":"explore", "run_in_background":true } { "subagent_type":"explore", "run_in_background":true } // затем БАРЬЕР: main дожидается обоих перед волной 2

First-wins / quorum / timeout — в UI нет прямых флагов; реализуются логикой оркестратора (описать в AGENTS.md: «продолжай, когда готовы 3 из 4»).

🤖 Claude Code — как задать синхронизацию

Синхронизация — на уровне процессов: ждать claude ... синхронно или запустить фоном и join'ить.

# Foreground — блокирующий вызов (pipeline) claude --agent implementer "..." # ждём завершения # Параллельная волна + barrier (join-all) claude --agent explorer "A" & # фон claude --agent explorer "B" & # фон wait # БАРЬЕР: ждём всех # Timeout-синхронизация timeout 120 claude --agent explorer "..." # Quorum/first-wins — логикой скрипта playbook

Правила «ждать N из M» / «взять первого» описываются в самом playbook (Command) или в CLAUDE.md как политика оркестратора.

Шаг 7. Куда класть шаги и оркестрацию

Оркестрацию нужно где-то зафиксировать, иначе каждый запуск придётся объяснять заново. Но сначала важно различать две вещи: сами шаги под конкретную задачу (меняются каждый раз) и устройство системы (порядок, роли, правила — стабильны).

Что такое «шаги» и что такое «оркестрация» — не путать

Шаги (подзадачи) — это результат декомпозиции конкретной задачи: «изучить middleware», «написать лимитер», «проверить безопасность». Они разные каждый раз и живут в плане текущей задачи (TodoWrite / план оркестратора), а не в файле проекта.
Оркестрация — это устройство системы: какой порядок волн, какие роли у воркеров, как они возвращают findings, какие ограничения. Это стабильно и повторяется от задачи к задаче → это и нужно записать в файлы проекта.

Разложим по местам, что именно куда класть:

Что такое playbook (Command / Skill)?

Playbook (дословно «сборник пьес/сценариев», термин из спорта и DevOps) — это записанный готовый сценарий действий, который можно запустить одной командой, не объясняя шаги заново. Как рецепт: «сделай A, потом B, потом C».
В агентских системах Command / Skill — это playbook оркестрации: файл, где записано «запусти explore → передай findings в implement → потом review». Один раз описал — дальше вызываешь как /feature-review вместо того чтобы каждый раз вручную расписывать всю цепочку воркеров.

Простое правило разделения

Rules = что нельзя никогда (постоянные правила). AGENTS.md = как система устроена (порядок + роли + протокол). Command/Skill = кнопка «выполнить этот сценарий». Prompt = разовый случай. Не смешивай: постоянное правило в prompt забудется на следующем запуске; устройство workflow в Rules будет мешать другим задачам.

Разделение ответственности: агенты / скиллы / AGENTS.md / Command / Rules

Частый вопрос: «если я делаю переносимые скиллы и агентов, нужен ли ещё AGENTS.md?» — Да. Скиллы и агенты — это переносимые capability («что вообще можно делать», их шарят между проектами). AGENTS.md — это привязка к конкретному проекту («как здесь всё собрано и когда что звать»). Это разные слои.

СлойЧто этоПереносимо между проектами?Отвечает на вопрос
Агент (роль воркера)Кто исполнитель: tools, readonly, формат output✅ Да«Кто делает»
СкиллПереиспользуемая процедура/playbook по шагам✅ Да«Как выполнить сценарий»
CommandКнопка запуска: вызвать скилл/цепочку одной строкой⚠️ Частично (может ссылаться на проект)«Чем запустить»
AGENTS.mdУстройство и контекст ЭТОГО проекта, диспетчеризация❌ Нет — своё в каждом проекте«Как устроен проект, когда что звать»
RulesПостоянные правила: что нельзя никогда⚠️ Инварианты переносимы, проектные — нет«Что запрещено всегда»

Что остаётся в AGENTS.md, даже когда есть скиллы и агенты

  • Контекст проекта: стек, команды запуска/тестов, структура, что нельзя трогать.
  • Оркестрация: порядок волн, какой агент на каком шаге, протокол findings.
  • Диспетчеризация: когда какой скилл/агент применять именно здесь.
  • Проектные конвенции, которые не входят в переносимый скилл.

Скиллы/агентов держи переносимыми (без хардкода под один проект), а AGENTS.md — тонкий слой, который ссылается на них и говорит, как применять здесь. Совсем убрать AGENTS.md обычно не стоит — теряется проектный контекст и точка входа для агента.

# AGENTS.md ССЫЛАЕТСЯ на переносимые скиллы/агентов, а не дублирует их ## Стек и запуск Python 3.11 · tests: pytest -q · lint: ruff ## Оркестрация фичи Волна 1: агент explorer (readonly) — исследовать зону Волна 2: агент implementer — код по findings Волна 3: агент security-reviewer (readonly) Запуск всего — скилл/команда /feature-review ## Когда что звать (диспетчеризация) - новая фича → скилл feature-review - разбор упавшего CI → агент ci-investigator - аудит безопасности → агент security-reviewer # сами роли/процедуры — в переносимых агентах и скиллах, не тут
🖱 Cursor — где живёт оркестрация
  • AGENTS.md в корне / поддиректории — порядок волн, роль оркестратора, протокол findings.
  • .cursor/rules/*.mdc — постоянные правила (readonly для reviewer, непересекающиеся файлы).
  • .cursor/commands — playbook: запуск всей цепочки одной командой (напр. /feature-review).

Соберём всё вместе на одной задаче: «Добавить rate limiting в публичное API». Ниже — какие файлы завести (AGENTS.md, Rule, Skill/Command, роли воркеров) и как это исполняется волнами. Слева Cursor, справа Claude Code — концепция одна, отличаются пути и синтаксис.

Важно: в Cursor роли воркеров — это НЕ файлы

Это главное отличие от Claude Code и частый источник путаницы. В Claude Code под каждую роль заводят файл (.claude/agents/explorer.md). В Cursor таких файлов нет. Роль воркера задаётся прямо в момент вызова встроенного Task tool — параметром subagent_type (выбор из готового списка) плюс текстом в prompt. Никакой конфигурации на диске создавать не нужно.

// В Cursor воркер = один вызов Task tool (не файл): { "description": "Explore middleware", "subagent_type": "explore", // ← готовый тип, не файл "prompt": "Role: Explorer. Files: api/middleware/**. Constraints: readonly. Output: findings с file:line", "readonly": true, "run_in_background": true // ← это и есть синхронизация }

То есть на скриншотах «Cursor — как задать синхронизацию» и «встроенные subagent_type» показаны не файлы, а поля одного JSON-вызова Task tool, который агент формирует на лету.

Тонкости построения систем

Coordination overhead может съесть выигрыш

Написать task description, дождаться, проверить findings, объединить — всё это время main агента. Для задачи на 10 минут overhead в 5 минут = 50%. Для задачи на 3 часа тот же overhead = 3%. Система оправдана на задачах достаточного размера.

Глубина иерархии — максимум 2 уровня

main → subagent. Глубже (subagent вызывает subagent вызывает subagent) — cost растёт экспоненциально, ошибки нижних уровней каскадят вверх с искажениями, debugging почти невозможен. Нужна глубина? Значит задачу надо декомпозировать иначе (pipeline вместо рекурсии).

Summary loss на каждом уровне

Воркер возвращает не весь свой context, а сжатый findings. В pipeline с несколькими звеньями потеря накапливается: то, что увидел Explorer, но не включил в summary, Implementer уже не узнает. Лечится строгим output format с обязательным evidence.

Cost параллели = сумма всех воркеров

5 параллельных воркеров = 5× токенов относительно одного. Быстрее, но дороже. Последовательно — дешевле, но медленнее. Оценивай бюджет заранее; если задача помещается в один агент — не строй систему.

Воркеры не знают Rules автоматически

Система — не гарантия соблюдения правил проекта. Каждый воркер стартует с чистого контекста и не видит Rules/AGENTS.md/Memory. Критичные constraints передавай в task description, а main обязан верифицировать findings против Rules перед применением.

Best practices — Делай / Не делай

✅ Делай

  • Начинай с дерева зависимостей, топология выводится из него
  • Параллель только для независимых подзадач на разных файлах
  • Explorer и Reviewer держи в readonly, правки — через одного воркера/main
  • Задай единый output format findings с evidence (file:line)
  • Main агрегирует и верифицирует против Rules перед применением
  • Оценивай cost-бюджет до запуска; сначала пробуй один агент

❌ Не делай

  • Не давай двум параллельным воркерам один и тот же файл
  • Не строй иерархию глубже 2 уровней
  • Не параллель зависимые шаги — получишь мусорные входы
  • Не принимай findings воркеров без проверки против Rules
  • Не строй систему для задачи на 1–2 файла
  • Не забывай точки синхронизации между волнами

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое агентская система?
Агентская система — это несколько AI-агентов, объединённых оркестратором (main agent) в единый рабочий процесс для решения одной большой задачи. Оркестратор разбивает задачу на подзадачи, раздаёт их специализированным воркерам (subagents), собирает их результаты и принимает финальное решение. Отличие от одного агента: работа распараллеливается и специализируется, а контекст изолируется по воркерам.
Junior Кто такой оркестратор?
Оркестратор — это главный агент (main agent), который управляет системой: декомпозирует задачу, назначает роли, запускает воркеров, ждёт и агрегирует их findings, разрешает противоречия и проверяет результат против Rules. Он не выполняет всю работу сам — как прораб на стройке раздаёт задачи специалистам и собирает результат.
Junior Что такое декомпозиция задачи?
Декомпозиция — разбиение большой задачи на маленькие атомарные подзадачи с чёткими границами и известными входами/выходами. Это первый шаг построения: пока не разбита задача и не построено дерево зависимостей, нельзя решить что параллелить, а что делать последовательно.
Junior Чем параллельная схема отличается от последовательной?
Параллельная (fan-out): несколько воркеров работают одновременно на независимых подзадачах, main агрегирует их результаты — быстрее, но дороже (cost × N). Последовательная (pipeline): output одного воркера становится input следующего — дешевле и предсказуемо, но без ускорения. Выбор зависит от зависимостей: независимые → параллельно, зависимые → pipeline.
Junior Что такое дерево зависимостей (DAG)?
Дерево (точнее — направленный ациклический граф, DAG) зависимостей показывает, какая подзадача использует результат какой. Если T3 нужны выходы T1 и T2 — значит T3 идёт после них, а T1 и T2 (не зависящие друг от друга) можно выполнять параллельно. Из DAG напрямую выводится топология системы.
Junior Почему параллельные воркеры не должны трогать один файл?
Потому что действует last-write-wins: тот воркер, который запишет файл последним, затрёт изменения другого. Автоматического merge нет. Поэтому параллельным воркерам назначают непересекающиеся наборы файлов, а общий файл отдают одному воркеру или обрабатывают последовательно через main.
Junior Что такое роль воркера?
Роль — специализация воркера в системе: Explorer (ищет и анализирует, readonly), Implementer (пишет код по spec), Reviewer (проверяет реализацию, readonly), Security/Performance (специализированные reviewer'ы). Роль задаётся в начале task description и определяет что воркер делает и какие у него ограничения.
Junior Что такое точка синхронизации?
Точка синхронизации — момент, когда оркестратор дожидается завершения всех воркеров текущей «волны» и собирает их findings, прежде чем запустить следующую волну. В гибридных системах между параллельными волнами всегда есть точки синхронизации: например, дождаться двух explorer'ов, потом запустить implementer.
Junior Что возвращает воркер оркестратору?
Воркер возвращает findings — структурированный результат работы: summary, конкретные факты с evidence (file:line), рекомендации и уровень уверенности. Он возвращает не весь свой context, а сжатую выжимку. Поэтому важно задать ему output format, чтобы нужные детали не потерялись.
Junior Когда НЕ стоит строить агентскую систему?
Когда задача маленькая (1–2 файла, <10 turns), когда шаги сильно зависят друг от друга и нечего параллелить, когда контекст критичен и его сложно передать воркеру, или когда overhead координации (написание briefing + verification + агрегация) больше выигрыша. В таких случаях эффективнее один агент.
Middle — 10 вопросов
Middle Как из дерева зависимостей вывести топологию системы?
Правило: узлы без взаимных зависимостей группируются в параллельную «волну»; узел, зависящий от других, ставится после них. Проход по DAG слоями (топологическая сортировка) даёт последовательность волн: волна 1 — все входные независимые задачи параллельно, точка синхронизации, волна 2 — зависящие от волны 1, и т.д. Так получается гибрид parallel+pipeline.
Middle Что такое гибридная (волновая) топология и когда она нужна?
Гибрид — чередование параллельных волн и точек синхронизации. Нужен почти всегда в реальных задачах: часть подзадач независима (explore backend ║ explore frontend), часть зависима (implement после explore). Волна 1 параллельно → синхронизация → волна 2 (single или parallel) → синхронизация → финальная агрегация. Это компромисс между скоростью параллели и корректностью pipeline.
Middle Почему параллелизм не всегда ускоряет?
Общее время параллельной волны = время самого долгого воркера. Если один заканчивает за 30 сек, а другой за 5 минут — выигрыш ограничен. Плюс если воркеры вернут противоречивые findings, main потратит время на reconciliation, что может свести ускорение к нулю. Нужно балансировать размеры подзадач и минимизировать пересечения.
Middle Как безопасно параллелить правки кода?
Два подхода: (1) непересекающиеся наборы файлов — каждый воркер владеет своей директорией; (2) read-write разделение — параллельные воркеры readonly, возвращают патчи/findings как текст, а физические правки применяет один main agent последовательно. Второй паттерн полностью исключает гонки записи ценой того, что применение изменений не параллелится.
Middle Что включать в task description воркера при построении системы?
Role (специализация), Goal (одно чёткое предложение), Context (findings предыдущих воркеров, если pipeline), Files (конкретные пути, непересекающиеся с другими воркерами), Scope (тематические границы), Constraints (readonly? запреты), Output format (единый для агрегации, с evidence file:line). Без единого output format собрать результаты нескольких воркеров будет тяжело.
Middle Зачем нужен единый протокол возврата findings?
Чтобы оркестратор мог механически собрать результаты нескольких воркеров, сравнить их и разрешить противоречия. Если один вернул prose, другой JSON, третий список — агрегация превращается в ручной разбор. Единый формат (Summary → Findings → Evidence → Next steps с обязательными file:line) делает результаты сопоставимыми и проверяемыми.
Middle Как оценить, оправдана ли система по стоимости?
Прикинуть токен-бюджет каждого воркера и сравнить с выигрышем. Параллель = сумма токенов всех воркеров (5 воркеров ≈ 5×). Если задача помещается в один context и решается за десяток turns — система не нужна. Правило: overhead координации в процентах от общего времени должен быть мал; для крупных задач (часы) это оправдано, для мелких — нет.
Middle Как роли Explorer/Implementer/Reviewer работают вместе?
Explorer (readonly) исследует и возвращает findings → main передаёт их как Context Implementer'у, который пишет код в узком scope → Reviewer (readonly) проверяет результат против требований и Rules и возвращает список проблем. Explorer и Reviewer безопасно параллелить с другими независимыми воркерами; Implementer стоит между ними в точке синхронизации. Правки идут через одного исполнителя, чтобы избежать конфликтов.
Middle Какие типичные ошибки при построении системы?
Слишком мелкая декомпозиция (overhead > benefit); параллель зависимых задач (мусорные входы); отсутствие output format (нельзя агрегировать); пересечение файлов у параллельных воркеров (гонки записи); иерархия глубже 2 уровней; blind-trust findings без verification против Rules; отсутствие точек синхронизации; передача воркеру недостаточного контекста.
Middle Как система помогает управлять контекстом (compaction)?
Вместо того чтобы main сам читал десятки файлов (и его context быстро заполнялся → compaction), исследование делегируется воркерам. Каждый работает в своём изолированном контексте, читает много и возвращает только summary. Main получает выжимки → его context растёт медленнее и остаётся структурированным даже на очень длинных задачах.
Senior — 5 вопросов
Senior Как спроектировать multi-agent workflow для сложной инженерной задачи?
(1) Декомпозиция на атомарные подзадачи + dependency graph. (2) Топологическая сортировка → волны parallel/sequential. (3) Назначение ролей (Explorer readonly, Implementer, Reviewer). (4) Распределение непересекающихся файлов. (5) Единый протокол возврата findings с evidence. (6) Точки синхронизации между волнами + checkpoint на завершении воркера. (7) Агрегация в main с разрешением противоречий и verification против Rules. (8) Cost-бюджет и error handling (retry/fallback/escalate).
Senior Как агрегировать результаты нескольких воркеров и разрешать противоречия?
(1) Прочитать все findings полностью перед агрегацией. (2) Категоризировать: confirmed facts / hypotheses / contradictions. (3) При противоречии смотреть scope (разные части кода?) и уровень evidence (у кого конкретнее file:line); при критичности — запустить tie-breaker воркера. (4) Взвешивать по confidence: findings с конкретными ссылками важнее общих утверждений. (5) Собрать unified summary. (6) Выявить gaps — что не покрыл ни один воркер → targeted воркер или прямое исследование.
Senior Как диагностировать coordination failures в системе?
Result gaps (ожидали X, получили Y) → неоднозначная task description. Contradicting findings → overlap в scope или разный доступ к данным. Missing evidence → output format не требовал ссылок. Timeout/hang → слишком широкий scope, поиск без ограничений. Rule violations → воркер не знал о Rule (не передали в task). Debug: попросить воркера объяснить reasoning step-by-step и проверить его входной briefing.
Senior Как контролировать затраты при параллельных воркерах?
(1) Оценить токен-бюджет до запуска; при перерасходе — пересмотреть архитектуру. (2) Узкий scope = меньше токенов. (3) Лимиты: max tool calls / max файлов на воркера. (4) Sequential fallback — сначала один агент, параллель только при бюджете. (5) Мониторинг токенов на воркера, выявление дорогих. (6) Кэширование findings, если задача не изменилась. Budget gate: не запускать систему, если estimate превышает порог без approval.
Senior Почему глубина иерархии ограничена двумя уровнями?
На каждом уровне: cost растёт мультипликативно, summary loss накапливается, ошибки нижних уровней каскадят вверх с искажениями, а debugging становится почти невозможным (непонятно, что пошло не так на 4-м уровне). Практический максимум — main → subagent. Если требуется бóльшая глубина, это сигнал декомпозировать задачу иначе: развернуть рекурсию в плоский pipeline или волновую топологию.
Lead — 5 вопросов
Lead Как выстроить governance построения агентских систем в команде?
(1) Use case policy: где системы одобрены (crossbase exploration, parallel review), где нет (мелкие/зависимые задачи). (2) Cost policy: бюджет на workflow, approval для дорогих. (3) Output standards: обязательный structured findings с evidence. (4) Review requirement: критичные findings (security, архитектура) → human review. (5) Audit trail: логировать какие воркеры запускались, что вернули. (6) Template library: готовые task descriptions. (7) Training. Документ governance — 1 страница с do's/don'ts и процессом approval.
Lead Как обучать команду паттернам построения систем?
(1) Foundations: декомпозиция, DAG, parallel vs pipeline, demo хорошего/плохого briefing. (2) Anti-patterns workshop: пересечение файлов, глубокая иерархия, blind trust. (3) Template library для частых сценариев (review, security scan, exploration). (4) Hands-on: каждый строит систему для реальной задачи. (5) Code review criterion: проверять task descriptions воркеров при AI-PR. (6) Retrospective на инцидентах: правильно ли построена система? Показать реальные costs.
Lead Когда запрещать автономные агентские системы (safety/compliance)?
Когда есть destructive actions без HITL (delete, deploy в prod, необратимое) — только через approval gate; при обработке PII — audit log и data policy; при cross-boundary access (secret files, prod credentials) — least privilege; при работе с непроверенными внешними данными — риск prompt injection; при high-stakes решениях (архитектура, security, compliance) — обязательный human review, не автономный воркер. Принцип: воркер получает ровно те права, что нужны для задачи.
Lead Как управлять costs параллельных систем в командной разработке?
(1) Cost visibility: dashboard спенда по команде/инженеру/проекту, системы отдельной строкой. (2) Budget allocation с alerts при 80%. (3) Approval gates для workflow с estimate > порога. (4) Обучение cost-efficient паттернам: узкий scope, чёткий output, без избыточных воркеров. (5) Cost attribution по tags. (6) Weekly review топ-5 дорогих workflow — что оптимизировать. Цель: максимум value per token.
Lead Как выстроить approval gates для действий воркеров в production?
(1) Risk classification: read (auto), write non-prod (auto), write prod/delete (approval). (2) HITL: при высокорисковом действии воркер создаёт approval request (описание, ожидаемый результат, rollback plan). (3) Dry-run first: воркер показывает план без выполнения, human одобряет → выполнение. (4) Audit log каждого approve/reject. (5) Emergency stop всех активных воркеров. (6) Post-action verification отдельным воркером.
Architect — 5 вопросов
Arch Какие архитектурные паттерны оркестрации существуют и как выбирать?
Hub-and-Spoke (оркестратор + воркеры, просто, но bottleneck на нём); Pipeline (output→input, для зависимых шагов); Hierarchical (multi-level, для очень крупных задач, max 2–3 уровня); Peer-to-Peer (агенты общаются через shared state, гибко но сложно); Competitive/Best-of-N (N решают независимо, выбирается лучший — для качества); Consensus (голосование, для critical decisions). Выбор: Hub-and-Spoke для большинства; Pipeline для sequential; Best-of-N при высоких требованиях к качеству. Детали — на странице Subagents.
Arch Как спроектировать платформу оркестрации воркеров?
Компоненты: Agent Registry (каталог типов агентов + capabilities/cost); Task Queue (очередь с priority/deadline/dependencies); Context Manager (передача входного контекста, сбор результатов); Communication Bus (shared memory / message queue / event stream); State Store (checkpoint каждого агента, resume); Observability (distributed tracing, cost tracking); Policy Engine (права, approval gates, rate limits); Scheduler (parallel/sequential/conditional). Стек: Celery/BullMQ, Redis/DynamoDB, Kafka/RabbitMQ.
Arch Как обеспечить fault tolerance в агентской системе?
Retry с exponential backoff для transient failures; circuit breaker для часто падающих типов агентов; fallback strategies (другой instance, более простой агент, graceful degradation с partial result); timeout enforcement с checkpoint того, что успел; partial success handling (3 из 4 succeeded → продолжить с пометкой missing); dead letter queue для failed tasks; health checks инстансов. Цель — система деградирует плавно, а не падает целиком.
Arch Как строить observability для multi-agent workflow?
Distributed tracing с единым trace_id через orchestrator → воркер → tool calls (OpenTelemetry, Jaeger/Zipkin). Structured logging (trace_id, agent_type, task_id, action, duration, cost_tokens). Agent activity timeline. Cost attribution по агенту/задаче/команде в реальном времени. Quality metrics (success rate, retry rate). Anomaly detection (слишком долгий агент, cost spike). Корреляция workflow-метрик с бизнес-результатами (время review, bug detection rate).
Arch Как проектировать communication protocols между воркерами?
Message schema: header (sender, receiver, trace_id, timestamp, type) + payload; JSON Schema или Protobuf. Паттерны: request-response, pub-sub, shared state. Consistency model: eventual для throughput или strong для critical data. Versioning с backward compat. Security: authentication (short-lived tokens) + authorization (кто что читает/пишет). Serialization: binary для performance, JSON для debuggability. Backpressure при перегрузке. Принцип: предпочитать простые паттерны (request-response) сложным, пока масштаб не требует иного.
