С нуля: что такое Agent Loop — Claude Code / Cursor

Прежде чем разобраться в subagents — нужно понять базу: как вообще работает агент изнутри. Это и есть agent loop. Концепция одинакова в Claude Code и Cursor — разница только в деталях реализации.

Уровень 0 — простыми словами

Когда ты пишешь агенту задачу — он не отвечает сразу. Он начинает цикл: думает → вызывает инструмент (читает файл, запускает команду) → смотрит на результат → думает снова → снова вызывает инструмент... И так повторяет пока задача не решена. Этот цикл и называется agent loop.

Снова оценивает — нужен ещё tool? │───┘ повтор │ ↓ нет │ │ 6. Пример: как агент исправляет тест — шаг за шагом

Задача: "Fix the failing tests in auth.ts" ── Turn 1 ────────────────────────────────────────── Claude вызывает: Bash("npm test") Результат: 3 теста упали, ошибки в auth.ts ── Turn 2 ────────────────────────────────────────── Claude вызывает: Read("auth.ts") + Read("auth.test.ts") Результат: содержимое файлов → видит где баг ── Turn 3 ────────────────────────────────────────── Claude вызывает: Edit("auth.ts", fix) → Bash("npm test") Результат: все 3 теста прошли ✅ ── Final Turn ─────────────────────────────────────── Claude: "Fixed the auth bug, all three tests pass now." → ResultMessage {subtype: "success", cost: $0.003} Итого: 4 turns, 3 с tool calls + 1 финальный текстовый

Выше effort → больше токенов → дороже и медленнее. Для subagents можно задавать effort отдельно через AgentDefinition.effort.

Почему context window растёт с каждым turn

Context не сбрасывается между turns внутри одной сессии — всё накапливается: system prompt + история всех turns + все tool inputs + все tool outputs. Большой файл прочитан → тысячи токенов добавлены навсегда в текущую сессию. Через 20–30 turns context может стать огромным → автоматический compaction (сжатие) или передача задачи subagent'у со свежим context.

🖱 Cursor — одинаково?

Да, концепция полностью та же: think → tool call → result → think. В Cursor это называется Agent mode. Cursor использует те же принципы loop — turns, tool calls, context accumulation.

Отличие от Claude Code SDK: в Cursor нет прямого доступа к параметрам типа max_turns или effort в коде — это управляется через UI и настройки. Но логика цикла идентична.

  • Tool calls: Read, Edit, Bash, Grep, Glob — те же
  • Subagents: через Task tool (встроен в агент)
  • Context accumulation: нарастает по turns
  • Compaction: автоматический при переполнении

Subagents: что это

Subagent — отдельный специализированный агент или изолированный image.png, которому main agent делегирует focused subtask. Он работает со своим контекстом и возвращает findings/summary.

Main agent видит основную задачу и принимает финальное решение
Task A → Code search subagent ищет классы/методы
Task B → Test analyst анализирует CI/test logs
Task C → Security reviewer проверяет риски
Summaries return main объединяет и проверяет

Cursor vs Claude Code — как запускать subagents

Механика запуска subagents отличается: Cursor использует Task tool, Claude Code — нативный синтаксис делегирования. Но концепция одна.

🖱 Cursor — Task tool

Cursor предоставляет встроенный Task tool в своём UI. Агент может создавать subagents через параметры:

// Как агент вызывает Task tool в Cursor { "description": "Security scan — auth module", "subagent_type": "generalPurpose", "prompt": "Role: Security Analyst. Goal: find SQL injection in api/auth/. Scope: api/auth/*.py only. Constraints: do not edit files. Output: list of issues with file:line", "readonly": true, // только чтение "run_in_background": false // foreground }

Типы subagent: generalPurpose, explore, shell, bugbot, security-review, best-of-n-runner, ci-investigator

🤖 Claude Code — нативное делегирование

Claude Code поддерживает TodoWrite + автономный режим. Subagents через AGENTS.md и --agent флаги. В Claude Max — параллельный запуск агентов.

# Запуск Claude Code как subagent (CLI) claude --agent "security-reviewer" \ --scope "api/auth/**" \ --readonly \ --output structured \ "Find SQL injection vulnerabilities. Return: file:line + severity for each." # Через AGENTS.md — роль subagent'а ## Role: Security Reviewer Subagent ## Scope: api/ directory only ## Allowed: Read files, grep, analyze ## Forbidden: Edit, delete, run commands ## Output: JSON {file, line, issue, severity}

AGENTS.md в поддиректории переопределяет роль для файлов этой директории — мощный механизм специализации.

Foreground vs Background выполнение

Subagent может работать в двух режимах. Выбор влияет на flow взаимодействия с пользователем и эффективность параллелизма.

Foreground (синхронный)

"run_in_background": false — main agent ждёт завершения subagent.

Main agent запускает subagent ↓ [BLOCKING — ждёт] ↓ Subagent выполняет задачу ↓ возвращает findings ↓ Main agent получает результат и продолжает работу

Когда: результат нужен для следующего шага. Задачи последовательные и зависимы.

Background (асинхронный)

"run_in_background": true — main agent продолжает работу, получает уведомление по завершении.

Main запускает subagent A ──┐ Main запускает subagent B ──┤ параллельно Main запускает subagent C ──┘ ↓ Main продолжает другую работу ↓ [уведомления при завершении] A: done → reading results B: done → reading results C: done → reading results ↓ Main агрегирует все findings

Когда: независимые задачи, нужен параллелизм. Ускоряет сложные workflows.

Background ≠ "fire and forget"

Main agent должен явно читать результаты background subagents и не заканчивать работу без получения их findings. Если main agent завершается раньше background subagent — результат теряется. Правило: в конце turn main agent явно ожидает все незавершённые background subagents.

Что наследует subagent от main agent — модель передачи контекста

Subagent — это чистый старт. Он не получает историю диалога, Memory Bank или Rules автоматически. Всё что ему нужно — main agent обязан передать явно в task description.

ПараметрНаследуется автоматически?Что происходит / как передать явно
История диалога (весь чат main агента) ❌ Нет Subagent не видит предыдущие сообщения. Передавать ключевые факты в prompt-е задачи.
Rules / AGENTS.md ⚠️ Зависит от платформы В Cursor — глобальные Rules могут загружаться, но лучше дублировать критичные constraints в task description. В Claude Code — AGENTS.md в рабочей директории загружается.
Memory Bank (файлы памяти) ❌ Нет Subagent не читает Memory Bank. Передать нужные выдержки явно в prompt.
System prompt / модель поведения ⚠️ Частично Базовый system prompt платформы — да. Кастомные инструкции main агента — нет. Ключевые правила поведения передавать в Role / Constraints.
Tool access (MCP, файловые tools) ⚠️ Зависит от readonly Cursor: если "readonly": true — только чтение. Разрешённые tools явно указывать в task description. По умолчанию — полный доступ платформы.
Files в context (открытые файлы, @-прикреплённые) ❌ Нет Явно перечислять нужные пути в task description: «Читай только api/handlers/*.py».
Текущий прогресс / todo list ❌ Нет Subagent не знает что уже сделал main agent. Включать релевантный статус в prompt.
Роль / Role definition ❌ Нет Main agent может называть себя «Senior Backend Engineer» — subagent об этом не знает. Задавать роль subagent'а в начале task description: «Role: Security Analyst».
MCP серверы / MCP tools ⚠️ Зависит от платформы и режима Подробнее — см. блок ниже.
Выбранная модель (model) ⚠️ Зависит от платформы В Cursor subagent использует ту же модель что main, если не задан другой model параметр.

MCP и subagents — что наследуется, что нет

MCP-серверы (tools, ресурсы) — одна из самых неочевидных частей наследования. Subagent не получает «доступ к MCP» автоматически в общем смысле — это зависит от режима запуска и платформы.

🖱 Cursor — MCP и subagents
  • Subagent запущенный через Task tool работает в той же среде Cursor → имеет доступ к тем же MCP-серверам что настроены в .cursor/mcp.json.
  • Но: если "readonly": true — MCP tools с write-операциями (create file, run command) заблокированы.
  • Subagent видит те же MCP tools что main agent, если они подключены глобально.
  • Если нужно ограничить MCP доступ — явно указать в Constraints: «Не использовать MCP tools, только чтение файлов».
// Task description — явно указать что MCP разрешён { "prompt": "Role: Data Fetcher. Goal: получить список issues через GitHub MCP. Allowed tools: mcp_github_list_issues only. Constraints: не редактировать, не создавать issues. Output: JSON array {id, title, labels}", "readonly": false // нужен для MCP write tools }
🤖 Claude Code — MCP и subagents
  • Claude Code subagent (через CLI) не наследует MCP-конфигурацию автоматически — нужно передать --mcp-config явно.
  • Каждый Claude Code процесс — отдельная сессия, MCP серверы инициализируются заново.
  • Если subagent должен использовать MCP — передавать конфиг файл или флаги при запуске.
  • В AGENTS.md подпапки можно указать разрешённые MCP tools для subagent этой директории.
# Claude Code subagent с MCP claude --mcp-config ./mcp-readonly.json \ --agent "data-fetcher" \ "Goal: fetch issues from GitHub MCP. Allowed: mcp__github__list_issues only. Constraints: no write operations." # mcp-readonly.json — минимальный конфиг # только нужные серверы, только нужные tools
СитуацияMCP доступен subagent'у?Что делать

MCP с доступом к базам / prod — принцип минимальных привилегий

Если main agent подключён к MCP с доступом к production БД или внешним API — subagent в Cursor автоматически наследует этот доступ. Это риск: subagent с широкой задачей может случайно вызвать write-операцию. Правило: если subagent делает только анализ → "readonly": true. Если MCP нужен — явно белым списком указывать разрешённые tools в Constraints: «Allowed MCP tools: только list/read операции».

Что можно переопределить в task description subagent'а

Переопределяемые параметры
  • Role — задать любую роль (Security Analyst, Performance Reviewer, Code Explorer)
  • Tone / style — «отвечай кратко», «только JSON», «без объяснений»
  • Tool access — ограничить до subset или разрешить конкретные MCP
  • Scope — явно сузить до нужных файлов/директорий
  • Output format — JSON, markdown, plain text, structured report
  • Model — в Cursor можно указать другую модель через model
  • Behaviour constraints — «не редактируй файлы», «не делай предположений»
Что нельзя переопределить
  • Безопасность платформы — системные ограничения Cursor/Claude Code (нельзя дать subagent право выполнять произвольный системный код за пределами разрешённого)
  • Billing / rate limits — платформенные лимиты не обходятся через task description
  • Базовый system prompt модели — конституция Claude/GPT остаётся
  • Глобальные Rules если они enforced платформой — в Cursor alwaysApply Rules могут применяться и к subagents

Главная ловушка: «он же должен знать наши правила»

Main agent работает с Rules, видит AGENTS.md, знает ограничения проекта. Subagent — нет. Инженеры часто предполагают что subagent автоматически «знает» правила («не удалять тесты», «не менять API контракт»). Он не знает. Критичные constraints нужно явно включать в task description или передавать содержимое AGENTS.md как часть prompt.

Files, Constraints, Scope — три ключевых параметра задачи

Любая хорошая task description для subagent строится на трёх концептах: Files (с чем работать), Scope (в каких границах), Constraints (что запрещено). Это три разных уровня ограничения — часто путают.

📂 Files

Что это: конкретные файлы или паттерны файлов которые subagent может / должен читать или изменять.

Зачем: без явного списка subagent начнёт сканировать весь проект, тратя tokens и время.

Files: - api/handlers/*.py # читать - api/models.py # читать - tests/test_auth.py # НЕ трогать # Паттерны (glob) Files: src/services/**/*.ts Exclude: **/*.test.ts, dist/

Частая ошибка: не указать Files → subagent читает всё → context overflow → дорого и медленно.

🔭 Scope

Что это: тематические границы задачи — что концептуально входит в задачу, а что нет. Отвечает на «о чём эта задача».

Зачем: subagent без scope может «поплыть» — найдя одну проблему, начать исследовать смежные, не связанные с заданием.

Scope: только аутентификация (auth, session, tokens) OUT of scope: - бизнес-логика заказов - UI компоненты - инфраструктура/деплой # Scope = тема, Files = файлы # Это разные вещи!

Частая ошибка: думать что Files = Scope. Можно дать файл в Files, но сказать «смотри только на функции авторизации».

🚫 Constraints

Что это: явные запреты на действия — что subagent не должен делать ни при каких условиях.

Зачем: защита от непреднамеренных изменений, нарушения архитектуры, работы за пределами полномочий.

Constraints: - readonly: не редактировать файлы - не удалять тесты - не менять публичный API - не обращаться к внешним URL - не запускать команды - не касаться prod configs - не предлагать смену стека

Частая ошибка: не указать constraints → subagent «помогает» редактируя файлы которые нельзя трогать.

Как они работают вместе — полный пример

Role: Security Code Reviewer Goal: Найти SQL injection уязвимости в слое работы с базой данных Files: ← конкретные файлы для чтения - api/handlers/*.py - api/repositories/*.py - api/models.py Exclude: tests/, migrations/ Scope: ← тематические границы Только SQL-запросы и ORM-вызовы. Не анализировать бизнес-логику, UI, конфиги. Фокус: raw queries, f-string interpolation, user input в SQL. Constraints: ← что запрещено - readonly: не редактировать файлы - не предлагать рефакторинг за пределами найденных уязвимостей - не запускать команды - не касаться тестов и migrations Output format: JSON array: [{file, line, code_snippet, severity: high|medium|low, fix_hint}] Max: 10 findings. Если больше — вернуть топ-10 по severity.

Files vs Scope — тонкая разница

Files = физические пути к файлам (что читать). Scope = концептуальные границы (о чём думать). Пример: Files включает api/orders.py потому что там есть auth-логика, но Scope = «только аутентификация» — subagent читает файл, но анализирует только auth-части, не всю бизнес-логику заказов. Без явного Scope subagent будет анализировать весь файл.

Тонкости и нюансы Subagents

Subagent не видит полный контекст главного агента

Main agent передаёт subagent только то, что явно указано в задаче: файлы, constraints, scope. Весь предыдущий чат, Memory Bank, все Rules — subagent не получает автоматически. Если subagent нужны правила — передавай их явно в задаче или убедись что AGENTS.md загружается в его контекст.

Summary loss — subagent возвращает не всё

Main agent получает от subagent сжатый результат (summary), а не весь его context window. Детали, которые subagent «видел», но не включил в summary — теряются. Поэтому в задаче явно указывай нужный output формат: «Верни 3 гипотезы с evidence: имя файла + строка + команда для проверки».

Несколько subagents редактируют один файл → конфликт

Если параллельные subagents редактируют один и тот же файл — последний write перезапишет предыдущий (last write wins). Архитектурное правило: параллельные subagents должны работать на непересекающихся наборах файлов. Для shared файла — последовательная обработка или merge через main agent.

Subagent может нарушить Rules если не передал их явно

Subagent не знает о Rules main agent'а если они не включены в его task context. Он может предложить удалить тест (если Rule запрещает — main agent не знает). Main agent всегда должен верифицировать findings subagent'а против Rules/AGENTS.md перед применением.

Cost и latency: параллельность стоит больше

Каждый subagent — отдельный LLM вызов со своим context. Параллельные subagents быстрее но дороже: 5 параллельных = 5x стоимость. Последовательные = медленнее но дешевле. Subagents оправданы когда выигрыш от изоляции контекста или параллельности перевешивает стоимость.

Best practices — Делай / Не делай

✅ Делай

  • Давай subagent чёткий scope: конкретные файлы, директории, ограничения
  • Указывай output формат: «Верни N гипотез с evidence (файл + строка)»
  • Передавай нужные Rules/constraints явно в task description
  • Main agent всегда верифицирует findings против Rules перед применением
  • Используй subagents для изоляции: читает много файлов → main получает summary
  • Assign непересекающиеся файлы для параллельных subagents

❌ Не делай

  • Не давай «Проверь весь проект» — слишком широко, scope не определён
  • Не позволяй нескольким subagents редактировать один файл параллельно
  • Не принимай subagent findings без проверки против Rules
  • Не используй subagents для простых однофайловых задач — overhead не оправдан
  • Не забывай: subagent не видит Memory Bank — передавай релевантный контекст явно

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое subagent?
Subagent — специализированный AI агент или изолированный agent loop, которому главный (main) агент делегирует конкретную подзадачу. Subagent работает в своём отдельном context window, выполняет задачу и возвращает результат (findings, summary, analysis) главному агенту. Аналог: руководитель (main agent) поручает специалисту (subagent) исследовать конкретный вопрос и доложить выводы.
Junior Зачем нужны subagents?
Subagents решают несколько проблем: (1) Изоляция контекста — subagent не загрязняет context main агента шумом из исследования. (2) Параллелизм — несколько subagents работают одновременно, ускоряя задачи. (3) Специализация — каждый subagent с определённой ролью (code reviewer, security analyst, test engineer). (4) Масштаб — задача слишком большая для одного context window → разделить между subagents. (5) Compaction management — subagent исследует много, возвращает только summary → main agent экономит context.
Junior Чем subagent отличается от main agent?
Main agent: ведёт основной диалог с пользователем, имеет полный context задачи, принимает финальные решения, проверяет findings против Rules. Subagent: получает narrow task description, имеет собственный изолированный context, фокусируется только на своей подзадаче, возвращает findings а не принимает финальные решения. Main agent несёт ответственность за финальный результат; subagent — execution специалист. Main может вызывать subagents, subagent обычно не вызывает других subagents (ограничение глубины).
Junior Что видит subagent в своём context?
Subagent видит только то что main agent включил в task description: конкретную задачу (роль, цель, scope), разрешённые tools, constraints (что не делать), output format (что вернуть и как). Subagent НЕ видит: полный диалог main агента с пользователем, предыдущие turns основной сессии, другие subagents и их findings, Global Rules/AGENTS.md если они не включены в task description. Изолированный context — особенность и ограничение одновременно: чистый фокус, но нет полной картины.
Junior Как main agent запускает subagent?
В Cursor: через Task tool с параметрами (description, subagent_type, prompt). В Claude Code: через специальный синтаксис делегирования. В custom системах: main agent генерирует task description → система создаёт новый agent loop с этим описанием → запускает execution. Task description = полный briefing для subagent: роль, задача, доступные данные, что вернуть. Main agent ждёт результат или продолжает работу пока subagent работает в background (async pattern).
Junior Что такое task description для subagent?
Task description — полный briefing который main agent передаёт subagent. Содержит: Role (кто ты — code reviewer, security analyst), Goal (что найти/сделать), Context (нужные файлы, данные, constraints), Allowed tools (что можешь вызывать), Output format (что вернуть и в каком формате). Плохая task description: 'проверь проект'. Хорошая: 'Role: Security Reviewer. Goal: найти SQL injection уязвимости в api/handlers/. Files: api/handlers/*.py. Return: список уязвимостей с файл:строка и severity'.
Junior Что возвращает subagent?
Subagent возвращает findings — структурированный результат своей работы. Обычно: конкретные факты (не 'проверил код' а 'найдено 3 проблемы в auth.py строки 45, 78, 102'), доказательства (конкретные файлы, строки, команды, примеры), рекомендации (что делать дальше), confidence (уверенность в findings). Subagent NOT: не принимает финальные решения, не применяет изменения без полномочий. Его output = data для main агента. Main agent верифицирует и принимает решения.
Junior Что такое isolated context subagent'а?
Isolated context означает что subagent начинает с пустым context window, получая только то что ему передаёт main agent в task description. Он не наследует историю диалога main агента. Преимущества: чистый фокус без distraction, не тратит context на нерелевантную информацию, может исследовать много файлов не загрязняя context main. Недостатки: не знает полный контекст задачи, может упустить важные детали которые main агент считал очевидными. Решение: main agent должен тщательно включить в task description всё что subagent'у нужно знать.
Junior Можно ли запустить несколько subagents параллельно?
Да, в поддерживающих платформах (Cursor) можно запустить несколько subagents параллельно. Условие: задачи subagents должны быть независимы (нет зависимостей между ними). Пример: subagent A исследует backend, subagent B — frontend, subagent C — infrastructure — все параллельно. Main agent ждёт все результаты и объединяет. Не параллельно: если subagent B ждёт результат subagent A (sequential dependency). Параллелизм ускоряет задачу, но умножает cost (токены суммируются).
Junior Что такое coordination overhead в subagent системах?
Coordination overhead — дополнительные затраты на управление subagents: написание task descriptions, ожидание результатов, проверка findings, объединение результатов от нескольких subagents. Если overhead > выигрыш от параллелизма или изоляции → subagents не оправданы. Пример: для задачи на 10 минут overhead написания task description + verification может занять 5 минут → 50% overhead. Для задачи на 3 часа → тот же overhead = 3% → выгодно. Правило: subagents оправданы для задач достаточной сложности и размера.
Middle — 10 вопросов
Middle Когда использовать subagents vs один агент выполняет всё?
Subagents оправданы когда: задача слишком большая для одного context window, части задачи независимы и можно параллелизировать, нужна специализация (разные роли для разных частей), исследование генерирует много шума который не нужен main agent. Один агент лучше когда: задача небольшая и умещается в context, части сильно взаимозависимы, context нужен shared (subagent не видит что сделал другой subagent без explicit sharing), overhead на coordination выше выигрыша. Практическое правило: задача > 2 часов с явными независимыми частями → рассмотреть subagents.
Middle Что нужно передавать subagent в task description для качественного результата?
Полный briefing включает: (1) Role — кто ты, твоя специализация. (2) Goal — одно чёткое предложение что найти/сделать. (3) Scope — конкретные файлы, директории, компоненты (не 'проверь всё'). (4) Context — ключевые факты которые subagent должен знать (стек, версии, ограничения). (5) Constraints — что НЕ делать, что не трогать. (6) Allowed tools — какие tools можно вызывать. (7) Output format — структура возвращаемого результата. (8) Evidence requirement — 'возвращай конкретные файлы:строки, не общие выводы'. Без этих элементов subagent может вернуть нерелевантный или слишком общий результат.
Middle Как main agent получает и использует результат subagent?
Flow получения результата: subagent возвращает findings → они попадают в context main агента как structured message. Main agent: (1) Reads findings — читает весь результат. (2) Verifies against Rules — проверяет что findings не нарушают Rules/AGENTS.md проекта. (3) Critically evaluates — не принимает слепо: 'subagent видел только часть картины'. (4) Cross-references — если несколько subagents → сопоставляет их findings на противоречия. (5) Makes decision — на основе findings принимает решение и действует. Subagent findings = data + evidence, не final truth. Main agent = decision maker.
Middle Когда параллелизм subagents эффективен, а когда создаёт больше проблем?
Эффективный параллелизм: задачи независимы (не зависят от результатов друг друга), примерно одинакового размера (один не заканчивается намного раньше другого), subagents работают с разными частями кодовой базы (нет конфликтов). Проблематичный параллелизм: задачи зависимы (нужно ждать result subagent A для запуска B), subagents работают с одними файлами (race condition на изменения), их findings противоречат друг другу (много overhead на reconciliation), один subagent быстро заканчивает и простаивает. Сигнал: если main agent тратит больше времени на merge results чем subagents на работу → пересмотреть архитектуру.
Middle Как использовать subagent для code exploration без загрязнения context?
Паттерн Code Explorer subagent: (1) Main agent создаёт subagent с задачей 'исследуй codebase, найди X'. (2) Subagent читает десятки файлов, выполняет grep/search, анализирует структуру — всё в своём изолированном context. (3) Subagent возвращает structured findings: 'AuthService в backend/auth/service.py строки 45–120, зависит от UserRepository'. (4) Main agent получает только summary — не весь шум от чтения файлов. Context main агента остаётся чистым. Бонус: если исследование было неуспешным (не нашли) → subagent отработал шум, main не тратил context на это.
Middle Какие преимущества и недостатки context isolation у subagent?
Преимущества isolation: (1) Чистый фокус — subagent не distracted чужим контекстом. (2) Context economy — subagent может исследовать много не загрязняя context main. (3) Параллелизм — изолированные contexts не конфликтуют. (4) Специализация — subagent optimal для своей задачи. Недостатки isolation: (1) Нет shared context — subagent не видит что сделали другие subagents. (2) Briefing burden — main agent должен тщательно передать весь нужный контекст в task description. (3) Blind spots — subagent может не знать важного constraint который main agent считал очевидным. (4) Result integration overhead — main agent должен объединять findings из изолированных contexts.
Middle Как subagent передаёт контекст обратно main агенту?
Subagent возвращает findings как structured message. Для качественного return: (1) Structured format — заголовки по секциям: Summary, Findings, Evidence, Recommendations. (2) Specific references — не 'нашёл проблему', а 'нашёл SQL injection в api/handlers/user.py:line 78 в функции get_user_by_id'. (3) Confidence level — 'высокая уверенность' vs 'возможно, требует проверки'. (4) Actionable next steps — что конкретно делать дальше на основе findings. (5) Size control — если findings большие, summarize ключевое, детали в appendix. Main agent должен просить subagent в task description: 'структурируй результат как: Summary (3 bullet) + Detailed findings + Next steps'.
Middle Как subagents соотносятся с compaction management?
Subagents как compaction management стратегия: вместо того чтобы main agent делал всё исследование (и context быстро заполнялся → compaction) → main agent делегирует исследование subagents. Каждый subagent работает в своём context, исследует много → возвращает summary. Main agent получает только summary → context наполняется медленнее. Практика: для длинных multi-step задач → создавать subagents для каждого major research/investigation шага. Main agent собирает findings и действует. Эффект: главный диалог остаётся структурированным и чистым даже для очень длинных задач.
Middle Какие типичные ошибки при проектировании subagent архитектуры?
Типичные ошибки: (1) Too fine-grained — subagent на каждую мелочь: overhead > benefit. (2) No scope constraints — 'проверь всё' → subagent делает всё, медленно и дорого. (3) No output format — subagent возвращает prose, main не может структурировано использовать. (4) Ignoring findings — main agent blind-trusts subagent без проверки против Rules. (5) No error handling — subagent вернул ошибку, main не заметил. (6) Sequential when parallel possible — запускают subagents по одному когда они независимы. (7) Too deep hierarchy — subagent вызывает subagent вызывает subagent → непредсказуемо и дорого. (8) Missing context — subagent не знает crucial constraint → возвращает нерелевантный результат.
Middle Какой depth limit для subagent иерархии?
Практическое ограничение: max 2 уровня (main → sub). Глубже: (1) Стоимость экспоненциально растёт. (2) Непредсказуемость — ошибки нижних уровней cascadе вверх с искажениями. (3) Debugging impossible — сложно понять почему что-то пошло не так на 4-м уровне. (4) Context fragmentation — каждый уровень теряет часть контекста. Исключение: иногда 3 уровня оправданы для очень специализированных задач. Если нужна глубокая иерархия → сигнал: декомпозируй задачу иначе, возможно нужна другая архитектура (pipeline, workflow), не рекурсивные subagents.
Senior — 5 вопросов
Senior Как проектировать multi-agent workflow для complex engineering задачи?
Multi-agent workflow design: (1) Task decomposition — разбить задачу на atomic, независимые subtasks. Dependency graph: что от чего зависит. Независимые → параллельно, зависимые → sequential. (2) Role specialization — assign roles: Code Explorer (readonly, ищет), Implementer (пишет код), Reviewer (проверяет), Security Analyst (ищет уязвимости). (3) Communication protocol — стандартный формат передачи findings между агентами. (4) Checkpoint per subagent — каждый subagent создаёт checkpoint при завершении. Main agent агрегирует. (5) Error handling — если subagent failed: retry, fallback, или escalate. (6) Result aggregation — main agent owned: merge, conflict resolution, final decision. (7) Cost budget — estimate tokens per subagent задо запуска, stay within budget.
Senior Как диагностировать coordination failures в multi-agent workflow?
Диагностика: (1) Result gaps — main agent ожидает X от subagent, получает Y → review task description: была ли задача однозначной? (2) Contradicting findings — два subagents возвращают противоречащие результаты → проверить scope: не было ли overlap или разного доступа к данным? (3) Missing evidence — subagent утверждает X но без конкретных references → task description не требовал evidence → исправить. (4) Timeout/hang — subagent завис → проверить: не слишком ли широкий scope? Бесконечный поиск без ограничений? (5) Rule violations — subagent предлагает нарушить Rule → он не знал о Rule → включать key constraints в task description. Debug: попросить subagent объяснить свой reasoning step-by-step.
Senior Как агрегировать результаты нескольких subagents и разрешать противоречия?
Result aggregation strategy: (1) Collect all findings — main agent читает все subagent results полностью перед aggregation. (2) Categorize — структурируй findings по категориям: confirmed facts, hypotheses, contradictions. (3) Contradiction resolution — если subagent A и B противоречат: смотреть их scope (разные части?), уровень evidence (у кого конкретнее), запросить tie-breaker subagent если критично. (4) Confidence weighting — findings с конкретными references (file:line) > findings без доказательств. (5) Unified summary — из всех findings создать единый structured summary. (6) Gap identification — что не было покрыто ни одним subagent → дополнительный targeted subagent или direct investigation.
Senior Как специализировать subagents — researcher, implementer, reviewer паттерн?
Specialized roles паттерн: (1) Researcher subagent (readonly) — ищет, анализирует, возвращает findings с evidence. Allowed tools: file read, search, grep. NO write access. (2) Implementer subagent — пишет код, создаёт файлы на основе spec. Gets: requirements + researcher findings. NO broader exploration. (3) Reviewer subagent — проверяет реализацию против requirements, Rules, best practices. Gets: implementation files + original requirements. NO changes, только feedback. (4) Security subagent — специализированный reviewer для security concerns. Реализация в Cursor: subagent_type определяет capabilities. Orchestration: main agent координирует всю цепочку, каждый subagent получает output предыдущего как input.
Senior Как контролировать затраты при использовании параллельных subagents?
Cost control strategies: (1) Estimate before launch — оценить приблизительный token budget каждого subagent перед запуском. Если общий > budget → пересмотреть архитектуру. (2) Scope constraints — узкий scope = меньше токенов. 'Проверь только api/handlers/' vs 'проверь весь backend'. (3) Max iterations — задать subagent max tool calls / max files to read. (4) Sequential fallback — если cost критичен: сначала попробовать одним агентом. Параллельные subagents только если есть бюджет. (5) Monitoring — логировать tokens per subagent, identify expensive subagents. (6) Result caching — если subagent задача не изменилась → можно переиспользовать previous findings. Budget gate: не запускать subagents если cost estimate > threshold без approval.
Lead — 5 вопросов
Lead Как выстроить governance использования subagents в команде?
Governance framework: (1) Use case policy — документировать approved use cases для subagents (code exploration, parallel review, security analysis) и НЕ approved (мелкие задачи, sequential dependent задачи). (2) Cost policy — budget per subagent task, approval для expensive workflows. (3) Output standards — стандарт что subagent должен возвращать: structured findings, конкретные references, confidence level. (4) Review requirement — critical subagent findings (security, architecture) → human review перед применением. (5) Audit trail — логировать subagent tasks: кто, что, когда, что вернул. (6) Training — инженеры знают когда и как использовать subagents. Governance document: 1 страница с do's, don'ts и approval process.
Lead Когда запрещать subagents с точки зрения safety и compliance?
Запрещать subagents когда: (1) Destructive actions без HITL — subagent не должен иметь право удалять файлы, деплоить в prod, выполнять irreversible operations без human approval gate. (2) PII обработка — subagent работающий с персональными данными требует audit log и data handling policy. (3) Cross-boundary access — subagent не должен выходить за пределы assigned scope (читать secret files, получать prod credentials). (4) Unverified external data — subagent не должен слепо доверять внешним источникам (prompt injection риск). (5) High-stakes decisions — финальные решения по архитектуре, security, compliance → только с explicit human review, не автономный subagent. Safety principle: least privilege — subagent получает ровно те права которые нужны для задачи.
Lead Как обучать команду эффективным subagent паттернам?
Training plan: (1) Foundations (45 мин) — что такое subagent, когда применять, demo хорошей task description vs плохой. (2) Anti-patterns workshop — разбор типичных ошибок: слишком широкий scope, нет output format, blind trust findings. (3) Template library — готовые task description шаблоны для частых use cases (code review, security scan, codebase exploration). (4) Hands-on — каждый инженер запускает subagent workflow для реальной задачи. (5) Code review criterion — при PR review с AI: проверить task description subagents. (6) Retrospective — при post-mortem инцидентов: был ли subagent использован правильно? Cost awareness: показать команде реальные costs subagent workflows.
Lead Как управлять costs при параллельных subagents в командной разработке?
Cost management: (1) Cost visibility — dashboard с AI spending per team, per engineer, per project. Subagent costs отдельной строкой. (2) Budget allocation — monthly budget per team для AI (включая subagents). Alerts при 80% использовании. (3) Approval gates — subagent workflows с estimated cost > $N → require tech lead approval. (4) Efficient patterns training — обучить команду cost-efficient паттернам: узкий scope, чёткий output format, избегать redundant subagents. (5) Cost attribution — каждый subagent task tagged с project/feature для attribution. (6) Review expensive workflows — weekly: top-5 самых дорогих subagent workflows → можно ли оптимизировать? Goal: максимальная value per token.
Lead Как выстроить approval gates для действий subagents в production?
Approval gates для subagent prod actions: (1) Risk classification — все действия subagents по risk: read (auto), write to non-prod (auto), write to prod или delete (require approval). (2) HITL integration — при высокорискованном действии subagent создаёт approval request: описание действия, ожидаемый результат, rollback plan. Human approves/rejects. (3) Dry-run first — subagent всегда сначала показывает план действий (dry-run) без выполнения. Human approves план → subagent выполняет. (4) Audit log — каждое approved/rejected действие logируется с: кто одобрил, время, контекст. (5) Emergency stop — механизм экстренной остановки всех active subagents. (6) Post-action verification — после выполнения → verification subagent проверяет результат.
Architect — 5 вопросов
Arch Какие архитектурные паттерны для multi-agent систем существуют?
Основные паттерны: (1) Hub-and-Spoke — один orchestrator (main agent) координирует множество специализированных subagents. Simple, но bottleneck на orchestrator. (2) Pipeline — output одного agent → input следующего. Sequential, хорошо для задач с зависимостями. (3) Hierarchical — multi-level: main → coordinators → workers. Для очень сложных задач. (4) Peer-to-Peer — агенты общаются между собой через shared state/message queue. Сложно, но гибко. (5) Competitive — несколько агентов независимо решают одну задачу, выбирается лучший результат (Best-of-N). (6) Consensus — agents vote on decision. Для critical decisions. Выбор паттерна: Hub-and-Spoke для большинства cases; Pipeline для sequential workflows; Competitive для высоких quality requirements.
Arch Как спроектировать платформу для orchestration subagents?
Orchestration platform components: (1) Agent Registry — каталог доступных agent types с их capabilities, tool access, cost parameters. (2) Task Queue — очередь задач для subagents с priority, deadline, dependencies. (3) Context Manager — управление context для каждого subagent: передача начального context, сбор results. (4) Communication Bus — как agents обмениваются данными: shared memory, message queue, event stream. (5) State Store — хранение checkpoint каждого agent, возможность resume. (6) Observability — distributed tracing, cost tracking, success metrics per agent type. (7) Policy Engine — что может делать каждый agent type, approval gates, rate limits. (8) Scheduler — когда и как запускать agents (parallel, sequential, conditional). Tech stack: task queue (Celery/BullMQ), state store (Redis/DynamoDB), messaging (Kafka/RabbitMQ).
Arch Как обеспечить fault tolerance в subagent системах?
Fault tolerance design: (1) Retry logic — transient failures (timeout, rate limit) → exponential backoff retry. Permanent failures → escalate. (2) Circuit breaker — если agent type часто падает → temporary disable, не retry бесконечно. (3) Fallback strategies — если primary subagent failed: (a) retry с другим agent instance; (b) fallback к более простому (дешёвому) agent; (c) graceful degradation — partial result + notice. (4) Timeout enforcement — каждый subagent task имеет deadline. По истечении → kill + checkpoint что успел. (5) Partial success handling — 3 из 4 subagents succeeded → можно продолжить с partial data, маркировав что missing. (6) Dead letter queue — failed tasks для manual review и recovery. (7) Health checks — периодический ping subagent instances.
Arch Как строить observability для multi-agent workflow?
Multi-agent observability stack: (1) Distributed tracing — единый trace_id через весь workflow: orchestrator → subagent → tool calls. OpenTelemetry spans. Visualize в Jaeger/Zipkin. (2) Structured logging — каждый agent action: structured JSON log с: trace_id, agent_type, task_id, action, duration, result_summary, cost_tokens. (3) Agent activity timeline — visual timeline: когда запущены subagents, сколько работали, что вернули. (4) Cost attribution — tokens per agent, per task, per team. Real-time и historical. (5) Quality metrics — subagent success rate, retry rate, findings quality score (если оценивается). (6) Anomaly detection — unusual patterns: agent работает слишком долго, excessive retries, cost spike. (7) Business metrics correlation — multi-agent workflow metrics → бизнес результаты (PR review time reduction, bug detection rate).
Arch Как проектировать communication protocols между subagents?
Inter-agent communication design: (1) Message schema — стандартизированный формат сообщений: header (sender, receiver, trace_id, timestamp, message_type) + payload (data). JSON Schema или Protobuf. (2) Communication patterns — request-response (subagent A запрашивает data у subagent B), pub-sub (events broadcast), shared state (чтение/запись в shared store). (3) Consistency model — для shared state: eventual consistency для high throughput или strong consistency для critical data. (4) Versioning — протокол версионируется, backward compat. (5) Security — authentication между agents (shared secret или short-lived tokens), authorization (что может читать/писать каждый agent). (6) Serialization — efficient serde для больших findings (binary для performance, JSON для debuggability). (7) Backpressure — если получатель перегружен → slow down sender. Design principle: prefer simple patterns (request-response) over complex (pub-sub) unless scale requires it.
