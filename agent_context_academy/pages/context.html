Формула для assessment

Сначала приоритет, потом конкретность, потом свежесть. System выше всех. Rules/AGENTS.md задают рамки. User request задаёт задачу. MCP/tool-result часто даёт самые свежие факты. Skill — способ выполнения, но не право нарушать правила.

Кратко: что такое context

Context — не только последний prompt. Это system instructions, AGENTS.md, Rules, чат, RAG, Memory, tool results и summaries в одном окне.

Самая короткая модель

Поведение: System, Rules, AGENTS.md.

Задача: current chat.

Знания: Memory, RAG, Checkpoints.

Действия: MCP / Tools.

Процесс: Skills.

Делегирование: Subagents.

Context window

  • В окно не помещается весь репозиторий — нужен RAG и @mention.
  • Длинный чат сжимается (compaction) — детали теряются.
  • Checkpoint фиксирует состояние задачи структурно.

Примеры контекста агента

Как выглядит реальный context window агента в разных сценариях.

Пример 1: Минимальный контекст (Junior задача)

System prompt: ~500 токенов (AGENTS.md + правила) User message: ~100 токенов ("Fix the bug in auth.py") Files in context: ~800 токенов (auth.py) ───────────────────────────────────── Total: ~1400 токенов (из 200k возможных)

Пример 2: Типичный рабочий контекст

System prompt: ~2000 токенов (AGENTS.md + Rules + Skills descriptions) Conversation: ~5000 токенов (история 10 turns) Tool results: ~8000 токенов (прочитанные файлы, grep результаты) RAG documents: ~3000 токенов (релевантные фрагменты) Current task: ~500 токенов ───────────────────────────────────── Total: ~18500 токенов (9% от 200k)

Пример 3: Контекст перед compaction

System prompt: ~2000 токенов Long conversation: ~80000 токенов (100+ turns, много файлов) Tool results: ~50000 токенов (накопленные) ───────────────────────────────────── Total: ~132000 токенов → приближается к лимиту → Автоматический compaction: summary заменяет историю

Тонкости и нюансы контекста

Lost in the middle — реальная проблема

LLM модели хорошо обрабатывают начало и конец context window, но информация в середине получает меньше attention. При 200K токенов контекста критический fact в позиции 100K может быть "проигнорирован". Именно поэтому важные инструкции кладут в начало (System/Rules), а не в середину длинного промпта.

Compaction уничтожает детали

При длинном чате Claude Code/Cursor автоматически сжимает историю (compaction). Детали теряются: конкретные имена переменных, точные error messages, решения принятые 10 сообщений назад. Защита: Checkpoint/Summary фиксирует состояние структурно ДО compaction. Без checkpoint агент "забывает" контекст задачи.

Весь репозиторий не помещается в context

Context window даже самых больших моделей (200K токенов) — это ~150K слов. Большой монорепозиторий может содержать миллионы токенов кода. Агент физически не может "видеть" всё. Поэтому нужны RAG (поиск релевантного кода) и @mention (явное добавление файла). Без них агент работает только с тем, что явно передали.

Каждый слой контекста потребляет токены

System prompt + Rules (alwaysApply) + AGENTS.md + Memory + текущий чат + RAG chunks + tool results — всё это накапливается. Каждый подключённый MCP сервер добавляет descriptions tools. 10 Rules по 500 токенов = 5000 токенов overhead на каждый запрос. Оптимизируй: только нужные Rules, компактная Memory, ограниченный RAG top-k.

Приоритеты слоёв — не абсолютны

Иерархия (System > Rules > AGENTS.md > Chat > Memory) описывает типичное поведение, но LLM может отклоняться при противоречиях. Если пользователь в chat явно просит нарушить правило из Rules — агент столкнётся с конфликтом. В Cursor: agentRespondedWithHarmful → safety system. Hooks в Claude Code устраняют этот риск для критических проверок на OS-уровне.

Best practices — Делай / Не делай

✅ Делай

  • Клади критичные инструкции в начало контекста (System/Rules), не в середину
  • Используй Checkpoint перед длинными задачами — фиксируй прогресс
  • @mention конкретные файлы вместо надежды что агент найдёт их сам
  • Ограничивай RAG top-k до 3-7 chunks для чистоты context
  • Используй Memory для устойчивых фактов, не засоряй чат повторениями
  • Явно разделяй инструкции и данные (retrieved docs = данные, не инструкции)

❌ Не делай

  • Не клади весь проект в context — выбирай через RAG и @mention
  • Не игнорируй эффект compaction — чекпойнтируй длинные задачи
  • Не добавляй все Rules как alwaysApply — только критичные
  • Не полагайся на агента "помнящего" детали из 30+ сообщений назад
  • Не подключай все MCP серверы — каждый добавляет token overhead
  • Не путай уровни: Chat prompt не отменяет System-level правила

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое context window агента?
Context window — это максимальный объём текста (в токенах), который языковая модель может «видеть» и учитывать при генерации ответа. Всё, что находится вне context window, для агента просто не существует. Современные модели имеют окна от 128K (GPT-4) до 200K (Claude Sonnet) и 1M+ токенов (Gemini). Чем больше окно, тем больше информации агент может учесть за один раз.
Junior Что входит в context window агента?
В context window одновременно помещаются: System Prompt платформы (Cursor/Claude Code), пользовательские Rules (alwaysApply), AGENTS.md/CLAUDE.md, вся история текущего чата, результаты tool calls, RAG chunks (найденные по смыслу фрагменты), Memory Bank (если подключён), Checkpoint/Summary и текущий запрос пользователя. Все эти слои конкурируют за ограниченное количество токенов.
Junior Что такое System Prompt?
System Prompt — это специальное сообщение, которое платформа (Cursor, Claude Code, claude.ai) отправляет модели перед началом диалога. Он содержит инструкции по поведению, описания доступных инструментов, safety guidelines. Пользователь не может напрямую редактировать System Prompt — это привилегия платформы. Rules (alwaysApply) добавляются к System Prompt как дополнительный слой, который уже контролируется пользователем.
Junior Что такое compaction и когда он срабатывает?
Compaction — автоматическое сжатие истории чата, которое срабатывает, когда context window приближается к лимиту токенов. Платформа заменяет ранние сообщения их кратким резюме. Детали при этом теряются: точные имена переменных, конкретные error messages, нюансы принятых решений. Именно поэтому нужен Checkpoint — структурированная фиксация прогресса до того, как произойдёт compaction.
Junior Что такое @mention файла?
@mention — явное включение конкретного файла целиком в context window. Когда вы пишете @filename.py, весь файл добавляется в context, и агент гарантированно его видит. Это детерминированный способ включения данных — в отличие от RAG, где поиск вероятностный. Каждый @mention занимает токены пропорционально размеру файла.
Junior Что такое alwaysApply в Rules?
alwaysApply: true — настройка в .mdc файле правила, которая заставляет правило загружаться в каждый запрос агента, независимо от контекста работы. Такие правила добавляются к System Prompt при каждом запросе. Используется для глобальных стандартов: язык ответа, запреты на определённые действия, стиль кода. Auto-attached правила загружаются только при работе с определёнными файлами по glob-паттернам.
Junior Как RAG добавляет данные в context?
RAG (Retrieval-Augmented Generation) работает в три шага: 1) Запрос пользователя превращается в embedding-вектор; 2) По этому вектору в индексированном corpus ищутся похожие фрагменты (chunks); 3) Найденные top-k chunks добавляются в context window перед запросом к модели. Агент «видит» эти chunks как часть context и может использовать их для ответа. RAG экономит токены по сравнению с @mention всего репозитория.
Junior Что такое token limit?
Token limit — максимальный размер context window в токенах для конкретной модели. Один токен ≈ 4 символа на английском, 2-3 символа на русском. GPT-4: 128K, Claude Sonnet: 200K, Gemini 1.5 Pro: 1M+. Агент физически не может учесть информацию, которая не поместилась в лимит. Для больших баз знаний используются RAG и subagents, чтобы работать в рамках ограничений.
Junior Что такое .cursorignore?
.cursorignore — файл, аналогичный .gitignore, но для AI-индексации. Паттерны в нём исключают файлы и директории из RAG-индекса Cursor. Файл может быть в git, но исключён из AI-поиска. Типичные кандидаты: build артефакты (dist/, target/), generated код, large binary files, migration scripts. В Claude Code аналог — .claudeignore.
Junior Как Memory Bank влияет на context?
Memory Bank — это набор markdown-файлов (например, memory.md), которые агент загружает в начало каждой сессии как постоянный контекст. В отличие от истории чата, Memory Bank не теряется при compaction — это файлы на диске. Содержит: стек технологий, принятые архитектурные решения, текущий статус проекта. Ограничение: всё содержимое занимает токены, поэтому Memory Bank должен быть компактным (≤800 слов).
Middle — 10 вопросов
Middle Чем @mention файла отличается от RAG?
@mention — явное детерминированное включение файла целиком. Агент гарантированно видит весь файл, но это дорого по токенам. RAG — вероятностный семантический поиск: возвращает chunks похожие по смыслу на запрос, экономит токены, но может вернуть нерелевантный фрагмент. Правило выбора: знаешь нужный файл → @mention; ищешь по смыслу в большом corpus → RAG. Комбинирование: RAG находит файл, @mention добавляет полностью если нужен весь контекст.
Middle Когда использовать alwaysApply vs auto-attached правила?
alwaysApply: true — только для абсолютно универсальных правил, которые нужны при ЛЮБОЙ задаче: язык ответа, запрет destructive операций, базовый стиль коммуникации. Максимум 3-5 таких правил, иначе overhead растёт и lost-in-the-middle ухудшается. Auto-attached (по glob-паттернам) — для правил специфичных для типа файлов: правила для *.py только при работе с Python, правила для *.sql только при SQL-задачах. Это экономит токены и фокусирует агента.
Middle Как избежать token overflow в длинных задачах?
Стратегии: 1) Checkpoint до compaction — фиксируй прогресс в файл, не в чат; 2) Новый чат + @checkpoint.md вместо бесконечного продолжения; 3) Subagent для изоляции — пусть читает 100 файлов, а main agent получает только summary; 4) @mention только нужных файлов, не директорий; 5) Компактный AGENTS.md (≤800 слов); 6) alwaysApply только для критичных правил; 7) RAG вместо @mention всего репозитория.
Middle Что такое «lost in the middle» и как это влияет на работу с context?
LLM attention mechanism неравномерно обрабатывает позиции в sequence: начало (primacy effect) и конец (recency effect) получают максимум attention, середина — значительно меньше. При 200K context ключевой факт на позиции 100K может быть фактически проигнорирован. Следствие: критические инструкции помещай в Rules (начало), Few-shot примеры — ближе к концу перед запросом. Длинный список RAG chunks: лучший chunk должен быть первым (reranking).
Middle Как строить context для длинной многоэтапной задачи?
Паттерн multi-session context engineering: 1) Разбей задачу на этапы с явными Checkpoint'ами между ними; 2) Checkpoint пиши в файл (checkpoint.md), не в чат — файл переживёт compaction; 3) Каждый новый этап начинай свежим чатом с @checkpoint.md; 4) Memory Bank содержит глобальное состояние (стек, решения), Checkpoint — текущее состояние этапа; 5) Subagents для параллельных независимых частей. Большие задачи = pipeline of sessions, не один бесконечный чат.
Middle Как compaction влияет на принятые ранее решения?
При compaction ранние сообщения заменяются summary. Конкретные решения («выбрали Redis вместо Memcached, потому что нужен TTL и persistence») превращаются в общие факты («использован Redis»). Причина выбора теряется. Агент может в дальнейшем предлагать Memcached, «забыв» trade-off. Защита: ключевые архитектурные решения с их причинами фиксируй в AGENTS.md или Memory Bank — они загружаются каждый запрос и не подвержены compaction.
Middle Каковы приоритеты слоёв context при конфликте?
Порядок приоритетов (от высшего к низшему): Platform System Prompt (safety, core capabilities) → Team/Project Rules (alwaysApply) → AGENTS.md/CLAUDE.md → Auto-attached Rules (по glob) → Memory Bank → RAG chunks → Tool call results → История чата → Текущий запрос. При конфликте: вышестоящий слой побеждает. Rules «не использовать библиотеку X» имеет приоритет над просьбой в чате «используй X».
Middle Как оптимизировать token usage в agentic workflow?
Ключевые техники: 1) Stable prefix для кеширования (System Prompt + Rules стабильны → cache hit rate >50%); 2) AGENTS.md ≤800 слов, без устаревших фактов; 3) alwaysApply максимум 3-5 правил; 4) RAG top-k ≤ 5-7 chunks; 5) @mention только конкретные нужные файлы; 6) .cursorignore для build артефактов и generated code; 7) Subagents для изоляции token-heavy операций (чтение 100 файлов); 8) Checkpoint как файл вместо длинной истории.
Middle Как Checkpoint помогает при compaction?
Checkpoint — структурированный снимок задачи: что сделано, какие решения приняты (с причинами), что осталось, на чём остановились. Записывается в файл (не в чат), поэтому compaction его не затрагивает. При следующей сессии: новый чат + @checkpoint.md в первом сообщении — агент мгновенно получает полный контекст задачи без длинной истории. Checkpoint как файл > Checkpoint как сообщение в чате.
Middle Когда использовать subagents для изоляции context?
Subagent оправдан когда: 1) Задача требует чтения 50+ файлов — context overflow для main agent; 2) Нужен параллельный анализ независимых частей (subagents работают параллельно); 3) Хочешь изолировать «шум» — subagent погружается в детали, main видит только summary; 4) Sensitive операции в изолированном context. Coordination overhead есть (передача задачи, получение результата), поэтому subagent оправдан когда isolation benefit > overhead.
Senior — 5 вопросов
Senior Как спроектировать многоуровневый context для сложного agentic системы?
Архитектурный подход: разделить context на слои с чёткой ответственностью. Слой 1 — Stable Core: System Prompt + alwaysApply Rules + AGENTS.md (кешируется, не меняется между запросами). Слой 2 — Task Context: Checkpoint + Memory Bank (обновляется при смене этапа). Слой 3 — Dynamic: RAG chunks + @mention для текущего запроса. Принципы: maximize cache hit rate для Слоя 1; minimize token usage для Слоя 2; maximize relevance для Слоя 3. Subagents получают только Слой 3 (задачу) + минимальный Слой 1 (правила). Main agent не передаёт весь Слой 2 — только distilled summary.
Senior Как тюнинговать RAG для кода vs документации?
Код и документация требуют разных стратегий chunking и retrieval. Для кода: chunk по функциям/классам (не по строкам), включать signature + docstring + первые строки реализации в chunk, использовать code-specific embeddings (CodeBERT, code-specific fine-tuned models). Для документации: semantic chunking по параграфам, иерархический индекс (заголовки → разделы → параграфы), freshness metadata с датой последнего обновления. Общее: reranking после retrieval — best chunk первым (primacy effect). Top-k ≤ 5 для кода (нужна точность), ≤ 7 для docs (нужна полнота).
Senior Как диагностировать context saturation в production?
Context saturation проявляется как: агент игнорирует инструкции, даёт поверхностные ответы, «забывает» ранее обсуждённые детали, повторяет уже сделанное. Диагностика: 1) Логируй usage.input_tokens при каждом запросе — при приближении к лимиту → compaction скоро; 2) Отслеживай cache_read_input_tokens / total_tokens — падение cache hit rate сигнализирует о нестабильном prefix; 3) Token composition audit: сколько токенов занимает каждый слой? Нет ли раздутого AGENTS.md или лишних alwaysApply правил? 4) Behavioral consistency test: один запрос 10 раз — насколько похожи ответы?
Senior Как оптимизировать layout context под attention механизм LLM?
Attention в transformer — не равномерный: primacy (начало) и recency (конец) эффекты реальны и измеримы. Оптимальный layout (сверху вниз): 1) Stable prefix: System + Rules (кешируется + максимум attention при примость); 2) AGENTS.md + Memory Bank (важные факты — близко к Rules для attention); 3) RAG chunks: лучший chunk ПЕРВЫМ после reranking; 4) Checkpoint (task state — близко к запросу); 5) Текущий запрос (конец, recency effect). Избегай: длинные промежуточные блоки между Rules и запросом — это «середина». Few-shot примеры: ближе к концу, не к началу.
Senior Как проектировать систему для 200K token window?
При 200K window стратегия меняется, но принципы остаются: 1) Не «бросать всё в окно» — lost in the middle работает и при 200K, только масштаб другой. Факт на позиции 100K всё равно получает меньше attention; 2) Пересмотреть что раньше шло через RAG только из-за лимита — некоторые docs выгоднее включить прямо (@mention); 3) Кеш: 200K stable prefix дороже кешировать, но при высоком hit rate экономия огромная; 4) Long-context reranking: при 200K особенно важно лучшие chunks в начало, худшие — убрать; 5) Subagents всё равно полезны для параллелизма, не только для лимита.
Lead — 5 вопросов
Lead Как выстроить стандарты context management для команды разработчиков?
Практический framework: 1) Документируй в AGENTS.md что агент «видит» и что «не видит» (.cursorignore); 2) Создай team Rules library — shared .mdc файлы в репозитории с алгоритмом: что alwaysApply, что auto-attached; 3) Установи лимиты: AGENTS.md ≤800 слов, alwaysApply ≤5 правил, Memory Bank ≤800 слов — это SLA для context overhead; 4) Checkpoint conventions: стандартный формат (что сделано / решения / следующий шаг) — template в AGENTS.md; 5) Quarterly Rules audit — каждый .mdc файл проверяется на актуальность. Rules Debt опаснее, чем их отсутствие.
Lead Как обучить junior разработчиков минимизировать потребление токенов?
Обучение через практику: 1) Покажи token composition на реальном запросе — сколько токенов занимает каждый слой (Cursor показывает в debug); 2) Упражнение «@mention диета»: ограничь на 1 неделю использование @mention до 3 файлов на запрос — учит думать что реально нужно; 3) Code review context: добавь чеклист «проверь что @mention только нужные файлы»; 4) Покажи разницу: запрос с 10 @mention vs 2 @mention + RAG — что агент отвечает лучше; 5) Обучи Checkpoint habit — создай team template, требуй Checkpoint при задачах >30 мин.
Lead Как организовать аудит context размера в CI/CD?
CI-интеграция context audit: 1) Скрипт проверки AGENTS.md — word count, wc -w agents.md, fail если >1000 слов; 2) Rules audit: count alwaysApply rules, warn если >5, fail если >10; 3) .cursorignore completeness check: build/, dist/, node_modules/ должны быть в ignore; 4) Тест на cache stability: запустить один и тот же prompt 3 раза, сравнить cache_read_input_tokens — должно быть >50% hit rate; 5) Memory Bank size check аналогично AGENTS.md. Эти checks быстрые (секунды) и предотвращают незаметное накопление context bloat со временем.
Lead Какие conventions по Rules и AGENTS.md нужны при onboarding нового разработчика?
Onboarding context conventions: 1) AGENTS.md — первый файл для чтения, должен объяснять что агент видит/не видит, ключевые Rules, примеры Checkpoint; 2) Guided tour: покажи как читать .cursor/rules/, объясни alwaysApply vs auto-attached разницу; 3) Practice задача: junior должен создать первый Checkpoint самостоятельно по template; 4) Shadow session: разработчик наблюдает за senior при работе с агентом — видит как выбирают @mention, когда используют RAG; 5) Rules ownership: каждый .mdc файл имеет OWNER и LAST_REVIEWED — при onboarding назначают владельца. Цель: за 1 день junior знает как не «засорять» context.
Lead Как выстроить governance контроля context в команде?
Governance framework: 1) Context Policy Document: что можно включать в context (типы данных, классификация), что нельзя (PII, secrets); 2) Rules change process: изменения alwaysApply Rules → PR review с 2 approvers (влияют на всех); 3) AGENTS.md ownership: team lead review при обновлении; 4) Quarterly context audit — все .mdc файлы, AGENTS.md, Memory Bank проверяются командой; 5) Incident process: если агент повёл себя неожиданно из-за context — root cause в context layers, fix в Rules/Hooks; 6) Метрики: average context size per request, cache hit rate, compaction frequency — в team dashboard.
Architect — 5 вопросов
Arch Как спроектировать context architecture для multi-agent системы?
Multi-agent context architecture строится на принципе isolation + shared state: 1) Shared State Layer: AGENTS.md/Memory Bank доступен всем агентам read-only; только main agent или orchestrator имеет write access (предотвращает race conditions); 2) Agent-specific context: каждый subagent получает только свою задачу + минимальный глобальный контекст; 3) Communication protocol: агенты общаются через structured output (JSON), не через свободный текст — это детерминирует передачу данных; 4) Context versioning: при параллельной работе — optimistic locking на shared files (проверяй версию до записи); 5) Observability: логируй context composition каждого агента для debugging variance.
Arch Как проектировать retrieval pipeline с учётом context constraints?
Production RAG pipeline для agentic систем: 1) Инgest: semantic chunking (не fixed size), code-aware splitting для кода, freshness metadata (дата, версия, source); 2) Retrieval: hybrid search (dense embeddings + BM25 keyword), query expansion для лучшего recall; 3) Reranking: cross-encoder для точного ранжирования, best chunk → position 0 в context (primacy effect); 4) Context assembly: top-k выбирается динамически на основе доступного budget (не фиксированное 5 всегда); 5) Freshness gate: chunk старше N дней → помечается stale, альтернативно → MCP live tool; 6) Evaluation: NDCG, MRR для retrieval качества в CI при каждом изменении corpus.
Arch Как рассматривать context window как архитектурный ресурс?
Context window — это finite shared resource, аналог памяти или CPU. Архитектурные принципы: 1) Budget allocation: явно определи квоты токенов для каждого слоя (Rules: 2K, AGENTS.md: 3K, RAG: 5K, chat: 10K); 2) Priority preemption: при нехватке — исключать слои от низшего приоритета к высшему; 3) Caching economics: stable prefix кешируется → амортизирует cost; нестабильный prefix → каждый запрос платит полную цену; 4) Capacity planning: при росте команды Rules и AGENTS.md растут → нужно проектировать «сжатие» заранее; 5) SLA: max context size constraint в архитектуре системы, нарушение → degraded mode (subagent isolation включается автоматически).
Arch Как реализовать кэширование context layers для снижения стоимости?
Prompt caching (Anthropic, OpenAI) работает на prefix matching: byte-for-byte совпадение кешируется. Архитектурная реализация: 1) Stable layer первым: System Prompt → Rules → AGENTS.md → Memory Bank (неизменны между запросами); 2) Dynamic layer в конце: RAG chunks, user query (меняются каждый раз); 3) Explicit cache breakpoints (Anthropic API): cache_control markers после stable части; 4) Rules ordering: алфавитный или по важности — главное, чтобы порядок не менялся; 5) Мониторинг: cache_read_input_tokens / total_tokens → target >60%; при падении — найди что изменилось в stable layer; 6) Versioned snapshots: при изменении AGENTS.md — invalidate cache намеренно.
Arch Что такое context isolation pattern и когда его применять?
Context isolation — архитектурный паттерн, при котором разные агенты или части системы работают в изолированных context windows, не разделяя состояние напрямую. Применение: 1) Security isolation: subagent для работы с sensitive данными не получает глобальный context (ограничение blast radius при prompt injection); 2) Noise isolation: агент-исследователь читает 200 файлов в своём окне, main agent получает distilled summary — его context чист; 3) Parallel execution: независимые subagents работают параллельно в изолированных контекстах, нет конкурентного доступа; 4) Failure isolation: сбой subagent не «загрязняет» context main agent; 5) Anti-pattern: isolation без structured communication → черный ящик, неотлаживаемый.
