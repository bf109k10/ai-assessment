Какой инструмент когда

Memory Bank, AGENTS.md, Rules, Skills, Hooks — пять инструментов управления поведением агента. Таблицы best practices: что куда класть, когда что выбирать, почему путаница стоит дорого.

Таблица решений По ситуациям

Почему важно выбирать правильный инструмент

У каждого инструмента свой тип содержимого, lifetime и механизм применения. Положишь факт о проекте в Rules вместо Memory — агент будет его получать каждый раз независимо от задачи, засоряя context. Положишь обязательный workflow в Memory вместо AGENTS.md — он может не загрузиться в Cursor без явного @include.

Главная ошибка: все пять инструментов выглядят как «напиши текст и агент это знает». Но они отличаются scope, приоритетом, lifetime, гарантиями загрузки и тем, кто их читает — агент или платформа.

🧠 Memory Bank (memory.md)

Что это: накопленные факты о проекте — решения, ADR, стек, глоссарий. Меняется со временем.

Для чего: не объяснять стек и историю решений заново в каждой сессии.

📋 AGENTS.md / CLAUDE.md

Что это: «контракт» агента — как работать здесь, роль, workflow, definition of done.

Для чего: агент всегда следует одному и тому же рабочему процессу в этом репо.

🔒 Rules (.cursor/rules/)

Что это: постоянные инструкции поведения — стайлгайд, запреты, coding conventions.

Для чего: агент ВСЕГДА или при определённых условиях следует конкретным правилам.

⚡ Skills

Что это: переиспользуемые процедуры — «как выполнить задачу X по шагам».

Для чего: стандартизировать повторяющиеся workflow, которые агент вызывает по запросу.

🪝 Hooks (Claude Code)

Что это: OS-уровень перехватчики событий агента — до/после каждого действия.

Для чего: гарантированное enforcement правил безопасности — агент не может обойти.

Memory Bank vs AGENTS.md — детальное сравнение

Оба файла попадают в context window при каждом запросе. Путаница между ними — главная причина дублей и устаревших команд. AGENTS.md (в Claude Code — CLAUDE.md) = «как работать здесь»; memory.md = «что мы накопили о проекте».

Rules vs Skills — когда что

Оба инструмента задают поведение агента. Rules — ограничения и стиль. Skills — процедуры и workflow. Ключевой вопрос: «это правило (как не надо) или процедура (как делать)?»

Rules — ограничения, конвенции, стиль

Применяются автоматически: alwaysApply или по glob/trigger. Агент не «вызывает» Rule — он просто следует ему.

  • Coding conventions (no var, camelCase)
  • Запреты: «не делать X без approval»
  • Стиль ответов: тон, язык, формат
  • File-specific: инструкции для *.test.ts
  • Security constraints: никаких секретов в коде

Skills — процедуры, workflow, алгоритмы

Вызываются по команде (/skill-name) или при совпадении trigger-фразы. Агент читает Skill как пошаговую инструкцию.

  • Deploy workflow: «как деплоить в staging»
  • PR creation: пошаговый процесс с чеклистом
  • Migration: алгоритм с ветвлением (if/else)
  • Code review: «как ревьюить по стандартам»
  • Onboarding script: «как настроить локальную среду»
ВопросRulesSkills

Правило большого пальца: если содержит слово «никогда/всегда/обязательно» — это Rule. Если содержит нумерованный список шагов — это Skill. Если и то, и другое — разбей на Rule (ограничение) + Skill (процедура).

Когда добавлять Hooks (Claude Code)

Hooks — единственный инструмент с гарантированным enforcement. Rules и AGENTS.md — это «агент должен следовать». Hooks — «платформа гарантирует выполнение на OS-уровне, агент не может обойти».

🪝 Hooks — когда не хватает Rules

Добавляй Hooks когда нарушение правила стоит дорого и нельзя доверять только тексту Rules.

  • Критические security запреты: «никогда не коммить .env» → PreToolCall hook на git commit
  • Обязательные проверки: запустить linter перед каждым сохранением файла
  • Аудит лог: логировать все tool calls для compliance
  • Approval gate: требовать подтверждение перед destructive действием (drop table, rm -rf)
  • Форматирование: автоформатирование кода после каждого изменения файла
КритерийRules достаточноНужны Hooks
Последствие нарушенияНеудобство, стиль не соблюдёнSecurity breach, потеря данных, compliance violation
Частота проверкиАгент помнит правило в сессииКаждый tool call, каждое сохранение — без исключений
Можно ли «попросить агента нарушить»?Технически да (User override)Нет — Hook выполнится независимо от сообщения
Пример«Пиши комментарии на английском»«Не коммить файлы с секретами»

Rules ≠ Hooks по надёжности

Rule в .cursor/rules/ — это текст в context window. LLM «пытается» следовать. Если Rule попал в середину длинного context (lost in the middle), вес снижается. Hooks — это код который выполняется платформой. Для критических ограничений: Rule + Hook (двойная защита), только Rule недостаточно.

Ответственность: кто управляет каждым инструментом

В команде важно договориться кто отвечает за каждый файл. Иначе — конфликты, дубли, устаревший контент.

ИнструментКто создаёт/поддерживаетКогда обновлятьРиск при игнорировании
AGENTS.md Tech Lead / Architect При смене workflow, процесса PR, onboarding Агент следует устаревшему процессу, пропускает DoD
Memory Bank Вся команда (каждый добавляет знания) После каждого ADR, Gotcha, sprint retrospective Агент предлагает отвергнутые решения, не знает истории
Rules Tech Lead + разработчики (по модулям) При смене coding standards, новые запреты Inconsistency в коде, security drift
Skills Разработчики (тот кто знает workflow) При смене процесса deploy/PR/migration Агент выполняет устаревший workflow, ломает деплои
Hooks Tech Lead / Security engineer При новых security требованиях Критические запреты не соблюдаются на OS-уровне

Тонкости и нюансы

Дублирование = главный враг

Один и тот же контент в AGENTS.md и в memory.md = дубль. При расхождении агент не знает чему верить. Правило: каждый факт живёт ровно в одном месте. AGENTS.md ссылается на memory.md, не копирует. Регулярный аудит: поищи одинаковые разделы — удали из менее подходящего файла.

Memory Bank не загружается автоматически в Cursor без @include

В Claude Code — memory.md из .claude/memory/ подхватывается автоматически. В Cursor — нужна явная строка в AGENTS.md: @docs/memory.md или через Rules alwaysApply. Без этого Memory Bank просто не попадёт в context. Проверь: агент видит твой memory.md, попросив его процитировать что-то оттуда.

Rules — не место для фактов о проекте

Частая ошибка: кладут в Rules «наш стек — Java 21, Maven» или «мы используем PostgreSQL». Rules — для поведения (как делать), не для фактов (что есть). Факты в Rules: (1) засоряют Rules файл; (2) не обновляются при смене стека; (3) смешивают поведение с контентом. Факты → memory.md. Конвенции → Rules.

Skill vs AGENTS.md workflow — разница в granularity

AGENTS.md workflow: «для PR — следуй PR checklist в SKILL.md». Skill PR-checklist: подробные шаги, ветвления, команды. AGENTS.md = «что делать» (высокоуровнево). Skill = «как именно» (детально). Если детали workflow растут — выноси в Skill, в AGENTS.md оставь ссылку. Иначе AGENTS.md раздувается и агент теряет focus.

Размер context window — чем меньше, тем лучше работает каждый файл

Memory Bank, AGENTS.md, Rules — все попадают в context window одновременно. Суммарный объём влияет на качество: чем больше нерелевантного контента, тем хуже attention на важном. Принцип: AGENTS.md ≤ 500 строк, memory.md ≤ 300 строк на секцию, Rules — конкретные и короткие. Альтернатива для большого corpus — RAG (Cursor @Docs) вместо прямой загрузки.

Best practices — Делай / Не делай

✅ Делай

  • Держи AGENTS.md кратким (<500 строк) — только роль, workflow, DoD
  • Факты о проекте — в memory.md, не в AGENTS.md
  • Конвенции и запреты — в Rules, не в Memory
  • Для повторяющихся процедур — Skills, не Rules
  • Критические запреты — Rule + Hook (двойная защита)
  • В Cursor: добавь @docs/memory.md в AGENTS.md явно
  • Регулярный аудит: проверяй дубли между файлами
  • Каждый инструмент — одна ответственность (SRP)

❌ Не делай

  • Не клади факты о проекте в Rules — они для поведения
  • Не дублируй контент между AGENTS.md и memory.md
  • Не полагайся на Rules для критических запретов без Hooks
  • Не раздувай memory.md без структуры — агент теряет важное
  • Не смешивай процедуры (Skill) и запреты (Rule) в одном файле
  • Не забывай обновлять Skills при смене workflow
  • Не используй Memory для живых фактов — для них MCP/RAG

Вопросы по уровням

Junior — 10 вопросов
Junior В чём главное отличие AGENTS.md от memory.md?
AGENTS.md = «как работать здесь» — роль агента, workflow, definition of done. Это процессный контракт, меняется редко. memory.md = «что мы накопили о проекте» — факты, ADR, история решений, глоссарий. Это живые накопленные знания, растёт со временем. Аналогия: AGENTS.md — это должностная инструкция сотрудника, memory.md — его рабочий дневник и база знаний. Если ты пишешь «как делать» — AGENTS.md. Если пишешь «что мы знаем» — memory.md.
Junior Что такое Rules и для чего они нужны?
Rules — это постоянные инструкции поведения агента. Хранятся в .cursor/rules/*.mdc (Cursor) или как секции в CLAUDE.md (Claude Code). Применяются автоматически: alwaysApply — при каждом запросе; glob — при открытии файлов по паттерну (например *.test.ts). Типичные примеры: coding conventions (camelCase, no var), запреты (не коммить секреты), стиль ответов (отвечай по-русски). Главный принцип: Rules — для ограничений и конвенций, не для процедур и не для фактов.
Junior Что такое Skills и чем они отличаются от Rules?
Skills — переиспользуемые пошаговые процедуры. Агент «вызывает» Skill по команде (/deploy) или trigger-фразе — и выполняет шаги из SKILL.md файла. Rules применяются автоматически и постоянно. Skills применяются только когда вызваны — экономят context. Rules описывают «чего не делать» или «как писать». Skills описывают «как выполнить задачу шаг за шагом». Пример Skills: «как создать PR», «как задеплоить в staging», «как делать code review».
Junior Зачем нужен Memory Bank если есть AGENTS.md?
AGENTS.md — статичный контракт, не предназначен для роста. Если класть в него всю историю проекта (ADR, Gotchas, решения) — он раздуется и агент будет работать с огромным context при каждом запросе. Memory Bank (memory.md) — отдельный файл для накопленных знаний. Разделение ответственности: AGENTS.md остаётся компактным (workflow + DoD), memory.md растёт с проектом (факты + история). Ещё: memory.md можно не включать в маленьких задачах, AGENTS.md — всегда.
Junior Что такое Hooks и когда они нужны?
Hooks (Claude Code) — перехватчики событий на OS-уровне. Срабатывают до (PreToolCall) или после (PostToolCall) действий агента — независимо от содержимого context window. Нужны когда Rules недостаточно: Rules — это текст, LLM «старается следовать» но может не заметить (lost in the middle). Hooks — это код, платформа гарантирует выполнение. Используй Hooks для: не коммить секреты, не выполнять rm -rf без approval, запускать linter после каждого сохранения, логировать все tool calls.
Junior Куда класть информацию «почему мы отвергли Redis»?
В Memory Bank (memory.md). Это архитектурное решение (ADR) с датой и обоснованием — накопленное знание о проекте. Именно для этого существует memory.md. Не в AGENTS.md (это не процесс), не в Rules (это не запрет поведения), не в Skills (это не процедура). Формат для memory.md: ## ADR: отказ от Redis (Q1 2025) — коротко, почему, что выбрали вместо. Агент, читая это, не будет предлагать Redis в будущих сессиях.
Junior Куда класть coding conventions (camelCase, no var)?
В Rules. Coding conventions — это постоянные инструкции поведения агента при написании кода. Rules идеальны: поддерживают glob (применяются только для *.ts файлов), поддерживают alwaysApply. Не в AGENTS.md — он для workflow, не конвенций. Не в memory.md — это не факт, а правило поведения. Пример Rule в .cursor/rules/typescript.mdc: - Always use const/let, never var, - Variable names: camelCase. Glob: **/*.ts, **/*.tsx.
Junior Нужно ли создавать все пять инструментов для каждого проекта?
Нет. Начни минимально: AGENTS.md (обязателен) + память только когда накопится что записывать. Добавляй инструменты по мере необходимости: Rules — когда команда хочет enforce coding standards; Skills — когда есть повторяющийся workflow; Memory — когда накопилось достаточно фактов; Hooks — когда нужны security гарантии. Для pet-проекта может хватить одного AGENTS.md. Для большой команды — все пять. Принцип: каждый инструмент добавляется когда появляется конкретная проблема которую он решает.
Junior Что будет если положить workflow в Memory вместо AGENTS.md?
В Cursor — workflow может вообще не загрузиться если memory.md не подключён явно через @include. Даже если загрузится — AGENTS.md читается агентом как основной контракт, memory.md как дополнительные знания. Workflow в memory «конкурирует» с другими фактами за attention. Риск: агент не выполнит DoD или пропустит шаги PR process. Правило: процессные требования, которые должны выполняться ВСЕГДА — в AGENTS.md. Знания которые полезны «иногда» — в memory.md.
Junior Где хранится memory.md в Cursor и в Claude Code?
В Cursor: обычно docs/memory.md или .cursor/memory.md — произвольное расположение, главное указать путь в AGENTS.md через @include. Claude Code: .claude/memory/ — стандартная директория, файлы оттуда загружаются автоматически без явного подключения. Три уровня: ~/.claude/memory/ (global — для всех проектов), .claude/memory/ (project — текущий проект), создаваемые в сессии (session memory). В Cursor иерархии нет — один файл, подключается явно.
Middle — 10 вопросов
Middle Как избежать дублирования между AGENTS.md и memory.md?
Правило одного источника истины: каждый факт и каждая инструкция живут ровно в одном файле. AGENTS.md ссылается на memory.md (строка @include), не копирует оттуда данные. Стек: в AGENTS.md — только текущий (Java 21, Maven) как шпаргалка для старта. История изменений стека — в memory.md. Если одинаковый раздел появился в обоих — удали из менее подходящего. Регулярный аудит: раз в квартал проверяй оба файла на дубли. Сигнал проблемы: агент цитирует разные версии одного факта в разных сессиях.
Middle Когда использовать glob в Rules vs alwaysApply?
alwaysApply: для инструкций которые нужны при КАЖДОМ запросе — стиль ответов, язык общения, общие запреты. Glob: для инструкций которые нужны только при работе с определёнными файлами — TypeScript conventions только для *.ts, SQL formatting только для *.sql, migration rules для db/migrations/**. Практика: сначала спроси «нужно ли это при каждом запросе?». Если нет — glob. alwaysApply засоряет context всегда, glob — только когда нужно. Ошибка: класть file-specific конвенции в alwaysApply — агент получает TypeScript правила когда пишет Python.
Middle Как структурировать Memory Bank чтобы агент не терял важное?
Структура — критична для attention агента. Рекомендованные секции: ## Stack (текущий стек), ## Architecture Decisions (ADR с датами), ## Gotchas (баги и ловушки), ## Glossary (доменные термины), ## Sprint outcomes (выводы ретроспектив). Важные факты — в начало файла (lost in the middle: середина читается хуже). Каждый ADR: дата, что решили, почему, что НЕ использовать. Ограничение размера: каждая секция ≤ 50 строк. При росте — разбить на несколько файлов: memory-stack.md, memory-adr.md, memory-glossary.md.
Middle Как Rules конкурируют с Skills при конфликте?
Rule выше Skill при конфликте. Skill описывает «как выполнить», Rule ограничивает «что можно». Пример: Skill «deploy to production» включает шаг «run migrations». Rule: «migrations требуют approval». Агент должен выполнить Skill, но адаптировать шаг: запросить approval перед migration. Неправильно: выполнить Skill и пропустить Rule потому что «в Skill написано запускать». Правило: Skill — способ достичь цели, Rule — ограничение на пути. Skill адаптируется к Rule, не обходит его. Best practice: в Skill явно указывать «проверь применимые Rules перед этим шагом».
Middle Когда Memory Bank недостаточно и нужен RAG?
Memory Bank загружается целиком в context window — ограничение по размеру. Для больших corpus (вся документация API, полные спецификации, большая кодовая база) — RAG эффективнее. Критерий: если memory.md > 500-1000 строк и большинство контента нерелевантно для типичного запроса — переходи на RAG (Cursor @Docs, векторные базы). Memory Bank → для «всегда нужно» (стек, ключевые ADR, Gotchas). RAG → для «нужно при конкретных вопросах» (API reference, схемы, спецификации). Hybrid: небольшой memory.md (ключевые факты) + RAG для деталей.
Middle Как убедиться что агент действительно читает memory.md в Cursor?
Тест: спроси агента процитировать конкретный факт из memory.md («какую версию Java мы используем по memory.md?»). Если агент не знает — файл не загружен. Причины: (1) нет строки @include в AGENTS.md — добавь; (2) AGENTS.md не читается — проверь что он в корне репо; (3) файл слишком большой — агент «теряет» середину. Дополнительная проверка: в начале сессии попроси агента «summarize memory.md» — если резюме точное, всё работает. В Claude Code проверяется аналогично, но файл должен быть в .claude/memory/.
Middle Как правильно описать Deploy workflow — в AGENTS.md, Skills или Rules?
Правильно: Skills. AGENTS.md: одна строка — «для deploy используй /deploy skill». Skills (deploy.md): полные шаги — git pull, run tests, build image, push to registry, update k8s, verify health. Rules: только ограничения — «deploy в prod только в рабочее время», «требует approval от 2 reviewers». Почему так: AGENTS.md остаётся компактным, Rules содержат только запреты/условия, Skill содержит процедуру целиком и обновляется независимо. Бонус: один Skill можно переиспользовать в разных проектах.
Middle Как Hooks в Claude Code усиливают Rules?
Rules в context — это «мягкое» ограничение: LLM читает текст и старается следовать. В длинном context (lost in the middle) Rule может потерять вес. Hooks — «жёсткое» ограничение: код выполняется платформой при каждом tool call независимо от context. Схема двойной защиты: Rule в .mdc описывает намерение (документация для агента и команды), Hook в settings.json реализует enforcement (код который нельзя обойти). Пример: Rule «не коммить .env файлы» + Hook PreToolCall который проверяет список файлов в git add и блокирует если есть .env. Rule объясняет почему, Hook гарантирует соблюдение.
Middle Что положить в global memory (Claude Code) vs project memory?
Global memory (~/.claude/memory/): предпочтения которые одинаковы во всех проектах — стиль ответов, предпочитаемый язык, личные workflow предпочтения, часто используемые команды. Project memory (.claude/memory/): всё специфичное для репо — стек, ADR, Gotchas, глоссарий команды, sprint outcomes. Session memory: временные заметки для текущей задачи, не нужны в следующей сессии. Правило: если факт нужен во всех проектах → global. Если только в этом репо → project. Если только сегодня → session.
Middle Как правильно организовать Rules когда их много?
Разбей по ответственности: один файл .mdc = одна тема. Примеры: typescript.mdc (TS конвенции, glob *.ts), testing.mdc (тестирование, glob *.test.*), git.mdc (git workflow, alwaysApply), security.mdc (запреты безопасности, alwaysApply), style.mdc (тон ответов, alwaysApply). Избегай: один большой rules.mdc на всё — агент теряет конкретные правила в большом файле. Лимит: каждый .mdc файл ≤ 100 строк. При росте — разбивай дальше. Регулярно: аудит на конфликты между файлами.
Senior — 5 вопросов
Senior Как спроектировать систему из пяти инструментов чтобы минимизировать context overhead?
Принцип Just-In-Time context: инструмент попадает в context только когда нужен. Стратегия: AGENTS.md — компактный (<500 строк), только workflow skeleton. Memory — структурированный, важное вверху. Rules — разбиты по glob, не всё alwaysApply. Skills — on-demand, не в context пока не вызваны. Hooks — OS-уровень, не занимают context вообще. Метрика: подсчитай токены которые ВСЕГДА в context (AGENTS.md + alwaysApply Rules + memory.md). Цель: <20% от context window. Остаток для кода, истории чата, task context. При превышении: Rules → glob, memory → RAG для объёмных частей, AGENTS.md → ссылки на Skills вместо деталей.
Senior Как обнаружить и разрешить конфликт между инструментами в production?
Сигналы конфликта: агент ведёт себя непредсказуемо при похожих запросах; цитирует разные версии одного факта; игнорирует явный запрет из Rule. Диагностика: (1) включи verbose logging в Claude Code (CLAUDE_CODE_DEBUG=1); (2) попроси агента «explain which rules and memory you're using for this»; (3) проверь overlap — ищи одинаковые темы в AGENTS.md и memory.md. Разрешение: найди конфликтующие инструкции → определи правильный инструмент → удали из неправильного → обнови. Профилактика: каждый инструмент — single responsibility, аудит раз в квартал, PR review для изменений в Rules/AGENTS.md.
Senior Когда Rules + Memory + AGENTS.md недостаточно и нужна архитектурная перестройка?
Сигналы необходимости перестройки: (1) context window заполнен ≥40% только служебными файлами; (2) агент регулярно «забывает» важные Rules несмотря на alwaysApply; (3) Memory Bank > 2000 строк и плохо структурирован; (4) одна и та же Skill используется в 10+ проектах. Архитектурные решения: Memory Bank → векторная RAG система для больших corpus; Rules → централизованный Rules registry с versioning; Skills → shared Skill library (git submodule или package); большой AGENTS.md → иерархические AGENTS.md (корень + поддиректории). Триггер для обсуждения с командой: когда maintenance cost инструментов превышает их пользу.
Senior Как обеспечить consistency Rules и Memory при работе нескольких агентов (subagents)?
Проблема: subagent получает свой context, может не читать те же Rules что main agent. Решение: явно передавать critical Rules и relevant Memory в системный промпт subagent или через AGENTS.md subagent-а. Паттерн: main agent передаёт subagent только те Rules которые релевантны задаче (не весь набор). Memory: subagent читает ту же память что main agent если работает в том же репо. Риск: subagent находит «другой ответ» потому что видел другой subset Rules. Защита: subagent findings = evidence, main agent верифицирует против Rules прежде чем применить (см. conflicts.html). Hook на OS-уровне применяется к любому агенту — это самое надёжное.
Senior Как оценить «quality» текущей системы инструментов и найти слабые места?
Метрики: (1) Context saturation — % context window занятый служебными файлами при типичном запросе. Цель <25%. (2) Rule compliance rate — % запросов где агент следовал критическим Rules. Проверяется через выборку из истории. (3) Memory freshness — дата последнего обновления каждой секции memory.md. Устаревшие разделы — риск. (4) Skill usage — какие Skills вызываются часто, какие — нет (obsolete). (5) Conflict frequency — как часто агент получает противоречивые инструкции. Инструменты: git blame на AGENTS.md/memory.md (когда последний раз обновлялись), grep на дубли (одинаковые слова в разных файлах), тестовые сессии с «provocative» запросами которые обнаруживают конфликты.
Lead — 5 вопросов
Lead Как выстроить governance для пяти инструментов в команде 10+ разработчиков?
Принципы governance: (1) Ownership — каждый файл имеет одного owner (AGENTS.md → Tech Lead, memory.md → команда через PR, Rules → per-domain owners, Skills → domain experts, Hooks → Security engineer). (2) Change process — изменения в AGENTS.md/Rules/Hooks через PR с обязательным review от owner. Memory — более свободный PR process. (3) Versioning — все файлы в git, история изменений видна. (4) Audit calendar — раз в квартал: conflict detection, freshness check, obsolete skills removal. (5) Onboarding — новый разработчик обязан прочитать AGENTS.md + memory.md как часть onboarding. (6) Breaking changes — изменение в Rules которое меняет поведение = обсуждение в команде, не unilateral change.
Lead Как провести onboarding команды на новую систему инструментов?
Фазы: (1) Foundation (week 1): всем прочитать AGENTS.md + memory.md. Задача: найти одну устаревшую секцию и создать PR с исправлением. (2) Practice (week 2-3): каждый добавляет один Gotcha или ADR в memory.md который знает. (3) Rules review (week 4): совместный просмотр всех Rules — есть ли конфликты, дубли, устаревшие. (4) Skills demo (week 5): каждый domain expert демонстрирует свой Skill или пишет новый для повторяющегося workflow. (5) Governance setup (week 6): назначить owners, договориться о PR process. Инструмент: создай team wiki страницу «Наши инструменты агента» с описанием каждого файла и кто owner.
Lead Как стандартизировать использование инструментов между несколькими репозиториями команды?
Паттерны: (1) Shared Rules library — git submodule или npm package с общими Rules (security, coding standards). Каждое репо делает extend + override. (2) Template AGENTS.md — базовый шаблон в team wiki, каждое репо кастомизирует для своих нужд. (3) Shared Skills — критические Skills (deploy, PR process, incident response) в shared repository, подключаются как submodule. (4) Memory structure template — стандартная структура секций memory.md для всех проектов — упрощает чтение при переходе между репо. (5) Global Rules — для Claude Code можно положить в ~/.claude/ — применяются ко всем проектам. Риск унификации: over-standardization убивает flexibility. Баланс: стандартизировать security и team process, оставлять свободу для project-specific Rules.
Lead Как измерить ROI от инвестиций в систему инструментов?
Метрики до/после: (1) Time to complete repetitive task — сравни время deploy/PR с Skills vs без. (2) Rework rate — % задач где агент предложил неправильное (отвергнутое) решение. С хорошей Memory должно снижаться. (3) Onboarding time — сколько дней до продуктивности нового разработчика. Memory Bank ускоряет. (4) Security incidents — нарушения Rules/Hooks. (5) Context-related bugs — баги из-за того что агент «не знал» о решении которое уже было принято. Методология: baseline measurement перед внедрением, track 30/60/90 дней после. Soft metrics: команда реже объясняет контекст агенту, агент реже предлагает уже отвергнутые решения.
Lead Как поддерживать актуальность Memory Bank при быстрых изменениях стека?
Проблема: стек меняется быстро, memory.md устаревает незаметно — агент продолжает думать что используется старый подход. Решения: (1) ADR с датами — каждое решение датировано, легко видеть устаревшее. (2) «Expires» пометка — [deprecated Q2 2025] рядом с устаревшим решением вместо удаления. (3) Sprint ritual — в конце каждого спринта 15 минут: обновить memory.md по sprint outcomes. (4) Bot check — скрипт или CI check который сравнивает даты обновления memory.md с датами PR в репо. (5) Memory owner rotation — один разработчик на sprint отвечает за актуальность memory.md. (6) Automated memory — в Claude Code агент сам предлагает обновить memory.md при обнаружении расхождений между памятью и текущим кодом.
Architect — 5 вопросов
Arch Как спроектировать систему инструментов для организации с 50+ репозиториями?
Архитектура уровней: (1) Organization level — shared Rules package (security, compliance) версионируется как npm/pip package, все репо подключают как dependency. Global CLAUDE.md / team-wide Hooks для критических запретов. (2) Team level — team-specific Rules и Skills в shared git submodule. Memory template для всех репо команды. (3) Project level — project-specific AGENTS.md, memory.md, project Rules. (4) Individual level — personal global memory в ~/.claude/memory/ для персональных предпочтений. Governance: Organization Rules → CODEOWNERS + mandatory security review. Team Rules → team lead approval. Project → обычный PR. Versioning: Rules package с semver, breaking changes требуют migration guide для всех репо.
Arch Как интегрировать систему инструментов агента с CI/CD pipeline?
Integration points: (1) Memory validation в CI — скрипт проверяет что memory.md не содержит устаревших ссылок (на deprecated packages, старые версии). (2) Rules linting — automated check на конфликты между .mdc файлами при PR. (3) Skills testing — для критических Skills (deploy, migration) — automated regression test: запускаем агент в sandbox с этим Skill и проверяем результат. (4) Hooks audit — в CI проверяем что hooks.json содержит обязательные security hooks (никто не удалил случайно). (5) Memory freshness — CI warning если секция memory.md не обновлялась > 90 дней. (6) Context size check — CI метрика: суммарный size AGENTS.md + alwaysApply Rules не превышает лимит (например 50KB). Pipeline: все проверки в pre-merge check, блокирующие только security (Hooks).
Arch Как спроектировать Memory Bank для мультиагентной системы где агенты работают параллельно?
Проблемы concurrent access: (1) Race condition — два агента одновременно обновляют memory.md → merge conflict. (2) Stale read — агент читает memory в начале сессии, к концу оно уже обновлено другим агентом. (3) Conflicting writes — агенты записывают противоречивые факты. Решения: (1) Read-only memory — агенты только читают, обновление только через human review (PR). (2) Append-only log — агенты добавляют записи в log, не редактируют существующее. Conflict resolution run отдельно. (3) Sharded memory — каждый агент работает со своим subset (memory-billing.md, memory-auth.md). (4) Versioned snapshots — memory snapshot в начале session, агент работает с snapshot, изменения через PR. (5) External memory store — Redis или vector DB с proper concurrency control вместо git файлов.
Arch Как обеспечить security и compliance для системы инструментов в regulated industry?
Compliance requirements: (1) Audit trail — все изменения в Rules, AGENTS.md, Hooks должны быть traceable. Git blame + signed commits + CODEOWNERS обеспечивают аудит. (2) Access control — Hooks (security critical) — только security team может изменять. Enforcement: CODEOWNERS + branch protection. (3) Rules для sensitive data — явные Rules о PII, PHI, PCI. Hooks для enforcement: блокировать любой tool call с sensitive data patterns. (4) Memory sanitization — перед добавлением в memory.md проверка на sensitive data (secrets, credentials, PII). (5) Skills approval — workflow Skills для production actions требуют approval gate. (6) Compliance Rules package — отдельный versioned package с compliance-specific Rules (GDPR, HIPAA, PCI). Подключается обязательно во все репо организации. (7) Quarterly compliance audit — проверка что Hooks не были обойдены, Rules не конфликтуют с compliance требованиями.
Arch Как проектировать cross-cutting concerns (logging, security, observability) через систему инструментов?
Cross-cutting concerns плохо ложатся в один файл — нужна многоуровневая стратегия. Logging: Hooks (PostToolCall) — логировать все tool calls в centralized logging service. Не в Rules (агент может забыть), только Hooks гарантируют. Security: Organization-level Rules package + mandatory Hooks для всех репо. CODEOWNERS → Security team owns эти файлы. Observability: Skills для incident response (structured процедура сбора диагностики). Hooks для critical alerts (при аномальных tool calls — уведомление). Memory для known incidents (Postmortem outcomes → memory.md). Архитектурный принцип: чем critical cross-cutting concern → тем ниже уровень enforcement (Hooks > Rules > Memory). Logging → Hooks. Security → Hooks + Rules. Best practices → Rules. Knowledge → Memory. Process → AGENTS.md + Skills.
