Agent Loop — как агент работает изнутри

Механика цикла агента: think → act → observe. Что происходит между твоим промптом и ответом агента, как растёт контекст, когда цикл останавливается и почему это важно для всего остального.

Что такое Agent Loop и зачем он нужен

Agent Loop — это повторяющийся цикл «подумать → вызвать инструмент → посмотреть результат», который позволяет агенту решать многошаговые задачи автономно, не ожидая инструкции на каждый шаг.

Agent Loop = конвейер на заводе

Представь завод, где робот получает задание «собери деталь». Он не делает всё за один раз — он берёт болт, проверяет размер, крутит гайку, снова проверяет... пока деталь не готова. Агент работает так же: каждый шаг использует инструмент, смотрит результат, решает что дальше.

  • Один проход цикла = один «turn»
  • Простая задача: 1–3 turns
  • Сложная задача: 10–50 turns
  • Каждый turn добавляет данные в контекст

Cursor: Agentic Mode

В Cursor Agent Loop включается в режиме Agent (вкладка в чате). Агент может читать файлы, запускать терминал, редактировать код. Каждое действие требует подтверждения или работает в YOLO-режиме автоматически.

Claude Code: встроенный loop

В Claude Code Agent Loop — основа всего. Каждый запрос запускает цикл. Агент сам решает, когда задача выполнена. Цикл останавливается, когда больше нечего вызывать или достигнут лимит turns.

Три фазы агентного цикла

Каждый цикл включает три фазы — они перетекают друг в друга, но логически различны.

Gather
Сбор контекста — агент читает файлы, ищет код, загружает CLAUDE.md / Rules, запрашивает память и RAG. Цель: понять, что уже известно и чего не хватает.
Инструменты: Read, Grep, LS, WebSearch, Memory, MCP (Jira, GitHub, Sentry, БД…)
Act
Действие — агент редактирует файлы, запускает команды, вызывает MCP-инструменты, создаёт subagents. Изменения реальны и немедленны.
Инструменты: Edit, Bash, MCP tools, Subagent spawn
Verify
Проверка — агент запускает тесты, читает вывод, сравнивает с ожидаемым результатом. Если что-то не так — возвращается к фазе Act или Gather.
Инструменты: Bash (тесты), Read (diff), анализ stderr
Полный цикл — пример: «Fix failing tests in auth.ts» ┌────────────────────────────────────────────────────────────────┐ │ USER → "Fix failing tests in auth.ts" │ ├────────────────────────────────────────────────────────────────┤ Turn 1 [Gather] Read(auth.ts) → код функции validateToken() Read(auth.test.ts) → 3 теста, 2 падают Bash("npm test") → Error: expected 403 got 401 Turn 2 [Gather + Reason] Grep("403", "src/") → нашёл middleware/errors.ts Read(errors.ts) → понял: HTTP-коды переопределены MCP github.search_commits("errors.ts") → PR #412: «403 для forbidden после рефактора» Turn 3 [Act] Edit(auth.ts) → исправил статус-коды: 401 → 403 Turn 4 [Verify] Bash("npm test") → ✓ 3 passed, 0 failed └────────────────────────────────────────────────────────────────┘ Итог: 4 turns, 8 вызовов инструментов (в т.ч. 1× MCP), задача решена
MCP в этом сценарии: встроенные Read/Grep/Bash закрывают репозиторий на диске. MCP подключает внешний контекст — историю PR, тикет, CI, прод-логи — без ручного копипаста. Вызов MCP — тот же паттерн, что ToolUse/ToolResult в контексте; часто фаза Gather (уточнить «зачем 403») или Verify (проверить remote CI через GitHub Actions MCP). В Act MCP бывает для create_issue, post_comment, query DB — когда действие вне git/терминала.

Инструменты (Tools) — движки agent loop

Без инструментов агент — просто текстовая модель: она может только сгенерировать текст. Tool (инструмент) — это оформленное «действие снаружи модели»: прочитать файл, выполнить команду, вызвать API. Хост (Cursor или Claude Code) исполняет действие и возвращает результат в контекст — так работает фаза act → observe в loop.

Tool = не магия модели, а контракт

Модель не «знает» ваш диск и терминал. Она выдаёт структурированный запрос: имя инструмента + аргументы (function calling / tool use). Клиент агента (IDE или CLI) проверяет права, выполняет код на вашей машине и кладёт ответ в историю как ToolResult.

Кто что поставляет

  • Вендор LLM (Anthropic, OpenAI…) — умеет в ответах вызывать tools по схеме; сами tools не шипит.
  • Хост агента (Cursor, Claude Code) — набор встроенных tools + рантайм исполнения.
  • MCP-серверы — внешние tools (БД, Jira, браузер…), подключаются через протокол MCP.
  • Вы — конфиг MCP, permissions, rules; иногда отключаете или ограничиваете вызовы.

Встроенные vs внешние

Встроенные — реализованы внутри Cursor / Claude Code (Read, Grep, Bash…). Список фиксирован версией продукта (может меняться в релизах).

Внешние (MCP) — отдельный процесс или сервис; хост только проксирует tools/list и tools/call. Подробнее — на странице MCP / Tools.

Три слоя в одном запросе

Цепочка ответственности 1. Модель решает: «нужен tool X с аргументами {…}» ← только текст/JSON в ответе 2. Хост агента маршрутизирует: встроенный handler ИЛИ MCP-сервер ← реальное исполнение 3. Tool возвращает данные (файл, stdout, JSON API) → ToolResult в контекст Один и тот же loop для Read("app.ts") и для mcp__linear__create_issue — разница только в том, кто зарегистрировал tool.

Список tools — часть контекста

Перед каждым turn агент видит описания всех доступных инструментов (имя, description, схема параметров). Чем больше MCP-серверов подключено, тем больше «шума» в контексте и тем важнее точные descriptions — см. как агент выбирает tool и какой инструмент когда.

«Встроенный» ≠ «безопасный»

Bash/Terminal и Edit — встроенные, но с реальными последствиями на диске и в shell. Хост добавляет подтверждения (Cursor) или permission rules (Claude Code). MCP-tools наследуют те же гейты, плюс доверие к стороннему серверу.

Когда loop останавливается

Задача выполнена

Агент сам решает, что цель достигнута. Возвращает финальный AssistantMessage без вызова новых инструментов.

Нужен ввод пользователя

Агент вызывает AskFollowUpQuestion — цикл приостанавливается и ждёт ответа. После ответа цикл продолжается.

Лимит turns / контекста

Достигнут максимум вызовов инструментов или заполнен контекст. Агент возвращает частичный результат с предупреждением.

Ошибка / Отказ

Инструмент вернул ошибку, которую агент не может обойти (нет прав, файл не существует). Агент сообщает о проблеме.

Тонкости Agent Loop

Контекст растёт необратимо — нельзя «удалить» прочитанный файл

Как только инструмент вернул результат — эти токены в контексте навсегда (до компакции). Поэтому важно не читать огромные файлы без необходимости: одно чтение на 50 KB = 12 000 токенов, и они остаются до конца сессии.

«Рассуждение» агента — тоже часть контекста

Каждый AssistantMessage между вызовами инструментов (когда агент «думает вслух») занимает токены. Модель, которая рассуждает многословно между инструментами, быстрее заполняет контекст. Хорошие agents пишут короткие мысли между actions.

Порядок инструментов влияет на результат

Агент видит результаты инструментов последовательно. Если сначала прочитать неправильный файл — это «заражает» понимание на все следующие turns. Хорошо сформулированный промпт с указанием конкретных файлов экономит turns и снижает вероятность ошибок.

Прерывание loop не откатывает уже сделанные действия

Если агент уже отредактировал 3 файла, а ты нажал Esc после 4-го действия — первые 3 правки остаются. Checkpoint позволяет откатиться к началу, но bash-команды (push, deploy) откатить нельзя. Именно поэтому опасные команды требуют явного подтверждения.

Agent Loop ≠ Chat — он не ждёт каждого шага

В обычном чате каждый ответ требует твоего следующего сообщения. Agent Loop работает автономно: агент сам вызывает следующий инструмент, не спрашивая тебя. Ты участвуешь только когда агент специально запрашивает ввод или ты прерываешь его.

Best Practices

Делай

  • Указывай конкретные файлы/директории в промпте — агент тратит меньше turns на поиск
  • Давай агенту способ проверить результат (тест, ожидаемый вывод)
  • Используй Plan Mode для сложных задач — сначала план, потом действия
  • Клади персистентные правила в CLAUDE.md / Rules, а не в чат
  • Прерывай (Esc) сразу, если видишь что агент идёт не туда

Не делай

  • Не давай читать огромные файлы без необходимости — токены не вернуть
  • Не запускай опасные команды (push, drop table) без явных разрешений
  • Не рассчитывай что ранние инструкции в чате сохранятся после компакции
  • Не оставляй агент без условия остановки — он будет крутиться пока не заполнит контекст
  • Не путай «агент подумал» с «агент сделал» — только ToolUse производит реальные изменения

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Agent Loop простыми словами?
Agent Loop — это повторяющийся цикл, в котором агент: думает что делать → вызывает инструмент (читает файл, запускает команду) → получает результат → думает снова. Цикл повторяется пока задача не выполнена. Это принципиально отличает агента от обычного чат-бота, который отвечает один раз и ждёт следующего сообщения.
Junior Сколько «шагов» нужно агенту, чтобы исправить небольшой баг?
Обычно 3–6 turns: 1) прочитать файл с ошибкой, 2) найти связанные файлы (если нужно), 3) внести правку, 4) запустить тест, 5) убедиться что тест прошёл. Если с первой попытки тест не прошёл — добавляется ещё 1–2 turns на диагностику. Простые задачи («добавь комментарий») решаются за 1–2 turns.
Junior Что происходит, когда агент вызывает инструмент?
Результат инструмента добавляется в контекст как ToolResult. Агент видит его при следующем «рассуждении» и использует для принятия решения о следующем шаге. Контекст постепенно заполняется историей всех вызовов и их результатов.
Junior Что такое «turn» в контексте Agent Loop?
Turn — один проход цикла: агент рассуждает → вызывает один инструмент → получает результат. Затем начинается следующий turn. Количество turns в задаче — это сколько раз агент «подумал и сделал что-то» прежде чем вернуть финальный ответ.
Junior Какие три фазы есть в каждом agent loop?
Gather (сбор контекста) — читает файлы, ищет информацию. Act (действие) — редактирует файлы, запускает команды. Verify (проверка) — запускает тесты, проверяет результат. Фазы перетекают друг в друга и могут повторяться внутри одной задачи.
Junior Чем Agent Loop отличается от обычного чата с AI?
В обычном чате AI отвечает один раз и ждёт следующего сообщения от тебя. В Agent Loop AI работает автономно: после получения задачи сам решает что делать дальше, вызывает инструменты, проверяет результаты и продолжает, пока задача не выполнена. Ты не участвуешь в каждом шаге.
Junior Какие инструменты чаще всего использует агент?
Наиболее частые: Read (прочитать файл), Grep/Glob (найти файлы/строки), Edit/Write (изменить файл), Bash (запустить команду). Для Claude Code ещё WebSearch. Выбор инструмента зависит от типа задачи.
Junior Как агент узнаёт, что задача выполнена?
Агент сам решает — на основе результатов инструментов. Если тест прошёл, файл изменён правильно, вывод команды соответствует ожидаемому — агент считает задачу выполненной и возвращает финальное сообщение без вызова новых инструментов.
Junior Что происходит если нажать Esc во время работы агента?
Текущий вызов инструмента отменяется, loop останавливается. Уже сделанные изменения (отредактированные файлы) остаются. Агент ждёт следующей инструкции. Если нужно откатить изменения — используй checkpoint или git.
Junior Почему нельзя «удалить» прочитанный файл из контекста?
Контекст — это последовательность сообщений. Однажды добавленный ToolResult (результат чтения файла) уже является частью этой последовательности. Нельзя «вырезать» токены из середины. Они остаются до тех пор, пока не произойдёт компакция всего контекста.
Middle — 10 вопросов
Middle Как компакция контекста влияет на поведение агента?
При компакции (/compact или авто) LLM суммаризирует историю, заменяя детальные сообщения кратким резюме. После компакции: ранние детальные инструкции могут быть потеряны или упрощены, агент «забывает» детали но помнит суть. Именно поэтому критичные правила должны быть в CLAUDE.md (он загружается заново в каждую сессию), а не написаны в начале чата.
Middle Как Plan Mode меняет поведение agent loop?
В Plan Mode агент может использовать только инструменты Gather-фазы (Read, Grep, LS, WebSearch) — он может исследовать но не может редактировать файлы или запускать команды с сайд-эффектами. Это позволяет безопасно изучить задачу, составить план, получить фидбек, и только потом перейти к действиям. В Claude Code: Shift+Tab дважды. Агент сам запрашивает подтверждение для перехода в Act-фазу.
Middle Почему важно давать агенту «условие проверки»?
Без условия проверки агент не знает точно когда остановиться. Он может остановиться слишком рано («код выглядит правильным») или крутиться в петле, пробуя разные подходы. Когда даёшь проверяемое условие (тест, ожидаемый вывод, конкретный файл) — агент может запустить инструмент и получить объективный результат.
Middle Какие задачи требуют больше turns и почему?
Больше turns нужно когда: 1) задача охватывает много файлов (рефакторинг), 2) нужно исследовать незнакомый код, 3) первая попытка даёт ошибку и нужна диагностика, 4) задача итеративна (написать тест → поправить код → тест → поправить). Простые задачи с чётким указанием «что и где» делаются за 1–3 turns.
Middle Как управлять тем, что агент читает первым?
Через конкретизацию промпта: «Посмотри src/auth/token.ts, там функция validateToken — она возвращает неправильный HTTP-код» гораздо лучше чем «почини баг с авторизацией». Первый вариант направляет агента к нужному файлу сразу, второй заставит его потратить 2–5 turns на поиск. Также можно использовать @-mentions (Cursor) или явно указывать пути.
Middle Что такое «thrashing» (петля компакции) и как с ним бороться?
Thrashing происходит когда один файл или ответ инструмента настолько большой, что сразу после компакции контекст снова заполняется. Агент компактирует → читает тот же большой файл → снова полный контекст → снова компакция. Claude Code показывает ошибку и останавливается. Решение: разбить задачу на части, использовать subagent для большого файла, или явно указать агенту читать только нужные секции файла.
Middle Как subagents помогают с ограничениями agent loop?
Subagent получает собственный свежий контекст, полностью изолированный от родительского. Его работа не заполняет контекст основного агента — только финальный результат (краткое summary) возвращается. Это позволяет делегировать тяжёлые задачи (анализ большого файла, сложный поиск) без «загрязнения» основного контекста.
Middle Как hooks интегрируются с agent loop?
Hooks — обработчики событий loop. PreToolUse срабатывает до вызова инструмента (можно заблокировать или модифицировать). PostToolUse — после (можно добавить авто-форматирование). Stop — когда агент завершает задачу (можно отправить уведомление). Hooks не меняют контекст LLM, но управляют поведением на уровне harness.
Middle Почему ранние инструкции в чате ненадёжны для долгих сессий?
При компакции LLM суммаризирует историю чата. Детальные инструкции из раннего сообщения («всегда использовать tabs, не spaces») могут упроститься до «используй правила стиля» или вообще потеряться. CLAUDE.md загружается в начало каждого нового system-промпта и не подвергается компакции — именно там должны быть постоянные правила.
Middle Как выглядит «хороший» agent loop с точки зрения токенов?
Хороший loop: использует точечные инструменты (Grep вместо чтения всего файла), читает только релевантные части, пишет короткие «рассуждения» между инструментами, останавливается как только цель достигнута. Плохой loop: читает все файлы подряд, многословно объясняет каждый шаг, делает redundant проверки. Разница в 5–10x по токенам на одну задачу.
Senior — 5 вопросов
Senior Как ограничить количество turns агента и когда это нужно?
В Claude Code: флаг --max-turns N при запуске или параметр в SDK. В Cursor: через Rules можно указать ограничения поведения. Зачем ограничивать: 1) в CI/CD — предотвратить бесконечный цикл при непредвиденной ошибке, 2) для дорогих операций — контроль стоимости, 3) для задач с чётким скоупом — агент не должен «уходить» в смежные проблемы. Компромисс: слишком мало turns = задача не выполнена.
Senior Как архитектурно решить проблему «lost in the middle» в длинном agent loop?
«Lost in the middle» — инструкции из середины контекста хуже обрабатываются моделью. Решения: 1) CLAUDE.md с Compact Instructions — ключевые правила, которые сохраняются при компакции, 2) Subagents — каждый работает с коротким свежим контекстом, 3) Checkpoint + /compact focus — явно указать что важно сохранить, 4) Skills — повторяемые workflow вне основного контекста, 5) разбить задачу на несколько коротких сессий.
Senior Как отличить «агент застрял в петле» от «агент решает сложную задачу»?
Признаки петли: агент вызывает одни и те же инструменты с теми же аргументами, сообщения между turns однотипны, количество turns значительно превышает ожидаемое. Признаки сложной задачи: каждый turn открывает новые файлы или возвращает новые данные, агент явно меняет стратегию («первый подход не работает, попробую иначе»). Мониторинг: /context показывает уникальность ToolUse вызовов.
Senior Как устроен agent loop в SDK (Agent SDK)?
В Claude Agent SDK loop реализован явно: agent.run(prompt) возвращает стрим событий. Каждое событие — либо AssistantMessage, либо ToolUse + ToolResult. Разработчик может: перехватывать события (observability), прерывать loop при условии (hooks), передавать custom tools (in-process MCP), управлять permissions (allow/deny rules). Loop продолжается пока агент вызывает инструменты; когда возвращает AssistantMessage без ToolUse — loop завершён.
Senior Какие side effects agent loop нельзя откатить и как это учитывать в дизайне?
Нельзя откатить: git push, деплой, изменения в БД, отправку email/webhook, API-вызовы с состоянием. Можно откатить: правки файлов (checkpoint + git). Стратегии: 1) Разделить loop на dry-run и apply фазы — сначала план без side effects, потом применение с подтверждением, 2) Permissions — блокировать опасные команды в allow/deny rules, 3) Staging environment — агент работает в изолированной среде, 4) Sandbox mode — в Claude Code sandboxed bash не имеет сетевого доступа.
Lead — 5 вопросов
Lead Как стандартизировать использование agent loop в команде?
1) Шаблонные CLAUDE.md — корпоративный шаблон с командными правилами, архитектурными решениями, запрещёнными командами. 2) Onboarding skills — команда /team-onboarding которая объясняет новичкам как работать с агентом в данном проекте. 3) Permission policies через managed settings — организационные allow/deny rules для всех. 4) Метрики — отслеживать avg turns per task, token usage, чтобы выявлять неэффективные паттерны промптинга.
Lead Как обучать разработчиков эффективно работать с agent loop?
Основные паттерны для обучения: 1) «Конкретизация промпта» — покажи разницу в turns между размытым и конкретным запросом, 2) «Давай условие проверки» — тест или ожидаемый вывод, 3) «Не читай весь репо» — используй Grep/Glob вместо Read всех файлов, 4) «Plan Mode для сложных задач» — исследуй перед действием, 5) «CLAUDE.md — не чат» — постоянные правила должны быть в файле. Форматы: парное программирование с агентом, code review агентских сессий.
Lead Как контролировать затраты на токены при командном использовании agent loop?
1) Мониторинг: /usage в Claude Code показывает что drive limits. Anthropic Console / analytics dashboard для командных данных. 2) Managed settings: spend limits на пользователя/день/месяц. 3) Практики: поощрять конкретные промпты, использование Grep вместо Read, короткие focused сессии вместо одной гигантской. 4) Модели: Sonnet для рутины, Opus для архитектурных решений — не наоборот. 5) Fast mode: для простых задач включить fast mode — дешевле и быстрее.
Lead Как организовать безопасный agent loop для production-критичных операций?
1) Разделение сред: агент работает только в dev/staging, деплой в prod — через отдельный pipeline с human approval. 2) Permission rules: явный deny для команд с production side effects (kubectl apply --prod, psql prod). 3) Dry-run phase: сначала Plan Mode с полным описанием изменений, потом подтверждение. 4) Audit log: сохранять транскрипты всех агентских сессий (Claude Code: ~/.claude/projects/). 5) Rollback plan: всегда git commit перед сессией с агентом.
Lead Как agent loop вписывается в CI/CD пайплайн?
Паттерны интеграции: 1) PR review: агент запускается на каждый PR, анализирует diff, оставляет комментарии — GitHub Actions + Claude Code. 2) Auto-fix: агент получает failing test, исправляет и создаёт новый коммит. 3) Security audit: ночной запуск, агент ищет уязвимости в новом коде. Ограничения: в CI нет interactive mode, нужно --headless, результат через exit code или вывод в файл. Безопасность: песочница, нет доступа к prod, read-only credentials где возможно.
Architect — 5 вопросов
Arch Как спроектировать multi-agent систему поверх agent loop?
Паттерны: 1) Orchestrator + Workers: главный агент (orchestrator) разбивает задачу, делегирует subagents, получает summary, интегрирует. 2) Parallel workers: несколько агентов работают над независимыми частями одновременно — в разных worktrees, итог сводится в main. 3) Pipeline: агент 1 исследует → агент 2 планирует → агент 3 реализует → агент 4 тестирует. 4) Dynamic workflows: orchestrator пишет скрипт запуска subagents на основе анализа задачи. Ключевой вопрос: как минимизировать inter-agent context transfer (только Summary, не полный контекст).
Arch Как обеспечить идемпотентность agent loop при сбоях?
Agent loop не идемпотентен по умолчанию: повторный запуск может создать дубли, конфликты, double-apply. Решения: 1) Checkpoint перед каждой сессией — возможность откатиться к исходному состоянию, 2) Git branch per session — агент работает в отдельной ветке, результат — PR, 3) Idempotent tools — проектировать custom MCP tools идемпотентно (upsert вместо insert), 4) State check в начале — агент читает текущее состояние и пропускает уже выполненные шаги, 5) Session resume — `claude --resume` продолжает с места остановки, не начинает заново.
Arch Как спроектировать наблюдаемость (observability) для agent loop в production?
Уровни наблюдаемости: 1) Трейсы: OpenTelemetry экспорт (Agent SDK поддерживает OTLP) — каждый ToolUse = span, весь loop = trace. 2) Метрики: turns per task, tokens per turn, tool usage distribution, failure rate по типам ошибок. 3) Логи: transcript.jsonl — полная история сессии, searchable. 4) Алерты: петли (N идентичных ToolUse подряд), превышение turns threshold, запрос опасных команд. 5) Дашборд: avg cost per feature, времяко выполнения, hit rate by developer — помогает находить неэффективные паттерны.
Arch Как agent loop взаимодействует с permission системой на архитектурном уровне?
Permission system — harness вокруг loop, не часть LLM. Архитектура: 1) Перед каждым ToolUse harness проверяет allow/deny rules (settings.json, managed settings). 2) Если deny — инструмент блокируется, агент получает отказ в ToolResult и должен найти обходной путь или запросить подтверждение. 3) Auto mode добавляет background safety classifier, который оценивает безопасность до выполнения. 4) Hooks PreToolUse могут динамически модифицировать или блокировать на основе runtime-логики. Разделение ответственности: LLM решает «что делать», harness решает «разрешено ли».
Arch Каковы архитектурные ограничения agent loop и как их обходить?
Фундаментальные ограничения: 1) Последовательность: по умолчанию инструменты вызываются последовательно — обходится через subagents в параллель, 2) Контекст: конечный размер окна — обходится компакцией, subagents, Skills-on-demand, 3) Stateless между сессиями: агент «забывает» — решается auto memory, CLAUDE.md, Session resume, 4) Недетерминизм: один промпт → разные actions — решается через Plan Mode review, детальные правила, конкретные тесты для проверки, 5) Latency: каждый turn = API call — решается fast mode, кэширование, параллельные subagents.
