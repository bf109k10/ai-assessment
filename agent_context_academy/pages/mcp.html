Что такое MCP и зачем он нужен

MCP = протокол

Model Context Protocol — открытый стандарт Anthropic для подключения AI-агентов к внешним данным и действиям. Как USB для AI-инструментов: один протокол, разные устройства.

  • Стандартный формат tool descriptions
  • JSON-RPC поверх stdio или SSE
  • Агент вызывает tools/call → сервер отвечает

MCP Server = набор tools

MCP сервер — процесс, который предоставляет инструменты. Агент подключается к нему и получает список доступных tools со схемами и описаниями.

  • Один сервер = одна область (filesystem, Jira, GitHub)
  • Может быть несколько серверов одновременно
  • Запускается локально или как remote сервис

Tool = отдельная функция

Каждый tool — это функция с именем, description и JSON schema параметров. Агент видит список всех доступных tools и их схемы — по этому и решает что вызвать.

  • read_file(path)
  • search_issues(query, project)
  • run_tests(command)

Как AI решает какой tool вызвать

Это один из самых важных вопросов для assessment. Агент не знает инструменты наперёд — он читает их описания в context и выбирает по смыслу.

Ключевой принцип: агент видит список tools как часть своего context window. Каждый tool имеет name, description, и inputSchema. Агент принимает решение на основе description — если description плохой, агент выберет не тот tool.

Что агент видит (tool list в context)

// Фрагмент context агента — список tools Available tools: [ { "name": "search_code", "description": "Search for code patterns in the repository using ripgrep. Use for finding usages, definitions, or specific strings in source files.", "inputSchema": { "type": "object", "properties": { "pattern": {"type":"string"}, "path": {"type":"string"} } } }, { "name": "read_file", "description": "Read contents of a file. Use when you need to see the full content of a specific file.", "inputSchema": { "properties": { "path": {"type":"string"} } } } ]

Алгоритм выбора tool агентом

1

Читает задачу

Агент понимает что нужно сделать.

2

Сканирует descriptions

Читает description каждого доступного tool и сопоставляет с задачей семантически.

3

Выбирает наиболее подходящий

На основе description и inputSchema определяет какой tool решит задачу.

4

Формирует аргументы

По inputSchema строит корректный JSON вызов.

5

Интерпретирует результат

Tool result попадает в context → агент решает следующий шаг.

Почему плохой description = сломанный tool

Агент не читает исходный код tool. Он видит только description. Если description расплывчатый ("does stuff with data"), агент не будет его вызывать. Если description конфликтует с другим tool, агент может выбрать неправильный. Правило: первые 2 предложения description — самые важные, они в "горячей зоне" attention агента.

Где хранится конфиг

  • Cursor: .cursor/mcp.json (проект) или Settings → MCP (глобально)
  • Claude Code: claude_desktop_config.json
  • Конфиг описывает: команду запуска сервера + env vars с секретами

Транспорт: stdio vs SSE

  • stdio: локальный процесс, обмен через stdin/stdout. Самый простой.
  • SSE (Server-Sent Events): remote сервер по HTTP. Для облачных и командных серверов.
  • Большинство серверов из реестра работают через stdio.

Установка MCP серверов: uvx vs npx/npm

MCP серверы могут быть написаны на Python (запускаются через uvx) или JavaScript/TypeScript (через npx/npm). Выбор зависит от экосистемы сервера.

🐍 Python серверы — uvx PyPI

uvx — это uv (быстрый Python package manager) в режиме run-without-install. Запускает Python MCP сервер изолированно.

# Запуск Python MCP сервера без установки npx @modelcontextprotocol/inspector \ uvx mcp-server-git \ --repository ~/code/my-project # В claude_desktop_config.json: { "mcpServers": { "git": { "command": "uvx", "args": ["mcp-server-git", "--repository", "/path/to/repo"] } } }
Когда uvx: сервер из PyPI, написан на Python. Не нужно устанавливать в систему — uvx создаёт временный venv.

📦 JS/TS серверы — npx/npm npm

npx запускает npm пакет без глобальной установки. Большинство официальных MCP серверов Anthropic написаны на TypeScript.

# Запуск через npx (без установки) npx @modelcontextprotocol/inspector \ npx @modelcontextprotocol/server-filesystem \ /Users/username/Desktop # В claude_desktop_config.json: { "mcpServers": { "filesystem": { "command": "npx", "args": [ "-y", "@modelcontextprotocol/server-filesystem", "/allowed/path" ] } } }
Когда npx: сервер из npm registry. -y флаг — автоматически соглашаться на установку без запроса.

Реальный mcp.json — шаблон конфигурации

Пример production-конфигурации с npm и Python серверами. Структура одинакова для .cursor/mcp.json и claude_desktop_config.json.

📦 npm/npx серверы Node.js

{ "mcpServers": { // Context7 — актуальная документация библиотек "context7": { "command": "npx", "args": ["-y", "@upstash/context7-mcp@latest"], "env": {} }, // GitLab MCP — PR, issues, commits "gitlab-mcp": { "command": "npx", "args": [ "-y", "@zereight/mcp-gitlab@latest", "--token=glpat--YOUR_TOKEN", "--api-url=https://gitlab.yourcompany.com/api/v4" ] }, // Task Master AI — управление задачами "task-master-ai": { "command": "npx", "args": ["-y", "task-master-ai"], "env": { "TASK_MASTER_TOOLS": "get_tasks,next_task,set_task_status" } }, // Chrome DevTools — отладка браузера "chrome-devtools": { "command": "npx", "args": ["-y", "chrome-devtools-mcp@latest"] }, // Webex — корпоративный мессенджер "webex": { "command": "npx", "args": ["-y", "@yuryev89/webex-mcp@latest"], "env": { "WEBEX_CLIENT_ID": "YOUR_CLIENT_ID", "WEBEX_CLIENT_SECRET": "YOUR_SECRET" } } } }
Ключевые поля: command: "npx" + args: ["-y", "package-name"]. Флаг -y — автоматически подтверждать установку. Секреты передаются через env.

🐍 Python/uvx серверы PyPI

{ "mcpServers": { // Atlassian — Jira + Confluence через uvx "mcp-atlassian": { "command": "uvx", "args": ["mcp-atlassian"], "env": { "JIRA_URL": "https://jira.yourcompany.com", "JIRA_PERSONAL_TOKEN": "YOUR_JIRA_TOKEN", "CONFLUENCE_URL": "https://confluence.yourcompany.com", "CONFLUENCE_PERSONAL_TOKEN": "YOUR_CONF_TOKEN" } }, // Jenkins через mcp-jenkins (uvx) "jenkins-deployer": { "command": "uvx", "args": [ "mcp-jenkins", "--jenkins-url=https://ci.yourcompany.com", "--jenkins-user=your-user", "--jenkins-password=YOUR_API_TOKEN" ] }, // Graylog — просмотр логов "graylog-prod": { "command": "npx", "args": [ "-y", "@alexbuzo/graylog-mcp", "--graylog-url", "https://graylog.yourcompany.com/", "--token", "YOUR_GRAYLOG_TOKEN", "--ssl-verify=false" ] }, // Кастомный Python MCP сервер "my-internal-mcp": { "command": "uv", "args": [ "--directory", "/path/to/my-mcp-server", "run", "my-mcp-server" ], "env": { "DATABASE_URL": "postgresql://..." } } } }
uvx vs uv run: uvx package-name — запуск из PyPI без установки. uv --directory /path run server — запуск локального Python проекта через uv.

Никогда не храни токены прямо в mcp.json

Файл .cursor/mcp.json может попасть в git. Используй переменные окружения через env vars системы или .env файл (в .gitignore). В enterprise: secrets manager (Vault, AWS SSM). Пример безопасного подхода: "JIRA_TOKEN": "${JIRA_PERSONAL_TOKEN}" — читается из переменной окружения системы.

Структура одной записи mcpServers

// Полная схема записи в mcpServers "server-name": { // уникальный ключ (произвольный) "command": "npx" | "uvx" | "node" | "python" | "uv", "args": ["arg1", "arg2"], // список аргументов командной строки "env": { // переменные окружения для процесса "API_KEY": "your-secret-key", "BASE_URL": "https://api.example.com" }, "disabled": false, // true = отключить сервер без удаления "alwaysAllow": ["tool_name"] // инструменты без запроса разрешения (Cursor) }

MCP Registry — официальный каталог серверов

Официальный реестр MCP серверов по адресу registry.modelcontextprotocol.io — место где публикуются верифицированные серверы сообщества и Anthropic.

Что такое MCP Registry: открытый каталог MCP серверов. Каждый сервер — это набор tools для конкретной системы (GitHub, Jira, filesystem, database...). Серверы публикуются через GitHub, доступны как npm пакеты или PyPI пакеты.
🗂

registry.modelcontextprotocol.io

Официальный реестр. Список серверов с описаниями, версиями и ссылками на GitHub / npm / PyPI.

Как пользоваться

  1. Найти нужный сервер по категории или поиску
  2. Посмотреть список tools с описаниями
  3. Скопировать конфигурацию для claude_desktop_config.json
  4. Запустить через uvx или npx

Популярные серверы

  • filesystem — read/write локальных файлов
  • github — Issues, PRs, repos
  • git — git log, diff, blame
  • fetch — HTTP запросы к URL
  • postgres / sqlite — SQL запросы
  • puppeteer — браузерная автоматизация

MCP Inspector — инструмент отладки

MCP Inspector — интерактивный web UI для тестирования и дебаггинга MCP серверов. Запускается через npx без установки.

Зачем нужен Inspector

Когда создаёшь или отлаживаешь MCP сервер, нужно проверить: правильно ли он отвечает, корректны ли описания tools, как выглядит фактический JSON ответ. Inspector даёт web UI для этого без написания тестового кода.

Запуск Inspector

# npm пакет npx @modelcontextprotocol/inspector \ npx @modelcontextprotocol/server-filesystem \ /Users/username/Desktop # Python пакет npx @modelcontextprotocol/inspector \ uvx mcp-server-git \ --repository ~/code/my-project # Локальный сервер (TypeScript) npx @modelcontextprotocol/inspector \ node path/to/server/index.js # Локальный сервер (Python) npx @modelcontextprotocol/inspector \ uv --directory path/to/server \ run package-name

Что показывает Inspector

📋 Resources tab

Список доступных ресурсов (файлов, данных). MIME типы и содержимое.

🛠 Tools tab

Все доступные tools с JSON Schema. Можно вызвать любой tool вручную и увидеть результат.

💬 Prompts tab

Шаблоны промптов сервера (если есть). Тестирование с кастомными аргументами.

🔔 Notifications pane

Все логи и уведомления от сервера в реальном времени.

Workflow разработки MCP сервера: Launch Inspector → проверь connectivity → добавь tool → проверь описание в Tools tab → протестируй вызов → смотри результат → исправь description если агент не понимает.

MCP vs RAG — чем отличаются

RAG

Достаёт фрагменты знаний из corpus: docs, code, wiki.

  • Поиск по смыслу (embeddings)
  • Статичный corpus (нужна переиндексация)
  • Нет side effects
  • Результат — текстовые chunks

MCP / Tools

Даёт live данные и действия через tool calls.

  • Live API / DB / CI
  • Актуальные данные всегда
  • Возможны side effects (write, trigger)
  • Нужны permissions и safety
Когда что: документация и архивные данные → RAG. Текущий статус CI, тикеты, live метрики → MCP. Часто используются вместе: RAG находит контекст из docs, MCP даёт актуальные данные из систем.

Тонкости и нюансы MCP

Description — это всё что видит агент

Агент выбирает tool исключительно по description. Нет UI, нет icon, нет hint — только текст. Плохое description ("does stuff with files") → агент не поймёт когда использовать. Хорошее: "Reads file content by path. Use when you need to inspect source code, config, or any text file. Returns full content as string." Первые 1-2 предложения — самые важные.

MCP сервер ≠ MCP tool

MCP сервер — это процесс (Python/Node.js), который предоставляет несколько tools. Один сервер может содержать 5-20 tools. В конфигурации ты подключаешь сервер, а агент автоматически получает доступ ко всем его tools. Если добавил новый tool в сервер — перезапусти сервер, не меняй конфиг хоста.

Slop descriptions засоряют context

Каждый подключённый MCP сервер добавляет descriptions своих tools в system prompt (context) агента. Если у тебя 5 серверов по 10 tools с длинными descriptions — теряешь ~2000-5000 токенов на описания. Подключай только нужные для задачи серверы. В production: ограничивай список tools явно, не подключай всё подряд.

uvx и npx: без глобальной установки, но с кешем

uvx/npx не устанавливают постоянно, но кешируют. Первый запуск — скачивание пакета. Последующие — из кеша. Это значит: нет сети → нет MCP если кеш пуст. В enterprise среде с ограниченным интернетом используй локальные пакеты или Docker-образ сервера.

Тестируй tool вызовы через Inspector перед production

MCP Inspector (npx @modelcontextprotocol/inspector ...) позволяет вызвать любой tool вручную и увидеть raw JSON ответ. Обязательный шаг при создании своего сервера: проверь description читаемость, убедись что JSON schema параметров корректна, протестируй edge cases. Это дешевле чем debuggить через агента в реальном чате.

Best practices — Делай / Не делай

✅ Делай

  • Пиши description в формате: что делает + когда использовать + что возвращает
  • Добавляй Use for X, NOT for Y чтобы различить похожие tools
  • Подключай только нужные для задачи MCP серверы (меньше tool bloat)
  • Тестируй каждый новый tool через MCP Inspector перед использованием
  • Используй uvx для Python MCP серверов, npx для Node.js
  • Храни конфиг в репо (claude_desktop_config.json или .cursor/mcp.json) для команды

❌ Не делай

  • Не пиши однословные descriptions: "file tool", "database" — агент не поймёт
  • Не подключай все известные MCP серверы сразу — засоришь context
  • Не давай write/action tools без Permissions или Hooks для safety
  • Не забывай chmod +x для кастомных скриптов-серверов
  • Не смешивай MCP и RAG для одних данных — выбери одно (live → MCP, archive → RAG)
  • Не экспонируй production DB без allow/deny rules в permissions

Junior Что такое MCP (Model Context Protocol)?
MCP — открытый протокол, разработанный Anthropic, стандартизирующий способ предоставления инструментов и контекста AI-агентам. MCP определяет формат описания инструментов (tools), вызовов и результатов. Вместо того чтобы каждая платформа изобретала свой формат tool calling, MCP предоставляет единый стандарт. Агент подключается к MCP-серверам, которые предоставляют tools — функции для взаимодействия с внешними системами.
Junior Что такое MCP server?
MCP server — процесс или сервис, который предоставляет набор tools агенту через MCP протокол. Сервер декларирует список доступных tools с их описаниями, схемами входных параметров и возвращаемых данных. Примеры: MCP server для GitHub (создание PR, чтение issues), для файловой системы (чтение/запись файлов), для базы данных (выполнение SQL запросов). MCP серверы могут быть локальными (процесс на машине) или remote (HTTP сервис).
Junior Что такое MCP tool?
MCP tool — именованная функция с описанием, которую агент может вызвать через MCP server. Каждый tool имеет: name (идентификатор), description (для агента — как понять когда вызывать), inputSchema (JSON Schema параметров), возвращаемый тип. Пример: tool 'create_pull_request' с параметрами title, body, branch. Агент видит descriptions всех tools в своём context window и на их основе решает, какой tool вызвать.
Junior Как агент использует MCP tool — опиши flow?
Flow: (1) Агент читает descriptions всех доступных tools (они в system context). (2) Агент решает что для задачи нужен определённый tool. (3) Агент генерирует tool call с параметрами. (4) MCP client (платформа) перехватывает tool call, передаёт его MCP серверу. (5) MCP сервер выполняет функцию (обращается к API, читает файл, выполняет запрос). (6) Результат возвращается агенту как tool result. (7) Агент продолжает reasoning с полученными данными.
Junior Что такое tool description и почему она критически важна?
Tool description — текстовое описание tool для LLM: что делает, когда вызывать, какие параметры ожидаются. Агент не 'знает' что умеет tool — он читает description и принимает решение по ней. Плохое description ('process data') → агент не знает когда применять. Хорошее description ('Use this tool to create a new GitHub Pull Request from a branch. Do NOT use for reading existing PRs.') → чёткие границы применения. Первые 2 предложения description критичны — они в 'горячей зоне' attention модели.
Junior Как MCP отличается от RAG?
RAG — поиск по статическому corpus документов (что написано в документации). MCP — вызов live инструментов для получения актуальных данных или выполнения действий (что происходит прямо сейчас). RAG даёт историческое/документальное знание; MCP даёт live state и позволяет совершать действия. Пример: 'какова схема таблицы users' — RAG найдёт в docs; MCP tool выполнит DESCRIBE TABLE и вернёт актуальную схему. Для live data → MCP; для archived knowledge → RAG.
Junior Где конфигурируется MCP в Cursor и Claude Code?
Cursor: в настройках (Settings → MCP) или в .cursor/mcp.json файле в корне проекта. Конфигурация задаёт список MCP серверов с командами запуска и переменными окружения. Claude Code: через claude_desktop_config.json (глобальная конфигурация) или через CLAUDE.md с ссылкой на настройки. В обоих случаях конфиг определяет: какие серверы запустить, какие команды выполнить, какие env vars передать (API keys). Проверить активные tools: запросить у агента 'list available tools'.
Junior Что такое tool call и tool result?
Tool call — структурированный запрос от агента к MCP серверу: имя tool + параметры в JSON формате. Tool result — ответ MCP сервера: данные, статус выполнения (успех/ошибка), metadata. И tool call, и tool result становятся частью conversation context — агент видит что он запрашивал и что получил. Важно: tool result занимает место в context window, особенно если возвращает большие данные (JSON dump базы данных).
Junior Что такое tool result и как он влияет на context window?
Tool result — ответ внешней системы, возвращённый через MCP, добавляется в conversation как отдельное сообщение. Проблема: большие tool results (например, полный список из 10000 файлов) занимают огромную часть context window. Решения: (1) Просить агента summarize результат перед добавлением; (2) Настроить MCP tool возвращать только нужные поля; (3) Pagination — получать данные по частям; (4) Фильтрация на стороне tool (передавай параметры фильтрации при вызове).
Junior Какие MCP серверы существуют и что они делают?
Основные категории MCP серверов: (1) File system — чтение/запись файлов, дерево директорий. (2) Version control — GitHub/GitLab: PR, issues, commits, branches. (3) Databases — SQL запросы, schema inspection. (4) Project management — Jira, Linear, Trello: tasks, sprints. (5) Communication — Slack, Email: отправка сообщений, чтение каналов. (6) Cloud/Infra — AWS, GCP: ресурсы, логи, metrics. (7) Browser — web scraping, playwright. Anthropic и сообщество публикуют open-source MCP servers на GitHub. Можно создавать кастомные под нужды проекта.
Middle — 10 вопросов
Middle Когда использовать MCP vs RAG vs прямое чтение файла?
Правило выбора по типу данных: Live/real-time state (текущий статус сервиса, активные PR, deployment статус) → MCP tool. Historical/archived knowledge (документация, старые решения, архитектурные решения) → RAG. Конкретный файл который точно знаешь → direct read / @mention. Combo: RAG для discovery ('что есть'), MCP для verification ('что сейчас'), direct read для precision ('полный файл'). Никогда не используй RAG для 'текущее состояние' — там устаревшие данные.
Middle Как tool result от MCP попадает в context агента?
После вызова tool агент получает tool result как отдельное сообщение в conversation history. Этот результат становится частью context window наравне с другими сообщениями. Если агент делает несколько tool calls подряд — все results накапливаются в context. Важно: большие results (dump таблицы, список файлов) могут заполнить context window быстро. Стратегия: просить агента summarize и filter results прежде чем продолжать. Tool calls + results видны в conversation как structured data.
Middle Зачем использовать MCP для live данных а не просить агента самостоятельно найти информацию?
Агент без tools не может получить live данные — он ограничен своими весами обучения и тем что ему передали в context. 'Найди сам' без tools = агент придумает (hallucination) или скажет 'не знаю'. MCP tool обращается к реальному API/файлу/БД и возвращает актуальный факт. Разница: 'Какой статус PR #123?' — без MCP агент не знает (PR создан после обучения). С MCP tool get_pull_request → реальный статус из GitHub API. MCP = глаза и руки агента в живом мире.
Middle Как обеспечить безопасность при использовании MCP — какие риски и как управлять?
Риски MCP: (1) Over-permissive tools — tool имеет доступ к production DB write когда нужен только read. (2) Credential exposure — API keys в tool config, могут утечь через logs. (3) Lateral movement — через один tool агент получает доступ к другим системам. Меры: (1) Least privilege — каждый tool только с необходимыми permissions (read-only DB tool). (2) Secrets management — API keys через env vars, не в config файлах. (3) Tool sandboxing — ограничить scope каждого tool (только определённые репозитории). (4) Audit logging — каждый tool call логируется. (5) Human approval gate — destructive операции требуют confirm.
Middle Как MCP используется для интеграции с GitHub/Jira/Slack в рабочем процессе?
GitHub MCP tools в workflow: list_pull_requests (найти открытые PR) → get_pull_request_diff (получить diff) → create_review_comment (добавить review комментарий) → merge_pull_request. Jira workflow: create_issue (создать задачу) → update_issue_status (перевести в In Progress) → add_comment (добавить progress update). Slack: send_message (нотификация о deploy), read_channel (получить последние сообщения команды). Мощь в цепочке: агент сам orchestrates multi-step workflow без ручных переключений между инструментами.
Middle Как latency MCP вызовов влияет на опыт использования агента?
Каждый tool call добавляет latency: network round-trip (100–500ms для cloud API) + processing time + context update. Сложный workflow с 10 tool calls = +1–5 секунд к общему времени. Компенсация: (1) Parallel tool calls — если tools независимы, агент может вызывать несколько одновременно. (2) Caching — кэшировать результаты idempotent запросов (list repos меняется редко). (3) Batching — один tool call возвращает сразу несколько сущностей. (4) Early termination — если ответ получен из первых tool calls, не делать остальные. UX: показывать пользователю прогресс 'calling GitHub...'.
Middle Как агент выбирает нужный tool среди множества похожих?
Агент читает name и description всех доступных tools и на основе LLM reasoning выбирает наиболее подходящий для текущего шага. Критически важно: (1) Distinct descriptions — 'Use for reading file content' vs 'Use for listing directory contents' — чёткое разграничение. (2) 'When NOT to use' в description — явно указать что tool НЕ делает. (3) Naming conventions — verb_noun pattern (create_pr, list_issues, get_file). Если два tool имеют похожие descriptions — агент может ошибиться. Тестируй: попроси агента объяснить почему он выбрал именно этот tool.
Middle Как обрабатывать ошибки MCP tools в workflow агента?
MCP error handling стратегии: (1) Retry с backoff — для transient ошибок (rate limit, timeout). (2) Fallback — если primary tool failed, попробовать альтернативный путь. (3) Graceful degradation — не знает live статус → сообщить пользователю и продолжить с cached/documented данными. (4) Error propagation — tool result содержит error message → агент должен явно обработать ошибку, не игнорировать. (5) User notification — если критический tool недоступен → сообщить пользователю, не молча давать неполный результат. MCP protocol определяет standard error format: error code + message.
Middle Как тестировать MCP интеграцию в проекте?
Тестирование MCP: (1) Unit tests для MCP server функций — mock external APIs, тестируй tool logic изолированно. (2) Integration tests — реальный MCP server против test environment (test GitHub repo, test DB). (3) Contract tests — проверить что tool schema соответствует реальному API response. (4) Agent behavior tests — задай агенту сценарий, проверь что он вызывает правильные tools в правильном порядке. (5) Error scenario tests — как агент ведёт себя когда tool возвращает ошибку. CI pipeline: запускай MCP integration tests при каждом изменении server кода.
Middle Как использовать MCP server в CI/CD pipeline?
MCP в CI: агент в pipeline вызывает tools для automated workflow. Примеры: (1) Code review agent — создать PR review comments через GitHub MCP. (2) Deploy agent — check deployment status, run smoke tests через cloud MCP. (3) Documentation agent — обновить docs в Confluence через MCP после deploy. Настройка: MCP server запускается как процесс в CI job, переменные окружения (API keys) из secrets manager. Важно: в CI используй read-only permissions где возможно, destructive actions (delete, merge) — только с explicit approval step. MCP transform CI от скриптов к intelligent automated workflow.
Senior — 5 вопросов
Senior Как архитектурно спроектировать MCP для complex multi-step workflow?
Complex workflow design: (1) Decompose в atomic tools — каждый tool делает одну вещь. Не create_pr_and_merge, а отдельно create_pr + merge_pr. (2) Stateless tools — tool не хранит state между вызовами. State в агенте, не в tool. (3) Idempotent operations — повторный вызов create_pr c теми же параметрами не создаёт дубль (check if exists first). (4) Tool composition — агент orchestrates sequence; tool не должен знать о других tools. (5) Error contracts — каждый tool возвращает structured error с recoverable/fatal flag. (6) Tool namespacing — github.create_pr, jira.create_issue — избегать name collision в multi-server setup. (7) Versioning — tools/v1/create_pr vs tools/v2/create_pr для backwards compat.
Senior Как диагностировать сложные MCP failures — агент не вызывает tools или вызывает неправильно?
Диагностика по слоям: (1) Tool visibility — агент видит tools? Запрос 'list all available tools' должен вернуть список. Если нет — проблема с MCP server connection или config. (2) Tool selection — агент выбирает неправильный tool → review descriptions на ambiguity. Добавь примеры вызова в description. (3) Parameter errors — tool вызывается с неверными параметрами → проверь inputSchema, добавь примеры в description. (4) Execution errors — tool вызван правильно, но server возвращает ошибку → check server logs, test tool напрямую через MCP inspector. (5) Result handling — агент получил result но игнорирует → посмотри conversation для evidence of processing. Enable verbose logging для трассировки.
Senior Как проектировать MCP tool descriptions для максимальной точности выбора агентом?
Принципы хорошего tool description: (1) Первая строка — самый важный: одно предложение 'что делает'. (2) Explicit trigger — 'Use when you need to X'. (3) Anti-triggers — 'Do NOT use for Y, use Z instead'. (4) Input semantics — описывай что значит каждый параметр в бизнес-терминах. (5) Output description — что вернёт tool и в каком формате. (6) Examples — 1–2 примера вызова. Тест на качество: дай агенту задачу и проверь выбрал ли он правильный tool и правильно ли сформулировал параметры. Итерируй descriptions на основе ошибок выбора.
Senior Как строить цепочки MCP вызовов без потери контекста и накопления ошибок?
Chain of tool calls best practices: (1) State tracking — после каждого tool call агент явно summarize полученное перед следующим. (2) Error gates — если критический tool failed → stop chain, report error пользователю. Не продолжать workflow на broken state. (3) Context compression — длинные tool results summarize перед следующим call (экономия context window). (4) Checkpoint pattern — после каждого значимого шага агент сохраняет checkpoint: 'PR создан (#123), переходим к review'. (5) Rollback design — деструктивные операции: сначала создай backup/draft, потом apply. (6) Validation — после create операции — verify: get созданный объект и проверь.
Senior Как строить MCP + RAG hybrid стратегию для enterprise AI assistant?
MCP + RAG hybrid: используй оба, маршрутизируй по типу запроса. Query classifier: (1) 'Что сейчас происходит / какой статус / live данные' → MCP first, потом RAG для контекста. (2) 'Как сделать / что написано / что решили' → RAG first, потом MCP для verification. (3) 'Сделай действие' → MCP only. Hybrid flow: RAG даёт документальный контекст (как работает AuthService), MCP даёт live state (какая сейчас версия задеплоена). Реализация: router agent определяет тип запроса → dispatches к RAG/MCP/hybrid subagent. Важно: явно в prompt агенту: 'For current facts use tools, for documentation use search'.
Lead — 5 вопросов
Lead Как выстроить governance MCP серверов в организации — кто одобряет, кто поддерживает?
Governance model: (1) Approval process — новый MCP server должен пройти security review (permissions, data access, secrets handling) и architecture review (нужен ли он, не дублирует ли существующий). (2) Registry — внутренний реестр всех одобренных MCP серверов с owner, purpose, status. (3) Ownership — каждый MCP server имеет designated owner (команда). Owner отвечает за availability, updates, deprecation. (4) Deprecation process — если server устарел: notify users, migration guide, sunset date. (5) Community servers — внешние open-source MCP: security audit перед adoption. Platform team — gateway для одобрения, product teams — requestors.
Lead Какой approval process установить для использования destructive MCP tools?
Destructive tool policy: (1) Классификация risk level — read (авто), write (авто для dev), delete/merge/deploy (require approval). (2) Approval gate types — human-in-the-loop confirm (для prod операций), PR-like review (для infra changes), dry-run first (показать что будет сделано). (3) Audit trail — каждый destructive call логируется: кто, что, когда, результат. (4) Rollback plan — destructive action должен иметь undo: backup перед delete, tag перед deploy. (5) Staging первым — обязательный dry-run в staging перед prod. (6) Rate limiting — ограничить количество destructive calls в unit time. HITL (Human-in-the-loop) обязателен для prod database changes.
Lead Как выстроить MCP security policy для enterprise?
Enterprise MCP security: (1) Zero-trust для tool permissions — каждый tool с минимально необходимыми правами. (2) Secrets management — все API keys/credentials через secrets manager (Vault, AWS Secrets Manager), никогда в конфиге. (3) Network isolation — MCP серверы в отдельном network segment, ingress/egress whitelist. (4) Audit logging — все tool calls в SIEM (Splunk, DataDog). Retention: 1 год. (5) Access control — RBAC на уровне MCP server: разные роли получают разные tools. (6) Data loss prevention — tool results проходят DLP scanner, чтобы PII не попало в conversation context. (7) Penetration testing — quarterly pentest MCP endpoints. Policy document: обновлять при каждом новом server.
Lead Как обучать команду эффективно использовать MCP и избегать ошибок?
Training program: (1) Foundations (1ч) — что такое MCP, как tool call работает, как читать tool result. (2) Tool catalog — демо всех доступных tools с примерами реальных use cases. (3) Anti-patterns (30 мин) — типичные ошибки: избыточные tool calls, игнорирование ошибок, неправильные permissions. (4) Security awareness — как защититься от tool misuse, почему нельзя давать write access когда нужен read. (5) Hands-on lab — практические сценарии: написать task для агента с tool calls, отладить failing workflow. (6) Office hours — weekly 30-min session для вопросов. Критерий успеха: engineer может написать task description вызывающий правильные tools.
Lead Как аудировать и логировать MCP использование для compliance?
Audit framework: (1) Structured logging — каждый tool call: timestamp, user/agent id, tool name, parameters (sanitized), result status, duration, error если есть. (2) PII scrubbing — в параметрах и results удалять/mask PII перед логированием. (3) Centralized log storage — все MCP logs в SIEM с retention policy (regulatory: 1–7 лет). (4) Real-time monitoring — алерты при: большом числе failed calls, unusual patterns (много destructive calls), new tool usage (security anomaly). (5) Regular audit reports — monthly: top tools by usage, error rates, permission violations. (6) Compliance reports — по запросу: все actions конкретного user/agent за период. SOC2/ISO27001: audit trail обязателен.
Architect — 5 вопросов
Arch Как спроектировать MCP как extensibility layer для enterprise AI platform?
MCP as extensibility platform: (1) Plugin architecture — MCP server как plugin: стандартный интерфейс для любого нового инструмента без изменения core agent. (2) Tool marketplace — внутренний registry одобренных MCP servers, self-service для команд. (3) Versioning — semantic versioning для tool schemas, backward compat для N-1 версий. (4) Discovery API — агент может запросить список доступных servers и их capabilities (dynamic tool loading). (5) Federated deployment — каждая команда деплоит свой MCP server, platform team managed gateway. (6) SLA framework — каждый MCP server имеет SLA: availability 99.9%, latency P95 < 200ms. Паттерн: API Gateway + service mesh для MCP серверов.
Arch Как строить внутренние MCP серверы для proprietary business logic?
Internal MCP server design: (1) Domain separation — по domain: billing-mcp, user-mcp, inventory-mcp. Каждый сервер — bounded context. (2) Thin adapter pattern — MCP server как тонкий адаптер над существующими internal APIs. Не дублируй бизнес-логику в MCP server. (3) Async tool support — для long-running operations: tool возвращает job_id, отдельный tool check_job_status. (4) Schema-first — define tool schemas в OpenAPI/JSON Schema первым, автогенерация server skeleton. (5) Testing pyramid — unit tests для business logic (не MCP), integration tests для tool API contract. (6) Documentation — каждый tool с usage examples и rate limits. Мониторинг: tool call metrics per domain сервера.
Arch Как проектировать MCP в enterprise security architecture?
MCP security architecture: (1) Zero-trust network — MCP серверы в DMZ, агент communicates через authenticated API gateway. (2) mTLS — mutual TLS между agent runtime и MCP servers. (3) Token-based auth — short-lived tokens (JWT, 1h TTL) per session, не static API keys. (4) Capability-based access — агент получает только tokens для tools которые ему разрешены (не все tools at once). (5) Request signing — каждый tool call cryptographically signed для non-repudiation. (6) Egress filtering — MCP server не может обращаться к arbitrary external URLs. (7) Threat model — MCP-specific threats: prompt injection через tool results, SSRF через URL parameters, data exfiltration через tool responses. Security review для каждого нового server.
Arch Как обеспечить observability для multi-agent workflow с множественными MCP вызовами?
Multi-agent MCP observability: (1) Distributed tracing — trace_id прокидывается через весь workflow: main agent → subagent → tool call → external service. OpenTelemetry spans на каждом уровне. (2) Agent activity log — структурированный log всех decisions и tool calls каждого агента в workflow. (3) Tool call graph — визуализация: какие tools вызывались, в каком порядке, сколько времени, что вернули. (4) Error propagation tracking — если subagent tool failed → как ошибка прошла вверх по цепочке к main agent. (5) Cost tracking — tokens + tool calls per workflow, cost attribution per agent/tool. (6) Business outcome correlation — workflow metrics → business metrics (PR review time, deployment success rate). Dashboard: timeline view workflow с drill-down на каждый tool call.
Arch Как проектировать MCP versioning и обеспечивать backward compatibility?
MCP versioning strategy: (1) Semantic versioning для tool schemas — major.minor.patch. Breaking changes только в major версии. (2) Version negotiation — при коннекте агент и server negotiates supported versions. (3) Backward compat policy — поддерживать N и N-1 major версии (deprecated = N-2). (4) Additive changes only для minor — новые optional поля, новые tools. Удаление = major. (5) Migration guides — для каждой major версии: что изменилось, как мигрировать, срок поддержки старой версии. (6) Feature flags — новые experimental tools за флагом, не ломают existing clients. (7) Contract testing — автоматические тесты совместимости: client v1 против server v1 и v2. Rollout: canary deployment нового server, monitor error rates перед full rollout.
