Context Engineering

Управление всем контекстным окном как архитектурная задача: что включить, что убрать, в каком порядке, как не потерять важное в середине. Уровень выше Prompt Engineering.

← Prompt Engineering Контекст агента

Что такое Context Engineering

Prompt Engineering = как написать один хороший запрос. Context Engineering = как управлять тем, что агент видит при каждом запросе. Это системный, архитектурный уровень.

Определение

Context Engineering — практика проектирования и управления контекстным окном агента: выбор источников, порядок их расположения, фильтрация шума, структурирование для максимального signal-to-noise ratio.

  • Что включить (нужные файлы, chunks, Memory)
  • Что исключить (.cursorignore, фильтры RAG)
  • В каком порядке (важное в начало / конец)
  • Как структурировать (Rules vs AGENTS.md vs prose)
  • Когда сжать (Checkpoint, compaction)

Почему это важнее промпта

Идеальный промпт в плохом контексте даёт плохой результат. Средний промпт в хорошем контексте — отличный результат.

  • Модель "видит" только то, что в окне
  • Лишний шум "разбавляет" важный сигнал
  • Lost in the middle: середина окна теряется
  • Stale context хуже чем отсутствие
  • Правильный контекст = предсказуемое поведение

Аналогия

Prompt Engineering — как правильно задать вопрос эксперту. Context Engineering — как правильно подготовить рабочий стол эксперта до встречи: положить нужные документы, убрать лишнее, выделить главное.

Ключевая идея: LLM — это функция от контекста. Лучший контекст → лучший ответ, даже с тем же промптом.

6 принципов Context Engineering

1. Signal over noise

Каждый токен в контексте должен помогать модели. Irrelevant файлы, старые сообщения, дублирующийся контент — это шум. Шум "разбавляет" сигнал и снижает внимание к важному.

❌ @весь_репо/
✅ @UserService.java @UserServiceTest.java

2. Primacy & Recency

LLM лучше воспринимает токены в начале и в конце контекстного окна (primacy и recency effect). Середина — "мёртвая зона" при длинном контексте. Верх окна — system instructions и Rules.

Порядок в prompt:
[System/Rules] ← начало (primacy)
[AGENTS.md / Memory]
[RAG chunks]
[User request] ← конец (recency)

3. Freshness

Stale context (устаревшие данные) опаснее отсутствия данных. Агент будет уверенно давать неверные ответы на основе устаревшей информации.

✅ Live source: MCP tool → актуальный state
❌ RAG doc from 6 months ago without date tag

4. Specificity

Конкретный, точный контекст лучше общего. Один нужный файл лучше десяти связанных. Точный chunk лучше целого документа.

❌ @src/  (500 файлов)
✅ @src/billing/PaymentProcessor.java

5. Hierarchy

Разные слои контекста имеют разный приоритет. Агент должен знать что важнее при конфликте.

Hooks (OS) > System Prompt
> Rules (alwaysApply)
> AGENTS.md
> Current chat
> Memory / RAG

6. Isolation

Subagents изолируют контекст: сложная задача со 100 файлами → subagent читает их, main agent получает только summary. Чистый контекст на каждом уровне.

Main agent: task + summary
Subagent: full corpus + analysis

Что убирать из контекста

  • Build артефакты — target/, dist/, __pycache__ → .cursorignore
  • Generated code — proto files, ORM mappings → .cursorignore
  • Secrets / .env файлы → никогда в context
  • Весь репозиторий → только нужные файлы через @mention или RAG
  • Устаревшие RAG chunks → freshness filter в pipeline
  • Длинная история чата → Checkpoint + compaction
  • Лишние MCP серверы → подключать только нужные (каждый сервер = токены)

Паттерны Context Engineering

Паттерн: Stable prefix + Dynamic suffix

Неизменяемое начало (система, правила, документы) + изменяемый конец (вопрос пользователя). Stable prefix кешируется → экономия стоимости и latency.

[System] stable ← кешируется
[Rules] stable ← кешируется
[Base docs] stable ← кешируется
[---separator---]
[User query] dynamic ← меняется

Паттерн: Subagent isolation

Сложная задача требует чтения 100 файлов → context overflow. Решение: subagent читает 100 файлов, возвращает 300-слов summary. Main agent работает с чистым контекстом.

Main agent (чистый context):
  → spawn subagent(files=[...100...])
  ← receive summary (300 words)
  → action based on summary

Паттерн: Checkpoint before compaction

Длинная сессия → история chat растёт → compaction теряет детали. Checkpoint фиксирует критические решения и состояние до того как compaction их сотрёт.

## Checkpoint (сессия 2026-07-01)
Задача: рефакторинг UserService
Сделано: методы findById, save
Решение: без Optional (Legacy runtime)
Следующий шаг: delete + update

Паттерн: RAG + MCP разделение

RAG для архивного знания (документация, API specs). MCP для live state (текущие метрики, CI статус, конфиги). Никогда не путай: live данные через RAG — stale risk.

RAG → "как работает auth flow"
MCP → "какой сейчас статус CI job #123"

Тонкости и нюансы

Lost in the middle — самый опасный эффект

При контексте 200K токенов информация на позиции 100K получает меньше attention даже если технически в окне. Следствие: критические инструкции в Rules (начало), важные факты в Memory (начало файла), Few-shot примеры (конец перед запросом). Что нельзя ставить в середину: safety constraints, ключевые технические решения, единственный релевантный chunk.

.cursorignore ≠ .gitignore — разные цели

.gitignore исключает файлы из version control. .cursorignore исключает из AI-индексации. Файл может быть в git но не в AI-контексте (например: migration scripts, large test fixtures). Аналог в Claude Code — .claudeignore. Без явного .cursorignore — Cursor индексирует всё включая build артефакты → шум в RAG + замедление поиска.

Каждый MCP сервер = токены в контексте

При подключении MCP сервера descriptions всех его tools добавляются в context при каждом запросе. 10 серверов × 10 tools × 200 символов = тысячи токенов overhead. Принцип: подключай только нужные серверы. Temporary задача (DB migration) → подключи MCP для DB на время, отключи после.

Compaction убивает детали, Checkpoint спасает

Compaction — автоматическое сжатие истории chat при достижении лимита. Конкретные имена переменных, принятые решения ("использовать Optional везде"), точные error messages — всё заменяется кратким summary. Checkpoint создаётся явно, до compaction, и фиксирует именно то что важно помнить: current state, decisions, next steps.

Порядок Rules влияет на поведение

alwaysApply Rules загружаются в context в порядке файлов. Первые правила получают primacy attention, последние — recency. Правила в середине при большом числе Rules получают меньше weight (lost in the middle для Rules). Совет: не более 3-5 alwaysApply Rules, критически важные — первыми.

Best practices Context Engineering

Делай ✅

  • Настрой .cursorignore / .claudeignore с первого дня
  • Структурируй AGENTS.md: важное — вверху (≤800 слов)
  • Используй @mention для конкретных файлов, не директорий
  • RAG top-k: 3-7 chunks (не 20)
  • Checkpoint перед compaction в длинных сессиях
  • Stable prefix → prompt caching для повторяющихся задач
  • Subagent для задач с большим объёмом чтения
  • Разделяй RAG (архив) и MCP (live state)
  • Документируй все слои контекста для команды

Не делай ❌

  • Не добавляй весь репозиторий в контекст
  • Не держи secrets в AGENTS.md или Memory
  • Не используй RAG для live state (статус CI, метрики)
  • Не пиши AGENTS.md на 2000+ слов
  • Не подключай 10 MCP серверов если нужны 2
  • Не добавляй timestamp в system prompt (ломает кеш)
  • Не смешивай temporary task state с долгосрочной Memory
  • Не игнорируй lost-in-the-middle для критических правил

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Context Engineering?
Context Engineering — дисциплина управления всем context window языковой модели как архитектурная задача. Это не просто написание хорошего промпта, а системное решение: что включить в context, что исключить, в каком порядке расположить, как структурировать для максимальной эффективности. CE охватывает Rules, AGENTS.md, RAG, Memory Bank, Checkpoints, .cursorignore — весь экосистем информации, которую видит агент.
Junior Чем Context Engineering отличается от Prompt Engineering?
Prompt Engineering — оптимизация одного запроса: формулировка, роль, примеры, CoT, формат. Работает с тем, что уже есть в context. Context Engineering — управление тем, что попадает в context: RAG chunks, Rules, AGENTS.md, Memory Bank, .cursorignore фильтры. PE отвечает на вопрос «как написать запрос», CE — на вопрос «что агент должен видеть». CE — уровень выше, системный подход. Хороший промпт в плохом context работает хуже, чем простой промпт в хорошем context.
Junior Что такое signal-to-noise ratio в context?
Signal — токены, которые помогают модели дать правильный ответ на текущий запрос: нужные файлы, релевантные правила, актуальные факты. Noise — всё остальное: нерелевантные файлы, устаревшие chunks, старые сообщения в чате, лишние Rules, раздутый AGENTS.md. Высокий SNR = агент сфокусирован на нужном. Низкий SNR = полезный сигнал тонет в шуме → hallucinations, lost-in-the-middle ошибки, непредсказуемое поведение. Цель CE — максимизировать SNR.
Junior Что такое context layers?
Context layers — слои информации, которые формируют context window агента. Основные слои (от высшего приоритета к низшему): System Prompt платформы → Rules (alwaysApply) → AGENTS.md/CLAUDE.md → Memory Bank → RAG chunks → Tool results → История чата → Текущий запрос. Каждый слой занимает токены и конкурирует за ограниченное окно. При конфликте вышестоящий слой побеждает. Context Engineering — это управление этими слоями: что включать, сколько токенов выделять каждому, как организовать.
Junior Что такое context poisoning?
Context poisoning — ситуация, когда в context попадают неверные, устаревшие или намеренно вредоносные данные, которые искажают поведение агента. Примеры: устаревший RAG chunk («порт 8080» вместо реального «9090»), противоречивые инструкции из разных Rules, prompt injection через RAG corpus (документ содержит «игнорируй предыдущие инструкции»). Poisoned context может заставить агента дать уверенно неверный ответ или выполнить нежелательное действие.
Junior Что такое context budget?
Context budget — явное распределение токенов между слоями context. Например: Rules: 2K токенов, AGENTS.md: 3K, Memory Bank: 2K, RAG chunks: 5K, chat history: 10K — итого 22K из 200K доступных. Context budget помогает контролировать чтобы один слой не «съел» все токены. Без явного budget AGENTS.md может незаметно вырасти до 10K токенов, вытесняя RAG и историю. Budget — архитектурный constraint, а не жёсткий лимит.
Junior Что включается в total context при запросе к агенту?
Total context = сумма всех слоёв при конкретном запросе: System Prompt (платформы) + Rules alwaysApply + AGENTS.md + Memory Bank + RAG chunks (retrieved для этого запроса) + Tool call descriptions + Tool results (из прошлых шагов) + История чата + Текущий запрос пользователя. Всё это суммируется в токенах и сравнивается с model context limit. Если сумма превышает лимит — compaction или ошибка. Именно поэтому CE оптимизирует каждый слой.
Junior Что такое relevance filtering?
Relevance filtering — механизм отбора информации для включения в context на основе релевантности текущей задаче. Примеры: RAG возвращает top-k chunks по семантической схожести (filtering по relevance score); auto-attached Rules применяются только к файлам по glob-паттерну; .cursorignore исключает нерелевантные файлы из AI-индекса. Без relevance filtering агент получает всё подряд → низкий SNR. Хороший filtering = агент видит только нужное для текущего запроса.
Junior Что такое context compression?
Context compression — техники сжатия информации для уменьшения объёма токенов без существенной потери смысла. Виды: 1) Compaction (автоматический): платформа сжимает историю чата при переполнении; 2) Summary: ключевые факты вместо полного текста в Memory Bank; 3) Checkpoint: структурированное резюме состояния задачи вместо длинной истории; 4) AGENTS.md компактность: 800 слов вместо 5000; 5) RAG chunking: chunk содержит только самое релевантное. Compression — trade-off: токены vs полнота информации.
Junior Что такое context overflow?
Context overflow — ситуация, когда суммарный объём всех слоёв context превышает model context limit (token limit). Последствия: платформа применяет compaction (сжимает историю чата, теряя детали), или API возвращает ошибку, или агент начинает давать деградированные ответы. Context overflow — не редкость при длинных сессиях, большом AGENTS.md, многих alwaysApply Rules и активном RAG. Context Engineering предотвращает overflow через явный budget management.
Middle — 10 вопросов
Middle Как расставить приоритеты слоёв при ограниченном context budget?
Приоритизация по убыванию важности: 1) System Prompt + alwaysApply Rules — нельзя убрать, они определяют поведение; 2) Текущий запрос пользователя — обязателен; 3) AGENTS.md + Memory Bank — постоянный контекст проекта; 4) RAG chunks — релевантные знания для задачи; 5) История чата — контекст текущей сессии; 6) Tool results — результаты предыдущих действий. При нехватке токенов: убирай от низшего приоритета. История чата → Checkpoint (сжатие). Много RAG chunks → top-3 вместо top-10. AGENTS.md раздут → compact.
Middle Когда стоит исключать информацию из context?
Исключать информацию стоит когда она: 1) Нерелевантна текущей задаче — build артефакты в .cursorignore; 2) Устарела — stale RAG chunk, outdated Memory; 3) Дублируется — одно и то же в Rules и AGENTS.md; 4) Добавляет шум без сигнала — весь репозиторий через @mention когда нужен один файл; 5) Слишком большая при низкой информационной плотности — raw logs вместо их summary. Правило: добавляй в context только то, без чего агент не может правильно выполнить задачу. Сомневаешься — исключи и посмотри влияние.
Middle Как измерять signal-to-noise ratio в context?
Практические подходы к измерению SNR: 1) Token composition audit: раздели context на слои и посчитай токены каждого. Какой % токенов напрямую релевантен текущей задаче? 2) Retrieval precision: из top-k RAG chunks — сколько реально использовал агент в ответе? Если 1 из 5 — precision 20%, остальное noise; 3) Behavioral test: убери по одному слою и проверь изменение качества — что реально влияет; 4) Ablation study на ответах: насколько изменяется качество при уменьшении context? Если не меняется — был noise; 5) Cache hit rate: unstable prefix → много noise в stable части.
Middle Какие методы context compression наиболее эффективны?
По эффективности: 1) Structured Checkpoint (лучший): явная структура «Что сделано / Решения / Следующий шаг» — заменяет 50+ сообщений истории при minimal потере; 2) Memory Bank distillation: факты вместо разговора — «stack: Python 3.11, FastAPI, PostgreSQL» вместо 10 сообщений об этом; 3) Hierarchical summarization: subagent делает детальный анализ → возвращает 500-слов summary → main agent работает с summary; 4) RAG вместо @mention: 3 релевантных chunks вместо 10 полных файлов; 5) AGENTS.md компактность: bullet points вместо прозы, только актуальные факты.
Middle Когда Context Engineering особенно критичен?
CE критичен в сценариях: 1) Длинные agentic задачи (>30 мин): без CE → compaction уничтожит детали; 2) Большой codebase: без RAG и .cursorignore → context overflow; 3) Multi-agent системы: без изоляции context → noise propagation между агентами; 4) Production automation: агент действует автономно → plохой context = неверные решения без human check; 5) Команда из 5+ разработчиков: без Rules/AGENTS.md стандартов → каждый агент ведёт себя по-разному. CE менее критичен для: коротких одноразовых запросов, простых задач с ограниченным контекстом.
Middle Как Rules влияют на качество context?
Rules как CE инструмент: 1) alwaysApply Rules добавляют стабильный сигнал при каждом запросе — языковой стандарт, запреты, workflow; 2) Auto-attached Rules добавляют контекст только релевантным файлам → экономят токены; 3) Хорошо написанное Rule = высокий SNR; плохо написанное = noise. Проблемы: слишком много alwaysApply → overhead растёт, lost in the middle для самих правил; устаревшие Rules → context poisoning; конфликтующие Rules → непредсказуемое поведение. CE best practice: alwaysApply ≤5, регулярный audit, одна идея — одно Rule.
Middle Как использовать RAG как инструмент Context Engineering?
RAG как CE: вместо «включить весь репозиторий» → включить только релевантные chunks. CE решения вокруг RAG: 1) Качество chunking: semantic chunking по функциям/классам лучше fixed-size — chunks сохраняют смысл; 2) Freshness: добавь metadata дату создания/обновления — старые chunks ранжировать ниже; 3) top-k management: top-3 при высоком precision лучше top-10 с noise; 4) Reranking: лучший chunk первым в context (primacy effect); 5) Hybrid search: BM25 + dense embeddings → лучший recall; 6) .cursorignore: исключи нерелевантные директории из индекса — меньше мусора в retrieval.
Middle Как писать компактные AGENTS.md с высоким SNR?
Принципы компактного AGENTS.md: 1) Максимум 800 слов — это жёсткий constraint; 2) Только актуальные факты: стек, ключевые решения, что нельзя делать; 3) Bullet points вместо прозы — информационная плотность выше; 4) Исключи очевидное (стандарты которые и так знает модель); 5) Ссылки вместо дублирования: «для Rules см. .cursor/rules/» а не копировать содержимое; 6) Регулярный audit: при ревью спрашивай «каждый факт здесь нужен агенту прямо сейчас?»; 7) Remove устаревшее: AGENTS.md с 2000 строк = накопленный долг. Меньше и точнее > больше и общее.
Middle Как оптимизировать context для длинных сессий?
Оптимизация для long sessions: 1) Checkpoint pattern: каждые 30-40 сообщений или при смене этапа — создавай Checkpoint файл; 2) Rolling window: при compaction — явно создай новую сессию с @checkpoint.md вместо продолжения деградирующей; 3) Memory Bank incremental update: после каждого значимого решения — добавляй в Memory (не накапливай в чате); 4) Subagent offload: token-heavy операции (анализ большого файла) → subagent, main получает summary; 5) Context health monitoring: отслеживай input_tokens при каждом запросе — рост сигнализирует о приближении к пределу; 6) Stable prefix guard: не трогай System + Rules состав в середине сессии — ломает кеш.
Middle Context Engineering vs fine-tuning: когда что выбирать?
CE: изменение контекста без изменения весов модели. Fine-tuning: изменение весов модели на специфическом dataset. Выбор: CE предпочтительнее когда: знания меняются часто (обновляемая документация), нужна быстрая итерация, задача domain-specific но не требует новых навыков, достаточен RAG + Rules. Fine-tuning нужен когда: нужен специфический стиль/формат которого нельзя достичь промптом, объём знаний огромен и в CE не помещается, нужна скорость/стоимость (fine-tuned модель меньше, дешевле). На практике: CE — первый выбор; fine-tuning — когда CE не справляется.
Senior — 5 вопросов
Senior Как проектировать context pipeline для сложной agentic задачи?
Context pipeline architecture: 1) Ingestion layer: определи источники (code, docs, memory, live data) и их freshness requirements; 2) Filtering layer: relevance gate (RAG retrieval + reranking), .cursorignore для нерелевантных директорий; 3) Assembly layer: собери слои в правильном порядке с учётом primacy/recency effect: Rules → AGENTS.md → Memory → RAG (best first) → task → query; 4) Compression layer: если budget превышен — Checkpoint вместо истории, summary вместо полных файлов; 5) Validation layer: проверь token composition перед отправкой — нет ли аномальных слоёв; 6) Caching layer: stable prefix помечается для кеширования. Pipeline явный и измеримый — каждый слой с explicit size limit.
Senior Как диагностировать context degradation в production?
Симптомы context degradation: агент игнорирует Rules, даёт противоречивые ответы, «забывает» ранние решения, повторяет уже сделанное. Диагностика: 1) Token timeline: логируй input_tokens при каждом запросе — нарастающий тренд = accumulation, скачок = что-то добавилось; 2) Layer audit: разбей context на слои и измерь каждый — нет ли аномально раздутого? 3) Compaction detection: при резком падении input_tokens — произошёл compaction → проверь потерю деталей; 4) Cache hit rate падение: нестабильный prefix → что-то изменилось в stable слое; 5) A/B test с чистым контекстом: новая сессия с только необходимым context → сравни качество. Degradation часто не очевидна — нужен мониторинг.
Senior Как A/B тестировать разные context layouts?
A/B testing context layouts: 1) Hypothesis: «Перемещение best RAG chunk первым улучшит точность ответов на X%»; 2) Test set: фиксированный набор задач с ожидаемыми категориями ответов (не exact match); 3) Control group: текущий layout; Treatment group: новый layout; 4) Metric: точность (правильный ответ?), format compliance (следует ли формату?), consistency (похожи ли ответы при повторном запуске?); 5) Sample size: минимум 50 задач для статистической значимости; 6) Confounding control: одинаковый model version, temperature, time of day при тестировании; 7) LLM-as-judge: используй отдельную модель как evaluator для scalable оценки. Без A/B testing оптимизация context — «чувство».
Senior Как оптимизировать context под attention heads LLM?
Attention optimization в CE: 1) Primacy effect (начало): System Prompt, alwaysApply Rules — помещай критические инструкции; 2) Recency effect (конец): текущий запрос, самые важные Few-shot примеры — максимум attention; 3) Middle penalty: история чата, вспомогательные RAG chunks — менее важное. При 200K window «середина» — это позиции 50K–150K, они реально получают меньше weight; 4) Chunk positioning: лучший RAG chunk всегда первым (не последним!), reranking обязателен; 5) AGENTS.md positioning: ближе к началу (после Rules), не в конце перед запросом; 6) Instruction repetition: для очень критичных инструкций (не делай X) — duplicate в начало (Rule) И ближе к запросу (в task description).
Senior Как распределить context budget для сложной multi-tool задачи?
Context budget allocation для multi-tool agentic задачи: 1) Резервируй output budget: если ожидается длинный output (план, код) — оставляй 20-30K под generation; 2) Tool overhead: каждый MCP сервер = N tool descriptions × M токенов → подключай только нужные серверы; 3) Tool results: результат tool call тоже занимает токены → truncate длинные outputs до summary; 4) Adaptive RAG: при большом tool history — уменьши top-k RAG (меньше chunks); 5) History pruning: при длинном tool chain — history компрессируй в Checkpoint раньше, чем платформа сделает это сама (хуже); 6) Priority queue: при бюджетном дефиците — инструмент выбора: Rules > Task context > RAG > History.
Lead — 5 вопросов
Lead Как выстроить framework контроля качества context в команде?
Context Quality Framework: 1) Context Policy: документ что можно/нельзя в context (PII policy, secrets policy, data classification); 2) Size SLA: AGENTS.md ≤800 слов, alwaysApply Rules ≤5, Memory Bank ≤800 слов — нарушение = PR blocker; 3) CI checks: automated проверка размеров файлов при каждом PR; 4) Quarterly audit: команда совместно проверяет все .mdc файлы, AGENTS.md, Memory на актуальность; 5) Rules ownership model: каждое правило имеет owner, last_reviewed дату; 6) Incident process: при «странном» поведении агента → root cause в context layers → post-mortem → fix в Rules/AGENTS.md; 7) Quality metrics dashboard: average context size, cache hit rate, compaction frequency.
Lead Как обучить команду Context Engineering?
CE обучение в команде: 1) Workshop «Token Awareness»: покажи реальный context в Cursor debug mode, посчитай токены каждого слоя вместе — видимость создаёт понимание; 2) Pairing session: senior разработчик показывает свой workflow — как выбирает @mention, как строит Checkpoint, когда использует subagent; 3) CE anti-patterns gallery: реальные примеры из команды — «вот что пошло не так из-за контекста»; 4) Checkpoint habit: team challenge — 2 недели каждый создаёт Checkpoint при задачах >30 мин. Ретро: что изменилось; 5) Context health check ritual: в начале недели — быстрый audit AGENTS.md и Rules; 6) Code review addition: в checklist добавить «не добавляет ли PR лишнего в alwaysApply или AGENTS.md?»
Lead Какие метрики context effectiveness использовать?
Context Effectiveness Metrics: 1) Cache hit rate: usage.cache_read_input_tokens / total_tokens → target >50%. Показывает стабильность prefix; 2) Compaction frequency: как часто происходит compaction в сессиях команды → высокая частота = context не управляется; 3) RAG retrieval precision: из top-k chunks — сколько реально цитировано в ответе. Если 1 из 5 → precision 20%; 4) Context size per task: сколько токенов при стандартной задаче → тренд роста = накопление долга; 5) Rules compliance rate: при code review — как часто агент нарушил Rules. Высокий rate → правила lost-in-middle или conflicting; 6) Agent restart frequency: как часто разработчики начинают новый чат «потому что агент деградировал» → indirect measure compaction damage.
Lead Как управлять governance context design decisions?
Context Governance: 1) Decision record: при значимых изменениях context architecture (новый Memory Bank паттерн, изменение Rules структуры) — ADR (Architecture Decision Record) с rationale; 2) Change control для alwaysApply Rules: изменения влияют на всю команду → mandatory review, не просто PR; 3) Experimentation process: хочешь изменить context layout → A/B test на 1 неделе → метрики → решение; 4) Rollback capability: при деградации поведения агента — git revert Rules/AGENTS.md; 5) Communication: при изменениях context, влияющих на всю команду — explicit announcement (Slack, team meeting), не просто merge; 6) External context audit: раз в квартал — кто-то из другой команды проверяет ваш context setup. Внешний взгляд находит то, что замылился.
Lead Как организовать checklist context review при code review?
Context Review Checklist для PR: 1) alwaysApply count: количество alwaysApply правил ≤5? Новое правило действительно universal или может быть auto-attached? 2) AGENTS.md size: слово-счёт ≤800? Каждый добавленный факт необходим? 3) .cursorignore coverage: новые build директории добавлены в ignore? 4) Memory Bank changes: если Memory обновляется агентом автоматически — review изменений; 5) Rules conflicts: новое правило не конфликтует с существующими? 6) Checkpoint template: если задача была долгой — создан ли Checkpoint? 7) RAG corpus: при добавлении документов — правильно ли структурированы для chunking? 8) Security: нет ли PII в примерах или AGENTS.md? Этот checklist занимает 2-3 минуты и предотвращает context debt.
Architect — 5 вопросов
Arch Как рассматривать Context Engineering как архитектурный слой системы?
Context Engineering как архитектурный слой: CE — это не «настройка агента», это infrastructure layer определяющий качество системы. Архитектурные свойства: 1) Scalability: как context architecture масштабируется при росте codebase, команды, числа агентов; 2) Reliability: насколько context предсказуем и воспроизводим между сессиями; 3) Observability: есть ли мониторинг token composition, cache rates, compaction; 4) Maintainability: как легко обновлять Rules, AGENTS.md при изменении системы; 5) Security: context как attack surface (prompt injection через RAG). В правильной архитектуре: CE решения документированы как ADR, мониторятся как infrastructure, тестируются как код.
Arch Как реализовать context routing между агентами в multi-agent системе?
Context routing architecture: разные агенты получают разные контексты в зависимости от роли. Implementation: 1) Orchestrator agent: полный context (AGENTS.md, Memory, все Rules) для принятия высокоуровневых решений; 2) Specialist agents: только domain-specific контекст (security agent видит security Rules + security docs; data agent видит data schemas + data docs); 3) Context injection protocol: orchestrator явно передаёт context пакет каждому subagent (не все видят всё); 4) Context versioning: при передаче context между агентами — версия snapshot чтобы агенты работали с consistent view; 5) Context registry: центральный сервис знает какой agent получает какой context — observability и governance; 6) Principle of least context: каждый agent получает минимально необходимый context.
Arch Что такое versioned context specifications и зачем они нужны?
Versioned context specifications — explicit versioning всего context setup: Rules, AGENTS.md, Memory schema, RAG corpus version, prompt templates. Зачем: 1) Reproducibility: воспроизведи точно то поведение агента которое было в moment X — нужно для debugging production incidents; 2) Rollback: при деградации — не «что изменилось?», а git revert context spec до предыдущей версии; 3) A/B testing: v1.2 context vs v1.3 context на split трафике; 4) Compliance: audit trail что агент видел в конкретный момент (важно для regulated industries); 5) Team onboarding: новый разработчик получает точно тот же context setup что и команда. Implementation: context spec как code в repo, versioned с semver, с changelog.
Arch Как организовать context monitoring в production?
Production context monitoring stack: 1) Metrics: input_tokens (total + per layer), cache_hit_rate, compaction_count, context_overflow_count, p50/p95 latency — в time series (Prometheus/Datadog); 2) Alerts: input_tokens растёт >10% в неделю → alert (накопление долга); cache_hit_rate <30% → alert (нестабильный prefix); compaction_count spikes → alert (сессии деградируют); 3) Tracing: каждый запрос с context composition breakdown — какие Rules загружены, сколько RAG chunks, размер Memory → для debugging; 4) Anomaly detection: резкое изменение average context size после deploy → correlation с изменением Rules/AGENTS.md; 5) User feedback loop: при «странном поведении» агента → автоматически capture context snapshot для анализа.
Arch Как применять Context Engineering для системы с RAG + Memory Bank + MCP?
Интегрированная CE архитектура для RAG + Memory + MCP: 1) Token budget allocation: явно разделить 200K window: System+Rules (5K), AGENTS.md (3K), Memory Bank (2K), RAG chunks (8K), MCP tool descriptions (5K), MCP results (10K), chat history (10K), output buffer (30K) — итого ~73K используется, 127K резерв; 2) Priority preemption: при перегрузке — MCP results truncate (summary), затем RAG chunks reduce, затем chat history → Checkpoint; 3) Freshness hierarchy: MCP > RAG > Memory для live данных; Memory > RAG для persistent знаний проекта; 4) Anti-duplication: не держи одно знание и в Memory и в RAG — выбери canonical source; 5) Circuit breaker: если MCP сервер возвращает огромный result → automatic truncation + summary hook; 6) Unified observability: мониторь все три источника как единый context budget.
