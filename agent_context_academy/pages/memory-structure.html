5 типов памяти в AI-системах

Не вся «память» одинакова. Каждый тип отвечает на свой вопрос и живёт в своём слое.

💬

In-context (session)

Текущий чат и история диалога. Исчезает при compaction или новом чате. Самая «свежая» — приоритет при конфликте.

Живёт: текущий чат
📁

External / Project memory

Файлы в репо: memory.md, .cursorrules/memory. Постоянна между сессиями. Это и есть «Memory Bank».

Живёт: файл в проекте
🔍

Semantic (RAG)

Векторное хранилище документов. Находит по смыслу, не хранит структурно. Это не Memory Bank — это retrieval.

Живёт: vector DB
📌

Episodic (Checkpoint)

Структурный снимок состояния задачи: что сделано, решения, риски. Не общие знания, а конкретная задача.

Живёт: файл checkpoint
👤

User / Preference

Предпочтения разработчика: стиль ответа, язык, детальность. Хранится в глобальном AGENTS.md или Memory Bank.

Живёт: глобальный Memory

Пример структуры Memory Bank — папка memory-bank/

Один файл memory.md достаточен для маленького проекта. Когда память растёт, её делят на роли: каждый файл отвечает за один тип знаний — агент подключает нужные через @memory-bank/… в AGENTS.md или по запросу.

memory-bank/ ├── projectbrief.md # что строим (продукт, цели) ├── activeContext.md # фокус прямо сейчас ├── progress.md # сделано / в работе / блокеры ├── systemPatterns.md # как принято писать код в репо ├── techContext.md # стек, команды, окружение └── roadmap.md # планы (необязательно)
ФайлДля чегоЧто писатьКак часто обновлять
projectbrief.md Смысл продукта для агента: границы, пользователи, MVP, что вне скоупа 1–2 экрана: проблема, цели, ключевые ограничения; ссылка на полный PRD в docs/ Редко (новый продукт / pivot)
activeContext.md «Где мы сейчас»: ветка, задача, открытые вопросы, что нельзя трогать в этой итерации Короткий список: current focus, next step, blockers; дата последнего обновления Часто — конец сессии или смена задачи
progress.md Хронология работ: что завершено, что в процессе, известные долги Done / In progress / Todo; можно по спринтам или по фичам Регулярно; устаревшее — вычищать, не копить вечный лог
systemPatterns.md Архитектурные и кодовые паттерны: слои, naming, error handling, типовые структуры модулей «В этом репо X делаем так» + ссылка на ADR; не дублировать весь design doc Средне — при новых ADR или рефакторинге стиля
techContext.md Технический контекст: стек с версиями, build/test, env, внешние сервисы Команды одной строкой; gotchas окружения; что поднимать локально При смене стека, CI или зависимостей
roadmap.md (необязательно) Ближайшие приоритеты релиза — чтобы агент не уезжал в «большую перестройку» Квартал / 2–3 эпика; детальный roadmap — в issue tracker или docs/ По планированию; не путать с учебным Roadmap академии

Как подключить агенту

В AGENTS.md перечисли обязательные файлы, например: @memory-bank/projectbrief.md, @memory-bank/techContext.md, @memory-bank/activeContext.md. Тяжёлые или редко нужные — только по @mention в задаче. Checkpoint и временное состояние задачи — отдельно, см. Checkpoint, не смешивать с projectbrief.

Один memory.md vs папка: до ~80–100 строк держи один файл. Дальше — разнеси по таблице выше; иначе срабатывает lost in the middle и агент «не видит» середину файла.

Структура memory.md — что и как писать

Хорошая структура

# Project Memory — order-service ## Last updated: 2025-06-15 ## Stack & Environment - Java 21, Quarkus 3.x, JPA/Hibernate - PostgreSQL 15 (no raw SQL) - Kafka for events - K8s namespace: order-ns ## Build & Test Commands - Test: `mvn test -pl order-service` - Build: `mvn clean install -DskipTests` - IT: `mvn verify -Pit -pl order-service` ## Architecture Decisions # ADR-001: No raw SQL — use JPA only # ADR-002: Kafka for cross-service events # ADR-003: Circuit breaker via Quarkus fault tolerance ## Team Conventions - PR summary: Problem → Fix → Tests → Risk - No dependency without tech-lead approval - Rollback plan required for DB migrations ## Known Issues / Gotchas # UserService.findById returns Optional, not null # BillingModule has 30s timeout — don't mock it away ## User Preferences - Respond in Russian - Show code examples for every explanation

Что НЕ класть в memory

Секреты: пароли, токены, ключи → только env vars
Краткоживущие статусы: "сейчас сломан CI" → устареет
Сырые логи / stack traces → слишком шум
Данные которые часто меняются → лучше через MCP/RAG
Большая документация → это RAG, не Memory

Что всегда класть в memory

Архитектурные решения — почему выбрали X, а не Y
Стабильные команды — mvn test, npm run, docker-compose
Team conventions — PR format, naming, review rules
Gotchas проекта — нюансы которые всегда забывают
Preferences — стиль ответа, язык, детальность

Memory lifecycle — 5 шагов

1

Capture

Агент или разработчик выделяет факт / решение, которое будет полезно позже

→
2

Curate

Сжать, убрать шум, сформулировать устойчиво. Один факт — одна строка.

→
3

Retrieve

Подтянуть memory только когда релевантна. Не грузить весь файл всегда.

→
4

Validate

Если факт мог измениться — проверить через MCP/файл/RAG. Memory может устареть.

→
5

Invalidate

Удалить или обновить устаревшую запись. Stale memory хуже чем отсутствие.

Главная ловушка: Memory никогда не обновляется автоматически. Если изменился стек, workflow или ADR — память устаревает молча. Результат: агент действует на основе старой информации. Добавьте в AGENTS.md: "если memory противоречит текущему файлу — доверяй файлу".

Best practices управления Memory

Пишем правильно

  • Один факт — одна строка. Краткость — залог читаемости.
  • Дата last-updated. Без неё неизвестно насколько свежа память.
  • Секции по типу: Stack, Commands, Decisions, Conventions, Gotchas.
  • Глагол + объект: "Use JPA, not raw SQL" а не "database stuff".
  • ADR-ссылки: почему, а не только что.

Используем правильно

  • @memory.md в запросе если нужен конкретный контекст.
  • Ссылка в AGENTS.md: @docs/memory.md для автозагрузки.
  • Memory + MCP: Memory хранит решение, MCP проверяет актуальное состояние.
  • Memory + Checkpoint: Memory — общее, Checkpoint — конкретная задача.
  • При конфликте: MCP/файл/RAG > Memory (свежее побеждает).

Обновляем правильно

  • После смены стека — первым делом обновить memory.
  • После ADR — добавить решение в memory и дать ссылку на ADR файл.
  • Раз в спринт — review: что устарело? Что удалить?
  • Pull request к memory.md — как любой документ, через review.
  • Дата изменения — обновлять в заголовке файла.

Тонкости структуры Memory

activeContext.md перезаписывается слишком часто → теряешь историю

Если агент обновляет activeContext.md при каждом чеке, предыдущий контекст исчезает безвозвратно. Решение: использовать append-режим (добавлять запись с датой вниз) или версионировать через git-историю. Либо разделить файл на «текущее» и «последние N сессий».

Слишком большой memory файл = не читается полностью

При compaction или при загрузке агент может не прочитать весь файл — срабатывает эффект «lost in the middle». Держи каждый файл Memory Bank < 100 строк. Всё что не помещается — выносить в специализированный файл (decisions.md, techStack.md) и добавлять в main-файл ссылку.

projectBrief.md ≠ AGENTS.md

projectBrief.md описывает что строим — продукт, бизнес-цели, stakeholders. AGENTS.md описывает как работает агент — workflow, команды, conventions, процессы. Не мешай их в один файл: разные жизненные циклы и разные аудитории.

Memory Bank не синхронизируется между сессиями автоматически

Если несколько агентов или несколько параллельных сессий — каждый пишет в свои файлы без автоматического merge. Нет встроенного механизма reconciliation. Нужен явный протокол обновления: один «главный» агент как memory-coordinator, либо human-review перед слиянием изменений.

Best practices — Делай / Не делай

✅ Делай

  • Структурируй по файлам: brief / context / progress / decisions — каждый файл отвечает за один тип знаний.
  • Держи файлы < 100 строк: агент читает их целиком; потеря середины при превышении лимита.
  • Обновляй activeContext в конце сессии: зафиксируй текущий фокус, открытые нити, принятые решения.
  • Фиксируй решения с обоснованием: не только «что», но и «почему» — в decisionLog с датой и альтернативами.

❌ Не делай

  • Не смешивай всё в один файл: god-memory.md с 500+ строками теряет эффективность и читабельность.
  • Не обновляй memory слишком часто внутри одной сессии: overhead на запись превысит пользу; обновляй раз — в конце значимой сессии.
  • Не дублируй AGENTS.md и projectBrief: разные документы, разные назначения — дублирование ведёт к расхождению при обновлении.

Вопросы по уровням

Junior — 10 вопросов
Junior Какие файлы обычно входят в Memory Bank?
Типичный набор файлов Memory Bank: memory.md (основной файл с устойчивыми фактами) или структурированный набор: activeContext.md (текущее состояние и фокус), projectBrief.md (обзор проекта), progress.md (прогресс задач и завершённые этапы), decisions.md (архитектурные решения с обоснованием). Некоторые команды добавляют: techStack.md, conventions.md, gotchas.md. Минимальный вариант — один memory.md с секциями. Структурированный набор удобнее для больших проектов: каждый файл отвечает за свой тип знаний.
Junior Что такое activeContext.md и для чего он нужен?
activeContext.md — файл с описанием текущего состояния работы: над чем ведётся работа прямо сейчас, какой контекст активен, текущие приоритеты, незавершённые нити. Он отвечает на вопрос "что происходит сейчас?" и обновляется чаще других файлов Memory Bank. Используется для continuity между сессиями: агент читает activeContext.md и сразу понимает на чём остановились. В отличие от progress.md (история завершённого), activeContext.md — про текущий момент.
Junior Что такое projectBrief.md?
projectBrief.md — высокоуровневое описание проекта: что это, зачем, ключевые бизнес-требования, stakeholders, ограничения. Аналог executive summary. Обновляется редко — только при изменении бизнес-контекста. Читается агентом для понимания "зачем мы вообще это делаем" перед принятием технических решений. Отличие от AGENTS.md: AGENTS.md = "как работать"; projectBrief.md = "что и зачем". Полезен при onboarding и при принятии архитектурных решений с учётом бизнес-контекста.
Junior Что такое progress.md в Memory Bank?
progress.md — журнал завершённых этапов и результатов: что было сделано, когда, с каким результатом. Отвечает на вопрос "что уже сделано?" Полезен при длительных проектах и при handoff между разработчиками. Структура: список завершённых задач с датами и ключевыми выводами. Отличие от activeContext.md: progress = история; activeContext = настоящее. Отличие от Checkpoint: Checkpoint создаётся внутри текущей задачи; progress.md накапливается долгосрочно.
Junior Как правильно называть файлы Memory Bank?
Нет жёсткого стандарта, но рекомендуемые конвенции: camelCase (activeContext.md, projectBrief.md) или kebab-case (active-context.md, project-brief.md) — выбрать один стиль для всего проекта. Имена должны быть описательными: что хранит файл, не где он используется. Избегать: memory1.md, notes.md (не понятно содержимое). Хорошие имена: decisions.md, gotchas.md, techStack.md. Разместить в одной директории (docs/memory/ или .memory/) для удобства навигации.
Junior Каков минимальный набор файлов для Memory Bank?
Минимальный набор: один файл memory.md со структурированными секциями — это достаточно для большинства проектов. Секции в минимальном memory.md: ## Stack (технологии, версии), ## Build & Test (команды), ## Architecture Decisions (ключевые ADR), ## Conventions (стиль, PR format). Расширенный минимум (2-3 файла): memory.md + activeContext.md + decisions.md. Больше файлов нужно только когда один файл становится слишком большим (>100-150 строк) или когда разные файлы нужны разным агентам.
Junior Как создать первый Memory Bank с нуля?
Шаги создания: (1) Создать файл docs/memory.md в корне репо; (2) Добавить заголовок с датой: "# Project Memory — [Project Name]\n## Last updated: YYYY-MM-DD"; (3) Заполнить базовые секции: Stack, Build & Test, Architecture Decisions, Conventions; (4) Добавить ссылку в AGENTS.md: @docs/memory.md; (5) Закоммитить в git. Начать с малого — 20-30 строк достаточно. Дополнять в процессе работы когда появляются новые факты достойные сохранения. Memory Bank улучшается использованием, а не идеальным планированием.
Junior Что писать в каждый файл Memory Bank?
memory.md: устойчивые факты — стек, команды, ADR, конвенции. activeContext.md: текущий фокус, незавершённые нити, последние решения. projectBrief.md: бизнес-контекст, stakeholders, ограничения, цели проекта. progress.md: хронология завершённых этапов с результатами. decisions.md: ADR с датой, обоснованием, альтернативами. Общий принцип: один факт — одна строка; конкретно, не абстрактно; с датой если факт может устареть. Никогда: секреты, временные статусы, volatile данные.
Junior Как файлы Memory Bank связаны между собой?
Связи между файлами через markdown-ссылки и cross-references: memory.md может ссылаться на decisions.md для деталей ADR; activeContext.md ссылается на конкретные задачи или файлы кода; progress.md ссылается на решения принятые при завершении этапов. Пример: в memory.md запись "ADR-002: Kafka for events — подробнее в decisions.md#adr-002". Агент при необходимости может запросить конкретный файл через @mention. Разумная cross-reference стратегия: memory.md = index, детали = в специализированных файлах.
Junior Где хранить Memory Bank в структуре репозитория?
Рекомендуемые места: docs/memory/ (группирует с другой документацией), .memory/ (скрытая директория для AI-специфичного контента), или просто memory.md в корне (максимально просто). Cursor: добавить путь в AGENTS.md через @docs/memory.md. Claude Code: @include в CLAUDE.md. Избегать: node_modules, src, test — логически неправильно. Не добавлять в .gitignore — Memory Bank должен версионироваться вместе с кодом. Выбор места зависит от проекта: для маленьких — корень; для больших — docs/memory/.
Middle — 10 вопросов
Middle Какова оптимальная структура Memory Bank для реального проекта?
Оптимальная структура для среднего проекта (5-20 разработчиков): (1) memory.md — главный файл с сводкой (≤100 строк), содержит Stack, Build Commands, ключевые ADR, ссылки на детальные файлы; (2) decisions.md — полный список ADR с датами и обоснованием; (3) activeContext.md — текущий фокус и незавершённые нити; (4) techStack.md — детальное описание технологий; (5) conventions.md — coding style, PR format, review rules. Связующий AGENTS.md: @docs/memory/memory.md для автозагрузки главного файла. Агент подгружает детальные файлы по @mention когда нужно.
Middle Как разбить Memory Bank на файлы для удобного использования?
Критерии разбивки: (1) По жизненному циклу — стабильные факты (techStack) vs изменчивые (activeContext); (2) По типу знаний — решения отдельно от состояния; (3) По доменам — billing-decisions.md, auth-gotchas.md для крупных модулей; (4) По аудитории — командные знания отдельно от модульных. Правило разбивки: разбивать когда файл >100-150 строк ИЛИ когда разным агентам нужны разные части. Не разбивать ради разбивки — сложность управления растёт. Минимально жизнеспособная структура: один файл пока проект позволяет.
Middle Какие секции должны быть в activeContext.md?
Рекомендуемые секции activeContext.md: ## Current Focus (над чем работаем сейчас, sprint goals), ## Open Threads (незавершённые исследования, открытые вопросы), ## Recent Decisions (решения принятые в последние 1-2 недели, пока не устоялись), ## Blockers (что мешает прогрессу), ## Next Steps (ближайшие действия). Обновлять в начале и конце рабочей сессии. Lifetime записи: до разрешения (Open Threads, Blockers) или переноса в decisions.md (Recent Decisions). activeContext.md — самый живой файл Memory Bank.
Middle Как структурировать decisions.md?
Структура decisions.md: каждое решение как отдельная секция с шаблоном: ## ADR-NNN: [Краткое название] (дата). Status: Accepted/Deprecated/Superseded. Context: почему встала проблема. Decision: что решили. Rationale: почему именно так. Alternatives considered: что отвергли и почему. Consequences: что изменится, trade-offs. Нумерация ADR: последовательная, не переиспользовать номера устаревших. При изменении решения: добавить новый ADR со ссылкой "Supersedes ADR-XYZ", старый пометить Deprecated.
Middle Как писать компактные и информативные записи в Memory Bank?
Принципы compact writing: (1) Verb + object: "Use JPA, not raw SQL" вместо "мы решили использовать JPA а не raw SQL по причине..."; (2) Include WHY: "не используем Redis (overhead для нашей нагрузки, Q1 2026)" — даёт контекст почему; (3) Числа вместо слов: "timeout: 30s" не "таймаут тридцать секунд"; (4) Ссылки на детали: "ADR-003 (см. decisions.md)" вместо полного описания; (5) Даты при возможной устарелости: "Java 21 (обновлено Jun 2026)". Цель: максимум информации на минимум токенов.
Middle Как обновлять Memory Bank в конце сессии?
End-of-session update ритуал: (1) Обновить activeContext.md — текущий фокус, что сделано, открытые нити; (2) Если принято новое решение — добавить в decisions.md; (3) Если обнаружен gotcha — добавить в memory.md секцию Known Issues; (4) Если изменилась команда тестов/сборки — обновить соответствующую секцию; (5) Обновить дату "Last updated" в заголовке. Агент может предложить обновления сессии ("по результатам нашей сессии я предлагаю добавить в memory...") — человек review'ит и принимает решение. Не обновлять memory бесконтрольно.
Middle Какой шаблон Memory Bank использовать для нового проекта?
Минимальный шаблон memory.md для нового проекта: "# Project Memory — [Name]\n## Last updated: YYYY-MM-DD\n\n## Stack\n- [tech] [version]\n\n## Build & Test\n- Test: `[command]`\n- Build: `[command]`\n\n## Architecture Decisions\n- ADR-001: [решение] — [почему]\n\n## Conventions\n- [главное соглашение]\n\n## Known Issues / Gotchas\n- (пока пусто)". Начать с заполнения Stack и Build & Test — это немедленно полезно агенту. Остальные секции заполнять по мере появления знаний. Не пытаться заполнить идеально с первого раза.
Middle Как структурировать Memory Bank для API, frontend, full-stack проектов?
Для API проекта: Stack (язык, фреймворк, DB), Endpoints convention, Auth approach, Error handling pattern, DB migration policy. Для Frontend: Stack (framework, state management), Component naming, Routing conventions, Build/Deploy команды, Accessibility rules. Для Full-stack: разделить на секции backend/frontend или отдельные файлы (backend-memory.md, frontend-memory.md) с общим memory.md как index. Для микросервисов: memory.md per-service в соответствующей директории, корневой memory.md содержит межсервисные соглашения и API contracts.
Middle Как избежать дублирования между файлами Memory Bank?
Дублирование — главный враг Memory Bank: приводит к рассинхронизации. Стратегии: (1) Single source of truth — стек описывается только в memory.md, решения только в decisions.md; (2) Ссылки вместо копирования — "подробнее в decisions.md#adr-002"; (3) Регулярный дублирование-audit при PR review; (4) Четкое разграничение ответственности файлов — что в каком файле; (5) При обнаружении дублирования — немедленно удалить из одного источника. AGENTS.md и Memory Bank тоже не должны дублировать друг друга: AGENTS.md = workflow, Memory Bank = project facts.
Middle Каковы guidelines по размеру каждого файла Memory Bank?
Size guidelines: memory.md (главный) — ≤100 строк / ≤800 слов; агент читает целиком при каждом запросе, lost in the middle при превышении. decisions.md — может быть больше (до 300-500 строк) — читается по запросу через @mention, не всегда в context. activeContext.md — ≤50 строк; обновляется часто, должен быть компактным. techStack.md — ≤80 строк. Общий принцип: файл который всегда в context должен быть компактным; файлы которые читаются по запросу — могут быть объёмными. Мониторить: если агент игнорирует факты из середины файла — файл слишком длинный.
Senior — 5 вопросов
Senior Как структурировать Memory Bank для monorepo?
Monorepo Memory Bank архитектура: (1) Root memory.md — общее для всего monorepo: organization conventions, shared tooling, cross-service contracts, CI/CD; (2) Per-service memory — billing/docs/memory.md, auth/docs/memory.md — специфика каждого сервиса; (3) Cross-references — root memory ссылается на service memories для деталей; (4) Shared decisions.md — архитектурные решения затрагивающие несколько сервисов; (5) Service-specific decisions в subdirectory decisions.md. Агент при работе в billing/ читает root memory + billing/memory. Важно: не дублировать org-level знания в каждом сервисе.
Senior Как Memory Bank эволюционирует вместе с проектом?
Эволюция Memory Bank через стадии: (1) Bootstrap (0-3 мес) — один memory.md с базовым стеком и командами; (2) Growth (3-12 мес) — добавляются decisions.md, activeContext.md; растёт список ADR; появляются gotchas; (3) Maturity (12+ мес) — разбивка на файлы по доменам; cleaning устаревших записей; появление deprecated ADR; (4) Large scale — federated memory с per-service файлами; automated validation; Memory Bank становится частью CI. Ключевое: не планировать сложную структуру заранее — она вырастает из потребностей. Сложность Memory Bank должна соответствовать сложности проекта.
Senior Как диагностировать и починить раздутый Memory Bank?
Признаки раздутого Memory Bank: агент цитирует только начало и конец файла; memory.md >200 строк; много устаревших или временных записей; дублирование между файлами. Диагностика: (1) Проверить каждую запись — актуальна? Нужна агенту? (2) Найти дубликаты; (3) Найти volatile данные (статусы, временные решения). Починить: (1) Удалить явно устаревшее; (2) Перенести детальные ADR из memory.md в decisions.md, оставить только ссылки; (3) Volatile данные → убрать; (4) Большие документы → RAG corpus. Цель: вернуться к ≤100 строкам в главном файле.
Senior Как проектировать schema Memory Bank?
Schema Memory Bank — формализованная структура записей для предсказуемого парсинга. Зачем нужна: автоматический validation, automated updates через CI, cross-referencing между файлами. Подходы: (1) Markdown conventions — зафиксировать формат ADR записей, naming conventions, обязательные секции в каждом файле; (2) YAML frontmatter — добавить структурированные метаданные к каждому ADR; (3) JSON sidecar — отдельный memory-schema.json описывает структуру. CI validation: проверять что все файлы соответствуют schema, обязательные секции присутствуют, ADR имеют номера и даты. Schema делает Memory Bank machine-readable, не только human-readable.
Senior Как организовать cross-reference стратегию между файлами Memory Bank?
Cross-reference стратегия: (1) Anchor links — "#adr-002" в decisions.md для точного указания; (2) Bidirectional references — если memory.md ссылается на decisions.md, decisions.md должен содержать тег для обратной навигации; (3) Dependency tracking — "ADR-005 depends on ADR-002" — при изменении ADR-002 проверить ADR-005; (4) File index — memory.md содержит раздел "## References" со ссылками на все специализированные файлы; (5) Agents — в activeContext.md ссылки на конкретные файлы кода которые релевантны текущей работе. Anti-pattern: circular references; undefined references (битые ссылки); reference на временные файлы.
Lead — 5 вопросов
Lead Как стандартизировать структуру Memory Bank для всей команды?
Стандартизация: (1) Team template — создать memory-template/ в shared репо с skeleton файлами; новые проекты копируют при старте; (2) Required sections checklist — задокументировать что обязательно в каждом файле; (3) CI validation — automated check что обязательные секции присутствуют; (4) Naming convention — зафиксировать camelCase vs kebab-case, не смешивать; (5) Size limits — договориться о max размерах каждого файла; (6) Update triggers — список событий при которых каждый файл обновляется. Документировать стандарт в AGENTS.md: "Memory Bank located in docs/memory/, структура — см. docs/memory/README.md".
Lead Как организовать governance файлов Memory Bank?
Governance: (1) Git workflow — все изменения через PR, не прямой commit в main; (2) CODEOWNERS — назначить owner для каждого файла Memory Bank; memory.md → tech lead, decisions.md → architect или tech lead; (3) Review criteria — что проверять при review Memory Bank PR: актуальность, корректность, отсутствие дублирования, наличие дат; (4) Automated checks — CI lint проверяет size limits, required sections, ADR numbering; (5) Quarterly audit — scheduled review всего Memory Bank на устаревшее; (6) Incident process — при ошибке агента основанной на stale memory → post-mortem и обновление Memory Bank.
Lead Что коммитить в git из Memory Bank?
Всё коммитить: memory.md, decisions.md, activeContext.md, progress.md — все файлы Memory Bank в git. Обоснование: (1) История изменений = история архитектурных решений; (2) Reproducibility — новый разработчик получает полный контекст; (3) Rollback — можно вернуться к предыдущему состоянию знаний; (4) Collaboration — команда работает с одной версией контекста. НЕ коммитить: локальные personal notes которые не готовы для команды (держать в ~/.claude/ или gitignore); секреты которые случайно попали (немедленно удалить из истории). Правило: если знание нужно команде — в git; если только тебе — в личный ~/.claude/.
Lead В чём разница между team Memory Bank и individual Memory Bank?
Team Memory Bank (в репо, git) — командные знания: архитектурные решения, team conventions, project-specific gotchas. Версионируется, review, доступен всем. Individual Memory Bank (в ~/.claude/ или gitignored) — личные предпочтения: стиль ответа агента, личные shortcuts, эксперименты. Не нужен команде. Когда переносить из individual в team: факт стал релевантен для всей команды ("обнаружил что модуль X имеет timeout 30s — всем нужно знать") → переносить в командный memory. Граница размытая: при сомнении спросить себя "это нужно агенту коллеги?" — если да, в командный.
Lead Как создать шаблоны Memory Bank для разных типов проектов?
Шаблоны по типам: API-only — акцент на endpoints conventions, auth pattern, error handling, DB migration policy, API versioning strategy. Frontend — компонентная архитектура, state management, routing, a11y rules, bundle optimization. Microservices — service mesh, event contracts, distributed tracing, circuit breaker patterns, inter-service communication. ML/AI — model versioning, dataset management, experiment tracking, deployment pipeline. CLI tool — argument conventions, output format, error codes. Для каждого шаблона: обязательные секции + типичные gotchas специфичные для типа проекта. Шаблоны хранить в organization shared репо и обновлять по мере накопления опыта.
Architect — 5 вопросов
Arch Как Memory Bank реализует паттерн structured knowledge store?
Memory Bank как structured knowledge store: явная структура (секции, markdown headers), curated content (только то что прошло human review), versioned (в git), contextually loaded (агент читает только релевантные файлы). В отличие от unstructured knowledge (wiki, конфлюенс) — Memory Bank оптимизирован под потребление LLM: компактный, structured, с высокой information density. Паттерны: (1) Hierarchical knowledge — summary → details через @include; (2) Domain partitioning — knowledge по доменам; (3) Temporal tagging — даты для staleness detection. Memory Bank реализует "knowledge as code" принцип: знания версионируются и проходят review как код.
Arch Как организовать linked files Memory Bank в multi-agent архитектуре?
Multi-agent linked Memory: (1) Main agent — читает root memory.md (index), при необходимости запрашивает детальные файлы; управляет writes в Memory Bank; (2) Specialized agents — читают только релевантные файлы (billing-agent → billing-memory.md); readonly или restricted write к своему домену; (3) Coordination layer — main agent как memory coordinator: решает что и куда записать из findings subagents; предотвращает concurrent write conflicts; (4) Event-driven updates — subagent completion event триггерит предложение обновления Memory Bank; main agent или человек approve'ит. Ключевое: один writer per файл для предотвращения conflicts.
Arch Как управлять versioning схемы Memory Bank?
Schema versioning для Memory Bank: (1) Semantic versioning файлов — YAML frontmatter: "schema_version: 2.1.0"; breaking changes → major version; (2) Migration scripts — при изменении schema, скрипт мигрирует существующие файлы к новому формату; (3) Backward compatibility — новая версия читает старый формат; (4) Git-based versioning — schema changes через PR с migration guide; (5) Schema registry — centralized definitions для organization. Когда нужен schema versioning: при automated processing Memory Bank (CI validation, LLM-based updates); для organization с множеством проектов использующих Memory Bank. Для small projects: human-managed schema без формального versioning.
Arch Как автоматически обновлять Memory Bank через Hooks и CI?
Automated Memory Bank pipeline: (1) Change detection — git hooks или CI watch на ключевые файлы (pom.xml, package.json, Makefile); (2) Extraction — скрипт extract'ирует версии, команды, изменения из конфиг-файлов; (3) Memory update proposal — LLM генерирует предлагаемые изменения в Memory Bank на основе diff; (4) Human review gate — PR создаётся с proposed changes; merge только после approval; (5) Validation — CI проверяет что обновлённый Memory Bank соответствует schema. Что НЕ автоматизировать: WHY (обоснование решений) — требует human judgment; deprecated decisions — требует понимания контекста. Автоматизировать: WHAT (версии, команды) — можно извлечь из файлов.
Arch Как превратить Memory Bank в living documentation system?
Living documentation система: (1) Automated freshness — CI периодически проверяет факты в Memory Bank против live sources; алертит при расхождении; (2) Definition of done integration — принятый ADR → автоматически попадает в decisions.md через template; (3) Code-Memory bidirectionality — аннотации в коде ("// see ADR-005 in decisions.md") + ссылки из decisions.md на соответствующий код; (4) Continuous validation — LLM агент как "memory auditor" запускается по расписанию, проверяет актуальность и предлагает обновления; (5) Living ADR — ADR могут менять статус (Proposed → Accepted → Deprecated → Superseded) с историей в git. Результат: Memory Bank который не устаревает потому что его обновление встроено в процесс разработки.
