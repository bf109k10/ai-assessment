Каверзные вопросы по Cursor

1. В Cursor Rules имеют один тип?

Слабый ответ: “Rule — это просто инструкция для агента”.
Сильный ответ: “В Cursor есть Project Rules, User Rules, Team Rules и AGENTS.md. Project Rules лежат в `.cursor/rules` как `.mdc`, User Rules глобальные, Team Rules задаются командой/админами, AGENTS.md — plain markdown альтернатива для простых инструкций.”
Почему каверзно: assessor может проверить не термин, а детали применения и scope.

2. Что выше в Cursor: Team, Project или User Rules?

Слабый ответ: “Последнее правило побеждает”.
Сильный ответ: “Cursor docs описывают порядок применения Team Rules → Project Rules → User Rules, при этом applicable rules merge’ятся, а более ранние источники имеют precedence при конфликте. Enforced Team Rules пользователь не может отключить.”
Ключ: это специфично для Cursor, не универсальная истина для всех IDE.

3. `.cursor/rules/api.md` будет работать?

Слабый ответ: “Да, это же markdown”.
Сильный ответ: “Нет как Project Rule: Cursor ожидает `.mdc` с frontmatter. Plain `.md` в `.cursor/rules` игнорируется rules system. Для plain markdown лучше использовать AGENTS.md.”
Ключ: формат файла важен.

4. Rules применяются всегда?

Слабый ответ: “Да, если rule есть, он всегда в контексте”.
Сильный ответ: “Нет. В Cursor rule может быть Always Apply, Apply Intelligently, Apply to Specific Files через globs, или manual через @mention. Поэтому rule может не попасть в контекст, если scope/description/glob не сработали.”
Ключ: надо проверять, был ли rule реально attached.

5. AGENTS.md и Project Rules — одно и то же?

Слабый ответ: “Да, оба инструкции”.
Сильный ответ: “Похожи по цели, но разные по механике. AGENTS.md — simple plain markdown instruction, удобен для простых случаев. Project Rules — `.mdc` с metadata: description, globs, alwaysApply; их можно точнее scope’ить.”
Ключ: одинаковая идея, разная управляемость.

6. Nested AGENTS.md в Cursor работает?

Слабый ответ: “Только один AGENTS.md в root”.
Сильный ответ: “Cursor поддерживает AGENTS.md в root и subdirectories. Инструкции родительских и nested AGENTS.md комбинируются, а более специфичные nested instructions имеют precedence для файлов в этой области.”
Ключ: в monorepo это критично.

Каверзные вопросы по Skills

7. Skill — это просто Rule с другим названием?

Слабый ответ: “Да, это одно и то же”.
Сильный ответ: “Нет. Rule задаёт persistent guidance/constraints. Skill — reusable capability/playbook: инструкции + scripts + references + assets. В Cursor Skills — open standard и progressive: ресурсы грузятся on demand, чтобы не забивать контекст.”

8. Когда dynamic rule лучше мигрировать в Skill?

Слабый ответ: “Никогда, rules удобнее”.
Сильный ответ: “Если правило стало процедурой: много шагов, шаблонов, скриптов, references. Cursor docs прямо описывают `/migrate-to-skills` для dynamic rules и slash commands.”

9. Skill всегда автоматически применяется?

Слабый ответ: “Да, агент сам всегда выберет”.
Сильный ответ: “Не обязательно. Skill может быть auto-discovered и выбран агентом, может иметь paths scope, а `disable-model-invocation: true` делает его explicit slash-command only.”

10. Где хранить длинный reference для Skill?

Слабый ответ: “Всё в SKILL.md”.
Сильный ответ: “Лучше держать SKILL.md focused, а длинные материалы — в `references/`, scripts — в `scripts/`, templates/assets — в `assets/`. Это экономит context благодаря progressive loading.”

Каверзные вопросы по MCP / Tools

11. MCP — это RAG?

Слабый ответ: “Да, оба подключают внешние знания”.
Сильный ответ: “Нет. RAG — retrieval фрагментов из corpus. MCP — протокол подключения tools/resources/prompts к внешним системам. Через MCP можно сделать retrieval, но MCP шире: tools могут выполнять действия, возвращать UI/images, обращаться к API.”

12. Если Claude/Cursor вызывает tool, модель сама выполняет действие?

Слабый ответ: “Да, модель пошла и выполнила”.
Сильный ответ: “В Claude Platform tool use — это контракт: модель формирует structured tool request, а приложение/сервер выполняет операцию и возвращает результат. Модель не исполняет внешний код сама.”

13. MCP tool можно запускать без approval?

Слабый ответ: “Да, MCP всегда автоматический”.
Сильный ответ: “Зависит от run mode, allowlists и tool approval. В Cursor MCP tools по умолчанию требуют approval; team admins могут настраивать allowlist, network controls и tool allowlists.”

14. Чем опасен MCP server?

Слабый ответ: “Ничем, это просто источник данных”.
Сильный ответ: “MCP server может иметь доступ к внешним сервисам и выполнять код/действия от имени пользователя. Надо проверять источник, permissions, API keys, sandbox/network mode и audit.”

Каверзные вопросы по Subagents

15. Subagent видит всю историю parent chat?

Слабый ответ: “Да, это же часть того же чата”.
Сильный ответ: “В Cursor subagent starts with clean context. Parent agent передаёт релевантную информацию в prompt. Поэтому хорошая постановка задачи критична.”

16. Subagent всегда быстрее?

Слабый ответ: “Да, потому что параллельно”.
Сильный ответ: “Не всегда. Subagents дают context isolation и parallel work, но у них есть startup overhead, отдельное token usage и latency. Для маленькой задачи main agent часто быстрее.”

17. Subagent и Skill — когда что выбрать?

Слабый ответ: “Subagent всегда мощнее”.
Сильный ответ: “Subagent нужен для отдельного context window, long research, parallel work, independent verification. Skill лучше для single-purpose повторяемого action/playbook без отдельного контекста.”

18. Может ли Cursor subagent использовать MCP?

Слабый ответ: “Нет, MCP только у parent”.
Сильный ответ: “В Cursor subagents inherit tools from parent, включая MCP, но cloud subagents используют MCP servers из team configuration на cursor.com/agents, а не из local session.”

19. В Claude Code subagents всегда грузят CLAUDE.md?

Слабый ответ: “Да, все subagents одинаковые”.
Сильный ответ: “Не совсем. Claude Code docs отмечают, что built-in Explore и Plan skip CLAUDE.md и parent git status для скорости/стоимости, а другие built-in/custom subagents грузят оба.”

20. Subagent result — финальная истина?

Слабый ответ: “Да, раз subagent проверил”.
Сильный ответ: “Нет. Это finding/evidence. Parent agent должен проверить конфликт с Rules/AGENTS.md, полноту evidence и применимость результата.”

Каверзные вопросы по Claude Code / Claude Platform

21. CLAUDE.md — это system prompt?

Слабый ответ: “Да, это системный промпт проекта”.
Сильный ответ: “В Claude Code docs сказано: CLAUDE.md delivered as user message after system prompt, not system prompt itself. Поэтому он важен, но не даёт строгой гарантии compliance, особенно при vague/conflicting instructions.”

22. Несколько CLAUDE.md override друг друга?

Слабый ответ: “Да, ближайший полностью заменяет root”.
Сильный ответ: “Claude Code walks up directory tree и concatenates discovered CLAUDE.md/CLAUDE.local.md, а не просто overrides. Порядок — от root к current directory; более близкие инструкции читаются позже.”

23. Memory в Claude Code бесконечная?

Слабый ответ: “Да, всё помнит”.
Сильный ответ: “Нет. Auto memory имеет limits: first 200 lines или 25KB MEMORY.md loaded at startup. Detailed topic files читаются on demand tools’ами. Поэтому memory нужно держать concise.”

24. Если instruction must run after every edit — писать в CLAUDE.md?

Слабый ответ: “Да, добавь в CLAUDE.md”.
Сильный ответ: “Лучше hook. Claude docs прямо говорят: если инструкция должна выполняться в конкретной lifecycle point, например before commit или after edit, используйте hooks, потому что hooks execute regardless of what Claude decides.”

25. Messages API и Managed Agents — одно и то же?

Слабый ответ: “Да, это просто Claude API”.
Сильный ответ: “Нет. Claude Platform RU описывает Messages как прямой доступ к модели: вы сами управляете состоянием разговора и tool loop. Managed Agents — управляемая инфраструктура автономных агентов с stateful sessions и event history.”

26. Prompt caching — это memory?

Слабый ответ: “Да, модель запоминает prompt”.
Сильный ответ: “Нет. Prompt caching снижает latency/cost за счёт повторного использования одинакового prompt prefix. Это не смысловая память. Cache может invalidated, если prefix изменился byte-for-byte, например tool order, timestamp или раннее сообщение.”

Ответ assessor’у одной сильной фразой

“Я различаю источник по роли: system/platform задаёт верхние ограничения; Cursor Team/Project/User Rules и AGENTS.md задают persistent guidance с разным scope; Skill — это reusable workflow, который грузится/вызывается по необходимости; RAG даёт factual chunks из corpus; MCP/tools дают live data и actions через permissions; Memory/Checkpoint дают continuity; Subagents изолируют контекст и возвращают findings. При конфликте я выбираю не ‘последнее указание’, а source of truth: priority → specificity → freshness → safety.”
