Назначение и роль в системе

Что такое AGENTS.md

Файл Project Instructions — инструкции для агента о том, как работать в этом конкретном репозитории. Читается автоматически при каждом чате.

  • Задаёт роль и контекст: кто агент, что делает проект
  • Описывает workflow: как тестировать, как делать PR
  • Хранит team conventions: стиль кода, язык, форматы
  • Ссылается на memory, rules, skills

Cursor vs Claude Code

🖱 Cursor Файл называется AGENTS.md. Читается Cursor Agent при каждом запросе.

🤖 Claude Code Файл называется CLAUDE.md. Тот же смысл, другое название.

✓ Оба Каскадность: global → repo-root → subdirectory.

AGENTS.md vs Rules

  • AGENTS.md — описывает как работать, workflow, контекст. Рекомендации.
  • Rules — политика, которую агент обязан соблюдать. Жёстче.
  • AGENTS.md → агент старается следовать. Rules → агент обязан соблюдать.
  • Правило: "как писать тесты" → AGENTS.md. "никогда не коммить .env" → Rule.

Тонкости использования — то что не очевидно

Subdirectory AGENTS.md — когда и зачем

Если в репо несколько модулей с разным стеком (например, billing/ на Java и portal/ на Python), каждый может иметь свой AGENTS.md. Агент читает root + subdir при работе в конкретной папке. Так conventions не смешиваются между модулями.

my-project/ ├── AGENTS.md # общий: проект, команда, PR-формат ├── billing/ │ └── AGENTS.md # специфично: Java ADRs, billing commands └── portal/ └── AGENTS.md # специфично: Python, React, другие команды

AGENTS.md не заменяет Rules

Критически важная политика ("не коммить .env", "всегда запускать тесты перед PR") лучше в Rules (.cursor/rules/), а не в AGENTS.md. Причина: Rules соблюдаются жёстче — в Claude Code их нарушение можно перехватить через Hooks. AGENTS.md агент может проигнорировать в длинном контексте ("lost in the middle").

Проблема "lost in the middle"

Если AGENTS.md большой и размещён в начале context window, его середина может не учитываться агентом при длинных диалогах. Решения: 1) Ключевые инструкции — в первые и последние 20% файла. 2) Критическое — в Rules (alwaysApply). 3) Чем короче AGENTS.md, тем лучше внимание.

@include и ссылки на другие файлы

В Claude Code CLAUDE.md может включать другие файлы через @path/to/file.md. Это позволяет держать memory, ADRs, conventions в отдельных файлах и инклудить их по необходимости. В Cursor — агент может читать упомянутые файлы если они указаны через @mention в запросе или AGENTS.md.

# В CLAUDE.md: @docs/architecture.md # архитектурные решения @docs/memory.md # project memory @docs/conventions.md # code conventions

AGENTS.md vs system prompt

В Cursor: System prompt задаётся harness (IDE), AGENTS.md добавляется сверху как project context. В Claude Code: CLAUDE.md включается в system prompt автоматически. Оба читаются раньше пользовательских сообщений — это выгодная позиция для attention, но не гарантирует что всё будет прочитано при очень большом контексте.

Best practices

Делай

  • Начинай с Role section — агент сразу знает контекст
  • Всегда указывай команды тестов — агент запустит правильную
  • ADRs с обоснованием: "используем JPA потому что..."
  • Обновляй дату last updated при изменениях
  • Держи файл коротким — <200 строк идеал
  • Критическое — в Rules, не в AGENTS.md

Не делай

  • Секреты, токены, пароли — никогда
  • Длинную документацию — это работа RAG
  • Противоречивые инструкции в разных секциях
  • Дублировать содержимое Rules в AGENTS.md
  • Забывать обновлять при смене стека
  • Оставлять без даты — нельзя понять насколько устарело

Anti-patterns

  • God AGENTS.md — 1000+ строк всего подряд. Агент перестаёт читать.
  • Противоречие с subdir — root говорит "Java", subdir не уточняет "здесь Python".
  • Только правила без контекста — "не делай X" без объяснения почему.
  • Нет команд тестов — агент угадывает или не проверяет.

Cursor и Claude Code — расположение и иерархия файлов

🖱 Cursor — AGENTS.md

Файл AGENTS.md в корне проекта или поддиректории. Cursor читает его и включает в context агента. Можно использовать иерархию: корневой + поддиректорный AGENTS.md.

📁 project/ ├── AGENTS.md ← главный ├── frontend/ │ └── AGENTS.md ← переопределяет для frontend/ └── api/ └── AGENTS.md ← переопределяет для api/
🤖 Claude Code — CLAUDE.md

Файл называется CLAUDE.md (не AGENTS.md). Загружается автоматически при старте сессии. Поддерживает иерархию директорий. Перезагружается при каждом запросе — не compaction-ится.

📁 project/ ├── CLAUDE.md ← главный (= AGENTS.md в Cursor) ├── frontend/ │ └── CLAUDE.md ← переопределяет для frontend/ └── api/ └── CLAUDE.md ← переопределяет для api/

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое AGENTS.md и для чего он нужен?
AGENTS.md — файл с инструкциями проекта для AI-агента, который читается автоматически при каждом запросе. Он описывает кто такой агент, в каком стеке работает проект, как запускать тесты, каковы workflow и конвенции команды. Без него агент начинает каждую сессию с нуля и вынужден угадывать контекст. В Claude Code аналогичный файл называется CLAUDE.md, но выполняет ту же функцию.
Junior Что такое CLAUDE.md и чем он отличается от AGENTS.md?
CLAUDE.md — эквивалент AGENTS.md для платформы Claude Code (Anthropic). Содержимое и назначение идентичны: описать агенту проект, стек, команды, workflow. Основное отличие — название файла и механизм загрузки: в Claude Code есть глобальный уровень ~/.claude/CLAUDE.md, который применяется ко всем проектам пользователя, а в Cursor такого нативного аналога нет. Оба поддерживают каскадную загрузку из поддиректорий.
Junior Где должен храниться файл AGENTS.md в репозитории?
Основной AGENTS.md хранится в корне репозитория (root level) рядом с README.md. Дополнительные файлы можно создать в поддиректориях для модулей с особой спецификой — например billing/AGENTS.md для модуля с отдельным стеком. Каскадная загрузка: агент читает root-файл плюс все AGENTS.md на пути к текущей директории. Файл версионируется в git как обычный markdown.
Junior Что обязательно писать в AGENTS.md?
Минимальный обязательный набор: (1) Role — кто такой агент и что за проект; (2) Tech Stack — языки, фреймворки, базы данных с версиями; (3) Build & Test — точные команды для сборки и тестирования (mvn test, npm test и т.д.); (4) Key Conventions — главные соглашения команды. Рекомендуется также добавить Workflow (как делать PR, DoD) и Architecture Decisions (почему выбраны те или иные решения).
Junior Чем AGENTS.md отличается от Rules (.mdc)?
AGENTS.md описывает workflow и контекст: "как работать в этом проекте", стек, команды, DoD — это рекомендации. Rules — жёсткая политика: "что нельзя делать никогда" (не коммитить секреты, не удалять тесты). Rules с alwaysApply:true всегда присутствуют в context и агент обязан им следовать. AGENTS.md агент может частично проигнорировать при очень длинном контексте (lost in the middle), Rules — надёжнее.
Junior Когда агент читает AGENTS.md?
AGENTS.md читается автоматически при каждом новом запросе к агенту. Файл включается в context window в начале сессии как часть system context (Cursor) или как часть системного промпта (Claude Code). Это означает агент "знает" о проекте ещё до того как пользователь напишет первое сообщение. При очень длинном контексте и compaction файл может быть сжат.
Junior Что такое project context и зачем он нужен агенту?
Project context — набор сведений о проекте, которые агент должен знать чтобы работать эффективно: стек, архитектура, команды, соглашения, ADR. Без project context агент не знает на каком языке писать тесты, какой build tool использовать, каков PR format. Это "ввод в должность" для агента — аналог инструктажа нового разработчика в команде. AGENTS.md является основным носителем project context.
Junior Какие секции принято включать в AGENTS.md?
Стандартные секции: ## Role (кто агент, что за проект), ## Stack (технологии, версии), ## Build & Test (команды), ## Workflow (процесс, DoD), ## Architecture Decisions (ADR с обоснованием), ## Conventions (стиль кода, язык, форматы), ## References (ссылки на memory.md, rules). Обязательны первые три; остальные рекомендуются. Секция References позволяет разбить большой файл на несколько через @include.
Junior Как AGENTS.md помогает агенту работать эффективнее?
AGENTS.md устраняет необходимость объяснять контекст заново в каждой сессии. Агент сразу знает команды тестов (не угадывает), технологический стек (не предлагает несовместимые решения), PR-формат (формирует правильные коммиты). Это снижает количество уточняющих вопросов, ошибок от незнания контекста и итераций на исправление. Качественный AGENTS.md может сократить время выполнения задачи на 20–40%.
Junior Что произойдёт если в проекте нет AGENTS.md?
Агент будет работать в "слепом" режиме: угадывать технологии, запускать случайные команды тестов, нарушать конвенции команды. Каждую сессию придётся вручную объяснять контекст — тратить токены и время. Риски: агент предложит добавить зависимость без одобрения, использует неверный PR-формат, сгенерирует код не совместимый со стеком проекта. Без AGENTS.md каждая сессия начинается с нуля.
Middle — 10 вопросов
Middle Как решить что класть в AGENTS.md, а что в Rules?
Правило: AGENTS.md — для всего что описывает "как работать" (workflow, стек, команды, DoD). Rules — для всего что описывает "что нельзя" (safety, compliance, архитектурные запреты). Пример: "запускай тесты перед PR" → AGENTS.md (рекомендация). "Не коммитить .env файлы" → Rules (запрет). Критически важные ограничения всегда в Rules (alwaysApply:true) — они надёжнее при длинном контексте. AGENTS.md соблюдается "по желанию", Rules — принудительно.
Middle Как писать компактный AGENTS.md не теряя важного?
Принципы компактного AGENTS.md: (1) Один факт — одна строка; (2) Команды точные, не объяснения ("mvn test" а не "запусти тесты через Maven"); (3) ADR — только решение и краткое обоснование, детали — в отдельном docs/ файле через @include; (4) Убрать всё что есть в README или документации — не дублировать; (5) Целевой размер ≤200 строк, идеал ≤100. Длинный AGENTS.md страдает от lost in the middle — критическое уходит из зоны внимания модели.
Middle Как AGENTS.md помогает при onboarding новых разработчиков?
AGENTS.md — живая документация проекта, доступная мгновенно через AI-агента. Новый разработчик + агент: агент читает AGENTS.md и сразу знает стек, команды, конвенции — не нужно объяснять вручную. Это снижает time-to-first-contribution. Дополнительно: AGENTS.md сам по себе — документ для нового разработчика ("прочитай AGENTS.md — поймёшь проект за 10 минут"). Версионируется в git — всегда актуален.
Middle Когда использовать AGENTS.md, а когда Memory Bank?
AGENTS.md — стабильный контекст проекта (стек, команды, конвенции, workflow) который меняется редко и нужен всегда. Memory Bank — накопленные знания и решения которые появляются в ходе работы (ADR принятые сессия назад, gotchas найденные при отладке). AGENTS.md создаётся при старте и обновляется при архитектурных изменениях; Memory Bank растёт постепенно. В маленьких проектах их можно объединить в один файл.
Middle Какова оптимальная структура секций AGENTS.md?
Рекомендуемый порядок: (1) Role & Context — вверху, агент читает первым; (2) Stack — технологии кратко; (3) Build & Test — команды точно; (4) Workflow — процесс и DoD; (5) Architecture Decisions — краткие ADR с обоснованием; (6) Conventions — стиль кода, язык ответа; (7) References — @memory.md, @docs/. Критически важное в позицию 1-2 (горячая зона attention). Финальная секция References позволяет декомпозировать файл на части.
Middle Когда и как обновлять AGENTS.md?
Обновлять при: смене стека или версий (Java 11 → Java 21), добавлении нового сервиса/модуля, принятии нового ADR, изменении workflow или PR-формата, изменении команд сборки/тестов. Процесс: обновление как обычный PR с review — не хуже изменения кода. Добавить в PR checklist: "если изменились архитектурные решения или команды — обновить AGENTS.md". Обновлять дату "Last updated" в заголовке. Устаревший AGENTS.md — технический долг.
Middle Как организовать AGENTS.md для командной разработки?
Правила для командного AGENTS.md: (1) Версионировать в git с обязательным review изменений; (2) Назначить "AGENTS.md owner" — человек ответственный за актуальность; (3) Включить team conventions, не личные предпочтения; (4) PR checklist с проверкой AGENTS.md при архитектурных изменениях; (5) Не включать имена людей — только роли (tech lead, DBA). Общие правила команды — в корневом AGENTS.md, модульные — в subdirectory AGENTS.md.
Middle Что происходит при конфликте инструкций в AGENTS.md и Rules?
Rules всегда побеждают AGENTS.md. Если AGENTS.md говорит "следуй TDD" (рекомендация), а Rule с alwaysApply:true говорит "никогда не запускай тесты в продакшн" — Rule соблюдается без исключений. Иерархия: Hooks (Claude Code) > alwaysApply Rules > AGENTS.md > auto-attach Rules > задача пользователя. Если конфликт не в пользу Rules — это ошибка проектирования: policy нужно выносить в Rules, не оставлять в AGENTS.md.
Middle Как использовать AGENTS.md для workflow инструкций?
Workflow секция в AGENTS.md описывает шаги выполнения типовых задач: последовательность (Research → Plan → Implement → Test → PR), формат PR-описания (Problem → Fix → Tests → Risk), Definition of Done (тесты зелёные, linter чистый, changelog обновлён). Это рекомендации — агент старается следовать, но не обязан при прямом противоречии с задачей. Жёсткие workflow требования лучше в Hooks (Claude Code) или Rules.
Middle Каков оптимальный размер AGENTS.md?
Целевой диапазон: 50–200 строк, идеал ≤100 строк. При превышении 300–500 строк появляется риск lost in the middle — факты в середине файла получают меньше attention. Признаки "раздутого" AGENTS.md: дублирование README, подробные ADR (должны быть в docs/), примеры кода длиннее 5 строк, история изменений. Решение: вынести большие блоки в отдельные файлы через @include, оставить в AGENTS.md только сводку.
Senior — 5 вопросов
Senior Как работает иерархия AGENTS.md файлов в разных директориях?
Загрузка каскадная снизу вверх по директориям: глобальный (~/.claude/CLAUDE.md) → корневой (repo/AGENTS.md) → поддиректория (repo/src/AGENTS.md) → текущая директория. Все файлы конкатенируются. При конфликте: более специфичный (глубже в дереве) побеждает. В Claude Code все уровни мержатся. В Cursor поддиректорный перекрывает корневой при конфликте. Молчание поддиректорного файла = наследование родительского. Subdirectory AGENTS.md нужен только когда инструкции существенно отличаются.
Senior Как оптимизировать AGENTS.md чтобы минимизировать token overhead?
Стратегии: (1) Декомпозиция — вынести большие блоки в docs/ файлы, в AGENTS.md оставить сводки и @include ссылки; (2) Prompt caching — стабильный AGENTS.md кешируется провайдером, реальный overhead = 0 со второго запроса; (3) Минималистичный контент — каждая строка должна добавлять information density; (4) Subdirectory AGENTS.md — агент загружает только релевантный для текущей директории контекст. Цель: минимум токенов при максимуме передаваемой информации.
Senior Как организовать AGENTS.md для монорепозитория с разными модулями?
Паттерн для монорепо: корневой AGENTS.md содержит общее (технологии платформы, PR-формат, общие команды, CI/CD); каждый модуль имеет свой AGENTS.md с модуль-специфичным контентом (свой стек, команды тестов, ADR). Пример: billing/AGENTS.md — Java/Quarkus специфика, portal/AGENTS.md — React/TypeScript специфика. Агент при работе в billing/ читает оба файла: корневой + billing-уровень. Не дублировать общее в каждом subdirectory — только специфику.
Senior Как диагностировать ситуацию когда агент игнорирует AGENTS.md?
Пошаговая диагностика: (1) Проверить расположение файла — в корне проекта? Правильное имя (AGENTS.md vs CLAUDE.md)? (2) Проверить размер — если >500 строк, инструкция в середине попала в lost in the middle зону; переместить вверх. (3) Проверить конфликты — есть ли Rule или chat-инструкция которая перекрывает AGENTS.md? (4) Проверить compaction — в длинной сессии context мог быть сжат. (5) Критические инструкции дублировать в Rules (alwaysApply:true) для гарантии.
Senior Что такое AGENTS.md inheritance паттерн и когда его применять?
Inheritance паттерн: каждый уровень AGENTS.md наследует от родительского и добавляет или переопределяет специфику. Root AGENTS.md = базовый класс: общие правила, stack платформы. Subdirectory AGENTS.md = дочерний класс: override для конкретного модуля. Применять когда: у модулей разный стек, разные команды тестов, разные ADR. Не применять для мелких отличий — overhead от поддержки множества файлов превысит пользу. Правило: subdirectory AGENTS.md только если есть ≥3 существенных отличия от родительского.
Lead — 5 вопросов
Lead Как использовать AGENTS.md как командный договор?
AGENTS.md как командный договор — явная документация того, что команда считает нормой для AI-агента. Включать: принятые ADR (не только что, но и почему), team conventions (язык кода, PR формат), запрещённые практики (почему не используем X). Ключевое: AGENTS.md должен быть результатом команды consensus, а не решением одного разработчика. Изменения — только через PR с review от всей команды. Это предотвращает конфликты когда разные разработчики дают агенту противоречивые инструкции.
Lead Как организовать governance AGENTS.md через git?
Git governance для AGENTS.md: (1) Требовать PR для любого изменения — не прямой commit в main; (2) Обязательный review от ≥1 разработчика (или tech lead); (3) CODEOWNERS файл — назначить AGENTS.md owner кто обязательно review'ит изменения; (4) CI lint — проверять что AGENTS.md не превышает допустимый размер и содержит обязательные секции; (5) Changelog в самом файле. История изменений через git log AGENTS.md — это audit trail архитектурных решений.
Lead Как выстроить процесс обновления AGENTS.md в команде?
Процесс обновления: (1) Триггеры — чёткий список событий требующих обновления: смена стека, принятый ADR, изменение команд, новый workflow; (2) PR checklist — "обновить AGENTS.md если изменились архитектурные решения или команды"; (3) Sprint review — раз в спринт проверять актуальность; (4) AGENTS.md owner — ответственный за напоминания команде; (5) CI hook при изменении pom.xml/package.json проверяет актуальность команд в AGENTS.md. Stale AGENTS.md = технический долг.
Lead Когда нужен shared AGENTS.md, а когда team-specific?
Shared (корневой) AGENTS.md — для всего что общее для всей команды и всего репо: платформа, CI/CD команды, PR-формат, safety требования. Team-specific (subdirectory) AGENTS.md — для модулей с разной спецификой: разный язык/фреймворк, разный владелец, разные ADR. Не создавать team-specific файл если отличия минимальны (2-3 строки). В enterprise: может быть organization-level AGENTS.md (как policy) + team-level (как workflow) + module-level (как spec).
Lead Как использовать AGENTS.md для onboarding новых членов команды?
AGENTS.md как onboarding артефакт: новый разработчик открывает проект, читает AGENTS.md — за 10-15 минут понимает стек, команды, workflow, ключевые решения. AI-агент с AGENTS.md даёт ответы в контексте проекта, а не общие советы. Рекомендации: (1) Добавить ссылку на AGENTS.md в README как первый шаг onboarding; (2) Включить раздел "Key files — где что лежит"; (3) ADR с обоснованием — новый разработчик понимает почему так; (4) Раздел "Common pitfalls" — типичные ошибки и как избежать.
Architect — 5 вопросов
Arch Как AGENTS.md реализует паттерн declarative agent configuration?
AGENTS.md — декларативная конфигурация: описываешь желаемое поведение агента, а не программируешь его шаги. Аналог Infrastructure as Code для AI: как terraform описывает желаемое состояние инфраструктуры, AGENTS.md описывает желаемое состояние агента в проекте. Преимущества: версионируется, reviewable, аудируемый. Ограничения: AGENTS.md — soft enforcement, не hard enforcement (как Hooks). Архитектурно: сочетать с Rules (политика) + Hooks (enforcement) для полной системы управления агентом.
Arch Как организовать AGENTS.md в multi-repo архитектуре?
Multi-repo паттерны: (1) Federated — каждый репозиторий имеет свой AGENTS.md, общие правила продублированы или шаблонизированы через script; (2) Centralized — organization-level репозиторий с "master" AGENTS.md, каждый репо его импортирует/копирует через CI; (3) Layered — глобальный ~/.claude/CLAUDE.md для личных предпочтений + organization template + repo-specific. Проблема multi-repo: синхронизация общих правил при изменении — автоматизировать через CI/CD. Правило: organization conventions → centralized, project specifics → per-repo.
Arch Какую стратегию versioning применять для AGENTS.md?
Versioning стратегии: (1) Semantic versioning в заголовке файла: "## Version: 2.3.0" — major при архитектурных изменениях, minor при новых секциях, patch при правках; (2) Date-based: "Last updated: 2026-06-15" — проще, но не показывает масштаб изменений; (3) Git tags на репозитории при значимых изменениях AGENTS.md. Дополнительно: CHANGELOG секция в самом файле — 5-10 последних изменений с датами. Это позволяет понять эволюцию конвенций и не использовать устаревшие практики.
Arch Как автоматически генерировать и обновлять AGENTS.md из кодовой базы?
Автогенерация паттерны: (1) CI pipeline — при изменении pom.xml/package.json автоматически обновляет секцию Stack в AGENTS.md и создаёт PR; (2) Git hooks (pre-commit) — проверяет что при изменении Makefile обновлены команды в AGENTS.md; (3) Script-based generation — скрипт parse'ит конфиг файлы и генерирует skeleton AGENTS.md; (4) LLM-assisted update — агент предлагает обновление AGENTS.md на основе diff PR, человек approve'ит. Полная автогенерация рискованна: AGENTS.md содержит "почему", что из кода не extract'ируется. Оптимум: автоматизировать "что" (стек, команды), сохранять ручное "почему" (ADR, rationale).
Arch Как превратить AGENTS.md в living documentation системы?
Living documentation через AGENTS.md: (1) ADR tracking — каждое архитектурное решение сразу попадает в AGENTS.md с датой и обоснованием; (2) Automated validation — CI проверяет что факты в AGENTS.md соответствуют кодовой базе (версии в pom.xml, команды в Makefile); (3) Bidirectional sync — изменения в коде триггерят обновление AGENTS.md; (4) Cross-referencing — AGENTS.md ссылается на ADR файлы. Результат: AGENTS.md становится single source of truth о проекте, который актуален потому что это часть процесса разработки.
