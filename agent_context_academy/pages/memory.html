Agent Context Academy

Memory Bank

Долговременная память агента: 5 типов памяти, как они работают, что хранить, как не сломать. Объяснение с нуля — без предварительных знаний.

На главную К assessment

С нуля: почему AI-агент забывает всё после разговора

Прежде чем говорить о Memory Bank — нужно понять главную проблему: LLM (языковая модель) не имеет встроенной памяти между сессиями.

Аналогия: агент — как человек с амнезией

Представь коллегу-разработчика, который каждое утро просыпается без воспоминаний о прошлых днях. Каждый день ты должен заново объяснять ему: какой у вас стек, какие команды запускать, какие решения уже приняты. Продуктивности ноль.

Именно так работает LLM-агент без Memory Bank. Каждый новый чат — чистый лист. Агент не помнит ни твоих предпочтений, ни архитектурных решений, ни того, что Redis вы отвергли два месяца назад.

Что такое "контекстное окно"

LLM работает с context window — это всё, что она "видит" прямо сейчас: твой вопрос + история чата + системный промпт. Когда чат закрывается — всё исчезает. Context window = кратковременная рабочая память.

  • Типичный размер: 100k–200k токенов
  • 1 токен ≈ 0.75 слова
  • После закрытия — полное забывание

Что такое Memory Bank

Memory Bank — внешний файл (обычно memory.md) с записанными знаниями о проекте. Агент читает его при каждом запросе и "знает" всё что там написано. Это искусственная долговременная память.

  • Физически: markdown-файл в репозитории
  • Загружается в context window каждый раз
  • Сохраняется в git — не исчезает

Зачем это нужно

Без Memory Bank агент каждую сессию работает как первоклассник: не знает стека, команд, принятых решений. С Memory Bank — как опытный тимлид: знает проект, историю решений, предпочтения команды.

  • Не надо каждый раз всё объяснять
  • Агент не предлагает отвергнутые решения
  • Onboarding новых разработчиков быстрее
БЕЗ Memory Bank: Сессия 1: "Наш стек — Java 21, Maven, Spring Boot" → агент работает нормально Сессия 2: (чистый лист) Агент предлагает npm install... Не знает про Maven Сессия 3: Предлагает добавить Redis — который вы уже отвергли С Memory Bank (memory.md): ## Stack - Java 21, Maven, Spring Boot 3.2 - НЕ используем Redis (overhead для нашей нагрузки, Q1 2025) - Build: mvn -pl billing test Сессия 1, 2, 3, 100... Агент всегда знает стек и историю решений ✓

Как Memory Bank попадает к агенту — механика

Простыми словами: агент читает файл как ты читаешь инструкцию

Memory Bank не «встроен» в модель — это обычный текстовый файл. Он добавляется в context window в начале каждого запроса. Агент видит его как часть своего «системного промпта» и принимает как данность всё что там написано.

Как агент получает Memory Bank

  • ↓ Ты отправляешь вопрос
  • ↓ Cursor / Claude Code читает: AGENTS.md → memory.md → rules → …
  • ↓ Собирает context window (слои снизу вверх — как стопка):
System prompt (роль агента)
AGENTS.md (workflow, стек)
memory.md ← Memory Bank
Rules / Skills
История чата
Твой вопрос ← здесь

↓ LLM видит всё это как один большой текст и отвечает с учётом memory.md

5 типов памяти агента — подробно

У AI-агента есть несколько разных "слоёв" памяти. Они отличаются по сроку жизни, что хранят и как обновляются. Знать разницу критично — иначе будешь класть временные данные в постоянную память и наоборот.

Аналогия из жизни

Представь мозг человека: рабочая память (что ты делаешь прямо сейчас), кратковременная (разговор час назад), долговременная (навыки и факты на годы). У агента — то же самое, только реализовано через файлы и базы данных.

💬
1. Session Memory — память текущего разговора
Живёт: пока открыт чат  |  Управляется: автоматически

Что это такое: История текущего диалога — всё что ты и агент написали в этом чате. Агент помнит что ты уточнял, какие файлы открывал, что уже обсуждалось.

Аналогия: Твоя рабочая память прямо сейчас. Ты помнишь последние 10 минут разговора, но если прервёшься на день — детали уже не так чёткие.

Проблема: Session memory исчезает при закрытии чата. Также при длинных диалогах она сжимается (compaction) — старые сообщения вытесняются новыми. То что было 50 сообщений назад — агент может уже "не помнить".

Практика: Если ты уточнил важное в начале длинного чата — напомни снова в середине, или зафиксируй в Memory Bank.

Срок: текущая сессия Автоматически Исчезает при закрытии Сжимается при длинном чате
📁
2. Project Memory — знания о проекте
Живёт: весь срок проекта (месяцы/годы)  |  Хранится: memory.md в репо

Что это такое: Главный тип памяти. Файл в репозитории с накопленными знаниями: стек, команды, архитектурные решения, конвенции команды, gotchas.

Аналогия: Корпоративная wiki, которую разработчики реально читают — потому что она компактная (≤100 строк) и всегда актуальная.

Что записывать:

  • Технологический стек с версиями: Java 21, Maven, Spring Boot 3.2
  • Команды сборки/тестов: mvn -pl billing test
  • Архитектурные решения с обоснованием: "Redis отвергнут — overhead для нашей нагрузки"
  • Конвенции команды: PR формат, naming conventions
  • Gotchas: нюансы которые всегда забывают
  • Негативные решения: "не используем X потому что..."

Критический риск: Устаревание. Если команда перешла с Maven на Gradle, а memory.md не обновили — агент будет давать неверные команды с абсолютной уверенностью. Это хуже чем незнание — это уверенная ложь.

Срок: месяцы / годы Версионируется в git Передаётся между разработчиками Устаревает без обновления
👤
3. User Memory — предпочтения пользователя
Живёт: постоянно  |  Хранится: ~/.claude/CLAUDE.md или Cursor Settings

Что это такое: Персональные предпочтения конкретного разработчика: язык ответов, стиль объяснений, уровень детализации, форматирование.

Аналогия: Личные настройки в IDE — тема, размер шрифта, горячие клавиши. Это твоё, а не командное.

Примеры записей:

  • "Отвечай на русском, технически точно"
  • "Объясняй через аналогии — я лучше понимаю"
  • "В коде всегда добавляй error handling"
  • "Я предпочитаю короткие ответы без лишних вступлений"
  • "При review кода — сначала критика, потом похвала"

Важно: User memory не должна попадать в командный memory.md. Твои личные предпочтения не должны влиять на то как агент работает у других разработчиков. Хранится в личном файле, не в репо.

Срок: постоянно Личный файл Не влияет на других Не класть в командный memory.md
🤖
4. Agent Memory — опыт subagent-а
Живёт: в рамках задачи  |  Хранится: в отчёте subagent-а

Что это такое: Знания которые накапливает специализированный агент (subagent) в процессе работы над задачей. Например: security-agent нашёл типовые уязвимости в кодовой базе — это его "опыт".

Аналогия: Подрядчик который изучил твой проект и написал отчёт о типовых проблемах. Его знания — ценны для следующего подрядчика в той же области.

Паттерн использования:

  • Subagent завершает задачу и передаёт findings главному агенту
  • Главный агент решает: что достаточно важно для Project Memory
  • Ценные паттерны → memory.md; временные результаты → не сохраняются

Риск: Bias / Staleness. Если subagent сделал неверный вывод, он может "научить" Project Memory неправильному. Всегда нужен human review перед записью в постоянную память.

Срок: жизнь задачи Только для subagent-ов Риск bias при автозаписи Human review перед сохранением
📍
5. Checkpoint Memory — состояние текущей задачи
Живёт: пока идёт задача  |  Хранится: checkpoint.md (временно)

Что это такое: Снимок состояния конкретной задачи прямо сейчас: что уже сделано, какие решения приняты в процессе, что осталось, какие риски. Нужен при длинных задачах или при передаче между агентами.

Аналогия: Стикер на мониторе: "Остановился на строке 47, нужно ещё добавить тест на edge case с null". Временная заметка для себя.

Типичная структура:

## Checkpoint: рефакторинг billing модуля Status: в процессе (75%) Done: - Разделил BillingService на BillingService + PaymentProcessor - Обновил все тесты для BillingService Decisions made: - Используем Builder pattern для PaymentRequest - Старый API оставляем для обратной совместимости Next: - Обновить тесты PaymentProcessor - PR + review Risks: - Billing интеграционные тесты могут сломаться — проверить

Ключевое отличие от Project Memory: Checkpoint удаляется после завершения задачи. Если ты засоришь memory.md старыми checkpoint-ами — он превратится в мусорную кучу.

Срок: до завершения задачи Удаляется после done Нужен для длинных задач НЕ смешивать с Project Memory

Как они отличаются: документация, Memory Bank и AGENTS.md

В репозитории часто смешивают три разных слоя. Разделение помогает не захламлять README и не дублировать одно и то же в трёх местах: люди читают docs, агент — AGENTS.md + Memory Bank.

Правило переноса: длинный текст и диаграммы → документация; устойчивые факты для каждого запроса агента → Memory Bank; каркас поведения и автозагрузка файлов → AGENTS.md. Подробная раскладка по файлам в папке memory-bank/ — на странице Структура Memory.

Memory Bank vs AGENTS.md — когда что выбирать

Оба файла попадают в context window при каждом запросе. Путаница между ними — главная причина дублей и устаревших команд. AGENTS.md (в Claude Code — CLAUDE.md) = «как работать здесь»; memory.md = «что мы накопили о проекте».

Быстрый тест

  • «Это инструкция агенту как вести себя в репо?» → AGENTS.md
  • «Это факт/решение, которое появилось в работе?» → memory.md
  • «Один и тот же факт уже есть в AGENTS?» → не дублировать, обновлять один источник

Типичная связка

AGENTS.md остаётся компактным (≈100–150 строк): Role, Stack, Build, Workflow, References.

В References: @docs/memory.md — детальные ADR, gotchas и история решений живут в Memory Bank.

Ещё похожие слои: жёсткая политика → Rules; пошаговый сценарий → Skills; матрица AGENTS / Rules / Skills → AGENTS / Rules / Skills; детали файлов memory → Структура Memory; формат AGENTS → AGENTS.md.

Что хранить, что не хранить — критерии

✅ Хранить в memory.md

  • Устойчивые факты — живут ≥ 3 месяца
  • Tech stack с версиями: Java 21, Maven 3.9
  • Точные команды: mvn -pl billing test
  • Архитектурные решения с обоснованием
  • Негативные решения: "не используем X потому что..."
  • Конвенции: PR формат, naming, coding style
  • Gotchas: нюансы которые всегда забывают
  • Project glossary: что значат специфические термины

❌ Не хранить в memory.md

  • Секреты, токены, пароли — никогда и ни при каких условиях
  • Временные статусы: "CI сейчас сломан"
  • Данные которые меняются часто (метрики, статусы PR)
  • Сырые логи и stack traces
  • То что лучше проверять live через MCP
  • Дубли того что уже есть в AGENTS.md / README
  • Checkpoint-данные текущей задачи
  • Личные предпочтения одного разработчика

Правило «3 месяца»

Спроси себя: «Этот факт будет актуален через 3 месяца?» Если нет — не кладёт в memory.md. Если да — кладёт. Этот простой фильтр исключает 80% лишнего.

Пример хорошего memory.md — шаблон

Рабочий шаблон с объяснением каждой секции:

# Project Memory — [Название проекта] # Last updated: 2025-09-15 | Owner: @team-lead ## Stack & Environment - Java 21, Maven 3.9, Spring Boot 3.2 - PostgreSQL 15 (не используем ORM — raw JDBC в DAO слое, ADR-003) - CI: GitLab CI, runners на ubuntu-22.04 - Prod: AWS ECS Fargate, staging на ECS с t3.medium ## Build & Test - Полный билд: mvn clean install - Тесты billing-модуля: mvn -pl billing test - Интеграционные: mvn -pl integration-tests test -P it - Деплой на staging: ./scripts/deploy-staging.sh [version] ## Architecture Decisions - НЕ используем Redis — overhead для нашей нагрузки (ADR-001, Q1 2025) - НЕ используем GraphQL — команда не знает, REST достаточно (ADR-002) - PaymentService — отдельный микросервис, не монолит (ADR-005, Q3 2024) - Всё через Builder pattern для request объектов — immutability ## Team Conventions - PR: Problem → Fix → Tests → Risk (обязательные секции) - API breaking changes → architecture review обязателен - Тесты обязательны для любого public метода - Коммиты: feat/fix/chore + scope, напр. feat(billing): add invoice export ## Known Gotchas - Timezone в billing: всегда UTC, конвертация только на UI (баг #234) - BillingService.calculateTax() не thread-safe — не вызывать параллельно - Тест PaymentIntegrationTest требует Docker (testcontainers) — без Docker упадёт - Переменная PAYMENT_GATEWAY_URL обязательна в .env.local (нет дефолта)
Размер: целевой — ≤ 100 строк, ≤ 500-800 слов. Всё что больше — разбей на секции или вынеси в RAG corpus. Длинный memory.md = агент читает хуже (lost in the middle).

Memory pipeline — жизненный цикл знания

1

Capture — поймать факт

Обнаружил важный факт: новое решение, новый gotcha, изменение стека. Зафиксируй сразу — не откладывай.

2

Curate — сжать и очистить

Не пиши длинные объяснения — одна строка с сутью. Убери шум. Сформулируй так, чтобы агент понял без дополнительного контекста.

3

Retrieve — получить когда нужно

Memory Bank загружается в context автоматически. Но если он большой — самое важное должно быть вверху (горячая зона attention LLM).

4

Validate — проверить актуальность

Если факт мог измениться — проверь через файл/MCP/RAG перед использованием. Агент должен знать что проверять, а не слепо доверять memory.

5

Invalidate / Update — обновить устаревшее

Стек изменился? Обнови memory.md. Решение отменили? Замени запись. Удалённая запись не накапливается — история изменений есть в git.

Memory Bank Что мы уже знаем/решили? ≤100 строк Вручную при изменениях Постоянные факты, решения, соглашения
RAG Что написано в источниках? Без ограничений Автоматически при обновлении источников Большие документы, API specs, wiki
Checkpoint Где мы остановились? 1-2 страницы Постоянно в ходе задачи Длинные задачи, handoff между агентами
Summary Что было в длинном диалоге? Авто-сжатие Автоматически при compaction Сжатие истории чата
Аналогия: библиотека

Memory Bank — закладки на самых важных страницах. RAG — вся библиотека с поиском. Checkpoint — стикер «читаю сейчас, страница 47». Ни один не заменяет другой.

Тонкости и нюансы — неочевидные ловушки

Главный риск: Memory устаревает, а агент этого не знает

Memory Bank записывает факты на момент записи. Если команда перешла с Maven на Gradle, но memory.md не обновили — агент будет давать неверные команды с абсолютной уверенностью. Он не сомневается — он просто "знает" что написано. Это хуже чем незнание — уверенная ложь может дорого обойтись. Решение: обновлять memory.md при любом изменении технологического стека, команд или архитектурных решений. Добавить "Last updated" в заголовок.

Memory ≠ RAG: не смешивай назначение

Memory Bank — устойчивые кураторные факты: "мы не используем X", "команда тестов такая-то". RAG — актуальное содержимое больших документов. Класть в memory.md весь текст README — ошибка: создаёт дублирование и засоряет context лишним. В memory — только те знания которые нужны агенту без поиска. Большие документы → RAG corpus.

Lost in the middle: длинный memory.md работает хуже

LLM лучше "видит" начало и конец контекста, чем середину (эффект "lost in the middle"). Если memory.md занимает 5000+ токенов — факты в середине файла могут игнорироваться. Решение: держи memory.md компактным (≤ 500-800 слов), организуй по секциям. Самое важное — вверху файла. Остальное — в отдельные файлы с @include или в RAG.

Checkpoint ≠ Memory — разные жизненные циклы

Memory Bank — долговременные знания о проекте, обновляются редко. Checkpoint — снимок состояния текущей задачи, создаётся и обновляется внутри задачи, удаляется после завершения. Путать их — значит засорять memory.md временными данными. Через 3 месяца будешь читать: "Done: пункт A из задачи #123" — и не понимать зачем это здесь.

Агент может записать неверный факт в memory

Если агент сам управляет memory.md через Write tool — он может записать неверный или устаревший факт с полной уверенностью: "проблема с модулем X решена" — хотя решение не проверено. Или записать предположение как факт. Стратегия: Human-in-the-loop для изменений в memory. Агент предлагает → человек review'ит → только потом merge в memory.md.

Best practices — Делай / Не делай

✅ Делай

  • Организуй memory.md по секциям: Stack, Build, Decisions, Conventions, Gotchas
  • Самое критичное — в самом начале файла (горячая зона attention)
  • Обновляй при изменениях: новая версия, новый инструмент, смена архитектуры
  • Добавляй дату к решениям: "отвергнут Redis (Q1 2025)"
  • Включай негативные решения: "решили не использовать X потому что..."
  • Версионируй memory.md в git — история изменений = история решений
  • Держи размер ≤ 100 строк — принудительная дисциплина компактности
  • Human review перед записью агентом

❌ Не делай

  • Не храни секреты, токены, пароли — никогда
  • Не дублируй то что уже есть в AGENTS.md / README
  • Не записывай временные статусы: "сейчас ветка feature/X в review"
  • Не позволяй агенту бесконтрольно писать в memory без review
  • Не делай memory.md больше 1000 строк — разбей или используй RAG
  • Не смешивай checkpoint и project memory
  • Не класть личные предпочтения в командный memory.md
  • Не забывать обновлять при смене стека — устаревший memory = ловушка

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Memory Bank и зачем он нужен?
Memory Bank — файл (обычно memory.md) с явно структурированными, долговременными знаниями о проекте: стек, команды сборки, архитектурные решения, конвенции команды. Он нужен чтобы агент не начинал каждую сессию с нуля — между сессиями LLM не помнит ничего, и без Memory Bank всё забывается. Memory Bank — мост между сессиями: "что мы знаем и решили" сохраняется в файле и читается при следующем запросе.
Junior Почему LLM-агент "забывает" всё после закрытия чата?
LLM работает с context window — текущим "рабочим полем" в котором умещается весь разговор. Когда чат закрывается — context window исчезает. Модель не хранит историю сессий нигде — нет встроенной долговременной памяти. Это фундаментальное ограничение архитектуры трансформеров: они stateless между сессиями. Именно поэтому Memory Bank — внешний файл — нужен как искусственная замена долговременной памяти.
Junior Сколько типов памяти у AI-агента?
5 типов: (1) Session memory — история текущего диалога, исчезает при закрытии; (2) Project memory — знания о проекте в memory.md, живёт месяцами; (3) User memory — личные предпочтения пользователя, постоянно; (4) Agent memory — опыт subagent-а, в рамках задачи; (5) Checkpoint memory — состояние текущей задачи, временно. Главный для работы с проектом — Project memory.
Junior Чем Session Memory отличается от Project Memory?
Session Memory — то что было в текущем чате прямо сейчас. Исчезает при закрытии чата. Сжимается при длинных диалогах. Project Memory — файл memory.md в репозитории. Существует постоянно, читается в каждой новой сессии, версионируется в git. Session = "что говорили сейчас"; Project Memory = "что знаем всегда". Session memory нельзя контролировать, Project Memory — создаётся и обновляется командой вручную.
Junior Где физически хранится memory.md?
Обычно: markdown-файл в корне проекта — memory.md или docs/memory.md. В Cursor: файл добавляется через @mention в AGENTS.md для автозагрузки. В Claude Code: через @include в CLAUDE.md. Файл версионируется в git как часть проекта — не личный, командный. Некоторые платформы имеют встроенную Memory: Claude.ai Projects (Project Knowledge), Cursor Settings → Features → Memory.
Junior Что записывать в memory.md?
Устойчивые факты (живут ≥3 месяцев): tech stack + версии, команды сборки/тестов/деплоя, архитектурные решения с обоснованием ("используем JPA, не raw SQL, потому что..."), project glossary, team conventions (PR format, naming), gotchas (нюансы которые всегда забывают), негативные решения ("не используем Redis потому что..."). Не записывать: секреты, временные статусы, данные которые меняются часто.
Junior Что такое Checkpoint Memory и когда он нужен?
Checkpoint Memory — временный файл (checkpoint.md) со снимком состояния текущей задачи: что уже сделано, какие решения приняты по ходу, что осталось, риски. Нужен при длинных многоступенчатых задачах или при передаче задачи между агентами. Ключевое: Checkpoint удаляется после завершения задачи. Это не Project Memory — это временная заметка "где мы сейчас".
Junior Чем User Memory отличается от Project Memory?
User Memory — личные предпочтения конкретного пользователя: язык ответов, стиль объяснений, уровень детализации. Хранится в личном файле (например ~/.claude/CLAUDE.md), не в репозитории. Project Memory — командные знания о проекте в memory.md в репо. User Memory = "как Петя хочет чтобы агент с ним общался"; Project Memory = "что знаем об этом проекте". Смешивать нельзя: личные предпочтения не должны влиять на работу агента с другими членами команды.
Junior Какой оптимальный размер memory.md?
Целевой размер: ≤ 100 строк, ≤ 500-800 слов. Обоснование: (1) Memory Bank читается целиком при каждом запросе — большой файл = большой token overhead; (2) LLM плохо "видит" середину длинного контекста (lost in the middle) — факты в середине 500-строчного memory.md могут игнорироваться; (3) Большой memory.md = признак неправильного использования (туда попало то что должно быть в RAG). При превышении — разбить на секции или вынести в отдельные файлы.
Junior Как создать memory.md для нового проекта?
Минимальные шаги: (1) Создать файл docs/memory.md или memory.md в корне репо; (2) Добавить секции: ## Stack, ## Build & Test, ## Architecture Decisions, ## Team Conventions, ## Known Gotchas; (3) Заполнить актуальными фактами — конкретно, не абстрактно; (4) Добавить ссылку в AGENTS.md или CLAUDE.md для автозагрузки. Начать с малого — 10-20 строк — и дополнять по ходу работы. Плохой заполненный memory.md лучше хорошего пустого.
Middle — 10 вопросов
Middle Что писать в Memory Bank, а что в AGENTS.md?
AGENTS.md — стабильный конфигурационный контекст: кто агент, workflow, стек, команды, DoD. Обновляется редко. Memory Bank — накапливаемые знания о проекте: конкретные ADR с датами, найденные gotchas, результаты исследований, эволюция решений. AGENTS.md = "как работать"; Memory Bank = "что знаем о проекте". В маленьких проектах их можно объединить. Если AGENTS.md растёт — декомпозировать: постоянный контекст оставить, накопленные знания → Memory Bank.
Middle Как структурировать эффективный memory.md?
Эффективная структура: (1) Заголовок с датой "Last updated" и owner; (2) ## Stack & Environment — технологии, версии; (3) ## Build & Test — точные команды; (4) ## Architecture Decisions — ADR с обоснованием и датой; (5) ## Team Conventions — PR format, naming; (6) ## Known Gotchas — нюансы которые всегда забывают. Критически важное — вверху (горячая зона attention). Каждая секция — конкретные факты, не абстракции. Одна строка = один факт.
Middle Как обновлять Memory Bank в процессе работы?
Триггеры обновления: принятый ADR, изменение стека или команд, обнаруженный gotcha, изменение конвенций. Три подхода: (1) Manual — разработчик обновляет при значимых изменениях; (2) Agent-assisted — агент предлагает обновление по результатам сессии, человек review'ит и approve'ит; (3) Automated — CI hook при изменении pom.xml/Makefile обновляет Build секцию и создаёт PR. Минимум: добавить в PR checklist "обнови Memory Bank если изменились архитектурные факты".
Middle В чём разница между Memory Bank и RAG — когда что применять?
Memory Bank — явные, кураторные, структурированные факты. Читается целиком. Для устойчивых знаний (решения, команды, ≤100 строк). RAG — семантический поиск по корпусу документов, возвращает релевантные chunks. Для больших объёмов (API specs, архитектурные документы, wiki). Правило: небольшой набор стабильных фактов → Memory Bank. Большие документы с поиском по содержимому → RAG. Комбинация: Memory Bank хранит ключевые выводы, RAG — полный контекст.
Middle Когда делать Checkpoint вместо записи в Memory Bank?
Checkpoint — снимок состояния конкретной текущей задачи: что сделано, решения по ходу задачи, следующие шаги, риски. Lifetime: до завершения задачи — потом удаляется. Memory Bank — долговременные знания о проекте, lifetime: месяцы/годы. Правило: статус задачи, промежуточные результаты, "где остановились" → Checkpoint. Принятые ADR, найденные gotchas, изменения стека, важные выводы после завершения → Memory Bank. Смешивание засоряет memory.md временными данными.
Middle Какие anti-patterns при использовании Memory Bank?
Главные anti-patterns: (1) God Memory Bank — 1000+ строк всего подряд → lost in the middle; (2) Volatile state в Memory — "сейчас CI сломан" → устаревает за часы; (3) Секреты в Memory — никогда; (4) Автообновление без human review — агент может записать неверный факт с уверенностью; (5) Дублирование AGENTS.md — один факт в двух местах рассинхронизируется; (6) Checkpoint данные в Project Memory — временное в постоянном.
Middle Как организовать Memory Bank для командной разработки?
Командный Memory Bank: (1) Хранить в git — version history = история архитектурных решений; (2) Изменения через PR с review; (3) Назначить Memory Bank owner — ответственный за актуальность; (4) Чётко разграничить: общекомандные знания в memory.md, личные предпочтения — в личном ~/.claude/CLAUDE.md; (5) Sprint review — проверять актуальность раз в спринт. Командный Memory Bank = shared understanding, не личный блокнот.
Middle Как сжимать Memory Bank чтобы экономить токены?
Стратегии сжатия: (1) Один факт — одна строка (максимальная информационная плотность); (2) Детальные ADR → ссылки на docs/adr/ файлы, в Memory Bank только суть; (3) Большая документация → RAG corpus, Memory Bank хранит только ключевые выводы; (4) Удалять устаревшие факты — для истории есть git; (5) Целевой размер: ≤100 строк. Длинный Memory Bank — технический долг.
Middle Как хранить архитектурные решения в memory.md?
ADR в Memory Bank: записывать не только WHAT но и WHY с датой — "не используем Redis для кэша, т.к. операционный overhead не оправдан для нашей нагрузки (Q1 2025)". Это позволяет агенту понять когда причина устарела. Структура: краткое решение + обоснование + дата + ссылка на детальный ADR файл. При изменении решения — обновить запись с датой. Негативные решения ("не используем X") — особенно ценны: агент не будет предлагать то что уже отвергнуто.
Middle Как автоматически обновлять Memory Bank через Hooks?
Автоматизация через Hooks: (1) PostToolUse Hook — после завершения значимой задачи агент предлагает обновление Memory Bank; (2) CI webhook — при изменении pom.xml/package.json запускает скрипт который обновляет секцию Build в Memory Bank и создаёт PR; (3) Pre-commit hook — проверяет что если изменился стек, Memory Bank тоже обновлён. Риск: агент может записать неверное. Оптимум: автоматизировать триггер, но требовать human review перед merge.
Senior — 5 вопросов
Senior Как спроектировать архитектуру Memory Bank для сложного проекта?
Архитектура для сложного проекта: (1) Federated Memory — несколько файлов по доменам (billing-memory.md, auth-memory.md, infra-memory.md); главный memory.md с @include ссылками; (2) Memory hierarchy — глобальный (organization conventions) + project (конкретный проект) + module (специфика модуля); (3) Versioned ADR — отдельные docs/adr/ файлы, в Memory только сводка с ссылками; (4) Separation по volatility — стабильные факты в одном файле, более изменчивые в другом. Каждый файл: ≤100 строк.
Senior В чём разница между Memory Bank и external state store?
Memory Bank — документ для чтения агентом; human-curated знания; обновляется вручную; не масштабируется на большие объёмы; нет built-in querying. External state store (Redis, PostgreSQL, векторная DB) — программный storage; обновляется автоматически; масштабируется; поддерживает сложные запросы; доступен через MCP tools. Memory Bank оптимален для небольшого набора постоянных знаний. External state store + MCP = для динамических, часто меняющихся данных (CI статус, метрики, live configuration).
Senior Как выстроить compression strategy для Memory Bank?
Стратегия сжатия: (1) Information density — каждая строка несёт максимум информации; (2) Hierarchical summary — детальные документы в RAG corpus, Memory Bank хранит только выводы и ссылки; (3) Time-based cleanup — факты старше X лет — на удаление или архив; (4) Deduplication — Memory и AGENTS.md не должны содержать одно и то же; (5) Progressive disclosure — в Memory Bank сводка, агент запрашивает детальный файл по необходимости. Метрика: information per token.
Senior Как диагностировать и починить устаревший Memory Bank?
Диагностика: (1) Сравнить факты Memory Bank с live sources — версии в pom.xml, команды в Makefile; (2) Попросить агента проверить Memory Bank против текущих файлов проекта; (3) CI check — скрипт сравнивает версию Java/Node в pom.xml с тем что в memory.md. Починить: (1) "Memory audit" — агент проходит по каждой записи и проверяет актуальность; (2) Обновить с датами; (3) Удалить неактуальные; (4) Добавить "Last validated: DATE" в заголовок.
Senior Как использовать Memory Bank в multi-agent workflow?
Multi-agent паттерны: (1) Shared read-only Memory — все агенты читают один memory.md, только main agent пишет → предотвращает concurrent write conflicts; (2) Federated Memory — каждый специализированный агент имеет свой memory файл; main agent читает все; (3) Memory handoff — при завершении subagent task, findings передаются main agent который решает что записать в проектную Memory; (4) Memory as context injection — main agent передаёт релевантную часть Memory как context для subagent. Last write wins — главная опасность concurrent writes.
Lead — 5 вопросов
Lead Как организовать командный Memory Bank с governance?
Командный governance: (1) Храни Memory Bank в git с обязательным review через PR; (2) Назначь Memory Bank owner — отвечает за актуальность, review PR, quarterly audit; (3) CODEOWNERS для memory.md — автоматически запрашивает review у ответственного; (4) Определи что попадает в командный Memory Bank vs личный (~/.claude/CLAUDE.md); (5) Sprint review — слот для проверки актуальности; (6) Definition of Done для ADR: "запись в Memory Bank". Governance делает Memory Bank живым инструментом, а не устаревающим артефактом.
Lead Как выстроить процесс обновления Memory Bank в команде?
Процесс обновления: (1) Триггеры — явно определить события: принятый ADR, смена стека, изменение команд; (2) PR checklist — "обнови Memory Bank если изменились архитектурные факты"; (3) Post-incident review — если агент ошибся из-за устаревшей memory → обновить как часть post-mortem; (4) Automated checks — CI проверяет версии в memory.md против pom.xml/package.json; (5) Quarterly "memory spring cleaning". Стабильный процесс важнее разовой хорошей записи.
Lead Как распределить ownership Memory Bank?
Модели ownership: (1) Single owner — tech lead отвечает за все обновления; минус: bottleneck; (2) Domain ownership — каждый модуль имеет своего owner для своего memory файла; (3) Collective ownership — любой разработчик может предложить изменение через PR, review от tech lead. Рекомендация: collective ownership с mandatory review для существенных изменений. CODEOWNERS с per-section responsibility. Главное: нет бесхозных строк — у каждой записи есть ответственный за актуальность.
Lead Как использовать Memory Bank для onboarding новых разработчиков?
Onboarding через Memory Bank: (1) Ссылка на memory.md в README как "прочитай это первым"; (2) Секции "Architecture Overview" и "Common Pitfalls" — самое нужное новичку; (3) Первая задача нового разработчика — проверить Memory Bank и добавить вопросы которые возникли; (4) AI-assisted onboarding — агент с Memory Bank отвечает на вопросы нового разработчика в контексте проекта; (5) После первого месяца — новый разработчик обновляет Memory Bank своими открытиями. Memory Bank улучшается от каждого нового взгляда.
Lead Как использовать Memory Bank как живую документацию проекта?
Memory Bank как knowledge base: (1) Делает знание явным — неявные знания ("у нас баг с timezone в billing") записываются и доступны всем; (2) Снижает key-person dependency — знание не умирает когда Петя уходит; (3) Живая документация — обновляется вместе с проектом, в отличие от Wiki которая устаревает; (4) AI-enhanced — агент использует Memory Bank для ответов в контексте проекта; (5) Onboarding — новый разработчик + агент с Memory Bank = быстрый ввод в проект. Ограничение: Memory Bank не заменяет детальную документацию — только фиксирует ключевые выводы.
Architect — 5 вопросов
Arch Как Memory Bank вписывается в архитектуру AI-first разработки?
AI-first архитектура — разработка где AI-агент является первоклассным участником команды. Memory Bank — его persistence layer: агент работает эффективно только с актуальным контекстом. В AI-first архитектуре Memory Bank становится частью project infrastructure наравне с CI/CD. Роль: context continuity (агент "помнит" проект), knowledge sharing (вся команда + агент на одной волне), decision traceability (ADR с датами). Без Memory Bank AI-first разработка деградирует: агент = умный, но amnesiatic assistant.
Arch Как спроектировать persistence layer для AI agents?
Persistence layer: (1) Hot memory — Memory Bank файл, читается при каждом запросе; ≤100 строк; (2) Warm memory — RAG corpus; большие документы через semantic search; (3) Cold memory — архив ADR, historical context; доступен через explicit @mention; (4) Live data — MCP tools для актуального состояния системы. Агент: Hot → Warm → Cold → Live. Архитектура позволяет масштабировать knowledge base без деградации производительности.
Arch В чём разница между Memory Bank и Knowledge Graph?
Memory Bank — flat markdown файл; простота; нет relationships; не масштабируется на тысячи записей. Knowledge Graph — structured graph database (Neo4j, RDF); explicit relationships; semantic querying; автоматическое inference; масштабируется; требует MCP интеграции. Memory Bank оптимален для small-medium projects с сотнями фактов. Knowledge Graph — для enterprise с тысячами взаимосвязанных знаний. Комбинация: Memory Bank как high-level index, Knowledge Graph как queryable backend через MCP.
Arch Как комбинировать Memory Bank и RAG в dual memory system?
Dual memory system: Memory Bank (explicit/curated) + RAG (implicit/retrieved). Архитектура: (1) Memory Bank — всегда в context; устойчивые факты, решения, конвенции; (2) RAG corpus — большие документы; поиск по запросу; (3) Memory Bank содержит ссылки и ключевые выводы из RAG документов; RAG обеспечивает детали. Паттерн: агент сначала проверяет Memory Bank ("что мы знаем?"), затем при необходимости RAG ("что в документации?"). Ни один не заменяет другой: Memory Bank даёт структуру и decisions; RAG — актуальный content.
Arch Как спроектировать масштабируемую архитектуру агентной памяти?
Масштабируемая архитектура: (1) Layered memory с разными TTL — hot (session, часы), warm (project, месяцы), cold (organization, годы); (2) Domain partitioning — memory по доменам, каждый с отдельным owner; (3) Automated maintenance — CI pipeline для validation и cleanup; hooks для триггеров обновления; (4) Memory as code — хранится в git, имеет schema, тестируется CI; (5) Federated multi-agent memory — главный агент координирует, специализированные агенты обновляют свои домены. Метрика здоровья: information freshness score.
