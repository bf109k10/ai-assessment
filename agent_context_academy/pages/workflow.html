Agent Context Academy · Фундамент (после Agent Loop)

Explore → Plan → Code

Три шага процесса работы с агентом: Explore (разобраться в контексте) → Plan (согласовать план) → Code (писать код маленькими шагами с проверкой и коммитом). Это не «как писать код на языке», а в каком порядке вести задачу после Agent Loop. Best practices — Cursor и Claude Code; параллельные сценарии — в связке с Subagents.

Что такое Explore → Plan → Code и зачем это нужно

Explore → Plan → Code = рецепт, а не импровизация

Представь повара: без рецепта — каждый раз разный результат, забытые ингредиенты. С рецептом — предсказуемое качество. Workflow для агента — тот же рецепт: фиксированная последовательность шагов, которая срабатывает надёжно.

  • Сначала понять задачу и контекст
  • Потом план — что и в каком порядке
  • Затем Code — правки, verify (тесты/линт), commit

Cursor: Plan mode + правила процесса

В Cursor воркфлоу строится вокруг Plan mode (сначала план, только после одобрения — Agent mode), Rules (зафиксировать процесс), Commands (запустить сценарий), Hooks (гейты) и Background / Cloud agents для параллельного и фонового исполнения.

Claude Code: тот же триптих

Документированный цикл Explore → Plan → Code → Commit (commit — часть этапа Code). Плюс /clear, TDD внутри Code, subagents — позже в треке.

С нуля: как это работает изнутри — Claude Code / Cursor

Концепция одинакова в Claude Code и Cursor — разница только в том, как называются кнопки и файлы. Начни отсюда, если раньше просто писал «сделай X» в чат.

Уровень 0 — простыми словами

Если попросить агента «сделай фичу» одним сообщением — он бросится писать код, не разобравшись, и часто наделает лишнего. Workflow разбивает это на этапы: сначала агент читает проект, потом предлагает план (ты его смотришь и правишь), только потом пишет код, и проверяет себя тестами. Меньше сюрпризов, легче откатить.

Базовый цикл: Explore → Plan → Code

Ядро процесса в Cursor и Claude Code. Verify и commit — не отдельные «философские» этапы, а обязательная часть Code.

1

Explore

Читает файлы, доки, тесты. Без правок — только понять задачу и контекст.

2

Plan

План шагами. Ты смотришь, правишь, одобряешь — gate до кода.

3

Code

Правки по плану → verify (тесты, линт, запуск) → commit/PR. Не прошло verify — чиним в Code.

Почему именно так: LLM отлично пишет код, но плохо угадывает намерение и границы задачи. Explore и Plan дают агенту контекст и заранее согласованные рамки, Verify — объективный сигнал вместо «уверенного, но неверного» ответа. Пропуск Explore/Plan — причина №1 переусложнённых и мимо-задачи изменений.

Вариант с TDD — когда есть чёткий контракт

# TDD-воркфлоу (best practice Claude Code и Cursor) 1. Explore — понять требование и существующий код 2. Tests — написать падающие тесты по контракту (код ещё НЕ трогаем) 3. Confirm — убедиться, что тесты действительно падают (red) 4. Implement— писать код, пока тесты не позеленеют (green) 5. Refactor — почистить структуру, тесты держим зелёными 6. Commit — тесты + код одним осмысленным коммитом # Тесты = объективный gate. Агент не может «схитрить» — критерий готовности внешний.

Параллельные воркфлоу — несколько агентов сразу

Когда задача распадается на независимые части — их можно вести параллельно. Сначала освой линейный Explore → Plan → Code; параллель и fan-out логичнее после Subagents.

# Fan-out → Fan-in: разложить, сделать параллельно, собрать ┌────────────────────┐ │ Задача целиком │ └─────────┬──────────┘ ┌───────────────┼───────────────┐ # FAN-OUT: делим на независимые части ▼ ▼ ▼ ┌──────────────┐┌──────────────┐┌──────────────┐ │ Агент A ││ Агент B ││ Агент C │ │ worktree/ ││ worktree/ ││ worktree/ │ # изоляция: своя ветка/каталог │ ветка feat-a ││ ветка feat-b ││ ветка feat-c │ └──────┬───────┘└──────┬───────┘└──────┬───────┘ └───────────────┼───────────────┘ # FAN-IN: собираем результаты ▼ ┌────────────────────┐ │ Merge + Verify │ # слить → прогнать тесты на объединённом └────────────────────┘
🖱 Cursor — параллельно
  • Несколько Background / Cloud agents — запускаешь разные задачи, они идут параллельно и не блокируют чат.
  • Git worktrees — каждый агент в своём рабочем каталоге/ветке, правки не пересекаются.
  • Best-of-N — несколько попыток одной задачи в изолированных worktree, потом выбираешь лучшую.
  • Параллельные subagents — несколько задач в одном сообщении запускаются одновременно.
📖 Cursor — Background Agents
🤖 Claude Code — параллельно
  • Параллельные subagents — например 3 агента исследуют разные части кода, у каждого свой контекст (изоляция контекста главного агента).
  • Несколько сессий / git worktrees на одну репу — независимые ветки задач.
  • Headless claude -p — fan-out задач в скриптах и CI (например, «пройди по 20 файлам»).

Пример из практики: три независимых модуля в монорепо

Ситуация: в монорепо нужно добавить unit-тесты в три пакета, которые не импортируют друг друга: packages/auth, packages/billing, packages/notifications. Один агент в одном каталоге — долго; три изолированных потока — быстрее и без гонок за одни файлы.

🔀

Fan-out: три агента, три worktree

Реальный сценарий в Cursor или Claude Code + git

parallel
# 0. Главный агент (ты) — Plan, один раз Задача: в каждом пакете — покрыть публичный API characterization-тестами. Критерий готовности: npm test -w packages/auth (и аналог для billing, notifications) зелёный. # Не параллелим, пока нет общего плана и критериев verify. # 1. FAN-OUT — изоляция (терминал, до агентов) git fetch origin git worktree add ../oral-auth -b feat/tests-auth origin/main git worktree add ../oral-billing -b feat/tests-billing origin/main git worktree add ../oral-notify -b feat/tests-notify origin/main # Три каталога = три агента не трогают одни и те же файлы. # 2. Параллельная работа # Cursor: три Background/Cloud agent ИЛИ три окна + Agent в каждом worktree. # Claude Code: три сессии в ../oral-auth, ../oral-billing, ../oral-notify # Или один чат — параллельные subagents (одно сообщение): «Запусти 3 subagents параллельно: A) только packages/auth — тесты по плану, без правок billing/notifications; B) только packages/billing; C) только packages/notifications. Каждый возвращает: список файлов, что добавил, вывод npm test -w ….» # 3. FAN-IN — слияние и общий verify (ты или главный агент) cd oral-auth && git push -u origin feat/tests-auth cd ../oral-billing && git push -u origin feat/tests-billing cd ../oral-notify && git push -u origin feat/tests-notify # Три PR или последовательный merge в main: git checkout main && git pull git merge feat/tests-auth # resolve conflicts если только в lockfile git merge feat/tests-billing git merge feat/tests-notify npm test # полный прогон на объединённом коде — обязательный gate # 4. Уборка git worktree remove ../oral-auth git worktree remove ../oral-billing git worktree remove ../oral-notify
Что здесь «реального»: worktree — обычный git, не магия IDE; параллельные subagents — один промпт с тремя непересекающимися зонами; fan-in — не «склеить тексты агентов», а merge веток + общий CI. Если бы billing зависел от нового API в auth — сначала auth (последовательно), потом billing, а не три агента сразу.

Изоляция состояния обязательна — иначе race на файлах

Два агента, редактирующие одни и те же файлы в одном рабочем каталоге, затирают правки друг друга и ломают сборку. Правило: один агент = своя ветка/worktree. Только так параллелизм безопасен. Сливаешь результаты в конце (fan-in) и там же прогоняешь общий verify.

Параллелить можно только независимые задачи

Если задача B зависит от результата A (B использует API, который A ещё пишет) — параллелизм не ускоряет, а создаёт рассинхрон. Параллель хороша для независимых кусков: разные модули, разные файлы, разные попытки одной задачи (best-of-N). Зависимые шаги — только последовательно (или через DAG-зависимости).

Параллелизм повышает стоимость контекста и сложность ревью

N агентов = N контекстов = больше токенов и N веток на ревью. Иногда быстрее и дешевле сделать последовательно. Параллель оправдана, когда выигрыш по времени важнее и части действительно независимы. Не «параллель ради параллели».

Новая функциональность от идеи до PR

Багфикс по репорту

От воспроизведения до регресс-теста

reproduce→fix→regress
1. Reproduce: воспроизведи баг, зафиксируй шаги 2. Isolate: бинарный поиск — где ломается 3. Regress: напиши падающий тест на баг (red) 4. Fix: чини корневую причину, не симптом → тест зелёный 5. Verify: полный прогон тестов, ничего не сломалось 6. Commit: fix: ... + ссылка на issue
♻️

Рефакторинг с тестами

Меняем структуру, не поведение

safe refactor
1. Характеризующие тесты ДО любых изменений (фиксируем поведение) 2. Одно маленькое изменение за раз 3. После каждого — тесты зелёные (иначе откат/checkpoint) 4. Поведение НЕ меняем, только структуру 5. Каждый логический шаг — отдельный коммит
🔀

Параллельный рефакторинг модулей

Fan-out по независимым пакетам

fan-out→fan-in
1. Split Раздели: pkg/api, pkg/ui, pkg/db — независимы 2. Fan-out Агент на модуль, каждый в своей ветке/worktree 3. Work Параллельно: каждый рефакторит свой пакет + тесты 4. Fan-in Слить ветки по очереди, решить конфликты 5. Verify Прогнать ВСЕ тесты на объединённом коде # Если модули зависят друг от друга — НЕ параллелить, делать по порядку.

Тонкости

Гигиена контекста: /clear и свежие сессии между этапами

Длинная сессия копит мусор: старые файлы, отменённые попытки, устаревшие рассуждения. Это ухудшает ответы (модель «тонет» в шуме). Best practice Claude Code — /clear между несвязанными этапами. В Cursor — начать новый чат или использовать subagent для изолированного куска. Правило: новый крупный этап = чистый контекст.

План перед кодом — не формальность, а экономия

Одобренный план ловит недопонимание до написания кода, когда правка стоит одну фразу, а не переписывание файлов. Plan mode (Cursor) и plan mode (Claude Code) существуют именно для этого. Пропустил план на «простой» задаче — чаще всего именно там агент и уходит не туда.

Узкие, проверяемые шаги — «Fix failing tests in auth.ts», а не «почини всё»

Чем конкретнее и проверяемее шаг, тем предсказуемее результат. У шага должен быть объективный критерий готовности (тест зелёный, линтер чист, эндпоинт отвечает 200). Расплывчатые задачи → расплывчатый результат и невозможность проверить.

Тесты ≠ approval gate — HITL для необратимого

Автоматический verify (тесты/линт) отлично ловит регрессии, но не заменяет человека на необратимых операциях: удаление данных, деплой в prod, миграции, финансовые действия. Такие шаги — всегда явный human gate. Тесты — про «код корректен», gate — про «мы действительно хотим это сделать».

Длинный воркфлоу → lost in the middle

Если воркфлоу — это один гигантский промпт на 15 шагов, средние шаги теряются (модель хуже помнит середину контекста). Лучше: короткие фазы, checkpoint/summary между ними, делегирование крупных кусков в subagents. Важное держи в начале и конце текущего контекста.

Best practices

Делай

  • Сначала Explore и Plan, только потом код
  • Одобряй план до реализации (Plan mode / plan mode)
  • Узкие проверяемые шаги с объективным критерием готовности
  • Verify после каждого шага: тесты, линт, типы, запуск
  • Зафиксируй процесс в Rules / CLAUDE.md / Command
  • Чисти контекст между этапами (/clear, новый чат)
  • HITL-гейт на необратимых операциях
  • Параллель — только для независимых частей, в изоляции (worktree)

Не делай

  • «Сделай фичу» одним сообщением без плана
  • Код без объективной проверки — «кажется, работает»
  • Гигантский монолитный промпт на 15 шагов
  • Параллелить зависимые задачи (race, рассинхрон)
  • Несколько агентов в одном рабочем каталоге на одних файлах
  • Автодеплой без human gate «на удачу»
  • Копить один бесконечный чат для всех задач подряд

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое workflow агента?
Workflow — это повторяемый процесс, по которому агент решает задачу: фиксированная последовательность шагов (собрать контекст → спланировать → написать код → проверить → закоммитить). Он превращает хаотичную «переписку с ИИ» в предсказуемый инженерный процесс с точками проверки.
Junior Из каких этапов состоит базовый цикл?
Explore (изучить контекст) → Plan (составить план и одобрить) → Code (правки, verify, commit/PR). Verify и commit — часть этапа Code, не отдельные «философские» шаги. Общее для Cursor и Claude Code.
Junior Почему нельзя просто написать «сделай фичу» одним сообщением?
Агент бросится писать код, не разобравшись в задаче и границах, и часто сделает лишнее или мимо цели. Без Explore он не знает контекст, без Plan — не согласовал с тобой рамки. Результат сложнее проверить и откатить. Разбивка на этапы убирает эти риски.
Junior Что делает этап Explore?
Агент читает нужные файлы, документацию, тесты — без правок кода. Цель — понять, как устроен проект, прежде чем что-то менять. Это read-only сбор контекста: grep, чтение, поиск, уточняющие вопросы.
Junior Зачем этап Plan и одобрение плана?
План — это список шагов, который ты смотришь и правишь до написания кода. Так недопонимание ловится, когда правка стоит одну фразу, а не переписывание файлов. Одобрение плана — это gate: код не пишется, пока человек не согласен.
Junior Что такое Verify и зачем он нужен?
Verify — объективная проверка после изменений: запуск тестов, линтера, проверки типов, сборки, самого приложения. Он даёт факт «работает/не работает» вместо «кажется, работает». Без Verify легко закоммитить сломанный код.
Junior Что такое Plan mode в Cursor?
Plan mode — режим только для чтения, где агент исследует и составляет план реализации, но не меняет файлы. Реализация начинается в Agent mode только после того, как ты одобрил план. Это встроенный способ сделать этап Plan.
Junior Что делает команда /clear в Claude Code?
/clear очищает контекст текущей сессии. Её используют между несвязанными этапами задачи, чтобы старый мусор (отменённые попытки, устаревшие файлы) не мешал модели. Свежий контекст = более точные ответы.
Junior Что такое параллельный воркфлоу простыми словами?
Это когда задачу делят на независимые части и делают их одновременно разными агентами/сессиями — каждый в своей изоляции (ветка/каталог). Потом результаты собирают вместе. Быстрее, но подходит только для частей, которые не зависят друг от друга.
Junior Где зафиксировать воркфлоу, чтобы агент его соблюдал?
В Cursor — в Rules (.cursor/rules/*.mdc) и Commands. В Claude Code — в CLAUDE.md и .claude/commands/. Там описываешь порядок шагов («сначала план», «после правок — тесты»), и агент следует ему автоматически (Rules/CLAUDE.md) или по вызову (Command).
Middle — 10 вопросов
Middle Когда использовать TDD-воркфлоу вместо обычного?
Когда есть чёткий контракт/спецификация: известны входы, выходы, поведение. Пишешь падающие тесты по контракту, убеждаешься что они красные, потом код до зелёного. Тесты становятся объективным gate — агент не может «схитрить», критерий готовности внешний. Для исследовательских задач без ясного контракта TDD менее удобен.
Middle Как сделать шаги воркфлоу проверяемыми?
У каждого шага должен быть объективный критерий готовности: «тесты в auth.test.ts зелёные», «линтер чист», «эндпоинт /login отвечает 200». Формулируй узко: «Fix failing tests in auth.ts», а не «почини авторизацию». Проверяемый шаг = предсказуемый результат + возможность автоматического Verify.
Middle Когда стоит параллелить, а когда нет?
Параллель оправдана для независимых частей: разные модули, разные файлы, попытки best-of-N. Не параллелить, если задача B зависит от результата A (использует API, который A ещё пишет) — получишь рассинхрон, а не ускорение. Зависимые шаги — последовательно или через DAG.
Middle Зачем git worktrees при параллельной работе?
Worktree даёт каждому агенту отдельный рабочий каталог и ветку. Без изоляции два агента редактируют одни файлы и затирают правки друг друга, ломая сборку. Правило: один агент = свой worktree/ветка. Результаты сливаются в конце (fan-in), там же общий Verify.
Middle Что такое fan-out / fan-in?
Fan-out — разложить задачу на независимые подзадачи и запустить их параллельно (по агенту на подзадачу). Fan-in — собрать результаты: слить ветки, разрешить конфликты и прогнать тесты на объединённом коде. Это стандартный паттерн параллельного воркфлоу.
Middle Как гигиена контекста влияет на качество?
Длинная сессия копит шум: старые файлы, отменённые попытки, устаревшие рассуждения — модель «тонет» в них и отвечает хуже (lost in the middle). Между несвязанными этапами делай /clear (Claude Code) или новый чат/subagent (Cursor). Новый крупный этап = свежий контекст.
Middle Чем subagent помогает воркфлоу?
Subagent изолирует контекст: тяжёлое исследование или отдельный кусок выполняется в своём контексте и возвращает только выводы, не раздувая контекст главного агента. Плюс несколько subagents можно запустить параллельно (например, исследовать 3 части кода одновременно).
Middle Как встроить HITL-гейты в воркфлоу?
Определи необратимые/опасные операции (delete, deploy, миграции, финансы) и требуй явного человеческого подтверждения перед ними. В Cursor — Permissions/Hooks и Plan mode; в Claude Code — permissions + approval. Тесты не заменяют gate: они про «код корректен», gate — про «мы действительно хотим это сделать».
Middle Как воркфлоу для багфикса отличается от воркфлоу фичи?
Багфикс начинается с воспроизведения и изоляции, затем пишется регресс-тест на баг (падающий), потом чинится корневая причина (не симптом) до зелёного теста. Фича начинается с Explore существующей архитектуры и Plan новой функциональности. Общее — Verify и Commit; разное — старт (reproduce vs explore-arch).
Middle Как использовать Commands/Skills для повторяемых воркфлоу?
Вынеси частый процесс в Command (/pr-review, /refactor) или Skill: там пошагово описан процесс и формат вывода. Тогда любой в команде запускает одинаковый воркфлоу одной командой, без необходимости помнить все шаги. Это стандартизирует «правильный способ» выполнения задачи.
Senior — 5 вопросов
Senior Как спроектировать воркфлоу, устойчивый к неудачным шагам?
Ключевое — обратимость и точки восстановления: (1) маленькие шаги с Verify после каждого, чтобы сбой локализовался; (2) checkpoint/commit на каждом зелёном состоянии — есть куда откатиться; (3) явный «возврат на этап» в цикле (не прошёл Verify → назад в Implement/Plan, а не вперёд); (4) для параллели — изоляция в worktree, чтобы падение одного агента не ломало остальных. Воркфлоу должен уметь откатываться, а не только идти вперёд.
Senior Как выбрать между последовательным и параллельным воркфлоу?
Считай две вещи: степень независимости частей и стоимость координации. Параллель выигрывает, когда части действительно независимы (разные модули/файлы) и выигрыш по времени превышает накладные расходы: N контекстов (больше токенов), N веток на ревью, стоимость fan-in (merge + конфликты). Если есть зависимости — либо последовательно, либо DAG с топологическим порядком. Best-of-N — частный случай параллели для одной задачи ради качества, а не скорости.
Senior Как диагностировать, что агент «уходит не туда» в воркфлоу?
Признаки: правит файлы вне плана, растёт diff без роста готовности, зелёные тесты но не та фича, зацикливание на одном шаге. Причины и лечение: пропущен Explore/Plan → вернуть план и approval; переполнен контекст (lost in the middle) → /clear/summary/subagent; расплывчатый шаг → переформулировать в проверяемый; нет объективного Verify → добавить тест как критерий. Диагностика = сверять фактические правки с одобренным планом.
Senior Как строить многоэтапный воркфлоу без потери контекста?
(1) Короткие фазы вместо одного мегапромпта; (2) checkpoint/summary между фазами — компактное состояние (что сделано, что дальше, ключевые решения); (3) делегировать тяжёлые куски в subagents, возвращая только выводы; (4) важное (цель, ограничения) держать в начале/конце текущего контекста; (5) процесс — в Rules/CLAUDE.md, чтобы не пересказывать его каждый раз. Так контекст остаётся релевантным на всех этапах.
Senior Как встроить Verify как жёсткий gate, а не «на словах»?
Сделай проверку недоступной для обхода из чата: Hooks (PostToolUse) запускают тесты/линт и блокируют дальше при красном; CI-гейт на PR не даёт мержить без зелёного; в Command/Rule явно прописан «коммит только при зелёных тестах». Разница с «просьбой проверить» — принуждение на уровне инструмента/пайплайна, а не намерения модели. Так Verify нельзя пропустить, даже если агент «уверен».
Lead — 5 вопросов
Lead Как стандартизировать воркфлоу в команде?
Зафиксируй канонический процесс в версионируемых артефактах: Rules/CLAUDE.md (обязательный порядок Explore→Plan→Verify→Commit + запреты) и общий набор Commands (/pr, /refactor, /debug). Опиши в AGENTS.md, какие воркфлоу есть и когда их применять. Онбординг включает «прогони /pr-review на любом PR». Цель — чтобы правильный процесс был путём по умолчанию, одинаковым для junior и senior.
Lead Как измерять эффективность воркфлоу?
Метрики: доля PR, прошедших CI с первого раза; время от задачи до мержа; количество откатов/hotfix; доля задач, где план был одобрен без переделки. Сравнивай до/после внедрения воркфлоу. Через PostToolUse hooks можно логировать, как часто запускается Verify и сколько шагов проходит без падений. Рост «зелёных с первого раза» и падение переделок = воркфлоу работает.
Lead Как управлять параллельной работой нескольких агентов в команде?
Правила координации: (1) один агент = своя ветка/worktree, единый naming (feat/agent-a-...); (2) разделение зон ответственности по модулям, чтобы не пересекались файлы; (3) явная точка fan-in с ответственным за merge и общим Verify; (4) лимит на число параллельных агентов (стоимость контекста и ревью); (5) для необратимого — единый gate, а не «каждый агент сам деплоит». Параллелизм — управляемый, а не стихийный.
Lead Как поддерживать воркфлоу в актуальном состоянии?
Воркфлоу — код, а не документация: при изменении процесса деплоя/архитектуры обновление Rules/Commands входит в Definition of Done. Практики: owner у каждого воркфлоу, PR-ревью изменений процесса, дата последнего обновления, квартальный аудит, алерт на Commands без изменений 6+ месяцев. Обратная связь: разработчик нашёл, что шаг устарел → issue, а не молчаливый обход.
Lead Как воркфлоу помогает онбордингу?
Новичок не должен помнить 15-шаговый процесс: он вызывает /pr-review и следует одобренному плану — воркфлоу кодирует накопленный опыт команды. В AGENTS.md/CLAUDE.md описано, какой воркфлоу для чего. Первая неделя: попробовать основные Commands, сделать задачу по циклу Explore→Plan→Verify→Commit под присмотром. Так знания не теряются при ротации людей.
Architect — 5 вопросов
Arch Как спроектировать воркфлоу как DAG зависимостей?
Представь шаги как узлы, зависимости — рёбра ( acyclic). Топологический порядок задаёт, что можно параллелить (узлы без общих предков) и что строго последовательно (по рёбрам зависимостей). Fan-out — параллельные ветви DAG, fan-in — узлы слияния, где сходятся зависимости и стоит общий Verify. Такой дизайн даёт максимум параллелизма без race: параллельно идёт только реально независимое.
Arch Как встроить воркфлоу в CI/CD и headless-автоматизацию?
Headless-режим (claude -p) и Cloud agents позволяют запускать воркфлоу без интерактива: в CI-джобе агент проходит Explore→Fix→Verify по заданному промпту, а гейты (тесты, security, approval) реализуются шагами пайплайна. Паттерн fan-out — распараллелить по файлам/пакетам (matrix job), fan-in — общий прогон тестов и единый PR. Human gate остаётся на необратимом (deploy) через ручное approval стадии.
Arch Как воркфлоу, subagents и permissions образуют единую систему?
Воркфлоу задаёт что и в каком порядке (процесс), subagents дают изоляцию контекста и параллелизм (масштабирование), permissions/hooks — границы безопасности (что агент может без подтверждения). Вместе: главный агент оркеструет план → делегирует независимые куски subagents в изоляции → каждый шаг ограничен permissions → необратимое проходит gate. Разделение ответственности: процесс / масштаб / безопасность.
Arch Как обеспечить воспроизводимость и аудит воркфлоу?
Воспроизводимость: процесс в версионируемых файлах (Rules/CLAUDE.md/Commands) + фиксированные критерии Verify → одинаковый результат независимо от исполнителя. Аудит: PostToolUse hooks логируют каждое действие в audit trail, коммиты по шагам дают историю, PR фиксирует план и Verify. Для необратимого — approval gate с записью, кто подтвердил. Так воркфлоу — не «магия ИИ», а прослеживаемый процесс.
Arch Как масштабировать воркфлоу на организацию из многих команд?
Стандартные воркфлоу выносятся в shared-репозиторий/пакеты (Agent Package Manager) и подключаются в проекты; обновление одного источника улучшает процесс во всех командах. Domain-специфичные воркфлоу — поверх стандартных. Governance: owner, версионирование (major.minor), каталог воркфлоу, CI-линт на структуру. Платформенная команда владеет воркфлоу как внутренним продуктом, метрики (зелёные с первого раза, время до мержа) измеряют ROI.
