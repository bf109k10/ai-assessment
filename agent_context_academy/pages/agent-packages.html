Agent Package Manager

Agent Package Manager (APM) — система упаковки, распространения и установки способностей агента: Skills, Rules, AGENTS.md, Hooks, MCP-конфиги. Как npm для кода, только единица — не библиотека, а готовый workflow AI-агента.

Объясняю с нуля: аналогия

Если ты никогда не слышал об APM — представь что у тебя есть два мира: мир кода и мир агентов.

📦 npm / pip (для кода)

  • Пишешь npm install react
  • Получаешь готовую библиотеку компонентов
  • Импортируешь и используешь в своём коде
  • Пакет = файлы JavaScript/Python
→
то же, но для
агента

📦 APM (для агентов)

  • «Установи мне пакет code-review-agent»
  • Получаешь готовый набор Skills + Rules
  • Агент умеет делать code review «из коробки»
  • Пакет = Markdown-инструкции для агента
Проще говоря: APM — это способ сказать «дай агенту умение X», не объясняя каждый раз как делать X с нуля. Кто-то уже упаковал это умение — ты просто «устанавливаешь» его.

Что входит в один «агент-пакет»

Пакет — это папка с набором файлов. Каждый файл даёт агенту что-то конкретное.

📋 AGENTS.md обязательный

Главный файл пакета. Описывает роль агента, контекст проекта, стек, ограничения. Агент читает его первым.

# code-review-agent/AGENTS.md ## Role Senior Code Reviewer. Focus on correctness, security, test coverage. ## Stack Java 17, Spring Boot 3, JUnit 5. ## Constraints - Never suggest removing tests - Always cite line numbers

⚡ Skills workflow-инструкции

Пошаговые инструкции для конкретных задач. Агент выбирает нужный Skill по description.

# skills/review-pr/SKILL.md --- name: review-pr description: "Full PR review: correctness, security, tests. Use when reviewing code." --- ## Steps 1. Read changed files top to bottom 2. Check logic correctness 3. Look for security issues 4. Verify test coverage ## Output Structured list: file:line + issue + fix

🔒 Rules политика

Жёсткие ограничения агента. Применяются всегда, обойти нельзя.

# rules/no-secrets.mdc --- alwaysApply: true --- NEVER commit files containing API keys, passwords or tokens. If found — stop and notify the user immediately.

🛠 MCP config опционально

Конфигурация инструментов — например, какие MCP-серверы нужны для этого пакета.

# mcp.json { "mcpServers": { "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"] } } }

Пакет ≠ код

В agent-пакете нет исполняемого кода в привычном смысле. Это набор текстовых инструкций — Markdown-файлов, которые агент читает и которым следует. Поэтому пакет работает на любой модели и не требует компиляции.

Жизненный цикл: от создания до использования

1

Создание

Команда пишет AGENTS.md, Skills, Rules. Упаковывает в папку с именем пакета.

2

Публикация

Пакет кладётся в git-репо, npm registry, или внутренний реестр команды.

3

Установка

Другой проект «устанавливает» пакет — копирует файлы в нужные директории.

4

Активация

Агент читает AGENTS.md и Skills при запуске — «знает» что умеет этот пакет.

5

Обновление

Изменили Skill → обновили версию → pull в проекте, которые его используют.

Реальные реализации APM

Стандарта APM пока нет — каждая платформа решает по-своему. Вот как это выглядит сегодня.

🖱 Cursor — .cursor/rules/ + Skills

«Пакет» — набор .mdc файлов в .cursor/rules/ и .cursor/commands/. Устанавливается вручную или через git submodule. Нет единого CLI.

# Структура пакета для Cursor my-review-package/ ├── .cursor/ │ ├── rules/ │ │ └── review-policy.mdc │ └── commands/ │ └── review-pr.md └── AGENTS.md

🤖 Claude Code — .claude/ + SKILL.md

Skills с YAML frontmatter живут в .claude/commands/. Глобальные — в ~/.claude/commands/. Claude Code читает все SKILL.md при запуске.

# Структура пакета для Claude Code my-review-package/ ├── .claude/ │ └── commands/ │ ├── review-pr.md │ └── security-scan.md └── CLAUDE.md # = AGENTS.md для Claude

📦 NPM-based APM (экспериментальный)

Некоторые команды уже публикуют agent-пакеты в npm. Установка через npx agent-install, файлы копируются в проект.

# Гипотетический CLI: npx agent-install @myorg/code-reviewer # Копирует в .cursor/rules/ и .claude/ # package.json агент-пакета: { "name": "@myorg/code-reviewer", "version": "2.1.0", "agentFiles": { "cursor": ".cursor/", "claude": ".claude/" } }
Тренд: рынок движется к стандартизации. Anthropic, Cursor и сообщество обсуждают единый формат «agent package» с манифестом, версионированием и registry. Сегодня это в основном соглашения внутри команд, а не единый стандарт.

Эталонная структура agent-пакета

Стандарт де-факто, используемый командами с несколькими агентами.

# Корневая директория пакета my-agent-package/ │ ├── package.json # манифест: name, version, description, agentType ├── AGENTS.md # или CLAUDE.md — роль, контекст, стек ├── README.md # для людей: что умеет, как установить │ ├── skills/ # Skills = workflow-инструкции агента │ ├── code-review/ │ │ └── SKILL.md │ ├── security-scan/ │ │ └── SKILL.md │ └── db-migration/ │ └── SKILL.md │ ├── rules/ # Rules = жёсткая политика │ ├── no-secrets.mdc │ └── test-coverage.mdc │ ├── hooks/ # (опц.) Claude Code hooks для принуждения │ └── hooks.json │ └── mcp/ # (опц.) конфиги MCP-серверов └── mcp.json

Как «установить» пакет сегодня

Пока нет единого CLI — вот три рабочих способа.

1️⃣ Git submodule

Пакет как вложенный репо. При обновлении — git submodule update.

git submodule add \ https://github.com/myorg/code-reviewer \ .agent-packages/code-reviewer # Файлы появятся, вручную скопируй нужные в: # .cursor/rules/, .claude/commands/ и т.д.

2️⃣ Скрипт установки

Python/bash-скрипт копирует файлы в нужные директории.

# install_agent_package.py import shutil, json from pathlib import Path pkg = Path(".agent-packages/code-reviewer") cfg = json.loads((pkg/"package.json").read_text()) for src, dst in cfg["agentFiles"].items(): shutil.copytree(pkg/src, Path(dst), dirs_exist_ok=True) print("✅ Installed:", cfg["name"])

3️⃣ npx agent-install (npm-пакет)

Опубликованный в npm пакет с postinstall-скриптом.

# Установка npx @myorg/code-reviewer-agent install # Под капотом: postinstall.js const fs = require('fs-extra'); fs.copySync('./cursor-rules', '.cursor/rules'); fs.copySync('./claude', '.claude/commands');

Версионирование agent-пакетов

Semver работает для агентов так же как для кода — только «breaking change» означает другое.

ВерсияЧто изменилосьПример
patch 1.0.1Уточнение формулировок, исправление опечатки в Skill«Верни JSON» → «Верни валидный JSON»
minor 1.1.0Новый Skill или Rule добавлен, ничего не удаленоДобавили skill «performance-review»
major 2.0.0Удалили Skill, сменили output-формат, переименовали RoleOutput теперь JSON вместо Markdown
# package.json агент-пакета { "name": "@myorg/code-reviewer-agent", "version": "2.1.0", "description": "Senior code reviewer for Java/Spring projects", "agentType": "code-quality", "compatibleWith": ["cursor", "claude-code"], "agentFiles": { "cursor": ".cursor/", "claude": ".claude/" }, "changelog": { "2.1.0": "Added performance-review skill", "2.0.0": "Output format changed to JSON" } }

Тонкости

APM ≠ менеджер Python/npm пакетов

Не путай. npm ставит JavaScript-библиотеки в node_modules. Agent Package Manager ставит инструкции для агента — SKILL.md, AGENTS.md, Rules. Это разные уровни: npm — для кода, APM — для поведения агента. Оба могут сосуществовать в одном проекте.

Конфликт пакетов — Rule всегда побеждает

Если два пакета ставят разные Rules с противоречивыми инструкциями — агент применяет оба, а при конфликте приоритет у правила с alwaysApply: true. Явно фиксируй зависимости и несовместимости в README пакета.

Пакет не изолирован от контекста проекта

В отличие от npm-модуля, agent-пакет не живёт в «песочнице». Его Rules и Skills видят весь проект, весь context window. Ставь только то, что нужно для задачи — лишние Skills засоряют context агента и могут вызвать нежелательные автосрабатывания.

Версионирование сложнее чем кажется

Если Skill изменился — не все проекты сразу обновятся. Стale SKILL.md в команде из 10 проектов — реальная проблема. Решение: фиксируй версию в README каждого проекта и настрой оповещения при обновлении пакета.

Приватный реестр vs публичный

Публичный npm/GitHub — хорошо для open-source пакетов. Для корпоративных агентов с внутренними процессами и данными нужен приватный реестр (Artifactory, verdaccio, GitHub Packages private). Не публикуй в публичный npm пакеты с внутренними URL, токенами или специфичными бизнес-процессами.

Best practices — Делай / Не делай

✅ Делай

  • Один пакет = одна чёткая роль (reviewer, migrator, auditor)
  • Semver: major при breaking change output-формата или роли
  • README с примером task description для активации
  • Тестируй пакет: задай агенту задачу и проверь применение Skill
  • Prerequisites секция: что нужно установить/настроить
  • Приватный реестр для корпоративных агентов

❌ Не делай

  • Не клади все Skills в один мегапакет — агент запутается
  • Не ставь пакеты с перекрывающимися Rules без аудита конфликтов
  • Не публикуй в npm внутренние URL, токены, бизнес-процессы
  • Не игнорируй версионирование — stale Skill = неверный workflow
  • Не создавай пакет без README — другие не поймут как его активировать

Вопросы по уровням

Junior — 10 вопросов
Junior Что такое Agent Package Manager (APM)?
APM — система управления «пакетами способностей» агента. Как npm для JavaScript-кода, только единица распространения — не библиотека, а набор инструкций: Skills (как делать задачу), Rules (что запрещено), AGENTS.md (роль и контекст). Пакет устанавливается в проект, и агент сразу «умеет» всё что в нём описано.
Junior Чем agent-пакет отличается от npm-пакета?
npm-пакет — JavaScript/TypeScript код, выполняется runtime. Agent-пакет — Markdown-инструкции для AI-агента, не выполняется как код. npm ставится в node_modules, агент-пакет копирует файлы в .cursor/rules/ или .claude/commands/. npm для кода — APM для поведения агента. Оба могут быть в одном проекте одновременно.
Junior Что входит в agent-пакет?
Типичный состав: AGENTS.md (роль агента, стек, контекст) + Skills (пошаговые workflow-инструкции) + Rules (жёсткие ограничения, политика) + опционально Hooks (shell-скрипты принуждения) + MCP config (конфигурация инструментов). Всё это текстовые файлы — Markdown и JSON. Никакого исполняемого кода внутри пакета нет.
Junior Зачем вообще нужен APM, если можно просто написать AGENTS.md?
Написать AGENTS.md для одного проекта — легко. А если у тебя 10 проектов и все должны использовать одинаковые правила code review? Каждый раз копировать вручную — ошибки, устаревание. APM решает: пакет один, все проекты его устанавливают, обновление в одном месте — все получают новую версию.
Junior Как agent-пакет «активируется»?
Никакого специального запуска не нужно. Агент (Cursor или Claude Code) при старте сессии читает все файлы из .cursor/rules/, .claude/commands/ и AGENTS.md в проекте. Если там лежат файлы из пакета — агент автоматически их видит и знает что умеет. Skills применяются при совпадении задачи с description, Rules — всегда.
Junior Как сейчас устанавливают agent-пакеты?
Единого стандарта нет. Три рабочих способа: (1) git submodule — пакет как вложенный репо, файлы копируются вручную; (2) скрипт установки — Python/bash копирует файлы в нужные директории; (3) npm-пакет с postinstall — при npm install пакет сам копирует файлы в .cursor/ и .claude/. Скоро ожидается единый CLI, но пока команды выбирают сами.
Junior Что такое манифест пакета (package.json для агента)?
Файл, описывающий пакет: name (уникальное имя), version (semver), description (что умеет агент), agentType (роль), compatibleWith (cursor, claude-code), agentFiles (куда копировать файлы при установке). Аналог package.json для npm, только добавлены agent-специфичные поля. Используется скриптом установки.
Junior Можно ли поставить несколько пакетов в один проект?
Да. Например: code-reviewer-agent + security-auditor-agent + db-migration-agent в одном проекте. Агент видит Skills и Rules всех пакетов одновременно. Важно: проверь что у пакетов нет конфликтующих Rules. Если два пакета запрещают одно и то же по-разному — агент применит оба, может быть неожиданное поведение.
Junior Что означают major/minor/patch для agent-пакета?
Patch (1.0.1): исправлена опечатка в Skill, уточнена формулировка — поведение не меняется. Minor (1.1.0): добавлен новый Skill или Rule, всё старое работает как прежде. Major (2.0.0): удалён Skill, изменён output-формат, переименована роль агента — существующие проекты могут сломаться, нужна миграция.
Junior Где хранятся agent-пакеты?
Варианты: git-репозиторий (GitHub, GitLab) — самый простой; npm registry (публичный или приватный) — для команд уже знакомых с npm; внутренний корпоративный реестр (Artifactory, verdaccio) — для enterprise. Для начала достаточно обычного git-репо с README: как клонировать и куда скопировать файлы.
Middle — 10 вопросов
Middle Как спроектировать хорошую структуру agent-пакета?
Принципы: один пакет — одна роль (не мегапакет «всего и сразу»). Структура: AGENTS.md (роль) + skills/ (каждый Skill — поддиректория с SKILL.md) + rules/ (отдельные .mdc файлы по темам) + README.md (как установить, как активировать, примеры task description). Размер: Skills суммарно не более 5-7 (иначе агент путается). Rules: только то что реально нужно принудить.
Middle Как тестировать agent-пакет перед публикацией?
Тест-план: (1) Установи пакет в тестовый проект. (2) Дай агенту задачу совпадающую с description каждого Skill — убедись что Skill активируется. (3) Дай задачу намеренно нарушающую Rule — агент должен отказать. (4) Проверь output-формат: соответствует ли ## Output в SKILL.md. (5) Граничный тест: задача похожая на Skill, но другая — Skill НЕ должен активироваться. (6) Protip: golden set — 10 пар (задача, ожидаемый результат) как регрессионный тест.
Middle Как управлять зависимостями между пакетами?
Agent-пакеты могут зависеть друг от друга. Например: security-auditor требует, чтобы был установлен code-reviewer (читает его Rules). Как объявлять: в package.json поле agentDependencies (аналог peerDependencies в npm). При установке скрипт проверяет наличие зависимых пакетов. Circular dependencies — запрещены. Лучше: минимальные зависимости, общие Rules выноси в shared-пакет.
Middle Как разрешать конфликты Rules из разных пакетов?
Конфликт: пакет A говорит «всегда добавляй unit тест», пакет B говорит «не добавляй тесты в hotfix». Приоритет: alwaysApply: true — выше, чем условные Rules. При одинаковом приоритете — побеждает тот что последний в алфавитном порядке файлов (поведение платформы). Лучше: явный conflict matrix в README каждого пакета. При несовместимости — документируй это, не устанавливай одновременно.
Middle Как обновлять пакет в 10 проектах одновременно?
Стратегии: (1) Git submodule — git submodule update --remote во всех проектах одной командой. (2) Shared repo с синхронизацией — центральный репо, все проекты делают pull периодически. (3) CI pipeline — при тeг новой версии пакета автоматически создаёт PR в каждом проекте-потребителе. (4) Package registry — npm update, verdaccio. Лучший вариант для команды 10+ проектов: CI с автоматическими PR на обновление.
Middle Когда делить один пакет на два, а когда объединять?
Разделять когда: Skills несовместимы по контексту (security-audit и onboarding — разные роли, разные Rules), одну роль нужно в части проектов, другую во всех, разный ownership (security team vs product team). Объединять когда: Skills дополняют друг друга и всегда нужны вместе, общая роль в AGENTS.md, один owner. Правило: если в README пишешь «ставь оба вместе» — возможно стоит объединить.
Middle Как документировать agent-пакет для команды?
Минимальный README: (1) Что умеет агент с этим пакетом. (2) Как установить (шаги). (3) Как активировать: примеры задач которые триггерят каждый Skill. (4) Rules — список что агент никогда не сделает. (5) Несовместимость с другими пакетами. (6) Версионирование (changelog). (7) Owner и контакт. Хорошая документация = команда понимает без вопросов к автору пакета.
Middle Как использовать APM в CI/CD pipeline?
CI-паттерн: (1) В начале pipeline — шаг install agent packages (скрипт копирует нужные Skills/Rules). (2) CI-агент запускается с установленными пакетами. (3) Задачи: code review PR, security scan, changelog generation — всё через Skills. Пример: PR открыт → CI запускает Claude Code с code-reviewer-agent → reviews PR → оставляет комментарии. Ключево: пакет зафиксирован в lockfile (хэш коммита или версия), не плавающий latest.
Middle Как хранить секреты нужные для пакета (API keys)?
Никогда не в пакете. Пакет хранит только инструкции. Секреты — через env vars, CI secrets, secrets manager. В AGENTS.md пакета указывай: «Требуется переменная окружения JIRA_TOKEN». В mcp.json: "env": {"JIRA_TOKEN": "${JIRA_TOKEN}"}. При установке скрипт проверяет наличие нужных env vars и предупреждает об отсутствующих. README: список required env vars.
Middle Как APM связан со Skills и Commands?
APM — уровень распространения и управления. Skills и Commands — единицы внутри пакета. Аналогия: npm-пакет содержит функции/классы; agent-пакет содержит Skills/Commands. Устанавливая пакет, ты устанавливаешь набор Skills (автоматически активируемые) и Commands (явный вызов). APM = способ поставить эти Skills и Commands в проект, не копируя вручную.
Senior — 5 вопросов
Senior Как спроектировать APM-систему с нуля для организации?
Компоненты: (1) Package schema — стандарт package.json для агентов: обязательные поля, agentFiles mapping, agentDependencies. (2) Registry — приватный npm (verdaccio) или git-based. (3) CLI-инструмент — install/update/list команды, копирование файлов, проверка конфликтов. (4) Discovery — catalog страница с поиском по роли/домену. (5) Versioning policy — semver enforcement через CI. (6) Validation — линтер SKILL.md и AGENTS.md при публикации. Итеративно: начни с простого git + скрипт, перейди к CLI когда пакетов >10.
Senior Как обеспечить backward compatibility при обновлении agent-пакета?
Стратегии: (1) Additive changes — добавляй Skills, не удаляй. Если нужно убрать — deprecate (пометь в README и frontmatter) и удали в следующем major. (2) Output format compatibility — если меняешь ## Output, поддерживай оба формата N+1 период. (3) Migration guide — в CHANGELOG для каждого major: что изменилось, как обновить проект. (4) Тест на compatibility: golden set запросов из предыдущей версии должен проходить с новой. Правило: никогда не делай breaking change в minor/patch.
Senior Как диагностировать проблему «пакет установлен, но агент не применяет Skills»?
Диагностика по слоям: (1) Файлы на месте? Проверь что SKILL.md скопированы в .cursor/commands/ или .claude/commands/. (2) YAML frontmatter валиден? Ошибки в --- блоке молча игнорируются. (3) Description совпадает с задачей? Попробуй описать задачу словами из description Skill. (4) Конфликт с Rule? Может Rule блокирует действие которое Skill пытается сделать. (5) Skills слишком много? Context window переполнен descriptions — агент не может обработать. (6) Перезапусти сессию — некоторые платформы читают Skills только при старте.
Senior Как версионировать agent-пакеты в монорепо?
В монорепо: используй независимое версионирование для каждого пакета (changesets или lerna). Структура: packages/code-reviewer/, packages/security-auditor/ — у каждого свой package.json и CHANGELOG. CI: при изменении файла в packages/X — автоматически bump version X. Зависимости между пакетами внутри монорепо — через workspace protocol (npm workspaces). Публикация: только изменённые пакеты публикуются в registry при merge в main.
Senior Как тестировать APM в CI автоматически?
CI тест-pipeline для agent-пакета: (1) YAML lint — проверка frontmatter всех SKILL.md и AGENTS.md. (2) Schema validation — package.json соответствует схеме. (3) Install test — скрипт установки работает без ошибок в чистом проекте. (4) Agent behavior test — sandbox: запускаем агента с пакетом на golden set (задача → ожидаемый Skill). (5) Conflict check — проверяем что Rules пакета не конфликтуют с common-rules. (6) Size check — суммарный размер Skills не превышает N токенов. Инструменты: pytest + mock LLM для поведенческих тестов.
Lead — 5 вопросов
Lead Как выстроить governance APM в организации?
Framework: (1) Approval process — новый пакет: security review (нет секретов, нет опасных Rules) + architecture review (нужен ли, не дублирует ли существующий). (2) Owner per package — каждый пакет имеет team-owner, ответственный за актуальность. (3) Internal registry — каталог с owner, version, статусом (active/deprecated/sunset). (4) Update policy — при major: announcement 30 дней, migration guide, sunset дата для старой версии. (5) Quality bar: минимальные требования к README, тестам, описанию Skills. Platform team — владелец registry и CLI; product teams — авторы пакетов.
Lead Как обучать команду созданию и использованию agent-пакетов?
Программа: (1) Workshop (2ч) — что такое APM, демо установки пакета и активации Skills. (2) Template — готовая структура пакета с README, AGENTS.md, одним Skill. (3) Hands-on — каждый инженер берёт реальный повторяемый процесс своей команды и упаковывает в пакет. (4) Review — парное ревью первого пакета. (5) Catalog demo — показываешь каталог существующих пакетов, предотвращаешь дублирование. Критерий успеха: через месяц команда сама предлагает упаковать новые процессы в пакеты.
Lead Как управлять lifecycle устаревающих пакетов?
Lifecycle stages: (1) Active — поддерживается, получает обновления. (2) Deprecated — пакет работает, но рекомендован заменитель. Помечается в registry и README. (3) Sunset (end-of-life) — дата после которой пакет удаляется. (4) Removed. Процесс: Owner объявляет deprecated → 90 дней на миграцию → напоминания потребителям → удаление из registry. Метрика: count проектов использующих пакет. При нуле — ускоренный sunset.
Lead Как измерять ROI от внедрения APM?
Метрики: (1) Время на onboarding нового инженера до первого самостоятельного task — до и после APM. (2) Consistency score — доля PR прошедших code review без отклонения из-за стиля. (3) Time to install agent capability — ручное написание AGENTS.md vs установка пакета. (4) Defect rate — находит ли security-auditor-agent реальные уязвимости. (5) Package adoption — сколько проектов используют пакет. Presenting to leadership: «пакет сэкономил X часов/мес потому что Y команд не писали Rules с нуля».
Lead Как строить inter-team collaboration через APM?
Модель: (1) Platform team публикует shared-rules-agent — основные правила безопасности и стиля для всей организации. Все команды устанавливают обязательно. (2) Domain teams публикуют domain-специфичные пакеты для своего сервиса. (3) Inner-source — любая команда может PR в пакет другой команды. (4) APM guild — встреча раз в 2 недели: новые пакеты, улучшения, cross-team Issues. (5) Recognition — команда создавшая самый используемый пакет месяца. Эффект: экспертиза одной команды доступна всем через пакет без консультаций.
Architect — 5 вопросов
Arch Как спроектировать APM как platform capability?
Architecture: (1) Package registry service — API для publish/install/search, metadata store, версионирование. (2) CLI — единая команда agentpm install/update/list/publish. (3) Schema service — валидация package.json, SKILL.md, AGENTS.md при публикации. (4) Discovery UI — веб-каталог с поиском, ratings, usage stats. (5) Integration hooks — webhooks при publish новой версии, CI triggers для автопропагации обновлений. (6) Observability — какие пакеты установлены где, adoption metrics. Tech stack: любой registry (npm-compatible), CLI (Node.js/Python), catalog (статика или SPA).
Arch Как APM вписывается в enterprise AI governance?
APM как governance layer: все agent capabilities проходят через registry → естественный контроль. Governance points: (1) Publish gate — security review перед публикацией (нет секретов, нет опасных Rules). (2) Mandatory packages — shared-security-rules обязателен для всех. (3) Audit trail — кто установил какой пакет когда. (4) Compliance reporting — все проекты с агентами имеют approved пакеты. (5) Emergency revocation — если в пакете найдена проблема → central revoke → все инсталляции оповещены. APM = контролируемая точка распространения AI capabilities в организации.
Arch Как обеспечить безопасность цепочки поставок agent-пакетов?
Supply chain security для APM: (1) Signing — каждый пакет подписан GPG ключом publisher. При установке signature проверяется. (2) Hash pinning — в lockfile фиксируется SHA256 каждого пакета. (3) Registry trust — только approved publishers могут публиковать. Whitelist publisher teams. (4) SBOM для агентов — список всех пакетов и их версий как software bill of materials. (5) Dependency scanning — проверка что пакет не зависит от deprecated/unsafe пакетов. (6) Reproducible installs — одни и те же версии во всех окружениях. Threat: compromised SKILL.md может изменить поведение агента — не менее опасно чем compromised npm пакет.
Arch Как APM масштабируется при сотнях пакетов и тысячах проектов?
Scale challenges: (1) Discovery — поиск нужного пакета среди 500 становится сложным. Решение: таксономия по ролям, тегам, доменам + semantic search по descriptions. (2) Conflict detection — при установке нескольких пакетов проверять Rule conflicts автоматически. (3) Update propagation — при major обновлении известить всех потребителей (webhook список). (4) Context budget — агент с 20 пакетами может перегрузить context window descriptions Skills. Решение: lazy loading — Skills загружаются только при матче. (5) Quality decay — пакеты устаревают. Автоматический staleness check по дате последнего обновления.
Arch Как APM эволюционирует к стандарту отрасли?
Текущее состояние: разрозненные подходы, нет стандарта. Векторы эволюции: (1) Anthropic / Cursor specs — вероятно появится официальный формат agent package manifest. (2) Model Context Protocol extensions — MCP как стандарт уже есть; agent-capabilities package format — следующий шаг. (3) Open registry — аналог npmjs.com для agent-пакетов. (4) Interoperability — один пакет работает в Cursor, Claude Code, GitHub Copilot. (5) AI-generated packages — агент сам упаковывает свои удачные workflows в пакет. Позиция architect: закладывай абстракцию сегодня (agentFiles mapping), чтобы безболезненно перейти на стандарт когда он появится.
