Hooks — принуждение

Назначение: проверить или заблокировать действие на границе tool call (lint, секреты, запрет rm -rf).

Ключевое свойство: выполняется платформой, не моделью. Hooks →

Rules — политика

Назначение: зафиксировать что нельзя делать никогда — независимо от того, что попросил пользователь. Safety, compliance, архитектурные ограничения.

Ключевое свойство: применяется автоматически, не обходится из чата (в норме).

AGENTS.md — проектный контекст

Назначение: рассказать агенту кто он, в каком стеке работает, как запускать тесты, какой workflow ожидает команда. Загружается при каждом запросе.

Ключевое свойство: постоянный контекст — агент «помнит» проект без @mention.

Skills — playbooks

Назначение: сохранить шаги типовой задачи чтобы не объяснять каждый раз. Вызывается по имени или автоматически.

Ключевое свойство: рекомендует как делать, не запрещает. Skill проигрывает Rule при конфликте.

Memory Bank — между сессиями

Назначение: не терять решения, статус фичи и договорённости после закрытия чата.

Ключевое свойство: эволюцирует и может устареть; policy остаётся в Rules. Memory →

Часто путают с конфигом — Memory и соседи

AGENTS.md описывает как команде работать в репозитории (версионируется, ревьюится). Memory Bank хранит эволюцирующий контекст сессий (решения, открытые вопросы) — может устареть и требует дисциплины обновления. Запреты и MUST — всегда в Rules/Hooks, не в memory-файлах.

AGENTS.md / CLAUDE.md

  • Стабильный workflow, стек, команды сборки
  • Один источник правды для всей команды в git
  • Загружается при каждом запросе (prefix)

Memory Bank

  • «Что мы уже решили по фиче X», заметки между чатами
  • Может жить в memory/, user rules, CLAUDE.md memory — см. платформу
  • Не заменяет RAG для больших доков и не заменяет Rules для запретов

AGENTS.md / CLAUDE.md — best practices

Структура (7 секций)

# AGENTS.md

## 1. Role — кем является агент
You are a senior Java engineer
at Netcracker, Order Management.

## 2. Tech Stack
Java 21 · Quarkus 3 · PostgreSQL 15
Kafka · Kubernetes · Helm

## 3. Commands
build: mvn clean install -DskipTests
test:  mvn test
lint:  mvn checkstyle:check

## 4. Project Structure
src/main/java/  — production code
src/test/java/  — JUnit 5 tests
docs/           — architecture, ADRs

## 5. Workflow
Research → Plan → ✅ Approve → Code
→ Test → Review → Docs

## 6. DoD (Definition of Done)
- [ ] mvn test passes
- [ ] No linter errors
- [ ] CHANGELOG updated

## 7. Agent Behaviour
If unsure — ask, don't guess.
If risky — explain risk, request OK.

Частые ошибки AGENTS.md

  • Секреты в файле — никогда. Только env vars.
  • Устаревшие команды — агент будет запускать неправильно.
  • Нет DoD — агент сдаёт без тестов.
  • Нет workflow — агент делает всё сразу без одобрения.
  • Запреты ("не делай X") — должны быть в Rules, не здесь.

Rules — лучшие практики

4 типа Cursor Rules

Always Apply 
---
alwaysApply: true
---
# Safety Core
Never commit secrets.
Never delete tests.

Всегда в контексте, каждый запрос. Используйте для core safety и compliance. Не перегружать — только то что нужно всегда.

Auto-Attach (globs) 
---
globs:
  - "**.sql"
  - "**/migrations/**"
---
# DB Migration Rules
DDL only via Liquibase.
DBA review required.

Применяется автоматически когда в контексте есть файл совпадающий с glob. Идеально для контекстно-зависимых правил (DB, API, auth).

Conditional (агент решает) 
---
description: "Release process checklist"
alwaysApply: false
---
# Release Checklist
1. Tag git commit
2. Update CHANGELOG
3. Notify on-call

Агент читает description и решает применять ли. Для workflow-специфичных правил. description критически важен — от него зависит точность выбора.

Manual 
---
# Нет globs и alwaysApply
# Пользователь пишет:
# @rule-name в чате
---
# Experimental Refactor
Do X and Y only
if user explicitly asks.

Применяется только по явному вызову через @. Для редких сценариев, опытных пользователей.

Правила написания Rules

Плохие Rules

# Слишком длинная — шум
This rule contains 200 lines of
general advice about good coding
practices that apply everywhere...

# Без frontmatter — игнорируется!
# my-rule.md (не .mdc!)
Don't commit secrets.

# Дублирует AGENTS.md
Use Java 21 and Quarkus 3.
Run mvn test before finishing.

Хорошие Rules

---
description: "Security constraints"
alwaysApply: true
---
# Security Rules

- Never log passwords/tokens
- Parameterized queries only
- Validate inputs at controller
- No secrets in environment strings

Короткий, конкретный, actionable. Без дублирования. Правильное расширение.

Rules в Claude Code: нет формата .mdc. Альтернативы: 1) явная секция ## Rules (MUST follow) в CLAUDE.md, 2) Hooks в .claude/settings.json для hard enforcement — они выполняются на lifecycle event независимо от решения модели.

Skills / Commands — лучшие практики

Структура хорошего Skill

---                          (Claude Code)
name: "triage-ci"
description: "Analyse a failing CI
  job and propose a minimal fix"
---

# CI Failure Triage

## When to use
CI job is failing and root cause unknown.

## Steps
1. Read failing job log: $ARGUMENTS
2. Find failed stage (build/test/lint)
3. Trace error to file:line
4. Check recent commits in that area
5. State root cause (1 paragraph)
6. Propose minimal fix (≤5 files)

## Output format
## Root Cause
[paragraph]

## Fix
[code block with file:line refs]

## Degrees of freedom
High: wording of explanation
Low:  output format (must follow)

Ключевые принципы Skills

  • Description — критично. Именно его читает агент чтобы решить когда применить skill. Начинать с глагола: "Analyse", "Generate", "Triage". Максимум 100 символов.
  • 3-е лицо. "Analyses a failing job" а не "I will analyse".
  • Degrees of freedom — явно укажи что фиксировано (Low), что гибко (High).
  • $ARGUMENTS — placeholder для параметров (Claude Code).
  • Один skill = одна задача. Megaskill → разбить.
  • Constraints в конце — что skill НЕ делает (не путать с Rules).

Anti-patterns

  • "Help with code" — слишком размыто, агент не знает когда
  • Запреты в Skill ("never add deps") → это Rules
  • 500+ строк → разбить на несколько
  • Нет output format → непредсказуемый результат
  • Описание в 1-м лице ("I will…") → не работает как ожидается

Тонкости и нюансы

Перегруженный AGENTS.md → агент игнорирует

Если AGENTS.md длиннее ~200 строк, агент начинает игнорировать его части из-за эффекта «lost in the middle». Держи AGENTS.md кратким и конкретным — только то, что нужно агенту при каждом запросе.

Rules vs AGENTS.md: не дублируй

Если правило уже задано в Rules с alwaysApply: true, не повторяй его в AGENTS.md. Дублирование создаёт расхождения при обновлении: правишь в одном месте — забываешь обновить в другом.

Skills не загружаются автоматически

Skills нужно явно вызывать или упоминать в задаче. Агент не знает о skill, если его не упомянули в запросе или не сослались в AGENTS.md. Не рассчитывай что агент сам найдёт нужный skill.

Порядок приоритетов: Rules > AGENTS.md > Skills

При конфликте Rules всегда побеждает. AGENTS.md — рекомендации и workflow; Rules — политика, которую нельзя нарушить даже по просьбе пользователя. Skills — инструменты, не ограничения.

Best practices — Делай / Не делай

✅ Делай

  • Держи AGENTS.md кратким — до 150 строк максимум.
  • Используй Rules для жёсткой политики, AGENTS.md — для workflow и контекста.
  • Явно упоминай нужный Skill в задаче или в AGENTS.md.
  • Разделяй роли: AGENTS.md = кто я и как работаю; Rules = что нельзя; Skills = как делать конкретное.

❌ Не делай

  • Не дублируй одно и то же в AGENTS.md и Rules — при рассинхроне Rule победит, но путаница останется.
  • Не добавляй в AGENTS.md всё подряд — только то, что агент должен знать при каждом запросе.
  • Не рассчитывай что агент сам найдёт нужный Skill — всегда указывай явно.

Вопросы по уровням

Junior — 10 вопросов
Junior Чем AGENTS.md отличается от Rules?
AGENTS.md описывает "как работать в проекте" — это workflow, stack, команды, DoD, конвенции. Rules определяют "что нельзя" — ограничения, safety, compliance. AGENTS.md — это инструкция нового сотрудника; Rules — кодекс поведения. AGENTS.md агент соблюдает как рекомендацию; Rule с alwaysApply:true агент обязан соблюдать, даже если пользователь просит обратного. При конфликте Rule всегда побеждает AGENTS.md.
Junior Чем Rules отличаются от Skills?
Rules — политика: что агенту запрещено или обязательно. Они применяются автоматически и не обходятся из чата (alwaysApply). Skills (Commands) — рекомендации как делать задачу: пошаговый playbook для типового сценария. Skill вызывается явно или по решению агента; Rule всегда в фоне. Если Skill конфликтует с Rule — Rule побеждает. Skill проигрывает Rule при конфликте всегда.
Junior Чем Skills отличаются от Commands?
В Cursor они называются Commands (файлы в .cursor/commands/), в Claude Code — Slash Commands (файлы в .claude/commands/). По сути это одно: markdown-файлы с пошаговыми инструкциями для типовой задачи, вызываемые по имени через /command. "Skills" — обобщённый термин для обоих. Технически в Claude Code есть YAML frontmatter (name, description), в Cursor — plain markdown с заголовком.
Junior Для чего используется каждый из трёх инструментов?
AGENTS.md — "кто я и как работать здесь": постоянный project context, загружается при каждом запросе. Rules — "что запрещено или обязательно": safety и compliance, не обходятся из чата. Skills — "как делать эту задачу": playbook для повторяющихся сценариев (debug CI, write PR, triage incident). Каждый работает на своём уровне и не заменяет другие.
Junior Что такое policy vs workflow в контексте конфигурации агента?
Policy (политика) — набор правил что нельзя нарушать: "не коммитить секреты", "DDL только через Liquibase". Это Rules. Workflow — описание процесса как делать задачи: "сначала Research, потом Plan, потом Code". Это AGENTS.md (как workflow-инструкция) или Skills (как конкретный playbook). Policy — жёстче и принудительнее; workflow — гибче и следуется "по возможности".
Junior Где физически хранится каждый артефакт: AGENTS.md, Rules, Skills?
AGENTS.md: корень репо (Cursor) или CLAUDE.md в корне (Claude Code). Rules: .cursor/rules/*.mdc (Cursor, обязательно расширение .mdc) или секция в CLAUDE.md + .claude/settings.json Hooks (Claude Code). Skills: .cursor/commands/*.md (Cursor) или .claude/commands/*.md (Claude Code). Глобальные Skills в Claude Code: ~/.claude/commands/. Расположение принципиально — файлы не в нужном месте игнорируются системой.
Junior Когда агент применяет каждый из трёх инструментов?
AGENTS.md — при каждом запросе автоматически (загружается в context всегда). Rules с alwaysApply:true — при каждом запросе автоматически. Rules с globs — автоматически когда в context есть файлы совпадающие с glob-паттерном. Skills — только при явном вызове (/command) или когда агент решает что skill релевантен по description. Порядок приоритета: alwaysApply Rules > AGENTS.md > globs Rules > Skills.
Junior Что первично при конфликте между AGENTS.md и Rules?
Rules первичны. Иерархия от высшего к низшему: Hooks (Claude Code) / Team Rules (Cursor Business) → alwaysApply Rules → AGENTS.md / CLAUDE.md → auto-attach Rules (globs) → текущая задача (чат) → Skills. AGENTS.md не может "отменить" Rule. Это фундаментальный принцип: policy (Rules) всегда выше workflow (AGENTS.md).
Junior Что такое always-active vs on-demand применение правил?
Always-active: AGENTS.md + Rules с alwaysApply:true всегда присутствуют в context при каждом запросе — агент не может их "не увидеть". On-demand: Rules с globs применяются только когда есть совпадение; Skills применяются только при вызове. Разница важна для token budget: always-active компоненты занимают токены при каждом запросе независимо от задачи. Поэтому alwaysApply Rules должны быть лаконичными.
Junior Как все три инструмента работают вместе в одном проекте?
Совместная работа: AGENTS.md задаёт контекст ("мы Java-проект, Quarkus, mvn test"); alwaysApply Rules гарантируют safety ("не коммитить секреты", "не удалять тесты"); globs Rules добавляют контекстные ограничения при работе с SQL или API-файлами; Skills предоставляют готовые playbooks для частых задач (/debug-ci, /write-pr). Агент использует все три одновременно, сохраняя иерархию приоритетов.
Middle — 10 вопросов
Middle Как построить матрицу выбора между AGENTS.md, Rules и Skills?
Матрица решений: (1) Это постоянный контекст проекта (стек, команды, workflow)? → AGENTS.md. (2) Это запрет или обязательное требование которое нельзя обойти? → Rules (alwaysApply или globs). (3) Это пошаговая инструкция для повторяющейся задачи? → Skill. (4) Это применяется только к определённым типам файлов? → Rules с globs. (5) Это вызывается только при конкретном сценарии? → Skill. Если всё ещё не ясно: запрещает → Rules; рекомендует → AGENTS.md или Skills.
Middle Когда стоит перенести инструкцию из AGENTS.md в Rules?
Перенести в Rules когда: (1) инструкция является запретом или обязательным требованием ("не делай X", "всегда делай Y"); (2) агент периодически её нарушает (lost in the middle в длинных сессиях); (3) нарушение влечёт safety-риски или compliance-нарушения; (4) требование должно применяться при работе с конкретными файлами (globs). Остальное — в AGENTS.md. Правило: если добавляешь "никогда" или "всегда обязательно" — это признак Rule, не AGENTS.md.
Middle Когда стоит перенести инструкцию из Rules в Skills?
Перенести в Skill когда: (1) инструкция описывает пошаговый процесс, а не ограничение; (2) применяется только при конкретном сценарии (debug CI, write PR, triage incident); (3) слишком длинная для AGENTS.md но не является запретом; (4) разные пользователи нуждаются в разных вариантах одного процесса. Skill — это playbook, не policy. Если инструкция остаётся запретом ("не добавляй зависимости") — она в Rules, даже если звучит как процессная.
Middle Как избежать дублирования между AGENTS.md, Rules и Skills?
Принцип единственного источника: каждое правило живёт ровно в одном месте. Признаки дублирования: "mvn test" и в AGENTS.md, и в Skill; "не коммитить секреты" и в AGENTS.md, и в Rule; одна и та же инструкция в нескольких Rules. Решение: AGENTS.md — постоянный контекст; Rules — ограничения; Skills — процессы. Если есть соблазн скопировать — это сигнал что нужно удалить из одного источника и оставить в правильном.
Middle Как организовать три артефакта для командной разработки?
Организация для команды: (1) AGENTS.md — версионировать в git с review; назначить owner; обновлять при архитектурных изменениях. (2) Rules — хранить в .cursor/rules/ в git; alwaysApply Rules требуют строгого review (safety-critical); globs Rules — совместно согласованные. (3) Skills — командные Skills в .cursor/commands/ в git; личные Skills — в ~/.claude/commands/. Для всех трёх: изменения через PR, не прямой commit в main.
Middle Каковы performance trade-offs использования каждого инструмента?
AGENTS.md: токены при каждом запросе (всегда в context). alwaysApply Rules: токены при каждом запросе. globs Rules: токены только при совпадении. Skills: токены только при вызове. Итого: чем больше alwaysApply компонентов, тем выше постоянный token overhead. Оптимизация: держать AGENTS.md и alwaysApply Rules лаконичными (≤100 строк каждый), детальные playbooks — в Skills (on-demand). Prompt caching нивелирует часть overhead для стабильного контента.
Middle Как рефакторировать "God AGENTS.md" на три слоя?
Признаки God AGENTS.md: файл >500 строк, содержит запреты ("не делай X"), пошаговые инструкции и постоянный контекст вперемешку. Шаги рефакторинга: (1) Извлечь все запреты и обязательные требования → создать Rules (.mdc с alwaysApply или globs); (2) Извлечь пошаговые процессы → создать Skills в .cursor/commands/; (3) Оставить в AGENTS.md только: роль, стек, команды, workflow-описание, references. Цель: AGENTS.md ≤100-200 строк. Результат: каждый слой работает эффективнее.
Middle Какие типичные ошибки при смешивании AGENTS.md, Rules и Skills?
Типичные ошибки: (1) Запреты в AGENTS.md вместо Rules — агент может проигнорировать при длинном контексте; (2) Workflow-инструкции в Rules вместо Skills — раздувает always-active context без пользы; (3) Дублирование stack info и в AGENTS.md, и в Skills — рассинхронизируется при изменениях; (4) alwaysApply Rule с 200+ строками общих советов — загрязняет context; (5) .md файл в .cursor/rules/ без frontmatter — игнорируется rules-системой полностью.
Middle Как тестировать каждый тип артефакта?
Тестирование: AGENTS.md — спросить агента "каков стек проекта?" или "как запустить тесты?" — должен ответить верно из AGENTS.md без @mention. alwaysApply Rule — попросить агента нарушить правило ("закоммить .env") — должен отказаться. globs Rule — открыть файл подпадающий под glob и проверить что правило активно. Skill — вызвать /command-name и проверить что агент следует шагам из файла. Smoke test для всей системы: сессия с типовой задачей и проверка что все три слоя работают согласованно.
Middle Какую version control стратегию применять для трёх артефактов?
Стратегия: все три артефакта в git вместе с кодом. AGENTS.md — в корне repo, версионируется как любой документ; обновлять при архитектурных изменениях. Rules (.mdc файлы) — в .cursor/rules/, обновлять при изменении policy; alwaysApply Rules требуют особого review. Skills — в .cursor/commands/, обновлять при изменении процессов; командные Skills — в репо, личные — в ~/.claude не версионируются командой. Общий принцип: изменение конфигурации агента = изменение кодовой базы. Тот же review process.
Senior — 5 вопросов
Senior Как спроектировать трёхуровневую архитектуру конфигурации агента?
Архитектура трёх уровней: (1) Policy layer (Rules) — stateless, всегда активный, неизменяемый из чата. Отвечает на "что нельзя". Минимальный по токенам. (2) Context layer (AGENTS.md) — постоянный project context. Отвечает на "кто я и в чём работаю". Компактный, структурированный. (3) Workflow layer (Skills) — on-demand playbooks. Отвечает на "как делать конкретную задачу". Детальный, может быть большим. Правило слоёв: нижний не нарушает верхний. Skills не нарушают Rules. AGENTS.md не нарушает Rules.
Senior Как диагностировать конфликты между слоями AGENTS.md, Rules, Skills?
Сценарии конфликтов: (1) Skill рекомендует "добавь зависимость" + Rule запрещает без approval → Rule побеждает, агент должен запросить approval; (2) AGENTS.md говорит "всегда TDD" + пользователь просит "без тестов" → пользователь может попросить, AGENTS.md — рекомендация; (3) alwaysApply Rule vs пользователь — Rule побеждает. Диагностика: если агент нарушает Rule — проверить что файл .mdc с frontmatter, alwaysApply:true установлен, файл не слишком большой. Если агент не следует Skill — проверить description файла (агент его читает для выбора).
Senior Как оптимизировать token overhead от трёх слоёв конфигурации?
Token optimization стратегия: (1) AGENTS.md + alwaysApply Rules — кандидаты для prompt caching (стабильный prefix); реальный overhead = 0 со 2-го запроса при неизменном контенте; (2) Минимизировать alwaysApply слой — только safety-critical rules; (3) Детальные инструкции → Skills (on-demand); (4) Subdirectory AGENTS.md — загружается только при работе в директории; (5) Мерить: log input tokens для задачи, постепенно убирать лишнее из always-active слоя, измерять качество. Цель: минимальный always-active overhead при максимальном покрытии задач.
Senior Что такое composability паттерны для AGENTS.md/Rules/Skills?
Composability — способность компонентов комбинироваться. Паттерны: (1) AGENTS.md @include ссылки — декомпозиция на модульные файлы; (2) Skills вызывающие другие Skills через инструкции в тексте; (3) globs Rules — контекстно-зависимая композиция (SQL файл → db-migration Rule активируется автоматически); (4) Base AGENTS.md + Subdirectory AGENTS.md — inheritance паттерн. Anti-pattern composability: Skills которые дублируют Rules (нарушают single-source); AGENTS.md который инклудит всё подряд превращаясь в "god file".
Senior Какой emergent behavior возникает от взаимодействия трёх слоёв?
Emergent behavior: (1) Контекстная адаптация — глобальный AGENTS.md + subdirectory AGENTS.md + globs Rule = агент автоматически меняет behaviour в зависимости от директории и типа файла без явных инструкций в чате; (2) Защитное поведение — alwaysApply Rule + User request = агент объясняет почему нельзя, предлагает alternative; (3) Guided workflow — AGENTS.md workflow + Skill playbook = агент следует процессу команды при типовых задачах; (4) Непредвиденные конфликты — три слоя могут создавать contradictory instructions, которые LLM разрешает по своему усмотрению. Нужен регулярный audit.
Lead — 5 вопросов
Lead Как выстроить team agreement на использование трёх инструментов?
Team agreement: (1) Матрица решений — задокументировать когда что использовать (в AGENTS.md или wiki); (2) Naming conventions — единый стиль имён Rules и Skills (verb-noun: check-secrets.mdc, triage-ci.md); (3) Review process — кто и когда review'ит изменения в каждом слое; (4) Ownership — назначить ответственных за каждый слой (tech lead → Rules policy, team lead → AGENTS.md, devs → Skills); (5) Onboarding — раздел в документации о трёх инструментах для новых разработчиков. Agreement должен быть живым документом, обновляться по мере опыта команды.
Lead Как организовать governance изменений в AGENTS.md, Rules, Skills?
Governance по уровням критичности: alwaysApply Rules — максимальный контроль: изменения требуют tech-lead approval, CI проверяет что ни один существующий тест не нарушается; AGENTS.md — team review, обновлять при архитектурных изменениях через PR checklist; Skills — более гибко, разработчики могут предлагать новые Skills, review от любого senior. Для всех трёх: CODEOWNERS в git, история изменений, запрет прямых commits в main для конфигурационных файлов агента.
Lead Как обучать команду правильному использованию трёх артефактов?
Обучение в три этапа: (1) Концептуальный уровень — объяснить разницу между policy/context/workflow с примерами из проекта; (2) Практический уровень — workshop: команда создаёт минимальный набор (AGENTS.md + 2 Rules + 1 Skill) для реального проекта; (3) Review-based learning — code review изменений в конфигурации агента с объяснением почему "это должно быть Rule, а не AGENTS.md". Основная ловушка: разработчики интуитивно кладут всё в AGENTS.md — нужно явно показать когда это проблема (lost in the middle, bypassed safety).
Lead Как проводить audit трёх компонентов конфигурации агента?
Audit checklist раз в квартал: AGENTS.md — актуальны ли команды? Соответствует ли стек реальному? Нет ли устаревших ADR? Не слишком ли длинный (>200 строк — разбить)? Rules — все ли .mdc файлы имеют frontmatter? alwaysApply Rules — минимально необходимы? Нет ли конфликтующих rules? Skills — все ли Skills используются? Нет ли Skills которые дублируют Rules? Описания (description) актуальны? Дополнительно: test агента на нарушение каждой alwaysApply Rule, убедиться что не обходится.
Lead Как балансировать clarity vs flexibility в трёхуровневой архитектуре?
Баланс clarity/flexibility: строгие Rules дают clarity (предсказуемое поведение) но снижают flexibility (агент не может адаптироваться). Мягкий AGENTS.md даёт flexibility но снижает reliability (агент может нарушить workflow). Стратегия: alwaysApply Rules — только для safety/compliance (высокая clarity, низкая flexibility); AGENTS.md — workflow рекомендации (средняя clarity, гибкость для задач); Skills — детальные playbooks (высокая clarity в конкретном сценарии, вызываются по потребности). Правило: чем выше риск нарушения — тем жёстче enforcement. Остальное — гибко.
Architect — 5 вопросов
Arch Как трёхуровневая архитектура AGENTS.md/Rules/Skills соотносится с паттернами проектирования?
Аналогии: (1) Policy/Context/Workflow напоминает разделение concerns в чистой архитектуре (policy = domain rules, context = application layer, workflow = use cases); (2) alwaysApply Rules = Immutable invariants (не нарушаются никогда); AGENTS.md = Configuration (environment-specific); Skills = Strategy pattern (алгоритм делегирован); (3) Каскадный AGENTS.md = Template Method: базовый класс (root AGENTS.md) + конкретизация (subdirectory). Знание паттернов помогает принимать архитектурные решения: когда нужна жёсткость (invariants), когда гибкость (strategy).
Arch Как применить трёхуровневую архитектуру в enterprise контексте?
Enterprise паттерн: (1) Organization-level Rules — shared policy для всей организации (security, compliance, regulatory); управляются центральной командой; распространяются через CI template или Cursor Team Rules; (2) Team-level AGENTS.md — специфика команды/продукта; управляется team lead; (3) Project-level Skills — конкретные playbooks проекта; управляются разработчиками. Governance: изменения в org-level Rules — архитектурный review board. Трёхуровневая иерархия позволяет scale: org policy строгий, team context гибкий, project workflows итеративные.
Arch Как выстроить cross-team стандарты конфигурации агентов?
Cross-team стандарты: (1) Shared Rules template — git-репозиторий с организационными Rules (.mdc файлы), каждая команда копирует через CI или submodule; (2) AGENTS.md skeleton — шаблон обязательных секций (Role, Stack, Build&Test, Workflow), команды заполняют специфику; (3) Skills library — каталог общих Skills (triage-ci, write-pr, incident-report) доступных для всех команд; (4) Naming convention — единый стиль имён файлов; (5) Validation CI — проверяет что каждый проект имеет обязательный минимум. Cross-team стандарты = снижение cognitive load при переходе между командами.
Arch Какую migration стратегию применить при переходе на новую архитектуру?
Migration от "все в AGENTS.md" к трёхуровневой: (1) Audit — инвентаризировать текущий AGENTS.md; категоризировать строки: policy/context/workflow; (2) Strangler Fig pattern — постепенно выносить policy в Rules, оставляя AGENTS.md как "проксирующий" файл с @include; (3) Skill extraction — выделить повторяющиеся workflow-паттерны в Skills (начать с самых часто используемых); (4) Validate по-итогам каждого шага — агент всё ещё правильно работает?
